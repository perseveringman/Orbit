# 02-人生规划系统

## 1. 设计定位

人生规划系统不是一个要求用户持续维护的大表格，也不是一套把愿景压扁成 KPI 的 OKR 机器。它是 Orbit 在“人生尺度”上的决策层：用户保有方向、价值判断与最终决定权，Agent 负责把愿景翻译成不同时间尺度上的阶段性方向、主题、目标与日常节奏，并在执行过程中持续拆解、重排、回顾。

本文的核心判断有三点：

1. **愿景是罗盘，不是清单。** 五年尺度承载的是“我想成为什么样的人、过什么样的生活、创造什么样的作品”，不是量化考核表。愿景必须由用户书写，Agent 只读、不改写。
2. **规划是连续推导，不是静态层级图。** 五年 / 年 / 季 / 月 / 周 / 日不是六张独立表，而是一条持续收敛的推导链：愿景给方向，方向形成阶段主题，主题约束项目组合，项目再落成周计划与日节奏。
3. **规划的价值在于降低维护成本。** 用户不需要手工维护庞杂树状结构。Agent 应该从项目进展、任务状态、研究材料、写作草稿、Journal 记录中自动识别变化，提出更新建议，并把规划结果回写成轻量对象与事件。

因此，Orbit 的人生规划系统本质上是一个 **Agent 驱动的多时间尺度对齐系统**：上承人生愿景，下接项目执行，中间通过主题、目标、回顾和节奏维持连续性。

## 2. 关键用户场景

### 场景 A：从模糊愿景到可推进方向
用户写下“我想成为能做产品、能写清楚复杂技术、也能长期稳定创造的人”。Agent 不会立刻把它拆成 20 条 KPI，而是先整理出未来 12 个月更合理的几个方向，例如“Agent Native 产品能力”“持续公开写作”“稳定研究输入节奏”，供用户确认。确认后再继续下钻到季度主题与月度目标。

### 场景 B：季度中途失焦，系统帮助重排
用户本季度主题原本是“完成 Orbit 新蓝图并开始公开输出”，但最近两周大量时间被零散合作与临时请求吞掉。Agent 结合任务事件、时间记录和 Journal，提示“最近工作更像救火而非建设，是否要把两个合作项目降级为机会型项目，并恢复每周一次公开输出？”用户确认后，Agent 才执行重排。

### 场景 C：研究、写作、项目彼此断裂时自动补桥
用户读了十几篇资料、做了高亮与研究笔记，但迟迟没有进入项目决策或写作产出。Agent 识别这些材料持续指向某个季度主题，便建议：“这些研究已足够支持一篇文章与一个产品决策，是否分别生成写作提纲和项目假设？”

### 场景 D：从周计划落到今天
周一，系统根据季度主题、月度重点、项目阻塞关系与用户精力模式，给出本周 3 个承诺；周三早上，Agent 再从这些承诺中挑出今天最值得推进的两个深度工作块，而不是把整周所有事项都压到今日列表里。

### 场景 E：通过回顾把人生规划变成活系统
每天结束后，Journal 自动沉淀事实；周末，Agent 自动起草周回顾；月末，系统总结“哪些投入真正推动了年度方向、哪些只是消耗”；季度结束时，用户据此决定延续、收缩还是更换主题。规划因此成为被使用、被修正、被反思的活系统，而不是建立一次就腐烂的模板。

## 3. 核心设计

### 3.1 规划层级

Orbit 采用“少对象 + 多 horizon”的规划体系。不是每个时间尺度都发明一套新模型，而是用少数核心对象在不同 horizon 下表达不同粒度。

| 时间尺度 | 规划角色 | 典型对象 | 产出形式 |
|---|---|---|---|
| 五年 | 方向感与身份想象 | `vision` | 一段用户手写愿景文本，1~3 条 |
| 年 | 阶段性方向 | `direction` | 未来 12 个月重点发展什么、放弃什么 |
| 季 | 聚焦主题 | `theme` | 本季度真正要让生活/工作发生什么变化 |
| 月 | 可检验推进面 | `goal` | 4~6 周内希望推进到的阶段结果 |
| 周 | 有限承诺 | `commitment` | 本周最重要的 3~5 个推进承诺 |
| 日 | 当日节奏 | `focus_block` / `today_plan` | 今天的深度工作块、轻任务、缓冲时间 |

这六层并不是刚性父子树，而是“逐层约束、逐层具体化”：

- `vision` 决定方向感，不提供任务。
- `direction` 把方向感转成一年内值得投入的几个发展轴。
- `theme` 把方向轴压缩成一个季度真正要聚焦的变化主题。
- `goal` 定义这个主题在一个月或一个季度内应当出现的阶段性结果。
- `commitment` 只保留本周真正承诺推进的少量事项。
- `today_plan` 则是 Agent 在当前上下文下给出的当天建议，不要求长期手工维护。

### 3.2 核心对象

#### 1）`vision`
- 用户手写，Agent 只读。
- 不带截止日期、优先级、完成百分比。
- 允许是叙事性的、带情感温度的表达。
- 用于“做选择时想起方向”，不是“检查达标没有”。

#### 2）`direction`
- 是对未来 12 个月的阶段方向定义。
- 形式通常是“强化什么 / 减少什么 / 暂不做什么”。
- 数量应极少，通常 2~4 条。
- 可由 Agent 提议，但必须用户确认。

#### 3）`theme`
- 是季度层的聚焦语言，例如“把 Orbit 蓝图从研究推进到公开表达”。
- 它比 goal 更宽，比 direction 更具体。
- theme 约束项目组合与资源分配，是跨项目的焦点，而不是单个项目里程碑。

#### 4）`goal`
- 是阶段性结果对象，可对应月或季。
- 它强调“推进到什么状态”，不是“做多少任务”。
- 一个 goal 可以由多个项目共同支撑，也可以派生出若干项目内任务。

#### 5）`commitment`
- 是周尺度的有限承诺。
- 它连接目标与实际执行，强调“这一周我真准备推进什么”。
- 数量被强约束，避免计划膨胀。

#### 6）`review`
- 覆盖日 / 周 / 月 / 季 / 年回顾。
- 是规划系统的校准器，不是附属页面。
- 既沉淀事实，也生成对下一轮规划的修正建议。

### 3.3 五年计划和日计划如何衔接

五年计划与日计划的衔接，不靠用户手工维护一棵巨大目标树，而靠 Agent 持续做四次翻译：

`vision → direction → theme/goal → commitment → today_plan`

具体规则：

1. **五年愿景只回答“北在哪”。**
   它描述长期想成为什么样的人、建立什么作品与生活结构。
2. **年度方向回答“今年往哪几条路走”。**
   Agent 从 vision 中提炼 2~4 条年度方向，供用户确认。
3. **季度主题回答“这个阶段真正聚焦什么变化”。**
   用户不会同时推进所有方向，theme 用来做取舍。
4. **月目标回答“最近 4~6 周推进到哪里算有实质进展”。**
   这里开始出现阶段结果，但仍不等于任务清单。
5. **周承诺回答“这周我实际愿意押注什么”。**
   它把目标压缩成少量真实承诺，避免“计划上想做十件、现实只能做两件”。
6. **日计划回答“今天在当前精力和上下文下，最值得做什么”。**
   Agent 再结合日历、能量、阻塞关系、前一天进展与 Journal 事实，生成今天的工作块建议。

因此，日计划不是直接从五年计划硬拆出来，而是经过连续收敛与多次用户确认后自然落地。这样既保留长期一致性，也保留短期灵活性。

### 3.4 规划对象如何与项目、任务、研究、写作、日志联动

规划系统不是孤立模块，而是 Orbit 全局对象网络上的“上层调度语义”。

- `theme -> project`：季度主题决定哪些项目是当前活跃项目，哪些项目应降级为机会型项目或暂停项目。
- `goal -> task`：阶段目标并不直接等于任务，但会为任务系统提供优先级约束、截止窗口和拆解方向。
- `theme / goal -> research`：研究不是无主输入。研究材料、高亮、问题卡片应能回链到其支撑的主题或目标。
- `research -> writing`：当研究积累足够支撑表达时，Agent 应主动建议生成文章提纲、观点卡或写作项目，而不是让研究永远悬空。
- `journal -> review -> planning`：Journal 记录事实，review 解释事实，planning 根据解释做调整。日志不是装饰，而是规划闭环里的事实来源。

这里的关键不是“所有对象都从规划树派生”，而是“所有对象都能被规划语义解释”。用户做的研究、写的文章、推进的项目，最终都应该能回答：它在支持哪个主题、哪个目标、哪条方向？如果回答不了，Agent 可以提醒，但不应自动否定它的价值。

### 3.5 规划节奏

人生规划系统不是“年初写一次，年底看一眼”，而是有明确节奏的连续校准系统：

- **五年愿景**：半年一次轻校准，或在重大人生变化时重写。默认不频繁打扰。
- **年度方向**：年初建立，季度交界处复看是否仍成立。
- **季度主题**：季初确认，双周做一次轻量检查，避免整个季度失焦。
- **月目标**：月初设定，周中允许 Agent 根据现实推进情况建议缩放。
- **周承诺**：每周一次正式建立，周中允许一次重排，不鼓励每天重写整周。
- **日计划**：前一晚或当天早晨 materialize，当天根据精力与突发情况动态微调。

这个节奏的设计目的，是让高层规划变化更慢、低层规划变化更快：越靠近人生愿景，越强调稳定和方向感；越靠近当日执行，越允许 Agent 根据真实上下文即时重排。

## 4. Agent 行为与自动化机制

### 4.1 Agent 的职责

Agent 在人生规划系统中的职责不是替用户定义人生，而是做以下五件事：

1. **抽取**：从愿景文本、项目变化、Journal 与执行事件中抽取新的规划信号。
2. **推导**：把高层方向推导成季度主题、月目标、周承诺和今日候选项。
3. **重排**：当事实变化时，主动提出延后、收缩、拆分、合并建议。
4. **补桥**：发现规划与项目、研究、写作、日志之间的断裂，并建议建立关联。
5. **回顾**：基于 events 与 Journal 起草回顾，帮助用户形成下一轮规划决策。

### 4.2 主动建议机制

Agent 的建议不靠固定提醒器，而靠“规划差值”触发：当计划表达的方向与事实记录的轨迹出现明显偏差时，系统再发声。

典型触发器：

- **方向稀释**：某季度主题对应的项目在两周内几乎无推进。
- **项目漂移**：大量活跃任务没有任何上层 theme / goal 归属。
- **研究悬空**：连续积累研究材料，但没有形成项目决策或写作动作。
- **写作断档**：年度方向强调公开表达，但近几周只有输入没有输出。
- **节奏失真**：周承诺长期超载，说明规划粒度或节奏设置错误。
- **回顾缺席**：事实很多，但最近没有形成 review，对下轮规划失去校准依据。

建议的表达方式必须遵循“温和提醒，不做审计”：

- 好的方式：
  - “你最近 10 天主要在响应外部请求，和本季度主题的主线有点脱节。要不要我帮你把两个临时项目降级，再恢复主线节奏？”
  - “这批研究已经足够支撑一篇文章，要不要先起一个提纲？”
- 不好的方式：
  - “你本周愿景对齐度仅 42%，未达标。”
  - “你已连续 3 天未完成写作 KPI。”

### 4.3 “建议”与“替用户做决定”的边界

这是人生规划系统最重要的护栏。

#### 用户独占决定权
以下事项 Agent 永远不能自行决定：
- 创建、修改、删除 `vision`
- 确认年度 `direction`
- 接受或放弃季度 `theme`
- 对重大项目做开始 / 取消 / 长期暂停决定
- 改变用户明确声明的生活边界与价值偏好

#### Agent 可提议、需用户确认
以下事项 Agent 可以高质量建议，但必须确认后执行：
- 季度主题调整
- 月目标新增或收缩
- 活跃项目组合调整
- 本周承诺重排
- 将研究转为写作项目或项目任务

#### Agent 可在已确认约束内自动完成
以下事项可由 Agent 默认执行，并保留可撤销记录：
- 生成今日候选计划
- 自动聚合 Journal 事实并起草日/周回顾
- 为任务补全上层 goal / theme 候选关系
- 根据阻塞关系重排同级任务顺序
- 为研究、写作、项目建立候选链接

边界原则是：**Agent 可以压缩复杂性，但不能替用户定义人生方向。Agent 可以组织选项，但不能偷换用户的承诺。**

### 4.4 回顾闭环

规划系统必须以回顾闭环收尾，否则上层愿景和下层执行会再次断裂。Orbit 的闭环路径是：

`Journal 事实 → review 草稿 → 用户修正判断 → planning 重排 → 新一轮执行`

分层来看：

- **日回顾**：确认今天真正发生了什么，哪些计划是误判，哪些想法值得进入明天或本周。
- **周回顾**：检查本周承诺是否过载、哪些项目在推进、哪些只是伪忙碌。
- **月回顾**：判断月目标是否仍有意义，是否需要拆小、延后或取消。
- **季回顾**：评估季度主题是否真的改变了工作与生活结构，而不仅是完成了一堆任务。
- **年回顾**：回看年度方向是否仍然服务于五年愿景，是否要进入新的阶段。

Agent 在这里负责自动汇总事实、起草解释、提出下一步建议；用户负责给出最终判断。回顾的结果不是归档在角落里，而是直接驱动 direction、theme、goal 与 commitment 的更新。

## 5. 数据模型 / 接口 / 能力边界

### 5.1 规划域最小对象

在全局对象系统之上，人生规划域只需要少数对象类型与关系：

#### 对象类型
- `vision`
- `direction`
- `theme`
- `goal`
- `commitment`
- `review`

#### 关键字段
- `horizon`：five_year / year / quarter / month / week / day
- `title`
- `body`
- `status`：draft / active / paused / done / archived
- `time_window`：起止区间
- `confidence`：仅用于 Agent 建议置信度，不用于评价用户
- `decision_mode`：user_written / agent_suggested / user_confirmed
- `review_at`
- `meta.success_signals`：允许记录阶段信号，但不是硬性 KPI

#### 关系类型
- `derived_from`：下层规划来自上层规划
- `focuses_on`：theme 聚焦某个 direction
- `supports`：项目 / 研究 / 写作 / 任务支撑某个规划对象
- `implemented_by`：goal 由项目或任务推进
- `informed_by`：规划调整由哪些日志、研究、回顾事实支撑
- `reflected_in`：某个规划对象在某次 review 中被评估

#### 事件类型
- `suggested`
- `confirmed`
- `reframed`
- `rescheduled`
- `dropped`
- `reviewed`
- `linked`

### 5.2 规划如何避免变成僵硬 OKR

Orbit 不采用“目标 = 指标 = 审计”的默认范式，而采用“方向 + 主题 + 阶段结果 + 节奏校准”的规划哲学。

具体护栏：

1. **愿景不量化。** 五年层没有完成率。
2. **方向可叙事。** 年度方向强调取舍与身份成长，不要求必须数字化。
3. **主题优先于指标。** 季度主题先定义变化，再决定是否需要少量信号辅助判断。
4. **goal 看阶段结果，不看任务堆积。** 结果可以是“完成一轮公开表达”“形成稳定研究机制”，不必都转成数值。
5. **允许主动降级与改写。** 如果现实发生变化，Agent 优先建议重写目标，而不是逼用户继续追一个过时承诺。
6. **回顾面向理解，不面向评分。** review 的问题是“什么在发生、为什么、接下来怎么调”，不是“你得了几分”。

换句话说，规划系统重视节奏、主题和选择质量，而不是把用户生活异化成季度审计。

### 5.3 能力接口

人生规划系统至少向上层 Agent 与其他专题暴露以下能力：

- `planning.get_context(horizon)`：获取指定尺度的当前有效规划上下文。
- `planning.derive_children(object_id, target_horizon)`：从 vision / direction / theme 推导下层候选。
- `planning.suggest_reprioritization(scope)`：根据项目、任务、日志、研究事实给出重排建议。
- `planning.link_object(plan_id, object_id, relation)`：建立规划对象与项目/任务/研究/写作/日志的关系。
- `planning.generate_review(period)`：基于 events + Journal 生成回顾草稿。
- `planning.materialize_today(date)`：在当前约束下生成今日候选节奏。

这些接口输出的是“建议与上下文”，不是“代替用户做出的最终决定”。

## 6. 与其他专题的依赖关系

### 与 01 专题的接口关系
01 负责全局对象模型、信息架构与跨端主导航。本文向 01 输出的是：
- 规划域对象类型（vision / direction / theme / goal / commitment / review）
- 规划关系语义（derived_from / supports / informed_by 等）
- 需要在全局导航中具备的多时间尺度入口：人生规划总览、Today、Review 视图

01 则需要保证这些规划对象能作为一等对象进入全局搜索、关系视图与对象详情页。

### 与 03 专题的接口关系
03 负责项目 / 任务执行系统。本文与 03 的边界是：
- 本文定义“为什么做、这阶段做什么、这一周押注什么”
- 03 定义“具体项目如何拆解、任务如何流转、执行如何记录”
- 接口通过 `theme -> project -> task` 与 `goal -> task` 关系打通
- 03 产生的事件与状态变化，反向成为本文做重排与回顾的事实来源

### 与 08 专题的接口关系
08 负责写作 / 创作工作台。本文给 08 提供：
- 哪些年度方向与季度主题要求表达输出
- 哪些研究已经具备转写作的成熟度
- 哪些写作项目属于长期品牌建设而非临时记录

08 则把草稿状态、发布节奏、写作阻塞与产出事实回传给规划系统，用于判断“输入—思考—输出”是否真正闭环。

### 与 09 专题的接口关系
09 负责 Journal / 日志系统。两者关系最紧密：
- 09 提供原始事实时间轴：做了什么、读了什么、写了什么、在哪些项目上花了时间
- 本文负责把这些事实转成 review、偏差识别与下一轮建议
- Journal 是事实层，规划是解释层，回顾是中间的编译层

如果没有 09，规划会失真；如果没有本文，09 只能记录发生过什么，却无法帮助用户调整方向。

## 7. 风险、边界与设计决策

### 风险 1：规划层级过多，重新变成维护负担
**决策**：控制对象数量，用少数对象 + horizon 表达；很多视图由 Agent 临时生成，不要求用户显式维护每一层。

### 风险 2：五年计划被误用为长期 KPI
**决策**：`vision` 保持叙事文本形态，禁止完成率、禁止自动评分、禁止 Agent 私自改写。

### 风险 3：规划过强，压制探索与机会
**决策**：允许“机会型项目”“探索型研究”“临时写作”存在，但 Agent 要明确提示它们是否偏离当前主题，由用户选择是否保留，而不是默认拒绝。

### 风险 4：Agent 建议过多，打断用户生活
**决策**：只有出现明显规划差值时才主动提醒；建议应以“可执行重排”形式出现，而不是泛泛洞察轰炸。

### 风险 5：用户把确认权让渡给 Agent，系统逐渐替用户决定人生
**决策**：在产品和协议层显式区分 `user_written`、`agent_suggested`、`user_confirmed`。所有高层规划对象都必须保留决策来源与确认痕迹。

### 最终设计结论

Orbit 的人生规划系统不是另一个目标管理模块，而是整个 Agent 原生助手的长期对齐层。它用五年 / 年 / 季 / 月 / 周 / 日的连续收敛机制，把愿景转成主题、把主题转成项目组合、把项目组合转成今天的节奏；同时又通过 Journal 与 review 把事实持续回灌回来。用户不用维护巨型规划表，只需要在关键节点表达意图、做出选择；剩下的拆解、联动、重排与回顾，交给 Agent 作为系统级能力完成。
