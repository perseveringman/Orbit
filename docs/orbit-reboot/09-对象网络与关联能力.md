# 09. 对象网络与关联能力

## 1. 设计定位

对象网络不是 Orbit 里一个“知识管理功能”，而是整款 Agent-first 应用的公共底座。只要某个东西满足以下任一条件，它就不该只是页面里的局部状态，而应成为可寻址、可回链、可查询、可被 Agent 消费的对象：

1. 用户会反复返回它；
2. 它会与其他内容形成稳定关系；
3. 它可能成为任务、研究、写作或回顾的上下文；
4. 用户未来会追问“它和什么有关”；
5. Agent 需要把它纳入长期推理，而不是只在当前对话里短暂使用。

因此，材料、笔记、研究空间、任务、项目、文章、AI 对话、写作产物、日志条目都应进入统一对象网络。它们不是“模块内数据”，而是 Orbit 的工作记忆与长期记忆节点。真正不应对象化的，是纯瞬时 UI 状态与底层技术遥测；那些信息应进入 `events`，而不是伪装成用户对象。

本专题需要主动消解一个核心冲突：一部分文档提出 `content_links / entity_links / tag_links` 分裂方案，另一部分文档提出“一张 `links` 表”方案。本文最终结论是：**物理真相层采用“类型化对象表 + object_index + 单一 links + events”**；`content_links / entity_links / tag_links` 只保留为**查询视图、接口别名或服务层 Facet**，不再作为独立持久化真相表。这样既保留 Capacities 式“对象 × 链接”的统一性，也更适合 Agent-first 架构对跨对象推理、解释与纠错的要求。

## 2. 关键用户场景

### 2.1 从材料到行动的连续关系链

用户读到一篇文章，划出两段高亮，补了一条批注，随后把它们拖入某个研究空间；研究对话里又生成了一条判断，判断被保存为研究产物，并进一步拆出一个任务，最后任务推进出一篇文章草稿。这里每一步都不该丢失上一跳：

`article → highlight → note → research_space → ai_chat → research_artifact → task → writing_output`

对象网络的职责不是“显示很多连线”，而是保证用户随时都能从任何一点回到证据、上下文与后续产物。

### 2.2 从任意对象反查全部上下文

用户打开一条任务“写 Orbit reboot 架构文档”，系统不应只展示任务描述，而应能立刻反查：

- 它由哪次研究结论触发；
- 它关联了哪些文章、高亮、笔记；
- 最近哪次 AI 对话讨论过它；
- 已有哪些草稿或文档块是它的产出；
- 最近 Journal / 日志里对它有哪些阻塞或进展记录。

这就是 Backlinks 在 Orbit 中的价值：不是“笔记回链面板”，而是“从任何对象回看整条工作链”。

### 2.3 AI 对话不再是聊天气泡，而是可复用对象

用户在阅读器里问“这篇文章与我正在做的项目有什么关系？”，这次对话本身应成为对象，因为它满足三件事：有上下文、有推理、有后续价值。它既要链接当前文章，也要链接被提及的项目、研究空间、笔记和后续写作产物；未来打开项目或文章时，都应看到“这里曾发生过一段关键 AI 推理”。

### 2.4 日志不是尾声，而是对象网络的时间维度

日志条目、Journal 记录、Review 摘要不是对象网络外部的流水账，而是为对象关系补上“何时发生、为何变化、谁确认过”的时间层。项目、任务、研究空间和写作产物都需要从日志中获得过程证据；反过来，日志也需要能嵌入当天涉及的对象集合，形成“按时间看对象、按对象看时间”的双向穿透。

## 3. 核心设计

### 3.1 什么应该成为对象，什么不应该

本文建议用统一判据区分对象与事件：

| 类型 | 是否对象 | 原因 |
|---|---|---|
| 文章、书籍、网页材料、附件 | 是 | 可被阅读、引用、进入研究与写作 |
| 高亮、摘录、片段卡 | 是 | 是最小证据颗粒度，需要回链与溯源 |
| 笔记、批注、总结、Journal 条目 | 是 | 是用户思考或记录本身 |
| 研究空间、问题、缺口、研究产物 | 是 | 是长期推理容器与产物 |
| 项目、里程碑、任务、回顾 | 是 | 是执行与决策对象 |
| AI 对话、AI 回答快照 | 是 | 是可再利用的推理资产 |
| 文档、草稿、发布成品 | 是 | 是输出对象与下游引用源 |
| 标签 | 是 | 作为轻量语义对象参与聚合与筛选 |
| 编辑器光标、当前 tab、滚动位置 | 否 | 属于瞬时 UI 状态 |
| “任务被完成”“链接被接受”等事实 | 否，对应 event | 是状态变化，不是稳定内容实体 |

因此 Orbit 的对象分类可归为五层：

1. **材料对象**：article、book、asset、highlight、quote；
2. **思考对象**：note、concept、research_question、research_claim、ai_chat；
3. **执行对象**：project、milestone、task、review、directive；
4. **产出对象**：document、draft、post、report、presentation；
5. **时间对象**：journal_entry、log_entry、daily_summary、weekly_review。

这些对象都保留各自类型表；统一性不来自“万物一张 objects 表”，而来自 **`object_index` 负责寻址，`links` 负责关系，`events` 负责事实变化**。

### 3.2 关系图谱：对象之间如何形成网络

Orbit 的关系图不是“知识对象”专用，而是把输入、思考、行动、输出与时间全部接到同一张网上。典型关系链如下：

- **材料 → 笔记**：`highlight --annotated_by--> note`
- **材料 / 笔记 → 研究空间**：`article/note/highlight --supports--> research_space`
- **研究空间 → AI 对话**：`research_space --hosts--> ai_chat`
- **AI 对话 → 研究产物**：`ai_chat --produces--> research_artifact`
- **研究产物 / 笔记 → 任务**：`research_artifact/note --suggests--> task`
- **任务 / 项目 → 写作产物**：`task/project --outputs--> document/draft/post`
- **日志 → 任何对象**：`journal_entry --reflects_on--> project/task/research_space/document`

这张图有两个关键特征：

1. **无中心目录，只有多重关系**。项目不是所有内容的唯一父目录；研究空间也不是。一个对象可以同时支撑项目、研究与写作。
2. **每条关系都应有语义，而不是只有“related”**。Orbit 需要的是可供 Agent 推理的关系，不是只能画图的弱连接。

### 3.3 对“单一 links”与“多张 link 表”冲突的最终推荐

最终推荐如下：

#### 物理真相层

- 各对象保留类型表；
- 一张 `object_index` 做全局索引；
- **一张 `links` 表做全部跨对象关系真相层**；
- 一张 `events` 表记录状态与决策事实；
- 可选一张 `link_evidence` 表存 AI 关联解释材料，但不再拆出 `content_links / entity_links / tag_links` 真相表。

#### 为什么不继续保留三张或更多 link 表

1. **Agent 编排需要统一遍历**。任务、研究、写作、日志、AI 对话之间的跳转本质一致，拆表会让查询、权限、解释与纠错都分叉。
2. **“内容 / 实体 / 标签”只是视角，不是底层本体**。文章是内容，也是实体；标签本身也是对象；AI 对话既是内容，也是关系节点。
3. **Backlinks、跨对象检索、关系建议都需要单一索引面**。否则每次都要 UNION 与多路去重。
4. **Capacities 的价值不在多表分类，而在统一可链接性**。Orbit 只是在此基础上进一步加入 Agent 来源、解释与审批。

#### 如何保留原方案的便利性

- `content_links`：改为 SQL View / Repository Facet，表示“目标是可消费内容对象”的链接集合；
- `entity_links`：改为 View，表示“跨对象通用关系”的默认入口；
- `tag_links`：不再独立存储，统一为 `links(relation='tagged_with', target_type='tag')`；
- 对外 API 仍可保留 `project.attach_content`、`tag.assign` 这类友好能力名，但底层都写入 `links`。

也就是说：**分裂方案保留为使用层语义，不保留为存储层真相。**

### 3.4 统一关系语义词表

关系语义必须可控，不允许页面随意发明。建议维护全局 relation vocabulary，并按语义族分层：

| 语义族 | relation | 用途 |
|---|---|---|
| 结构关系 | `contains` / `parent_of` / `child_of` | 空间、项目、阶段性容器关系 |
| 来源关系 | `derived_from` / `excerpted_from` / `annotated_by` / `evidenced_by` | 追溯来源与证据链 |
| 支撑关系 | `supports` / `informs` / `context_for` | 材料、笔记、研究对任务/项目/写作的支撑 |
| 执行关系 | `blocks` / `depends_on` / `suggests` / `decides` | 任务与研究中的行动约束 |
| 产出关系 | `produces` / `outputs` / `published_as` / `feeds` | 从任务、研究到写作成品 |
| 讨论关系 | `discusses` / `asks_about` / `answers_for` | AI 对话与对象之间的上下文关系 |
| 反思关系 | `reflects_on` / `reviews` / `reframes` | 日志、回顾对对象的解释关系 |
| 轻量聚合 | `tagged_with` / `about` / `relates_to` | 标签与弱语义连接 |

词表设计原则：

1. `relates_to` 只作为兜底关系，不应成为主干；
2. 有方向的关系必须保留方向；
3. 若需要 UI 双向展示，可在词表中定义反向显示名，而不是插入两条重复边；
4. 强约束结构关系可在类型表保留本地外键，但必须同步投影到 `links` 供全局图谱查询。

### 3.5 Backlinks、跨对象查询与关联建议

#### Backlinks

Backlinks 面板必须统一支持：

- 入链 / 出链双视角；
- 按 relation 语义族过滤；
- 按 `origin = human | ai` 过滤；
- 按对象类型过滤；
- 以“为何关联”的解释卡展示，而不是只列标题。

#### 典型查询模式

1. **对象上下文包**：给某个 task 拉取其 2 跳内的笔记、研究产物、AI 对话、草稿与最近日志。
2. **跨对象工作链**：查找“某篇文章最终影响了哪些任务与输出”。
3. **孤岛检测**：找出 30 天内新增但没有任何有效 relation 的 note / article / ai_chat。
4. **证据追溯**：从某段写作成品反查其引用的高亮、笔记、研究结论与对话。
5. **时间反查**：找出过去 7 天所有 `reflects_on` 某项目的日志与 review。
6. **纠错视图**：找出所有 `origin='ai' AND status='rejected'` 的链接，避免重复建议。

#### 关联建议

关联建议不是“自动把所有相似对象连起来”，而是一个可审计的建议系统。建议来源主要有四类：

- 文本语义相似；
- 共现上下文（同项目、同研究空间、同会话）；
- 溯源链邻近（同一来源、同一引用链）；
- 时间邻近与行为模式（同一天被连续阅读、提问、写作）。

建议结果默认以 `status='proposed'` 写入链接层，等待用户接受、改写或拒绝；高置信、低风险关系才允许自动升级为已生效链接，但仍必须以 `origin='ai'` 明确标注。

## 4. Agent 行为与自动化机制

### 4.1 human / ai link source 的最终设计

所有链接统一保留：

- `origin`: `human | ai`
- `source_channel`: `manual | wikilink | drag_drop | quick_capture | agent_chat | ingest | review | import`
- `status`: `proposed | active | rejected | archived`

其中最重要的不是“由哪个模块创建”，而是**这是用户主张，还是 AI 建议**。这延续了研究基线中的主体性保护原则：AI 可以大量发现连接，但用户永远可以一键切回“只看我的关联”。

### 4.2 AI 自动关联的可解释性

AI 链接若不可解释，就不能成为长期记忆。每条 AI 生成的链接至少需要可展示以下解释字段：

- `why_summary`：一句话解释为什么建议关联；
- `evidence_refs`：指向触发关联的高亮、段落、对话轮次、日志片段；
- `signals_json`：相似度、共现、来源链、时间邻近等信号分解；
- `agent_run_id / model_hint`：由哪次 Agent 运行产生；
- `confidence`：供排序使用，但不直接替代用户判断；
- `supersedes_link_id`：若此链接是在纠错后替代旧链接，需可追溯。

UI 上必须能回答三个问题：

1. 它为什么出现；
2. 它依据了哪些对象或片段；
3. 如果我不同意，应该在哪里改。

### 4.3 用户纠错机制

对象网络不是只读图谱，必须允许用户快速校正 AI 的错误联想。建议提供五种纠错动作：

1. **接受**：保留 relation，并可把 `origin` 提升为 `human_confirmed` 的展示态；
2. **改写语义**：例如把 `supports` 改成 `relates_to` 或 `derived_from`；
3. **拒绝**：将链接置为 `rejected`，并记录 rejection reason；
4. **静音规则**：对某类建议模式设置局部 suppress，避免重复骚扰；
5. **拆分 / 合并**：当 AI 把多个对象混成一条关系或关系粒度过粗时，允许用户重构。

纠错后的结果必须反向进入 Agent 的后续建议：

- 被拒绝的链接不能在短期内再次原样推荐；
- 被频繁改写的 relation 说明词表或模型提示有问题，应进入系统级优化；
- 用户确认过的链接应在后续上下文召回中获得更高权重。

### 4.4 面向 Agent-first 的“对象 × 链接”适配

Capacities 的“对象 × 链接”主要服务人类浏览与手工建网；Orbit 需要把这套能力升级为 Agent 的工作内存总线：

1. **链接是 Agent 的上下文装配器**：Planner、Research、Writer、Review Agent 都应从同一图谱取材。
2. **对象是 Agent 的交接单元**：子 Agent 不应继承整段聊天，而应继承对象包与关系包。
3. **AI 对话本身进入图谱**：对话不再只是入口 UI，而是推理过程对象。
4. **链接既服务发现，也服务执行**：`supports`、`blocks`、`outputs` 直接参与任务编排与写作引用，而不只是展示关系图。
5. **图谱必须是可审批的**：Orbit 不是自动吸纳一切的黑箱知识库，而是“AI 可提议、用户可校正、系统可回溯”的协作网络。

因此，Orbit 里的对象网络不是 Capacities 的桌面版复刻，而是 **Agent-first、可解释、可纠错、可执行的关系基础设施**。

## 5. 数据模型 / 接口 / 能力边界

### 5.1 推荐数据模型

```sql
-- 各对象类型仍保留各自类型表
-- 全局索引
object_index(
  object_type,
  object_id,
  title,
  summary,
  status,
  layer,
  created_at,
  updated_at
)

-- 统一关系真相层
links(
  id,
  source_type,
  source_id,
  target_type,
  target_id,
  relation,
  origin,          -- human | ai
  source_channel,  -- manual | wikilink | drag_drop | agent_chat | ingest ...
  status,          -- proposed | active | rejected | archived
  confidence,
  why_summary,
  context_json,
  created_by,
  created_at,
  updated_at,
  UNIQUE(source_type, source_id, target_type, target_id, relation, status)
)

-- 可选：AI 解释与证据
link_evidence(
  id,
  link_id,
  evidence_type,   -- quote | block | message | event | similarity
  evidence_ref,
  payload_json,
  created_at
)
```

关键设计点：

- `tag` 也是对象类型，标签关系统一走 `links(relation='tagged_with')`；
- `ai_chat`、`journal_entry`、`writing_output` 都进入 `object_index`；
- `content_links / entity_links / tag_links` 若需要兼容旧接口，只做 view，不做真相表。

### 5.2 关键能力接口

建议把对象网络暴露为系统级能力，而不是页面私有逻辑：

- `object.get / object.search / object.bundle`
- `link.create / link.update / link.reject / link.list`
- `link.backlinks / link.neighbors / link.trace_path`
- `link.suggest / link.explain / link.confirm`
- `graph.query_cross_objects`
- `graph.find_orphans`
- `graph.activity_timeline`

这些接口同时服务本地 UI、内部 Agent 与 MCP 能力层。

### 5.3 能力边界

本专题明确不做三件事：

1. 不把所有 event 都提升为对象，避免对象网络被底层遥测污染；
2. 不允许页面私自新增 relation 字符串，必须走全局词表；
3. 不把 AI 建议静默固化为用户主张，除非是低风险自动链接且保留清晰来源。

| **15** | 15 的统一能力层与外部 Agent 接入必须通过 09 的受控图谱接口暴露 `object.*`、`link.*`、`graph.*` 能力，不能绕过词表、来源标注与纠错机制直接写库。 |

## 7. 风险、边界与设计决策

1. **最大风险不是“连得不够多”，而是“连得过于随意”**。因此必须把 relation 词表、AI 解释与用户纠错做成一等机制。
2. **单一 `links` 表不等于无约束**。统一的是存储层，不是语义随意性；强结构关系仍可有本地外键与系统投影。
3. **对象网络不是自动知识真理机**。AI 可发现、可建议，但最终可用图谱必须允许用户驳回、改写与隐藏。
4. **日志对象与事件层必须分开**。Journal 条目是对象；“今天完成了任务”是事件。把两者混同会导致图谱失真。
5. **最终设计决策**：
   - 采用“类型化对象表 + `object_index` + 单一 `links` + `events`”作为最终基础结构；
   - `content_links / entity_links / tag_links` 降级为兼容视图或接口 Facet；
   - 所有关联统一具备 relation vocabulary、`origin`、`status`、解释证据与纠错闭环；
   - Orbit 对象网络的目标不是展示图谱，而是让 Agent 和用户都能在任意对象上回答“它从哪里来、支持什么、影响了什么、接下来该连到哪里”。
