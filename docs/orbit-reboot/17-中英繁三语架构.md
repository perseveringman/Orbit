# 中英繁三语架构

## 1. 设计定位

Orbit 的三语能力不是“做完产品后再补一个语言包”，而是从系统边界开始就内建的基础设施。它同时覆盖五层对象：

1. **产品界面语言**：导航、按钮、空状态、系统提示、错误文案。
2. **内容对象语言**：文章、书籍、笔记、研究产物、任务描述、Agent 结果本身的语言归属。
3. **阅读翻译语言**：Reader 中原文、译文、术语注释、双语对照的呈现与缓存。
4. **Agent 输出语言**：主 Agent、子 Agent、自动摘要、自动命名、自动标签建议的语言策略。
5. **检索与排版语言**：搜索分词、同义词扩展、简繁转换、日期格式、字体回退、行高与标点策略。

因此，本专题的目标不是定义一套 React Web 的 i18n 用法，而是定义一套在 **Electron / iOS / Web 三端一致生效**、并可进入对象网络与 Agent 编排层的语言系统。Orbit 从第一天起只支持三个正式产品 locale：`zh-CN`、`en-US`、`zh-TW`；但对象内容语言、翻译目标语言与 Agent 输出语言可以彼此不同，系统必须显式区分，而不是偷懒地用一个 `locale` 字段混写全部语义。

## 2. 关键用户场景

### 场景 A：首次启动即完成三语落地

用户第一次在任一端打开 Orbit，系统读取设备偏好语言、地区、24 小时制、数字格式与字体可用性，生成一份 `locale_profile`。若系统语言为繁中，UI 直接进入 `zh-TW`；若为英文，进入 `en-US`。桌面端、Web 端、iOS 端都使用同一份语言资源主干与同一套 fallback 规则，避免“桌面端翻得完整、移动端还残留英文 key”的分裂。

### 场景 B：中文界面下阅读英文材料并生成繁中解释

用户 UI 语言为简中，但正在阅读英文论文，希望保持原文不变，同时开启繁中术语解释与段落级对照翻译。系统需要识别：UI 语言 = `zh-CN`、内容语言 = `en`、翻译目标 = `zh-TW`，并把译文作为派生层保存，而不是改写原始内容。

### 场景 C：Agent 始终按当前工作语境回答

用户界面是英文，但在一个繁体中文项目空间里工作，且明确要求“以后在此项目里都用繁體中文回答”。此时 Agent 不能简单跟随 UI 语言，而应遵守工作区语言策略与用户显式偏好：界面英文、对象标题可能混合、Agent 输出繁中、引用保持原文。

### 场景 D：搜索必须跨语言、跨简繁可召回

用户输入“第二大脑”“第二大腦”或 “second brain”，都应能召回相关中文、繁中、英文内容；但结果排序要优先与当前工作语言、术语表和对象上下文一致，不能只靠粗暴的全文字符串匹配。

## 3. 核心设计

### 3.1 Locale 体系：拆开 UI、内容、输出与检索语义

Orbit 采用 BCP 47 风格 locale 标识，但在产品层只维护三个 canonical locale：`zh-CN`、`en-US`、`zh-TW`。系统内部至少区分以下字段：

| 字段 | 含义 | 示例 |
|---|---|---|
| `app_locale` | 产品界面语言 | `zh-CN` |
| `system_locale` | 设备/浏览器原始语言偏好 | `zh-HK` |
| `content_locale` | 对象主内容语言 | `en` |
| `content_script` | 文字脚本 | `Latn` / `Hans` / `Hant` |
| `agent_output_locale` | Agent 默认输出语言 | `zh-TW` |
| `translation_target_locale` | Reader/摘要译文目标语言 | `zh-CN` |
| `search_locale` | 查询解析与排序的首选语言 | `zh-CN` |

设计决策如下：

1. **UI locale 不代表内容语言。** 用户可用英文 UI 阅读中文资料。
2. **内容语言不等于翻译目标。** 英文文章可以翻译为简中或繁中。
3. **Agent 输出语言独立于 UI locale。** 项目、空间、会话都可覆写。
4. **locale fallback 只用于 UI 文案，不用于内容真相。** 若某条对象没有繁中版本，不能自动把简中内容当繁中真相替代。

UI 文案 fallback 链建议为：

- `zh-TW -> en-US`
- `zh-CN -> en-US`
- `en-US -> en-US`

不做 `zh-CN -> zh-TW` 或 `zh-TW -> zh-CN` 的默认文案回退，避免术语与语气漂移。简繁互转只用于检索扩展、辅助理解和机器候选，不作为产品正式文案来源。

### 3.2 三端 monorepo 下的翻译资源组织

国际化资源按“**共享主干 + 平台适配层 + 自动校验**”组织，而不是每个端各存一套文案：

```text
apps/
  desktop/
  ios/
  web/
packages/
  i18n-core/          # locale 解析、fallback、Intl 包装、hooks/adapter
  i18n-resources/     # 三语 UI 文案，按 domain namespace 组织
  termbase/           # 术语表、禁用译法、别名、prompt glossary
  linguistics/        # 语言检测、简繁归一、分词器、查询扩展
  agent-language/     # Agent 语言策略、prompt locale contract
```

资源组织原则：

1. **按领域 namespace，不按平台复制。** 如 `common`、`reader`、`search`、`agent`、`settings`、`billing`。
2. **一个 key 只表达一个产品意图。** 不允许同 key 在桌面和移动端承载不同语义。
3. **平台差异走 override。** 如 iOS 原生权限弹窗文案、桌面快捷键提示，可在 adapter 层做少量覆写，但主翻译源仍来自共享包。
4. **编译产物面向三端。** Web/Electron 直接消费 JSON bundle；iOS 端消费同源编译后的静态资源或 codegen 结果，避免再维护一套 `Localizable.strings` 作为第二真相。

旧 `ARCHITECTURE.md` 中的 i18next 基础可保留，但升级为“共享 locale runtime + 资源编译管线”，而不是仅服务桌面渲染层。

### 3.3 术语表治理：把产品词汇、阅读术语与 Agent 提示词统一到 concept 层

三语系统最容易坏掉的不是缺翻译，而是**同一概念在不同端、不同页面、不同 Agent 提示词里被翻成不同词**。因此 Orbit 必须有一套独立于 UI 资源的 `termbase`：

| 字段 | 说明 |
|---|---|
| `concept_id` | 稳定概念 ID，如 `concept.reader.inbox` |
| `domain` | 所属域：reader / research / agent / task / system |
| `preferred_terms` | 三语首选词 |
| `aliases` | 可识别别名与历史叫法 |
| `forbidden_terms` | 禁止使用的旧译法或歧义词 |
| `definition` | 该概念的产品定义 |
| `usage_note` | 语境限制，如按钮、标题、Agent 用语 |

术语表用途有三类：

1. **UI 翻译校验**：新文案提交时检查是否违背首选术语。
2. **Agent 生成约束**：系统 prompt 明确要求 Agent 优先使用术语表首选词。
3. **搜索扩展**：同义词、别名、简繁词形映射进入查询扩展。

这样“收件箱 / Inbox / 收件匣”“高亮 / Highlight / 劃線”不会在不同模块里飘移成多套说法。

### 3.4 内容语言检测：对象级 + 片段级双层识别

Orbit 的内容不是只有一整篇文章。标题、正文、摘录、高亮、笔记 block、Agent 回复片段都可能是不同语言。因此语言检测必须至少支持两层：

1. **对象级检测**：对文章、笔记、文档、研究产物给出主语言、脚本、置信度。
2. **片段级检测**：对段落、Block、Selection Context、高亮 quote 给出局部语言标签。

建议写入如下元数据：

- `language_detected`
- `script_detected`
- `language_confidence`
- `mixed_language_flg`
- `dominant_locale`
- `segment_language_map`

识别流程：

1. 导入对象时先做对象级检测。
2. 若检测到混合内容（如英文正文夹中文术语），再按段或 block 补做片段级检测。
3. Reader 选中文本、Agent 引用、翻译请求、搜索索引都直接复用片段级结果。

关键边界：**语言检测只是元数据，不改写原文。** 对象真相仍留在 `sources/`；检测结果进入数据库与索引层；派生翻译进入 `wiki/` 或 `.orbit/cache/` 的译文层。

### 3.5 Reader 翻译层：原文真相不动，译文作为可版本化派生层

阅读翻译不是“把文章翻完显示一下”，而是一层可缓存、可引用、可回溯的派生内容系统。

建议将 Reader 翻译拆成四层：

1. **原文层**：抓取到的 article/book/document 原始内容，只读。
2. **片段映射层**：段落 ID、句子 ID、锚点与原文版本号。
3. **译文层**：按 `target_locale` 生成的句段译文，可包含术语注释、专名保留策略。
4. **呈现层**：原文、双语对照、仅译文、悬浮译注、术语卡片等不同阅读模式。

译文对象建议至少包含：

- `source_object_id`
- `source_version`
- `segment_id`
- `source_locale`
- `target_locale`
- `translation_provider`
- `translation_mode`（machine / glossary-aware / human-edited）
- `translation_text`
- `quality_score`
- `created_at`

设计边界：

- **不覆盖原文，不反写 sources。**
- **允许多目标语言并存。** 同一英文文章可同时有简中与繁中译文。
- **支持术语表优先。** 产品、研究、专业词汇先走 Orbit termbase，再走模型翻译。
- **支持 Agent 联动。** 用户可从某段译文直接追问“为何这样翻”，Agent 返回解释时必须引用原文片段与术语决策。

### 3.6 Agent 语言策略：内部控制面稳定，用户表面可切换

Agent 系统里必须把“内部运行语言”和“用户可见输出语言”分开。建议采用双层 contract：

1. **控制面语言**：系统 prompt、工具 schema、状态对象字段、日志 reason code 使用稳定内部语义，不随用户语言切换而改写结构。
2. **表面层语言**：最终输出、步骤说明、建议标题、按钮 label、摘要结果按 `agent_output_locale` 本地化。

策略优先级建议为：

`用户显式指令 > 当前空间语言策略 > 当前对象语言 > 用户长期偏好 > app_locale`

具体规则：

- Agent 默认跟随当前空间的 `agent_output_locale`。
- 引用外部材料时保留原文，并可附目标语言解释。
- 多 Agent 协作时，子 Agent 可以内部用稳定控制语言执行，但交回 orchestrator 时附带 `response_locale` 元数据；由 orchestrator 统一决定是否本地化后再呈现。
- 自动命名、自动标签、自动摘要都必须标注其生成语言，避免英文标题被错误塞进中文工作区。

这与 Hermes 式 Agent 架构兼容：system prompt 稳定、上下文动态注入、tool registry 不随语言改 schema，但最终回答层可以被 locale policy 约束。

### 3.7 多语言搜索索引：原文索引、归一索引、语义索引三层并存

三语搜索不能只靠一个全文索引。建议统一搜索层至少维护三种索引视图：

1. **原文倒排索引**：忠实保留原始 token，服务精确匹配与引用定位。
2. **归一化语言索引**：做大小写、全半角、标点、简繁、词形、术语别名归一，用于高召回。
3. **语义索引**：按对象或片段生成 embedding，服务跨语言语义检索。

中文相关设计：

- 简中、繁中分别保存原文字段。
- 同时生成 `zh_hans_normalized` 与 `zh_hant_normalized`，支持双向召回。
- 术语表中的首选词、别名、英文术语进入 synonym graph。
- 查询时先检测语言，再决定分词器、简繁扩展与排序 boost。

排序信号应综合：

- 当前 `search_locale`
- 对象 `content_locale`
- 是否命中术语表首选词
- 是否命中用户工作区语言
- 是否存在对应译文/摘要
- 对象关系强度（项目、研究、最近阅读、最近编辑）

这样用户搜“知识库”“knowledge base”“知識庫”时，都能找到同一批对象，但当前语言更一致、上下文更相关的结果排在前面。

### 3.8 日期、排版、字体：把语言差异上收到 design token 与 formatter 层

三语一致不只靠翻译，还包括可读性。Orbit 需要一套 locale-aware 的展示 token：

- **日期/时间**：统一使用 shared formatter，支持 `2026-04-08`、`Apr 8, 2026`、`2026年4月8日` 等区域化输出。
- **数字/单位**：千分位、小数点、百分比、货币格式按 locale 渲染。
- **字体回退链**：为 `zh-CN`、`zh-TW`、`en-US` 分别定义首选字体与 fallback，不允许浏览器/系统随机回退导致 UI 风格漂移。
- **行高与字重**：中文与英文在密度、断词、按钮长度上不同，应由 token 控制，而非页面临时打补丁。
- **截断与换行策略**：英文更适合按单词换行，中文更适合按字/短词断行；按钮、列表、卡片标题都要有 locale-aware 约束。

这部分必须进入共享 design token 与 formatter API，而不是留给各端各自处理。

## 4. Agent 行为与自动化机制

### 4.1 自动语言决策流程

每次 Agent 执行前，编排层都应生成 `LanguageContext`：

- 当前 app locale
- 当前空间/项目语言策略
- 当前对象内容语言
- 用户显式输出要求
- 翻译目标语言
- 术语表 domain

Agent 先决定“内部执行语言是否需要切换”，再决定“最终展示语言是什么”。默认不让模型自己猜。

### 4.2 自动翻译与自动命名的边界

系统可自动做这些事：

- 为导入内容生成语言标签
- 为 Reader 提供译文候选
- 为对象生成目标语言摘要或别名标题
- 在搜索时做简繁/术语扩展

系统不应静默做这些事：

- 未经标注就把原文标题替换成译文标题
- 未记录来源就把机器翻译写回用户正文
- 因 UI 语言变化而改写项目、任务、笔记真相内容

### 4.3 Agent 输出语言守卫

当模型输出语言偏离策略时，系统应有后处理守卫：

1. 检测输出语言是否符合 `agent_output_locale`。
2. 若不符合，优先做轻量重写，而不是重新跑整轮推理。
3. 对引用、代码、专有名词保持原样，仅本地化包裹说明。
4. 将最终语言决策记录到 event/log，供后续质量评估。

## 5. 数据模型 / 接口 / 能力边界

### 5.1 关键对象字段

```ts
interface LocaleProfile {
  appLocale: 'zh-CN' | 'en-US' | 'zh-TW';
  systemLocale?: string;
  agentOutputLocale: 'zh-CN' | 'en-US' | 'zh-TW';
  searchLocale: 'zh-CN' | 'en-US' | 'zh-TW';
  timeZone: string;
  hourCycle: 'h12' | 'h23';
}

interface ContentLanguageMeta {
  objectId: string;
  dominantLocale?: string;
  dominantLanguage: string;
  dominantScript?: 'Hans' | 'Hant' | 'Latn';
  confidence: number;
  mixedLanguage: boolean;
}

interface TranslationArtifact {
  id: string;
  sourceObjectId: string;
  segmentId: string;
  sourceLocale: string;
  targetLocale: 'zh-CN' | 'en-US' | 'zh-TW';
  provider: string;
  mode: 'machine' | 'glossary-aware' | 'human-edited';
  text: string;
}
```

### 5.2 核心能力接口

| 接口 | 用途 |
|---|---|
| `locale.resolveProfile()` | 生成用户当前语言画像 |
| `locale.setAppLocale()` | 切换产品界面语言 |
| `language.detect(objectOrSegment)` | 检测对象或片段语言 |
| `termbase.resolve(domain, locale)` | 获取术语首选词与别名 |
| `translation.generate()` | 生成段落/句子译文 |
| `translation.getOverlay()` | 获取 Reader 对照层 |
| `agent_language.resolvePolicy()` | 生成本轮 Agent 语言策略 |
| `search.expandMultilingualQuery()` | 做简繁/别名/术语扩展 |
| `format.date/number/relativeTime()` | 三端统一格式化输出 |

### 5.3 能力边界

1. **本专题不负责决定哪家翻译模型或分词器厂商。** 它定义的是能力接口与治理边界。
2. **本专题不把译文当真相内容。** 真相仍在原始对象；译文是派生资产。
3. **本专题不允许平台私有翻译资源成为第一真相。** 平台只可做适配，不可各自维护主文案。
4. **本专题要求所有语言自动化都可追踪。** 包括检测来源、译文来源、术语命中、Agent 输出 locale。

## 6. 与其他专题的依赖关系

| 依赖专题 | 关系说明 |
|---|---|
| **10（三端应用骨架与共享运行时）** | 17 依赖 10 提供 Electron / iOS / Web 的共享包装边界、启动期 locale 注入、设置同步与资源分发机制。没有 10，三语能力只能停留在单端前端层，无法形成统一 runtime。 |
| **04（Reader 与内容消费层）** | 17 需要 04 提供文章/书籍/网页的段落锚点、句段切分、阅读模式与浮层容器，Reader 翻译层才能把译文稳定挂到原文片段上，并支持双语对照、术语解释与引用追溯。 |
| **11（LocalFirst 同步与 GDPR）** | 17 的 locale 偏好、术语表权限、翻译缓存、外部翻译出境与敏感内容保护，需要由 11 规定同步、加密与数据边界。没有 11，三语能力无法满足本地优先与合规要求。 |
| **13（文件系统优先数据架构与索引管线）** | 17 定义的语言检测、简繁归一、术语别名与多语 embedding 策略，需要由 13 真正落到索引管线、查询扩展与排序模型中；否则三语只会体现在 UI，不会体现在检索质量上。 |
| **15（MCP 与开放能力层）** | 外部翻译器、分词器、术语库、Agent 工具都通过 15 接入。17 负责规定：外部能力只能产生候选检测/候选译文/候选术语，最终对象化、缓存、权限审计与用户呈现仍由 Orbit 内部语言层控制。 |
| **18（设计系统与排版展示层）** | 17 的字体回退、字号密度、按钮长度、日期格式、断行策略，都必须进入 18 的 token、组件约束与 formatter API。没有 18，三语文案会在不同端出现截断、跳字重、行高失衡与视觉风格漂移。 |

## 7. 风险、边界与设计决策

1. **最大风险是把“界面语言”误当成“全部语言策略”。** 本文明确拆分 UI、内容、翻译、Agent、搜索五类语言语义。
2. **最大一致性风险是术语漂移。** 因此术语表必须脱离页面文案独立治理，并同时服务 UI、Agent 与搜索。
3. **最大数据风险是机器翻译污染原文。** 设计上明确原文只读，译文只作为派生层保存。
4. **最大检索风险是只做字符串简繁转换。** Orbit 必须把简繁归一、分词、别名、embedding 一起纳入搜索架构。
5. **最大跨端风险是各平台维护各自语言包。** 本文明确共享资源为第一真相，平台只做 adapter，不做第二翻译源。
6. **Agent 必须可被语言策略约束。** 不允许子 Agent 各自随意切语言，最终输出必须受 orchestrator 的 `LanguageContext` 收敛。
7. **三语不是附加功能，而是对象网络与 Agent-first 产品的一部分。** 只有当语言元数据、译文层、术语表、搜索索引与排版系统都进入同一架构，Orbit 才能真正做到从第一天完整支持简中、English、繁體中文。
