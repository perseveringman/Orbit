# 19. 服务端架构

## 1. 设计定位

Orbit 是本地优先应用，但"本地优先"不等于"没有服务端"。用户需要跨设备同步、账号恢复、设备管理和数据删除，这些都需要一个后端服务。

本专题的核心原则是：**服务端是加密中转站，不是内容主库。** 服务端永远不接触用户内容明文，它的存在只为解决本地设备做不到的事：跨设备数据传输、设备信任协调、账号生命周期管理。

### 设计约束

1. **零知识**：服务端存储的全是密文，工程师在生产环境无法查阅用户内容
2. **极简**：单体服务 + 对象存储，不搞微服务，不做消息队列
3. **可自托管**：架构足够简单，用户未来可以把整个后端跑在自己的 VPS 上
4. **桌面端优先**：MVP 阶段只需服务桌面端同步，Web 和 iOS 的特殊需求后补

## 2. 服务端职责边界

### 服务端负责

| 职责 | 说明 |
|------|------|
| 账号管理 | 注册、登录、密码重置、账号删除 |
| 设备管理 | 设备注册、配对确认、撤权、设备列表 |
| 加密对象存储 | 接收/返回加密后的 blob，不解密 |
| 同步游标 | 记录每台设备的同步进度 |
| 变更通知 | 通知其他设备"有新数据可拉取" |
| GDPR 执行 | 处理导出工单和删除工单 |

### 服务端不负责

| 不做的事 | 原因 |
|----------|------|
| 存储或检索用户内容明文 | 零知识原则 |
| 全文搜索 | 搜索在本地执行 |
| Agent 推理 | Agent 运行在客户端，调用外部 LLM 由客户端直连 |
| 内容推荐 | 推荐逻辑在本地基于对象图谱执行 |
| 实时协作 | 个人产品，不需要 |
| 复杂权限系统 | 个人用户，没有团队/角色概念 |

### 技术栈

| 层 | 选型 | 理由 |
|---|---|---|
| 运行时 | Node.js / Bun | 和前端共享 TypeScript 类型定义 |
| 框架 | Hono | 轻量、跨运行时、类型安全 |
| 数据库 | PostgreSQL | 账号、设备、游标、审计，成熟可靠 |
| 对象存储 | Cloudflare R2 / AWS S3 | 存加密 blob，按量付费 |
| 实时通知 | WebSocket（桌面/Web）+ APNs（iOS） | 通知设备拉取新数据 |
| 部署 | 单容器，Fly.io / Railway / VPS | 单体够用，不需要 K8s |

## 4. API 设计

### 4.1 Auth API

```
POST   /auth/register          注册（邮箱 + 密码）
POST   /auth/login             登录，返回 JWT
POST   /auth/refresh           刷新 token
POST   /auth/reset-password    密码重置
DELETE /auth/account            删除账号（触发 GDPR 删除流程）
```

注册时服务端只存：
- 邮箱
- 密码哈希（bcrypt / argon2id）
- 创建时间
- 账号状态

**不存**恢复短语、主密钥、任何与加密内容相关的信息。恢复短语完全在客户端生成和管理。

### 4.2 Device API

```
POST   /devices/register       注册新设备（设备名、平台、公钥）
POST   /devices/pair/initiate  已信任设备发起配对，生成一次性配对码
POST   /devices/pair/confirm   新设备提交配对码完成配对
GET    /devices                获取设备列表
DELETE /devices/:id            撤销设备信任
```

设备配对流程：

```
已信任设备                     服务端                      新设备
    │                          │                           │
    ├─ POST /pair/initiate ──→│                           │
    │  返回 pairing_code       │                           │
    │                          │                           │
    │          （用户在新设备上输入配对码或扫码）              │
    │                          │                           │
    │                          │←─ POST /pair/confirm ────┤
    │                          │   (pairing_code + 新设备公钥)
    │                          │                           │
    │←─ WebSocket 通知 ────────│                           │
    │  "新设备已配对，请传递密钥"│                           │
    │                          │                           │
    ├─ 端到端加密传输主密钥 ──→│──→ 转发给新设备 ──────────→│
    │  (用新设备公钥加密)       │                           │
```

服务端在这个过程中**只做中转**，不解密主密钥。

### 4.3 Sync API

```
POST   /sync/push              上传一批加密变更
GET    /sync/pull?cursor=xxx   拉取自上次游标以来的变更
GET    /sync/cursor             获取当前设备的同步游标
```

#### Push 请求体

```ts
interface SyncPushRequest {
  deviceId: string
  changes: SyncChange[]
}

interface SyncChange {
  changeId: string           // 全局唯一变更 ID
  objectId: string           // 对象 ID（密文内部也有，但这里用于路由）
  channel: 'object' | 'document' | 'blob'   // 对应 11 号文档的三条通道
  version: number
  timestamp: number
  encryptedPayload: string   // Base64 编码的 AES-256-GCM 密文
  checksum: string           // 密文的哈希，用于去重
}
```

#### Pull 响应体

```ts
interface SyncPullResponse {
  changes: SyncChange[]
  nextCursor: string
  hasMore: boolean
}
```

#### 服务端对 Sync 数据的处理

服务端收到 push 后只做四件事：

1. 验证 JWT 和设备 ID
2. 把 `encryptedPayload` 原样存入 S3/R2，key 为 `{accountId}/{objectId}/{changeId}`
3. 更新 PostgreSQL 中的同步游标
4. 通知该账号的其他在线设备"有新数据"

**不解密、不解析、不索引内容。**

### 4.4 Blob API

大文件（图片、PDF、音视频）走独立的 blob 通道，避免 Sync API 被大文件阻塞：

```
PUT    /blobs/:hash            上传加密 blob（内容寻址）
GET    /blobs/:hash            下载加密 blob
HEAD   /blobs/:hash            检查 blob 是否存在（去重）
DELETE /blobs/:hash            删除 blob
```

blob 存储规则：

- key 是客户端计算的 `sha256(明文内容)`
- 上传的是加密后的内容
- `PUT` 前先 `HEAD` 检查是否已存在，避免重复上传
- 引用计数由客户端管理，服务端不感知对象间引用关系

### 4.5 Notification API

```
WebSocket  /ws/notifications    桌面端 / Web 端长连接
POST       /notifications/apns  注册 iOS 推送 token
```

通知内容极简，不含任何用户数据：

```ts
interface SyncNotification {
  type: 'sync_available'
  changeCount: number        // 有几条新变更
  latestTimestamp: number    // 最新变更时间
  // 不包含：对象 ID、对象类型、内容摘要
}
```

客户端收到通知后自行调用 `/sync/pull` 拉取。

### 4.6 Admin / GDPR API

```
POST   /admin/export/request    发起数据导出（服务端打包该账号所有密文）
GET    /admin/export/:id        下载导出包
POST   /admin/erase/request     发起账号删除
GET    /admin/erase/:id/status  查询删除进度
```

#### 删除流程

```
用户发起删除请求
    → 服务端标记账号为 pending_deletion
    → 7 天冷静期（可撤销）
    → 冷静期结束后执行：
        1. 删除 S3/R2 中该账号所有 blob
        2. 删除 PostgreSQL 中该账号所有记录
        3. 撤销所有设备 token
        4. 发送确认邮件
        5. 记录删除审计日志（仅保留"账号 X 于时间 Y 完成删除"）
```

## 5. 数据库设计

PostgreSQL 只存元数据，不存用户内容：

```sql
-- 账号
CREATE TABLE accounts (
  id TEXT PRIMARY KEY,
  email TEXT UNIQUE NOT NULL,
  password_hash TEXT NOT NULL,
  status TEXT NOT NULL DEFAULT 'active',  -- active | pending_deletion | deleted
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  deleted_at TIMESTAMPTZ
);

-- 设备
CREATE TABLE devices (
  id TEXT PRIMARY KEY,
  account_id TEXT NOT NULL REFERENCES accounts(id),
  name TEXT NOT NULL,
  platform TEXT NOT NULL,           -- desktop | ios | web
  public_key TEXT,                  -- 用于设备配对时的端到端加密
  trust_state TEXT NOT NULL DEFAULT 'pending',  -- pending | trusted | revoked
  last_sync_at TIMESTAMPTZ,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- 同步变更索引（不存内容，只存元数据供游标查询）
CREATE TABLE sync_changes (
  change_id TEXT PRIMARY KEY,
  account_id TEXT NOT NULL REFERENCES accounts(id),
  device_id TEXT NOT NULL REFERENCES devices(id),
  object_id TEXT NOT NULL,
  channel TEXT NOT NULL,             -- object | document | blob
  version INTEGER NOT NULL,
  storage_key TEXT NOT NULL,         -- S3/R2 中的 key
  checksum TEXT NOT NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX idx_sync_changes_account_created
  ON sync_changes(account_id, created_at);

-- 同步游标
CREATE TABLE sync_cursors (
  device_id TEXT NOT NULL REFERENCES devices(id),
  last_change_id TEXT,
  last_sync_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  PRIMARY KEY (device_id)
);

-- 配对请求（短期存在）
CREATE TABLE pairing_requests (
  code TEXT PRIMARY KEY,
  account_id TEXT NOT NULL REFERENCES accounts(id),
  initiator_device_id TEXT NOT NULL REFERENCES devices(id),
  status TEXT NOT NULL DEFAULT 'pending',  -- pending | completed | expired
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  expires_at TIMESTAMPTZ NOT NULL
);

-- GDPR 工单
CREATE TABLE gdpr_requests (
  id TEXT PRIMARY KEY,
  account_id TEXT NOT NULL REFERENCES accounts(id),
  type TEXT NOT NULL,                -- export | erase
  status TEXT NOT NULL DEFAULT 'pending',  -- pending | cooling | processing | completed
  requested_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  cooldown_until TIMESTAMPTZ,        -- 删除的 7 天冷静期
  completed_at TIMESTAMPTZ
);

-- 审计日志（最小化，不含内容）
CREATE TABLE audit_logs (
  id TEXT PRIMARY KEY,
  account_id TEXT,
  event_type TEXT NOT NULL,          -- login | device_paired | device_revoked | export | erase
  metadata_json TEXT,                -- 不含用户内容，只含事件元数据
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

**注意：`sync_changes` 表不存 `encryptedPayload`**——密文存在 S3/R2，这张表只存索引信息供游标查询用。

## 6. 对象存储结构

S3 / R2 中的存储布局：

```
orbit-sync-bucket/
├── {account_id}/
│   ├── objects/
│   │   └── {change_id}.enc        # 加密的对象变更
│   ├── documents/
│   │   └── {change_id}.enc        # 加密的文档变更（含 baseVersion）
│   └── blobs/
│       └── {content_hash}.enc     # 加密的大文件（内容寻址）
```

存储策略：

- 对象变更保留 90 天历史，之后只保留最新版本（压缩）
- blob 按引用计数管理，无引用 30 天后清理
- 账号删除时整个 `{account_id}/` 前缀下的内容全部删除

## 7. 安全设计

### 7.1 传输安全

- 全部 API 强制 TLS 1.3
- JWT token 短有效期（15 分钟），refresh token 长有效期（30 天）
- 设备撤权后立即失效所有该设备的 token

### 7.2 存储安全

- PostgreSQL 中不存任何用户内容
- S3/R2 存的全是客户端加密后的密文
- 服务端 SSE（Server-Side Encryption）作为第二层保护，但不作为安全假设
- 数据库连接使用 SSL，密码走环境变量

### 7.3 运维安全

- 生产环境没有"查看用户数据"的接口或工具
- 数据库 break-glass 只能看到账号元数据，看不到内容
- 所有管理操作记入 `audit_logs`
- 不用真实用户数据做开发测试

### 7.4 速率限制

```
注册：        5 次/小时/IP
登录：        10 次/分钟/IP
同步 push：   100 次/分钟/账号
同步 pull：   200 次/分钟/账号
blob 上传：   50 次/分钟/账号
```

## 8. 部署策略

### MVP 阶段

```
单实例部署：
  Fly.io / Railway / 自建 VPS
  ├── Orbit API Server（单容器）
  ├── PostgreSQL（托管实例或同机）
  └── Cloudflare R2（对象存储）
```

成本估算（个人用户 < 1000 人）：

| 项目 | 月成本 |
|------|--------|
| 服务器（2 vCPU / 1GB） | ~$5-10 |
| PostgreSQL 托管 | ~$5-15 |
| R2 存储（10GB） | ~$0.15 |
| R2 请求 | 免费额度内 |
| **合计** | **~$10-25/月** |

### 扩展阶段

当用户量增长时：

1. API Server 水平扩展（无状态，加实例即可）
2. PostgreSQL 读写分离
3. WebSocket 通知可独立部署为通知服务
4. R2/S3 天然可扩展

不需要提前为扩展做架构准备——单体 + 对象存储的上限足够支撑到几万用户。

## 10. 风险与决策

### 10.1 为什么单体而不是微服务

Orbit 服务端的业务逻辑极简：存密文、转通知、管账号。拆微服务只会增加部署复杂度、网络延迟和运维负担，没有任何收益。

**决策：单体服务，直到用户量明确需要拆分为止。**

### 10.2 为什么不做服务端 Agent

有些产品把 Agent 放在服务端运行。Orbit 不这样做，原因：

1. Agent 需要读取用户明文内容做推理，放服务端就破坏了零知识原则
2. 客户端直连 LLM API 延迟更低
3. 本地 Agent 可以在离线时使用本地模型

**决策：Agent 100% 运行在客户端。服务端只做数据中转。**

### 10.3 为什么不做实时协作基础设施

WebSocket 只用于通知"有新数据"，不用于实时文档协同。个人用户不需要两台设备同时编辑同一段文字。

**决策：通知用 WebSocket，同步用 HTTP pull。不做 OT / CRDT / 实时协作。**

### 10.4 自托管可行性

架构的简单性使自托管成为可能：

```bash
# 用户自托管只需要：
docker compose up
# 包含：API server + PostgreSQL + MinIO（S3 兼容）
```

客户端在设置中指定 `syncServerUrl` 即可切换到自托管实例。这也是对"数据主权"承诺的最终兜底：如果用户不信任 Orbit 的云服务，可以自己跑。
