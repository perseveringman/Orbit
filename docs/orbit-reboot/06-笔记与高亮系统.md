# 06. 笔记与高亮系统

## 1. 设计定位

笔记与高亮系统不是阅读器里的一个附属面板，而是 Orbit 把“输入 → 理解 → 连接 → 产出”串起来的材料转化层。用户在文章、书籍、笔记、研究文档、创作文档、Agent 回答中选中的任何一段文字，都应当立即进入统一捕捉管线：它可以先只是一个上下文片段，也可以升级为高亮、边注、闪念、研究笔记、任务线索或写作素材。  

因此，本专题的核心不是“如何记一条笔记”，而是定义一套跨阅读器与编辑器通用的最小单位：**可定位的文本片段、围绕片段产生的笔记对象、以及片段如何进入对象网络并持续流向研究、任务、写作与 Journal**。

系统必须满足四个原则：

1. **统一捕捉入口**：所有可选中文本的内容面都共享同一套选择菜单与上下文提问能力。
2. **统一笔记对象**：闪念、阅读批注、研究笔记、Journal 手记不再是四套系统，而是同一类 Note 对象在不同场景下的表现。
3. **可追溯**：高亮、笔记、写作引用都必须能回到原始片段、原始内容与中间转化链路。
4. **可流动**：高亮/笔记创建后必须天然进入对象网络，成为研究空间证据、任务上下文、写作素材与 Journal 时间线的一部分。

## 2. 关键用户场景

### 2.1 阅读中划线、写边注、继续提问

用户在 Reader 中选中一段文字后，浮动菜单直接提供：

- 高亮
- 添加边注
- 继续问 Agent
- 保存到某条笔记
- 创建任务

若用户只想先问问题，不应被迫先创建高亮；若用户稍后决定保留这段上下文，可以一键把当前提问上下文升级为高亮或笔记。

### 2.2 在任意内容中选中一段文字继续追问

同样的交互不只发生在 Reader，也发生在：

- 笔记编辑器
- 创作文档
- 研究结果页
- Journal 页面
- Agent 回答面板

用户选中一句话后，Agent 会拿到“选中文本 + 上下文 + 来源对象 + 锚点”，从而围绕具体片段继续推理，而不是基于整页模糊聊天。

### 2.3 随时记下闪念，但不打碎系统

用户可能在阅读时写一句边注，也可能在全局快速捕捉里写一条闪念，也可能在 Journal 里记当天感受。这些都应被保存为统一 Note 对象，只是 `note_kind`、`origin`、`links` 与默认视图不同。

### 2.4 从片段到研究、任务、写作

一条高亮最初只是阅读痕迹，但后续应能被：

- 加入研究空间，作为证据材料
- 提炼为行动项，成为任务背景
- 插入创作文档，形成带溯源的素材 Block
- 出现在 Journal 中，形成“今天我读到并思考了什么”的轨迹

## 3. 核心设计

### 3.1 统一捕捉单元：Selection Context

所有可选中文本的界面都先产出同一个运行时对象 `SelectionContext`，而不是各自发明高亮模型：

```ts
interface SelectionContext {
  surfaceType:
    | 'reader_article'
    | 'reader_book'
    | 'note_editor'
    | 'document_editor'
    | 'research_view'
    | 'journal'
    | 'agent_reply';
  sourceObjectType: 'article' | 'book' | 'note' | 'document' | 'research_artifact' | 'journal_note' | 'agent_message';
  sourceObjectId: string;
  selectedText: string;
  surroundingText?: {
    prefix: string;
    suffix: string;
  };
  anchor: AnchorPayload;
  blockRef?: {
    blockId?: string;
    startOffset?: number;
    endOffset?: number;
  };
  metadata?: Record<string, unknown>;
}
```

后续无论用户选择“高亮”“边注”“继续问 Agent”“加入笔记”，都只是对同一个 `SelectionContext` 执行不同动作。这样 Orbit 才能保证阅读器、编辑器与 Agent 共享上下文语义。

### 3.2 高亮锚点模型：先稳定定位，再允许回附

高亮不是“把一段文本存下来”这么简单；它必须能在内容更新、重排、跨端同步后仍尽量回到原位。因此锚点模型采用四层结构：

1. **对象层定位**：该片段属于哪个对象  
   `source_object_type + source_object_id`
2. **结构层定位**：该片段位于对象内部哪里  
   - Article/网页：`paragraph_index`、`block_id`、`dom_path`
   - EPUB：`epub_cfi`
   - PDF：`page + rects + text_layer_offsets`
   - Note/Document：`block_id + text_offset`
3. **文本指纹层**：用于回附  
   `quote + prefix + suffix + text_hash`
4. **状态层**：当前锚点是否稳定  
   `active | fuzzy | detached`

建议高亮对象持有 `anchor_json`，内部同时保存主定位与回退定位：

```ts
interface AnchorPayload {
  sourceVersion?: string;
  locator: Record<string, unknown>;
  quote: string;
  prefix?: string;
  suffix?: string;
  textHash: string;
  state: 'active' | 'fuzzy' | 'detached';
}
```

其中：

- `active`：可直接命中原位置
- `fuzzy`：结构定位失效，但文本指纹重新匹配成功
- `detached`：无法可靠重建原位置，只保留引用文本与原始来源

这套锚点模型不仅服务 Reader 高亮，也服务“在笔记/文档里选中一段文字继续问 Agent”。

### 3.3 高亮对象模型：高亮是对象，不是样式

高亮本身必须进入对象网络，成为可搜索、可回链、可引用的材料对象，而不是 Reader 私有状态。

建议保留 `highlight` 作为独立对象类型，核心字段包括：

- `id`
- `source_object_type`
- `source_object_id`
- `anchor_json`
- `quote_text`
- `color`
- `highlight_kind`：`highlight | question_seed | evidence | quote`
- `created_by`：`manual | import | agent`
- `created_at / updated_at`

关键点：

1. **高亮不直接承担长文本写作职责**。它负责保存“哪一段被标出来了”。
2. **边注不是 `highlights.note` 的一段孤立字符串**。用户输入的批注应自动创建真正的 Note 对象，再通过关系与该高亮相连。
3. **高亮默认就是对象网络中的一等公民**，因此必须进入 `object_index`，并通过 `links` 与文章、笔记、项目、研究空间、行动、文档相连。

### 3.4 笔记对象模型：一套 Note，承载五种写作时刻

Orbit 不再把“闪念”“阅读批注”“研究笔记”“Journal 手记”拆成四个模块，而是统一为 `note` 对象，只通过语义字段区分：

- `note_kind`
  - `fleeting`：闪念/快速捕捉
  - `annotation`：阅读边注/批注
  - `research`：研究笔记
  - `journal`：日志手记
  - `synthesis`：已经沉淀成相对稳定观点的笔记
- `maturity`
  - `inbox | linked | synthesized | reusable`
- `origin`
  - `manual | highlight | quick_capture | agent | journal`
- `file_path`
  - 所有 Note 仍以 Markdown 文件为真相来源

这意味着：

1. 用户在 Reader 边栏里随手写的一句话，本质上就是 `note_kind='annotation'` 的 Note。
2. 全局快速记录的一条闪念，是 `note_kind='fleeting'`。
3. Journal 页面顶部的自由输入区，不产生第二套“日记存储”，而是写入 `note_kind='journal'` 的 Note。
4. Research 页中的整理内容，本质上也是 Note，只是它与研究空间的关系更强。

统一之后，所有笔记都能被同一套编辑器、搜索、回链、对象关系与 Agent 能力处理。

### 3.5 上下文提问交互：先以片段发问，再决定是否沉淀

“继续问 Agent”是 Selection Context 的一级动作，交互流程如下：

1. 用户在任意内容中选中文本
2. 浮动菜单点击“继续问 Agent”
3. 右侧或底部打开上下文提问面板
4. 顶部固定展示一个 **Context Card**
   - 选中文本
   - 来源对象标题
   - 当前位置（段落/页码/Block）
   - 已有关联高亮/笔记/任务（如存在）
5. Agent 回答后，用户可以将结果：
   - 仅保留在会话里
   - 保存为 Note
   - 加入研究空间
   - 创建任务
   - 插入当前文档

默认情况下，**提问不自动制造高亮污染**。只有当用户明确选择“保存为高亮/保存为笔记/加入材料”时，片段才变成持久对象。

### 3.6 Block 溯源关系：从“引用了什么”升级为“这段内容从哪里长出来”

Block 溯源不能只服务创作文档，也必须覆盖笔记与研究沉淀。这里建议将现有 `block_provenance` 从“文档专用”推广为“任意 Block 容器可用”：

```ts
interface BlockProvenance {
  ownerType: 'note' | 'document' | 'research_artifact' | 'agent_message';
  ownerId: string;
  blockId: string;
  sourceObjectType: 'highlight' | 'note' | 'research_artifact' | 'agent_message';
  sourceObjectId: string;
  sourceBlockId?: string;
  sourceAnchor?: AnchorPayload;
  transformation: 'verbatim' | 'edited' | 'ai_rewrite' | 'merged';
  lineageRootType?: string;
  lineageRootId?: string;
}
```

这样可以形成完整链路：

`文章片段 → 高亮 → 批注 Note → 研究笔记 Block → 创作文档 Block`

在 UI 上，用户查看创作文档某个段落时，不只看到“来源是一条笔记”，还应能继续追到那条笔记最初引用的高亮，最终回到 Reader 中的原文位置。

### 3.7 与阅读器/编辑器的连接方式：同一条桥，不做两套入口

#### 与 Reader 的连接

- Reader 提供原生选区能力与锚点解析器
- Reader 侧边栏使用统一的 Note Composer
- 高亮列表、边注列表、相关提问列表都基于同一对象查询，而不是 Reader 本地状态
- 从高亮卡片可直接：
  - 打开原文
  - 打开关联 Note
  - 加入研究空间
  - 创建任务
  - 插入写作文档

#### 与 Block 编辑器的连接

- Note/Document/Research 编辑器都共享 TipTap Block 内核
- 在编辑器中选中文本时，出现与 Reader 一样的“继续问 Agent / 保存为引用 / 创建高亮式摘录”菜单
- `/from-highlight`、`/from-note`、拖拽素材卡片，都走统一溯源写入逻辑
- 轻量笔记模式默认可以不为所有 Block 生成 ID，但**一旦某个 Block 被选中提问、被引用、被插入其他内容，就立即补发 `block_id` 并进入溯源系统**

这解决了“笔记模式需要轻量”与“系统必须可追溯”之间的冲突。

### 3.8 如何流向研究、任务、写作与 Journal

| 去向 | 进入方式 | 形成的关系 | 最终价值 |
|---|---|---|---|
| 研究 | 高亮/笔记加入 Research Space，或由 Agent 建议归组 | `highlight -> research_space (evidence_for)`、`note -> research_space (supports)` | 从片段进入主题化研究 |
| 任务 | 从高亮/笔记/提问结果创建 Action | `action -> note/highlight (derived_from / references)` | 保留“为什么会有这个任务” |
| 写作 | 从素材面板插入高亮/笔记到文档 | `block_provenance` + `links(used_in)` | 所有观点都可回溯 |
| Journal | 当天的高亮、边注、提问、笔记自动写入事件流 | `event_log` + `note_kind='journal'` | 形成可回看的一天认知轨迹 |

关键原则：**06 的输出永远不是“留在 06 内部”**，而是持续变成其他专题的上游材料。

## 4. Agent 行为与自动化机制

### 4.1 Agent 在高亮与笔记系统中的角色

Agent 不是替代用户划线，而是围绕用户已选中的材料提供四类增强：

1. **上下文问答**：围绕选中片段回答“这段在说什么”“和我当前研究有什么关系”
2. **关系建议**：建议把某条高亮连接到已有项目、研究空间、文档或行动
3. **聚类提炼**：检测多个相近高亮，建议合并为一条研究笔记或观点笔记
4. **转化建议**：当某段批注包含明确行动意图时，建议创建任务；当某组材料已足够完整时，建议开始写作

### 4.2 自动化原则

- **默认轻自动化**：可推荐，不静默扩写用户内容
- **自动建链优先于自动建任务**：系统优先建议关系与归档，任务创建必须更谨慎
- **任何 AI 生成内容必须保留来源**：Agent 回答若被保存为 Note 或写作 Block，必须写入溯源信息
- **上下文召回以对象网络为主**：Agent 不只看当前选区，还应补充该高亮已有的笔记、相关项目、研究空间与历史提问

### 4.3 典型自动化触发

- 同一篇文章出现 5 条以上高亮且集中在同一主题 → 建议生成“文章要点笔记”
- 某条边注中出现“要做/待验证/需要查” → 建议创建 Action
- 一组高亮已被多个文档反复引用 → 建议升级为 `synthesis note`
- Journal 回顾时发现当天有连续的阅读→提问→记录链路 → 自动生成“今日学习摘要”

## 5. 数据模型 / 接口 / 能力边界

### 5.1 核心对象与关系

#### 高亮对象

```ts
interface HighlightObject {
  id: string;
  sourceObjectType: string;
  sourceObjectId: string;
  anchorJson: AnchorPayload;
  quoteText: string;
  color: string;
  highlightKind: 'highlight' | 'question_seed' | 'evidence' | 'quote';
  createdBy: 'manual' | 'import' | 'agent';
  createdAt: string;
  updatedAt: string;
}
```

#### 笔记对象

```ts
interface NoteObject {
  id: string;
  filePath: string;
  title: string;
  noteKind: 'fleeting' | 'annotation' | 'research' | 'journal' | 'synthesis';
  maturity: 'inbox' | 'linked' | 'synthesized' | 'reusable';
  origin: 'manual' | 'highlight' | 'quick_capture' | 'agent' | 'journal';
  createdAt: string;
  updatedAt: string;
}
```

#### 关键关系语义

- `highlight --annotated_by--> note`
- `note --derived_from--> highlight`
- `note --supports--> project / research_space / document`
- `action --derived_from--> note / highlight`
- `document --uses--> highlight / note`
- `journal_note --captures--> highlight / note / question`

这些关系统一进入 `links`，并同步到 `object_index` 以支持全局搜索、回链与 Agent 召回。

### 5.2 文件系统与数据库分工

- **Note 正文**：Markdown 文件，留在文件系统，保证可迁移与外部编辑兼容
- **Highlight 与 Anchor**：结构化保存在数据库，保证定位、排序、过滤与快速渲染
- **对象索引与关系**：进入 `object_index + links`
- **Block 溯源**：进入 `block_provenance`
- **行为轨迹**：进入 `event_log`，供 Journal/Review 使用

### 5.3 对外能力接口（IPC / MCP）

本专题至少暴露以下能力：

- `selection.capture(context)`
- `highlight.create(context, options)`
- `note.create(payload)`
- `note.attachToHighlight(noteId, highlightId)`
- `context.ask(context, prompt)`
- `material.promote(objectRef, targetType, targetId)`
- `provenance.trace(ownerType, ownerId, blockId)`
- `anchor.resolve(sourceObjectType, sourceObjectId, anchorJson)`

### 5.4 能力边界

1. **不是全文评论系统**：只围绕“用户显式选中的片段”工作，不对整篇内容自动逐段生成批注。
2. **不是孤立知识库**：高亮/笔记不单独形成封闭模块，必须通过关系进入研究、任务、写作、Journal。
3. **不是自动任务引擎**：高亮和批注可以导向任务，但不会因用户随手划线而自动产生大量 Action。
4. **不是静态引用系统**：溯源不仅记录“来自谁”，还要记录锚点、转化方式与原始来源链。

## 6. 与其他专题的依赖关系

以下按本轮专题编号约定说明：

### 6.1 依赖 04：阅读器系统

06 依赖 04 提供可选区阅读面、稳定的段落/页码/CFI/DOM 锚点能力与阅读侧边栏容器；反过来，04 通过 06 获得高亮、边注、上下文提问与材料流转能力。

### 6.2 依赖 09：对象网络与全局关系系统

06 产出的高亮与笔记必须进入 09 定义的 `object_index + links + events` 体系，否则无法被搜索、回链、Agent 召回，也无法成为研究/任务/写作材料。

### 6.3 连接 05：研究系统

05 需要 06 提供“证据颗粒度”对象：高亮、批注、研究笔记与提问记录。06 是研究空间的材料入口，05 则负责把这些材料组织成主题、假设与研究产物。

### 6.4 连接 03：任务系统

03 不能只接受一句裸任务标题；它应能接收由高亮、边注、提问链路导出的 Action，并保留来源关系。06 负责提供任务产生的语境，03 负责执行编排。

### 6.5 连接 07：写作系统

07 的素材面板、引用插入、写作溯源依赖 06 提供高质量的高亮与笔记对象。06 把材料沉淀好，07 把材料组织成成品。

### 6.6 连接 08：Journal 系统

08 不应另建“日记笔记模块”。06 负责统一 `note_kind='journal'` 的对象模型，并把高亮、边注、提问、捕捉事件写入事件流；08 负责按日聚合与叙事化展示。

### 6.7 连接 15：Agent / MCP 能力层

15 需要把 06 暴露为可组合能力：选择上下文、追溯来源、将回答保存为笔记、把材料推进研究/任务/写作。没有 15，06 只是本地 UI；有了 15，06 才成为 Agent-first 系统中的通用认知输入层。

## 7. 风险、边界与设计决策

### 7.1 锚点漂移是首要风险

网页正文重排、文章重新解析、外部编辑器改动 Markdown，都会导致锚点失效。因此必须采用“结构定位 + 文本指纹 + 状态机”的复合锚点，而不是只存起止 offset。

### 7.2 笔记泛滥的风险高于高亮不足

如果系统把每次捕捉都变成长文笔记，用户会迅速失控。设计上应允许：

- 先仅高亮
- 先仅提问
- 先仅快速记一句

之后再由 Agent 或用户把它们提升为更成熟的 Note。

### 7.3 必须避免双存储分裂

Journal、Reader 边注、Research 笔记若各存一份，系统会迅速失真。这里的明确决策是：**所有用户写下来的内容统一收敛为 Note 对象；不同页面只提供不同视图与默认关系，不再复制存储。**

### 7.4 轻量编辑与强溯源的冲突

Block 编辑器文档中笔记模式默认偏轻量，但本专题要求可追溯。设计决策是：  
**笔记模式默认不强制为所有 Block 生成 ID；只有当 Block 被选中提问、被引用、被外链、被插入其他内容时，才按需补发 Block ID 并接入 `block_provenance`。**

### 7.5 Agent 必须尊重“材料优先于结论”

Orbit 的优势不在于让 Agent 直接输出答案，而在于保留用户如何划线、如何提问、如何形成观点的路径。因此 Agent 不能跳过高亮、笔记与溯源链，直接把一切压扁成结论文本。
