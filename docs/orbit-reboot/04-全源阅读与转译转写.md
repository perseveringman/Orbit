# 04-全源阅读与转译转写

## 1. 设计定位

本专题定义 Orbit Reboot 的“输入到阅读”主干系统：它不是一个只负责展示文章列表的 Reader，而是一条从**来源发现 → 订阅/保存 → 内容摄取 → 提取/转写/翻译 → 阅读理解 → 流转到笔记/研究/任务/写作**的完整输入管线。系统目标不是把一切内容都塞进同一种容器，而是让文章、书籍、播客、视频、网页快照、社交媒体日报都能以统一的认知界面进入 Orbit，同时保留各自媒介特性。

Orbit 在这里承接两条核心原则。第一，**文件系统是真相之源，数据库是索引与状态层**：外部内容进入系统后，原始文件、可阅读表示、元数据索引、AI 衍生物必须明确分层。第二，**Agent-first 但不偷换来源**：Agent 可以帮用户发现信源、生成日报、做正文提取、转写、翻译、摘要和路由建议，但 AI 结果永远不能覆盖原始材料本身。

因此，04 专题不是单一页面设计，而是 Orbit 整个知识入口的基础设施。它回答四个问题：用户该从哪里发现值得读的东西；不同格式内容如何在一个阅读器里被承载；翻译、转写、多语言阅读如何成为默认能力；以及这些内容进入系统后如何被对象化、索引化并安全地流入后续的高亮、研究、写作与行动系统。

## 2. 关键用户场景

### 场景 A：用户想开始阅读，但不知道读什么
用户说“我今天想看 AI 产品设计相关内容”，Feed Agent 不只返回一个 RSS 列表，而是同时给出三类候选：可订阅 RSS 源、适合加入站点监控的网站栏目、以及来自 X / YouTube / 播客频道的“社交媒体日报包”。用户可以直接订阅长期信源，也可以只保存本次候选内容进入 Inbox。

### 场景 B：用户从任意地方保存内容
用户可能粘贴一篇网页、一条帖子、一段视频链接、一本 EPUB、一个 PDF，或者直接导入本地音视频。Orbit 不要求用户先决定它属于“文章”“素材”还是“研究资料”，而是先创建一个外部内容对象，记录来源、格式、抓取能力与许可状态，再在后台完成抓取、正文提取、字幕拉取或音轨转写。

### 场景 C：用户在同一个 Reader 中切换媒介
用户上午读文章，下午听播客，晚上看视频课程。Orbit 的 Reader 都提供统一结构：标题区、来源区、正文/字幕主区、翻译层、高亮与批注层、相关项目/研究侧栏。差异只体现在主内容渲染器：文章显示段落，书籍显示章节，播客/视频显示时间码分段文本并与播放进度同步。

### 场景 D：用户需要一键翻译、一键转写与跨语言阅读
用户保存一篇英文长文，点击“一键翻译”，系统生成逐段双语表示；用户导入播客或视频，点击“一键转写”，系统生成带时间戳 transcript；用户在中英混合材料中阅读时，可以按段切换原文、译文、双语对照、仅摘要模式，并把任一段落直接转成高亮、笔记或研究卡片。

### 场景 E：阅读不是终点，而是流向后续工作
用户在阅读过程中划线、写批注、提出问题。Orbit 不把这些动作困在阅读页，而是允许将高亮发送到 Notes、Research、Project Space、Today Action、Writing Draft。Agent 会基于当前项目上下文提示：“这段内容更适合成为研究问题”“这条观点可以转成待验证假设”“这期播客里 03:14-08:20 可转成一条写作素材”。

## 3. 核心设计

### 3.1 全源输入管线

Orbit 的输入管线分为八个阶段：

1. **来源发现（Discover）**：Feed Agent 通过 RSS 自动发现、站点结构识别、作者/频道扩展、用户粘贴 URL、社交平台列表监控、保存过往行为推荐等方式提出候选信源。
2. **订阅/保存（Subscribe / Save）**：用户可以创建长期订阅，也可以只保存单条内容。长期订阅生成 `source_subscription`，一次性保存生成 `capture_request`。
3. **抓取（Fetch）**：系统根据来源类型执行 RSS 拉取、网页抓取、附件下载、字幕抓取、媒体下载或页面快照。
4. **规范化（Normalize）**：把 URL、作者、发布时间、语言、媒体时长、canonical URL、许可信息、来源链条统一为标准元数据。
5. **内容提取（Extract）**：网页走正文提取；书籍走章节索引；视频优先读取原字幕，失败后转写；播客与本地音视频统一走 transcription orchestrator。
6. **可阅读表示生成（Render Build）**：为不同媒介生成统一的阅读表示，包括 paragraph blocks、chapter tree、timecoded segments、封面/缩略图、引用锚点。
7. **语言层处理（Translate Layer）**：翻译不是覆盖正文，而是附着在阅读表示上的平行层；用户可拥有多个语言版本并在 Reader 中切换。
8. **知识流转（Route）**：阅读动作、高亮、摘录、问题、摘要、研究卡片、待办建议通过 links 与 Notes / Research / Project / Action / Document 相连。

这条管线的关键不是“抓到内容”，而是让每个阶段都可重建、可审计、可替换。正文提取失败不影响原始文件保留；转写失败不影响原媒体进入库；翻译删除不影响原文阅读。

### 3.2 来源发现与订阅设计

来源发现分成三种模式：

- **RSS 资源发现**：输入站点、作者主页或栏目页后，系统优先尝试 feed autodiscovery、sitemap / archive page 识别，并推荐可长期订阅的 feed。用户也可按主题订阅“信源包”，例如“AI 产品设计”“独立开发商业化”。
- **网站资源保存**：针对没有 RSS 的网站，Orbit 提供 `site watch` 机制，把“页面模板 + 发现规则 + 更新检测规则”视作一种可订阅源。它并不伪装成 RSS，而是被建模为网站监控订阅。
- **社交媒体日报**：对于 X、YouTube、播客频道、论坛帖子流等高噪声来源，Orbit 不鼓励直接把原始流全量塞进 Reader，而是优先生成“日报对象”：按列表、频道、主题或关键词做日级汇总，日报中的单条帖子/视频/节目再可继续保存为具体内容对象。

因此 Orbit 的“订阅”并不等于传统 RSS reader 的 feed 列表，而是一个统一的 source network：feed、site watch、channel digest、saved search、manual capture 都是合法来源。

### 3.3 统一内容承载：同一 Reader，不同渲染器

Orbit 不采用“所有内容强行变成 article 表”的偷懒方案，而是采用**媒介分型 + 阅读抽象统一**：文章、书籍、播客、视频保留自己的类型字段与专属能力，但都向 Reader 暴露同一组阅读接口：标题、作者/发布者、语言、段落/片段序列、锚点、阅读进度、高亮范围、相关对象。

统一 Reader 体验包含：

- 顶部：标题、来源、保存时间、语言、项目关联、阅读模式切换
- 主区：文章段落 / 章节内容 / transcript segments / 字幕块
- 下层能力：高亮、批注、AI 问答、翻译、摘要、术语解释
- 右侧上下文：相关项目、研究空间、已有笔记、可转行动、来源链路
- 底部或浮层：媒体播放控制、时间码跳转、段落定位、导出/发送入口

统一的关键不是 UI 长得一样，而是**所有内容都拥有可引用、可高亮、可路由的最小阅读单元**。文章以 paragraph block 为单位，书籍以 chapter + paragraph 为单位，播客/视频以 timecoded segment 为单位；它们在 links、highlights、event_log 中都能被同一种引用机制消费。

### 3.4 原始内容、可阅读表示、元数据索引、AI 衍生物四层分离

这是本专题最核心的结构边界：

| 层 | 角色 | 存储形态 | 是否可重建 | 是否可被 AI 覆盖 |
|---|---|---|---|---|
| **原始内容文件** | 来源真相 | HTML / EPUB / PDF / MP3 / MP4 / 原字幕 / 页面快照 | 否，应尽量保留 | 否 |
| **可阅读表示** | Reader 消费层 | Markdown、规范化 HTML、chapter JSON、segment JSON | 是，可由原始内容重建 | 否 |
| **元数据索引** | 查询与状态层 | SQLite 表 + `object_index` + `links` + `event_log` | 是，可通过扫描重建大部分 | 不适用 |
| **AI 生成衍生物** | 增值层 | transcript、translation、summary、digest、question set、tag suggestion | 是，可再生成 | 可以被替换，但需版本化 |

建议的本地目录形态如下：

```text
library/
  raw/
    web/{content-id}/source.html
    books/{content-id}/book.epub
    media/{content-id}/source.mp3|source.mp4
    subtitles/{content-id}/source.vtt
  readable/
    article/{content-id}/content.md
    book/{content-id}/chapters.json
    media/{content-id}/segments.json
  derived/
    transcripts/{derivative-id}.json
    translations/{derivative-id}.md
    digests/{digest-id}.md
    summaries/{summary-id}.md
```

其中：原始文件偏向不可变；可阅读表示是渲染缓存；AI 衍生物明确带 provider、prompt、版本、生成时间与可删除标记；SQLite 只保存路径、状态、预览、语言、进度、关系与权限设置。

### 3.5 翻译层、转写层与多语言阅读

**转写层**服务于音频/视频等非文本媒介。系统优先复用已有字幕或 transcript，缺失时再发起转写。转写结果必须是带时间戳、说话人信息（若可得）、段落切分和置信度的结构化对象，而不是一整段大文本，这样才能支持播放联动、高亮定位、片段引用与后续翻译。

**翻译层**服务于所有可阅读单元。翻译结果不是“替换正文”，而是挂在 `readable_representation` 之上的平行版本：同一段 paragraph 或 segment 可以有 `zh-CN`、`en-US`、`zh-Hant` 等多个 target locale。用户可切换四种模式：原文、双语、仅译文、摘要视图。

**多语言阅读**不是简单的 UI i18n，而是内容层能力：

- 保留源语言字段与自动检测结果
- 允许多个翻译版本并行存在
- 高亮默认锚定原文单元，同时可显示对应译文
- 用户在译文上写的批注仍回链到原文 anchor
- Agent 默认引用原文 + 用户界面语言版本，避免语义漂移

### 3.6 从阅读到笔记 / 研究 / 任务 / 写作

阅读对象必须天然可流转，不能靠复制粘贴维持工作流。Orbit 在 Reader 中提供四条一等出口：

1. **转高亮与批注**：保留原文片段、anchor、来源对象、语言映射。
2. **转笔记**：把一组高亮、一个问题或一个摘要沉淀到 Notes，并建立 `derived_from`。
3. **转研究**：把文章、视频片段、播客主题或日报项送入 Research Space，作为待综合证据。
4. **转行动 / 写作**：把可执行结论送入 Action，把可表达观点送入 Writing Draft 或项目文档。

Agent 只做建议与编排，不越权替用户决定。例如它可以建议“这篇文章与你的 Orbit Reboot 项目高度相关，是否加入研究空间”，但不能自动把所有阅读都强行转成任务。

## 4. Agent 行为与自动化机制

### 4.1 Feed Agent：发现与分拣
- 根据用户主题、项目、历史收藏与信源质量，推荐 RSS、站点监控与社交日报来源。
- 负责去重、聚类与低价值噪声过滤，尤其是在社交媒体场景中优先输出日报对象而非原始信息洪流。
- 对新入库内容生成轻量摘要、项目相关度与处理建议，写入 Inbox，而不是直接打断用户当前工作。

### 4.2 Reader Agent：提取、翻译、转写、伴读
- 为网页选择最佳正文提取器；失败时回退到原始快照阅读。
- 为音视频决定“优先读取字幕 / 发起转写 / 使用已有 transcript”。
- 支持术语解释、段落问答、双语切换、章节摘要、关键片段定位。
- 在用户阅读时只读取当前对象与显式关联上下文，不默认扫描整个库。

### 4.3 Research / Notes / Writing Agent：后续流转
- 将高亮聚类成研究问题、候选卡片或写作素材。
- 识别可操作结论，建议生成 action，但必须让用户确认。
- 在写作场景中优先引用原始来源与高亮证据，不只引用 AI 摘要。

### 4.4 自动化边界
- Agent 可以自动抓取、提取、转写、翻译、摘要、生成日报。
- Agent 不可以改写原始内容文件，不可以伪造来源，不可以把 AI 摘要冒充正文。
- 用户必须能关闭任意自动步骤，例如关闭自动翻译、仅手动触发转写、禁止云端模型读取原文。

## 5. 数据模型 / 接口 / 能力边界

### 5.1 关键对象

建议在现有“类型表 + `object_index` + `links`”体系上补足以下对象：

- `source_endpoints`：信源定义，类型可为 `rss_feed | site_watch | channel | digest_list | saved_search | manual_url`
- `source_subscriptions`：用户级订阅关系、抓取频率、优先级、静音规则
- `content_items`：外部内容主对象，统一记录来源、媒体类型、语言、canonical URL、许可状态
- `raw_blobs`：原始文件索引，只存路径、hash、mime、size、capture_method
- `readable_representations`：正文 Markdown、章节树、segment JSON、字幕块等可阅读缓存
- `derivative_assets`：transcript、translation、summary、digest、qa-set 等 AI 衍生物
- `reader_sessions` / `reading_progress`：进度、最近位置、最后打开语言、播放位置

如果继续沿用分类型对象表，则 `articles`、`books`、`podcasts`、`videos` 仍保留各自专属字段；`object_index` 负责统一查询，`links` 负责表达 `belongs_to`、`captured_from`、`derived_from`、`translated_from`、`transcribed_from`、`references`、`highlights` 等关系。

### 5.2 状态机

外部内容从进入系统到可被消费，至少经过以下状态：

`discovered → subscribed/saved → queued → fetched → normalized → extracted/transcribed → translated(optional) → ready_to_read → highlighted/routed → archived`

同时允许失败分支：

- `fetch_failed`
- `extract_failed`
- `transcribe_failed`
- `translate_failed`
- `quarantined`（版权/权限/安全原因不可继续处理）

状态写入 SQLite，重型内容保存在文件系统。这样在同步、恢复或跨端重建时，移动端可以先拿到元数据与状态，再按需拉取原文件或可阅读缓存。

### 5.3 接口与 MCP 能力

MCP-first 能力层至少暴露以下工具：

- `reader.discover_sources`
- `reader.subscribe_source`
- `reader.capture_url`
- `reader.import_local_media`
- `reader.fetch_content`
- `reader.extract_readable`
- `reader.start_transcription`
- `reader.start_translation`
- `reader.open_in_reader`
- `reader.create_highlight`
- `reader.route_to_note`
- `reader.route_to_research`
- `reader.route_to_action`

这些接口必须遵守最小权限原则：抓取是写入外部内容对象；翻译与转写是创建衍生物；删除衍生物不影响原内容；外部 Agent 若无授权，不应读取原文或导出媒体文件。

### 5.4 能力边界

1. **Orbit 不是公共网络爬虫**：系统只抓取用户显式订阅、保存或授权监控的来源，不做无限制全网采集。
2. **Orbit 不是内容再分发平台**：原始内容主要用于个人阅读、研究与引用，不提供模糊版权边界的对外分发能力。
3. **翻译与转写默认是可重建缓存**：它们提升体验，但不是唯一事实源。
4. **社交媒体日报是降噪对象，不是全部原帖镜像**：必要时只存摘要、引用与回链，并尊重平台权限与可见性限制。
5. **云端模型调用必须显式可控**：用户可以限制“只上传摘要”“上传当前段落”“永不上传原始媒体”。

## 6. 与其他专题的依赖关系

### 对 05 专题的依赖：Inbox / 输入分拣编排
04 负责把来源发现、订阅与抓取后的内容送入系统；05 需要承接这些对象在 Inbox 中如何被分拣、延后、丢弃、关联项目或送入笔记池。没有 05，04 的输入管线会停在“抓到很多内容，但不知道如何处理”。

### 对 06 专题的依赖：高亮、批注与卡片化沉淀
04 定义阅读单元、锚点、翻译映射与 transcript segment；06 需要在此之上实现高亮、批注、摘录卡片与 `derived_from` 链路。尤其双语高亮必须以 04 的原文 anchor 为基准。

### 对 07 专题的依赖：研究空间与多源综合
04 提供可进入研究空间的来源对象、证据片段、日报项与媒体片段；07 负责把这些输入组织成研究问题、证据库、综合报告与论证结构。没有 07，阅读只能停留在消费层，无法形成研究资产。

### 对 08 专题的依赖：Journal / Review / 阅读回顾
04 产生的阅读、翻译、转写、高亮、路由动作都要进入事件系统；08 负责把这些事件转成日记、周期回顾与阅读复盘，例如“今天读了什么”“哪些阅读转化为项目推进”。

### 对 09 专题的依赖：写作与发布工作台
04 输出的是可引用材料、来源链路与已翻译/已转写文本；09 需要消费这些材料，在写作时提供可追溯引用、素材抽取、段落回链与发布前核验。没有 09，阅读到写作的闭环不完整。

### 对 11 专题的依赖：搜索、推荐与知识检索
04 生成大量来源对象、内容对象与衍生物；11 需要定义全文检索、语义检索、来源质量排序、相关推荐与跨媒介召回，否则“全源阅读”会迅速变成信息堆积。

### 对 13 专题的依赖：后台任务、通知与任务中心
抓取、正文提取、转写、翻译、日报生成都属于长任务。13 需要提供统一任务状态、失败重试、通知中心与资源调度，否则 04 无法在桌面、Web、移动端稳定落地。

### 对 15 专题的依赖：同步、隐私与权限治理
04 涉及最大规模的外部内容摄入与数据出境。15 需要明确原始媒体是否同步、翻译/转写是否可跨端复制、云端模型调用授权、缓存保留期、导出/删除策略与版权敏感内容处理。

### 对 17 专题的依赖：MCP capability layer 与外部连接器
04 的抓取器、feed 发现器、站点监控器、社交日报生成器、翻译引擎、ASR 服务都应作为 MCP 工具暴露；17 负责统一能力注册、权限模型、连接器生命周期与外部 Agent 调用边界。

## 7. 风险、边界与设计决策

1. **风险：把所有媒介硬塞进同一种对象，最后谁都支持不好**  
   - 决策：坚持“媒介分型 + 统一 Reader 抽象”，而不是做一个巨型万能 content blob。

2. **风险：AI 衍生物反客为主，用户只看到摘要、译文和转写，忽略原始内容**  
   - 决策：Reader 永远可回看原始内容；所有摘要、翻译、转写都必须标记来源、版本与生成方式。

3. **风险：社交媒体摄入导致噪声爆炸**  
   - 决策：默认以日报/列表汇总为主，不把高频短内容原样倾倒进主阅读流。

4. **风险：多语言锚点漂移，导致高亮与引用失真**  
   - 决策：所有高亮与引用默认锚定源语言单元；译文只是映射层，不拥有独立事实地位。

5. **风险：转写、翻译、抓取任务太重，破坏本地优先体验**  
   - 决策：采用状态机 + 可恢复任务 + 分层缓存；移动端优先获取索引和轻量表示，重型媒体按需拉取。

6. **风险：隐私与版权边界被“智能体验”掩盖**  
   - 决策：任何云端调用都必须显式可见、可撤销、可审计；网站抓取、字幕提取与社交内容保存遵守用户授权与平台边界。

7. **关键设计决策：把“来源发现”纳入阅读系统，而不是只做订阅后的消费页**  
   - 原因：在 Agent 时代，用户的核心痛点已经不是“有内容可读”，而是“如何找到值得读、和当前项目相关、能进入后续工作流的内容”。阅读器必须前移到发现环节。

8. **关键设计决策：把翻译与转写定义为阅读层能力，而不是独立工具页**  
   - 原因：用户真正需要的是在阅读现场低摩擦理解外语与媒体内容，而不是先去另一个工具里做处理再返回阅读。
