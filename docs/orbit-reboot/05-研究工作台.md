# 研究工作台

## 1. 设计定位

研究工作台不是一个“带上下文的聊天框”，而是一个围绕**问题、来源、分析与产物**长期演进的工作空间。它的目标不是让用户问完一个问题就离开，而是让一个研究主题可以持续几天、几周甚至几个月被反复进入、补充、校验、导出，并持续和项目、任务、写作、Journal、对象图谱互相供给。

研究工作台的第一原则是：**会话只是过程，研究空间才是容器；回答只是中间态，产物才是可复用资产。** 因此系统必须把“研究”拆成四层：

1. **问题层**：用户真正要回答的问题、子问题、假设与判断标准。
2. **来源层**：内部阅读材料、素材、笔记、外部检索结果所构成的资料池。
3. **推理层**：Agent 的研究会话、证据整理、知识缺口识别与交叉验证过程。
4. **产物层**：摘要、分析报告、对比矩阵、结论 memo、写作提纲、PPT 大纲等可导出的成果。

研究工作台既可以挂靠在项目之下，也可以作为独立研究存在；但无论入口来自 Reader、Notes、Project 还是 Journal，它最终都应落到统一的研究空间对象模型，而不是散落成一次次不可复用的聊天记录。

## 2. 关键用户场景

### 场景 A：从阅读与笔记中批量组装资料池

用户在 Reader、Notes、Assets 或对象详情页中，多选若干文章、高亮、书籍片段、历史笔记、图片/PDF 素材，执行“加入研究空间”。系统不是简单把这些对象附加到聊天上下文，而是创建或更新一个**资料池快照**：保留来源对象 ID、版本、摘要、关键片段、来源时间与可引用锚点。用户随后可以围绕“AI Native IDE 的差异化能力有哪些？”持续开展多轮研究。

### 场景 B：从一个问题出发，让 Agent 在应用内与网络中补足材料

用户输入“帮我研究 2026 年 agent memory 架构，重点看长期记忆、上下文压缩与 tool routing”，系统先在 Orbit 内检索已有文章、笔记、历史研究产物与项目文档，再识别知识缺口，最后才通过外部 MCP 进行检索。整个过程会形成：问题树、候选来源、已采纳来源、待验证来源、知识缺口列表，而不是只返回一段答案。

### 场景 C：把研究持续沉淀成多个产物

同一个研究空间里，用户可能先得到一份“竞品综述”，随后生成“项目决策 memo”，再进一步导出“文章提纲”与“汇报 PPT 大纲”。这些产物都共享同一个证据底座，但面向不同消费场景。研究工作台要支持一份资料池衍生多个产物，并保留它们之间的继承关系。

### 场景 D：研究驱动任务、写作与回顾

用户在研究过程中发现“还缺少 3 个竞品的定价数据”，可直接生成任务；在写作时可将研究结论、引用块、图表插入文档；在 Journal 中则自动记录“今天推进了什么研究、形成了哪些判断、还有哪些未解问题”。研究不孤立存在，而是 Orbit 里连接输入、思考与行动的中枢。

## 3. 核心设计

### 3.1 研究空间是长期容器，不是聊天线程

一个 `Research Space` 代表一个持续存在的研究主题，例如“Agent 产品形态研究”“品牌重启定位研究”“蛋白粉品类选择”。它包含但不等于对话。空间内部至少有四个稳定区域：

1. **问题区**：研究目标、子问题树、假设、成功判断标准。
2. **资料池**：内部来源、外部候选来源、已采纳证据、待验证来源。
3. **会话区**：多次研究会话与 Agent 执行记录，每次会话都基于某个问题和一版资料池快照。
4. **产物区**：报告、表格、卡片、提纲、PPT、决策 memo 等。

因此研究空间的核心不是“最近一次回复”，而是“当前我们对这个主题已经积累了什么，还缺什么，已经产出了什么”。

### 3.2 研究空间对象模型

| 对象 | 作用 | 是否长期保存 | 主要关系 |
|---|---|---|---|
| `research_space` | 研究主题总容器 | 是 | 关联 project / journal / artifact / question |
| `research_question` | 主问题、子问题、假设 | 是 | 属于 space，可指向 project、document、action |
| `research_source_set` | 某次研究所使用的资料池快照 | 是 | 包含多个 source item，供 session / artifact 引用 |
| `research_source_item` | 单条来源记录 | 是 | 可 derived_from article/note/highlight/asset/url |
| `research_session` | 一次研究执行或对话 | 是 | 回答某个 question，使用某个 source set |
| `research_gap` | 被识别出的知识缺口 | 是 | 由 session 发现，可触发外部检索或任务 |
| `research_claim` | 结论/判断/洞察 | 是 | 必须被 source item 或 artifact 引用支撑 |
| `research_artifact` | 报告、PPT、大纲、表格等产物 | 是 | answers question / supports project / feeds writing |

这里最关键的设计是：**研究会话、资料池、产物三者解耦，但可追溯。**

- 会话回答“这次研究做了什么”。
- 资料池回答“这次研究基于什么证据”。
- 产物回答“这次研究最终留下了什么可复用资产”。

### 3.3 资料池不是附件列表，而是可编排证据集合

资料池需要同时支持两种来源进入方式：

1. **用户显式选入**：从文章、高亮、笔记、素材、已有研究产物中批量选择。
2. **Agent 建议引入**：根据问题树与知识缺口，从应用内对象和外部检索结果中提出候选来源。

每个来源条目至少包含：来源类型、来源对象 ID 或 URL、标题、摘要、关键片段、可信度、更新时间、采纳状态、引用锚点。资料池内部要区分：

- **已采纳来源**：可直接进入 grounding。
- **候选来源**：Agent 找到但还未进入正式证据池。
- **待验证来源**：信息有用但真实性/时效性不足。
- **派生来源**：由其他来源提炼出的笔记、摘录、结构化表格。

这样研究空间不会退化成“扔很多链接进去然后继续聊天”，而是形成一个逐步净化、逐步增强的证据池。

### 3.4 会话围绕问题与资料池快照运行

每次 `research_session` 都必须绑定：

- 当前研究问题（可为主问题或子问题）
- 一份资料池快照
- 期望输出类型（综述、对比、决策、提纲、PPT 等）
- 执行策略（仅内部检索 / 可外部检索 / 需要多轮交叉验证）

这意味着未来回看时，用户可以知道：

- 当时用了哪些证据
- 当时有哪些缺口尚未补足
- 哪些结论是基于旧版本资料池做出的

研究空间因此具备版本感与可审计性，而不是被一长串聊天历史淹没。

### 3.5 产物架是研究的真正出口

研究产物必须是研究空间里的第一等对象，而不是“复制回复到别处”。建议支持以下产物类型：

- 摘要卡 / 洞察卡
- 对比矩阵 / 决策表
- 综述报告
- 结论 memo
- 文章提纲 / 演讲提纲
- PPT 结构稿
- 可回写到项目的任务建议清单

产物默认保存在本地文件系统的长期内容层（如 `research/` 或 `documents/`），元数据进入数据库并纳入 `object_index + links`。因此研究空间只是这些产物的组织与追踪视图，而不是它们唯一的存储位置。

## 4. Agent 行为与自动化机制

### 4.1 Source grounding 流程

研究 Agent 的默认顺序必须是“**先内部、后外部；先 grounding、后生成**”：

1. 解析当前 `research_question` 与已有 `research_space` 上下文。
2. 从项目、阅读库、笔记池、历史研究产物、对象图谱中检索候选对象。
3. 将被采纳来源编译成 `grounding bundle`：包含来源 ID、摘要、关键片段、引用锚点、可信度与版本信息。
4. 仅基于 grounding bundle 进行初步分析，显式标记“已有证据能回答什么、不能回答什么”。

换言之，Agent 不能跳过证据整理直接生成“看起来很完整”的答案。研究模式下，任何核心结论都必须可回溯到来源条目。

### 4.2 知识缺口识别

知识缺口不是错误状态，而是研究工作台中的稳定对象。Agent 在每轮研究后都应生成 `research_gap`，并按以下维度分类：

- **事实缺口**：缺少具体数据、案例、时间线、定义。
- **比较缺口**：样本不足，无法做横向比较。
- **论证缺口**：已有观点但证据链不完整。
- **时效缺口**：来源过旧，需要更新验证。
- **反例缺口**：缺少相反证据，结论可能过拟合。

每个缺口可被标记为：待检索、检索中、已补足、人工搁置、无需补足。这样研究空间会自然形成一张“未解问题地图”，帮助用户跨多次会话持续推进。

### 4.3 外部检索与 MCP 边界

外部 MCP 只负责**拿回候选信息或执行外部动作**，不直接成为 Orbit 的长期知识真相。边界如下：

1. **MCP 可做的事**
   - 搜索网页、学术数据库、行业 API、知识库
   - 抓取页面正文、结构化元数据、财报/价格/论文信息
   - 调用外部工具生成 PPT、表格或可视化中间结果

2. **MCP 不应直接做的事**
   - 不绕过 Orbit 的对象层直接写入本地知识图谱
   - 不把外部原始返回直接当作已采纳证据
   - 不跳过用户权限与来源审查自动发布最终产物

3. **进入 Orbit 的标准路径**
   - 外部返回 → 候选来源
   - 候选来源经去重、摘要、可信度标注、必要时用户确认
   - 再转为正式 `research_source_item`
   - 正式来源才能参与 grounding 与长期引用

因此，MCP 是研究工作台的“外部传感器与执行器”，不是内部知识模型本身。

### 4.4 结果沉淀机制

每次研究会话结束后，Agent 需要至少做四类沉淀：

1. **更新问题状态**：哪些子问题已回答，哪些仍开放。
2. **沉淀证据**：新增来源、去重、摘要、引用锚点、可信度。
3. **沉淀结论**：将关键判断提炼为 `research_claim` 或产物段落。
4. **沉淀后续动作**：生成任务、写作提纲、待补充检索、需要人工决策的事项。

当会话较长时，系统应使用上下文压缩保留结构化摘要，而不是单纯保留全部消息。被压缩掉的不是研究本身，因为真正重要的问题、缺口、结论、来源与产物都已对象化。

## 5. 数据模型 / 接口 / 能力边界

### 5.1 存储分层

遵循“文件系统优先 + 元数据数据库”原则：

- **数据库中保存**：空间、问题、会话、资料池快照、来源元数据、缺口、结论、关系、状态。
- **文件系统中保存**：长文本研究报告、PPT 大纲、综述、图表说明、导出文档。
- **对象索引中保存**：所有研究对象的可检索入口与跨域引用能力。

研究空间是查询视图，不是目录树。即使一个报告属于某个研究空间，它的长期文件仍应进入统一内容层，再通过链接回到研究空间、项目、写作文档与 Journal。

### 5.2 关键关系语义

建议在统一 `links` 体系中支持以下关系：

- `research_space -> project : supports`
- `research_space -> research_question : contains`
- `research_space -> research_artifact : produces`
- `research_space -> article/note/highlight/asset/document : references`
- `research_source_item -> article/note/highlight/url : derived_from`
- `research_claim -> research_source_item : evidenced_by`
- `research_gap -> research_question : blocks_answering`
- `research_artifact -> document : feeds`
- `research_artifact -> action : suggests`
- `journal_entry -> research_space : reflects_on`

这保证研究不是孤岛，而是 Orbit 全局对象网络中的一个高密度枢纽。

### 5.3 核心能力接口

研究工作台至少需要以下内部能力接口：

| 接口 | 用途 |
|---|---|
| `research_space.create/update/archive` | 创建与维护研究空间 |
| `research_question.upsert/tree` | 管理问题树与假设 |
| `research_source.select/import/promote` | 从内部批量选源、从外部候选转正式来源 |
| `research_session.run` | 基于问题与资料池快照执行一次研究 |
| `research_gap.list/resolve` | 管理知识缺口 |
| `research_artifact.generate/export` | 生成报告、提纲、PPT 等产物 |
| `research_trace.get` | 查看结论-证据-来源的追溯链 |
| `research_to_action` | 从研究缺口或结论生成任务 |
| `research_to_writing` | 将研究产物插入写作工作台 |

### 5.4 与外部 MCP 的能力边界

研究工作台面向 MCP 暴露的是**受控能力层**，而不是数据库直通车：

- 对外开放：检索请求、来源抓取、结构化转换、导出执行。
- 对内保留：对象入库、关系创建、资料池采纳、结论入图、任务生成。
- 审批要求：涉及大规模导入、覆盖现有结论、向外发布、调用付费数据源时，必须具备明确权限与用户可见记录。

这样可以避免外部 Agent 把 Orbit 变成一个不可治理的“无限写入黑箱”。

## 6. 与其他专题的依赖关系

| 依赖专题 | 关系说明 |
|---|---|
| **04（阅读/材料输入层）** | 研究资料池最重要的内部来源来自文章、书籍、高亮、稍后读与阅读上下文；04 必须提供多选、批量“加入研究空间”、可引用锚点与阅读对象版本信息。 |
| **03（项目与任务上下文）** | 研究空间既可独立存在，也可附着在项目、阶段与行动链路之中；03 负责提供项目入口、目标/里程碑/任务上下文注入，以及“项目 > 研究 > 行动”的闭环组织。 |
| **07（写作工作台）** | 研究产物是写作的上游证据层；07 需要消费研究报告、结论卡、引用块、PPT 大纲，并把写作过程中产生的新问题反写回研究空间。 |
| **08（Journal）** | Journal 负责把“今天研究了什么、形成了什么判断、还卡在哪里”沉淀为时间性记录；研究工作台则提供结构化对象与可回链产物。两者是时间视角与主题视角的互补。 |
| **09（对象图谱）** | 研究空间、问题、来源、缺口、产物都必须进入统一对象系统与 `links` 语义关系，否则研究无法获得 backlinks、全局搜索与跨域引用。09 是本专题的底层前提。 |
| **14（Agent 编排层）** | Source grounding、知识缺口识别、上下文压缩、子 Agent 分发、结果沉淀都依赖 14 提供的 Orchestrator / ResearchAgent / Tool Registry / 安全网关能力。 |
| **15（MCP 与开放能力层）** | 外部检索、学术数据库接入、网页抓取、PPT 导出等能力通过 15 提供；本专题定义其边界：MCP 负责外部获取与执行，Orbit 内部对象化与入库仍由本地能力层完成。 |

## 7. 风险、边界与设计决策

1. **不把研究等同于聊天记录**：聊天历史可压缩、可归档，但问题树、资料池、缺口和产物必须对象化保存。
2. **不把外部检索结果直接当真相**：所有外部结果先成为候选来源，再进入正式证据池。
3. **不让资料池失控膨胀**：需要候选/采纳/待验证/废弃分层，否则研究空间会退化成“链接垃圾场”。
4. **不让研究空间替代项目或写作空间**：研究空间负责证据与推理；项目空间负责推进；写作空间负责面向读者输出。
5. **允许一个来源服务多个研究空间**：来源对象不复制，关系可复用；避免因为主题变化产生重复存储。
6. **允许一个研究空间产出多个产物**：报告、memo、提纲、PPT 是同一研究主题的不同出口，不应被迫拆成多个空间。
7. **研究的完成不是“没有更多聊天”，而是“问题、证据、结论与产物达到当前可用闭环”**：空间可以转入 dormant/archived，但其链接与产物仍长期可检索、可复用、可再次唤醒。
