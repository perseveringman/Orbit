# 01. 愿景写入与长期记忆

## 1. 设计定位

本专题定义 Orbit Reboot 中“愿景”这一最高层用户对象：它不是目标清单，不是 KPI，不是 Agent 用来打分的审计器，而是用户亲手写下的方向感、身份感与长期偏好。系统的职责不是把愿景转译成一套硬考核，而是在关键选择时把它温和地带回现场，让 Agent 的建议更像“帮你想起你想成为什么样的人”，而不是“提醒你又没达标”。

因此，本设计采用四条硬原则：

1. **愿景必须由用户首写与最终确认**：Agent 可以提问、整理、润色、提出草稿，但不能代写最终版本，更不能偷偷改写。
2. **愿景是文本对象，不是指标对象**：核心载体是一段或数段用户文本，可带标题、语气标签、时间跨度，但没有对齐分数、置信度、配额或自动 KPI。
3. **愿景进入 Agent 上下文，但不等于长期记忆主表**：愿景是高优先级、低频变化的用户源事实；Agent 记忆中的内容只能是愿景的受控摘录、版本引用与回顾片段，不能替代原文。
4. **愿景位于分层顶端**：愿景定义“往哪边走”；directive 定义“做选择时坚持什么原则”；项目定义“这段时间要完成什么交付”；计划定义“接下来怎么推进”。

这同时消解现有文档冲突：较新的《修正：愿景是罗盘，不是清单》优先。此前把愿景做成 belief、metric、alignment score、drift 报警的方案，在本轮设计中一律降级或移除。Orbit 可以保留 Agent 的记忆系统、回顾系统、对象系统，但这些系统只能围绕愿景服务，不能把愿景吞没成机器可审计的结构化信念集合。

## 2. 关键用户场景

### 场景 A：第一次进入应用，写下“我想成为什么样的人”
用户首次启动应用时，不会先看到项目、收件箱、模板库，而是进入一个低压力的 onboarding：欢迎页说明“愿景是给你自己的罗盘，不是给系统的指标”；随后用户通过自由书写、语音转写或回答 3 个引导问题，写下第一版愿景。用户可以只写 80 字，也可以写 800 字；系统不要求 SMART，不要求时间承诺。

### 场景 B：愿景保存后，Agent 在创建 directive / 项目时引用它
用户随后新建 directive、项目或导入一批待办时，Agent 会在需要取舍的节点引用愿景，例如：“你写过‘我希望把复杂问题讲清楚’，这个项目更像研究型项目，还是更像对外输出项目？”这里的作用是帮助澄清方向，而不是强制对齐。

### 场景 C：几个月后用户改写愿景
用户的人生阶段变化后，希望保留旧版本并写新版本。系统应允许“在原文上继续写”“另起一版”“比较两版差异”，且旧版仍可被历史 journal、项目复盘引用。Agent 读取时默认只读当前生效版本，但回顾某段历史时可以切回当时版本。

### 场景 D：用户连续一段时间埋头执行，逐渐偏离自己在意的方向
Orbit 不弹“偏离警报”。它只会在周回顾、项目创建、长期停滞或用户明确求建议时，用温和语言提醒：“最近你大部分精力都在交付侧，上次写下的愿景里你很在意表达与连接，要不要留一个小出口给它？”

### 场景 E：Agent 在复杂会话中需要理解用户长期方向
当用户询问“接下来这周我该推哪个项目”“这条线索要不要继续”“我是不是又开始乱收集了”，Agent 需要把愿景作为高层背景注入 system prompt / memory context / journal 摘要，让建议更稳定，但仍以当前上下文、项目状态、用户最新表态为准。

## 3. 核心设计

### 3.1 首启 onboarding 流程

首启流程分为 6 步，目标是帮用户写下“第一版真实愿景”，而不是生成漂亮宣言。

1. **定位说明页**
   - 文案明确：愿景是罗盘，不是清单；以后随时可改；不会被系统拿来考核你。
   - 给出 1 个正例与 1 个反例。
   - 正例：“我想成为能把复杂技术讲清楚、并做出真正有帮助产品的人。”
   - 反例：“每周输出 2 篇内容、每月涨粉 1000。”

2. **愿景书写页**
   - 三种输入方式：自由写作、语音输入、问答引导。
   - 引导问题只服务于表达，不服务于结构化打分：
     - 你想长期成为什么样的人？
     - 你希望自己创造的东西如何影响他人？
     - 什么样的工作状态会让你觉得“这就是我想过的生活”？

3. **Agent 整理但不接管**
   - 用户可点“帮我整理表达”，Agent 只生成候选草稿。
   - UI 明确显示“原文 / Agent 整理版 / 最终采用版”。
   - 只有用户点击“采用为当前愿景”，才会生成正式版本。

4. **愿景确认页**
   - 用户确认标题、正文、语气标签（如“创造”“连接”“研究”“健康”）、可选时间尺度（长期 / 现阶段）。
   - 不出现指标字段、优先级字段、完成期限字段。

5. **提醒姿态设置页**
   - 不是设置“监控强度”，而是设置“被想起的方式”：
     - 仅在回顾时提醒
     - 在项目选择时提醒
     - 在我主动求建议时提醒
     - 暂不提醒，仅保存在背景中

6. **落地承接页**
   - 保存后只做两件事：
     - 邀请用户可选地写 1-5 条 directive 候选
     - 邀请用户把一个当前项目挂到该愿景下
   - 这一步是桥接，不是自动拆解。用户可以跳过。

### 3.2 愿景对象模型

为同时满足“文件优先”“对象系统统一索引”“版本可追溯”，愿景采用“Markdown 原文 + 关系型元数据 + 追加式版本”的三层模型。

#### 1）主对象：`visions`

```ts
Vision {
  id: string
  slug: string
  title: string
  currentVersionId: string
  status: 'active' | 'archived'
  reminderMode: 'review_only' | 'decision_points' | 'on_request' | 'silent'
  scope: 'life' | 'career' | 'theme'
  ownerUserId: string
  createdAt: string
  updatedAt: string
  lastReaffirmedAt: string | null
}
```

#### 2）版本对象：`vision_versions`

```ts
VisionVersion {
  id: string
  visionId: string
  versionNo: number
  sourceFileId: string
  bodyMarkdown: string
  summaryForAgent: string
  changeNote: string | null
  authoredBy: 'user' | 'user-confirmed-agent-draft'
  createdAt: string
}
```

#### 3）关联关系：`links`

- `directive -> vision`：`derived_from`
- `project -> vision`：`aligned_to`
- `plan -> directive`：`implements`
- `journal_entry -> vision`：`reflects_on`

#### 4）文件层

- 原始正文以 Markdown 保存在用户本地文件系统，例如：`Spaces/Life/Vision.md`
- 每次正式改写都会生成不可变版本快照，可映射到隐藏版本文件或数据库快照。
- 数据库负责索引、版本号、关系和权限；Markdown 负责保留用户真正写下的话。

### 3.3 愿景、directive、计划、项目的分层

这是本专题最核心的概念边界：

| 层级 | 对象 | 回答的问题 | 时间尺度 | 谁可修改 |
|---|---|---|---|---|
| L1 | 愿景 Vision | 我想朝哪种人生/身份/方向走？ | 年 / 阶段 | 仅用户确认 |
| L2 | Directive | 当我做选择时，坚持什么原则？ | 月 / 季 / 阶段 | 用户确认，Agent 可建议 |
| L3 | 项目 Project | 这段时间我要交付什么结果？ | 周 / 月 / 季 | 用户与 Agent 共建 |
| L4 | 计划 Plan | 下一步怎么推进、何时推进？ | 日 / 周 | Agent 可动态编排 |

关键规则：

- 愿景**不能直接分解成任务**，中间必须经过 directive 或项目语义层。
- directive 是“可操作原则”，不是愿景的结构化替身。
- 项目可以不挂愿景（例如机会型、维护型项目），但系统会显式标注其归属：`aligned_to_vision` / `maintenance` / `opportunistic`。
- 计划是最易变化的层，不反向改写愿景。

### 3.4 版本化与演化机制

愿景的版本化不是 Git 式工程追踪，而是服务于“人生方向会变，但不该丢失历史上下文”。

- 每次点击“保存为新版本”时创建 `vision_versions` 新快照。
- 旧版本永不被覆盖，只能归档。
- Agent 默认读取当前版本；在复盘历史项目或 journal 时，读取当期版本。
- Agent **不得自动升级版本**；最多只能生成“建议改写稿”，等待用户确认。
- 当愿景发生重大变化时，系统触发一次“重新挂靠检查”：提示用户决定哪些 directive / 项目继续继承、哪些应解除关联。

## 4. Agent 行为与自动化机制

### 4.1 Agent 读取方式

愿景进入 Agent 的路径分三层，且职责不同：

#### A. system prompt 层：写入不可违背的使用原则
system prompt 中只放规则，不放长文原文：

- 当前存在用户愿景，它是方向感，不是 KPI。
- 不得基于愿景输出审计式评分、红黄灯或强制拦截。
- 当用户求助于优先级、取舍、项目命名、周回顾时，可以温和引用愿景。
- 若愿景与用户最新明确表态冲突，以用户当前明确意图为准，并建议是否需要更新愿景。

#### B. memory context 层：注入当前有效愿景摘要
`<memory-context>` 中注入受控字段：

- 当前 active vision 标题
- 当前版本 `summaryForAgent`
- 1-5 条用户确认过的 directive
- 最近一次 reaffirm 时间
- 与当前任务直接相关的愿景-项目链接

这里注入的是**摘要与引用**，不是一整套 belief/metric/observation。Agent 记住的是“用户长期在意什么”，不是“系统推测用户应该达成什么比例”。

#### C. journal / review 层：在回顾时读取原文与历史版本
在日回顾、周回顾、项目复盘中，Agent 可以额外读取：

- 当前愿景原文
- 与本回顾时间段匹配的愿景版本
- 最近 7/30 天与该愿景相关的 journal 片段、用户自述、项目进展

这使回顾更像“把最近发生的事放回你在意的方向里理解”，而不是“检查你有没有照做”。

### 4.2 温和提醒策略

提醒策略必须避免“审计感”。只允许在以下四类触发点出现：

1. **选择前提醒**：用户在两个项目、两个机会之间犹豫时。
2. **创建时提醒**：用户新建项目、directive、长期计划时。
3. **回顾时提醒**：日/周/月回顾与项目复盘。
4. **主动求助时提醒**：用户问“我是不是偏了”“我该选哪个”。

提醒话术遵守三条规则：

- **先描述事实，再轻触愿景**：先说“最近两周你主要在做 X”，再说“你之前写过 Y”。
- **给邀请，不给裁决**：使用“要不要”“是否考虑”“我可以帮你”而不是“你应该”“你偏离了”。
- **给最小行动，不给道德压力**：如“要不要把今天的进展写成 100 字对外说明”。

明确禁止：

- 愿景对齐分数
- 偏离警报
- 按愿景自动扣减项目优先级
- 基于愿景生成硬性配额
- 以“连续 N 天未达成”为口径的提醒

### 4.3 Agent 写入边界

- Agent 可读取愿景、引用愿景、基于愿景生成 directive 草稿。
- Agent 不可直接创建、覆盖、合并愿景版本。
- Agent 可提出“是否需要更新愿景”的问题，但只有用户确认后才创建新版本。
- Agent 对愿景的任何摘要都必须带版本引用，避免摘要漂移反客为主。

## 5. 数据模型 / 接口 / 能力边界

### 5.1 核心数据模型

建议最小化为以下对象与接口：

#### 表 / 对象
- `visions`
- `vision_versions`
- `directives`
- `projects`
- `plans`
- `links`
- `journal_entries`
- `review_snapshots`

#### 核心接口

```ts
vision.createFromOnboarding(input)
vision.saveVersion(visionId, bodyMarkdown, changeNote)
vision.getActive()
vision.getVersionHistory(visionId)
vision.linkObject(visionId, objectType, objectId, relation)
vision.buildAgentContext(taskContext)
vision.generateDirectiveDrafts(visionId)
vision.reaffirm(visionId)
```

### 5.2 Agent 上下文装配接口

`vision.buildAgentContext(taskContext)` 应按场景裁剪：

- `planning`：愿景摘要 + directives + 相关项目
- `project_creation`：愿景摘要 + 相似历史项目 + 当前机会分类
- `review`：愿景原文 + 当期版本 + journal 摘要 + review 快照
- `general_chat`：仅愿景摘要，不注入历史长文

### 5.3 隐私边界

愿景属于高敏感个人数据，默认按最高等级保护：

1. **本地优先**：原文默认仅保存在本地 Markdown 与本地数据库。
2. **最小上传**：若调用云端模型，默认只上传 `summaryForAgent`，不上传完整原文；上传完整原文必须显式征得同意。
3. **可见性控制**：用户可设置“允许 Agent 读取原文 / 仅读取摘要 / 完全不用于云端推理”。
4. **可导出与可删除**：删除愿景时，系统需提供“删除对象索引”“保留本地 Markdown”“连同版本快照彻底删除”三种路径。
5. **不用于统计画像**：愿景文本不得进入产品分析、推荐训练、跨用户统计画像。

### 5.4 与 journal 和记忆的边界

- 愿景是**用户源文本**，不是回顾摘要。
- journal 是**过程记录**，可以反思愿景，但不替代愿景。
- review snapshot 是**阶段性理解缓存**，可过期、可重建。
- memory context 是**当前会话的受控注入块**，不是永恒真相。

换言之：愿景是原点，journal 是沿途，review 是回望，memory context 是当下会话的压缩视图。

## 6. 与其他专题的依赖关系

### 对 02 专题的依赖：directive 体系
本专题为 02 提供上游语义边界：directive 不是愿景拆成 checklist，而是用户确认后的行动原则。02 需要复用这里定义的 `derived_from` 关系、用户确认门槛与“Agent 可建议、不可代替确认”的规则。

### 对 03 专题的依赖：项目 / 计划分层
03 需要承接这里的层级：项目承接愿景，计划承接项目或 directive。03 不应允许计划直接声称“代表愿景完成度”，也不应把项目对齐度做成硬评分。

### 对 08 专题的依赖：Journal / Review 回顾体系
08 需要使用本专题定义的回顾读取方式：回顾时可读取愿景原文、历史版本、reaffirm 时间与相关 journal 片段，但输出应以“重新看见方向”为目标，而不是“对齐审计报告”。

### 对 14 专题的依赖：Agent prompt / memory orchestration
14 需要把本专题的三层注入法落地：system prompt 写原则，memory context 写摘要引用，review/journal 读取原文与历史。14 还要继承本专题的安全边界：不得把愿景摘要漂移成 belief 系统，不得自动写回用户愿景。

## 7. 风险、边界与设计决策

1. **风险：愿景被过度结构化**
   - 决策：只保留标题、正文、版本、提醒姿态、少量标签；不引入 metric、alignment score、confidence。

2. **风险：Agent 摘要逐渐取代原文**
   - 决策：所有摘要必须引用 `visionVersionId`；回顾场景始终可回看原文；摘要仅作为上下文压缩，不作为事实源。

3. **风险：提醒过多，重新制造任务管理负担**
   - 决策：仅在选择、创建、回顾、求助四类高价值节点触发；默认语气为邀请式；用户可设为 silent。

4. **风险：愿景更新导致历史混乱**
   - 决策：采用追加式版本；历史项目、journal、review 绑定当期版本；当前建议默认读取 active version。

5. **风险：隐私敏感内容暴露给云端模型**
   - 决策：默认仅上传摘要；完整原文上传需单独授权；本地模型可直接读原文。

6. **关键设计决策：愿景保留为一等对象，而不是被并入 directive 或 belief**
   - 原因：最小数据结构文档强调减少系统复杂度，这一点对 directive、plan、event 成立；但在全局对象系统里，愿景有独立的创作方式、版本节奏、隐私等级与 Agent 使用边界，值得保留为 first-class object。

7. **关键设计决策：Orbit 不做“愿景对齐审计”**
   - 原因：一旦愿景被转译成打分、配额、红线，系统就从陪伴式 Agent 退化为绩效工具。这与最新设计原则直接冲突，也会破坏用户的主体性。
