# Agent 驱动任务管理

## 1. 设计定位

Orbit 的任务管理不是让用户维护一套越来越复杂的待办系统，而是让 Agent 代替用户维护任务系统本身。用户主要负责四类动作：表达意图、确认关键决策、进入专注、做结果判断；澄清、拆分、排程、追踪、回顾、再组织，默认由 Agent 持续完成。

这里需要主动消解两类看似冲突的设计：

1. **最小数据结构视角**：规划域的不可约真相只有三类——对象、关系、事件。
2. **全局对象系统视角**：Orbit 不是一张万能 `items` 表，而是类型化对象表 + `object_index` + `links` + `events`。

因此本专题的结论是：

- **逻辑层**坚持最小模型：任务管理只关心“有什么对象、它们如何关联、发生了什么”。
- **物理层**接入全局对象系统：vision、directive、project、milestone、task、review、resource、note、document 等维持各自类型表；
- **统一编排层**只通过 `object_index`、`links`、`events` 做跨对象查询与推理。

这让任务管理既不会退化成“庞大任务宇宙”，也不会与 Orbit 的对象网络割裂。任务系统不是孤岛，而是把**愿景 → 项目 → 行动 → 输出 → 回顾 → 再决策**串成闭环的执行中枢。

## 2. 关键用户场景

### 2.1 从模糊意图到可执行结构

用户说“我想系统研究 Agent-first 产品设计，并写一篇真正能发出去的文章”。Agent 不应要求用户先自己建项目、列里程碑、补资源、排时间，而是自动完成：

- 识别这是“研究 + 写作”的复合型项目；
- 从现有阅读、高亮、笔记、研究空间中抽取已有材料；
- 生成项目、里程碑与初步任务树；
- 给出需要用户确认的关键分歧点：目标读者、完成口径、时间边界。

### 2.2 从输入材料到下一步行动

用户在阅读器里高亮了 5 段内容、在 Journal 里记了一条灵感、在研究空间里得到一份对比矩阵。Agent 需要判断这些新增内容应当：

- 关联到已有项目；
- 升级为任务；
- 仅作为 support resource 保留；
- 进入下一次 Review 再处理。

核心不是“多收集”，而是让输入在 24-48 小时内找到归宿。

### 2.3 从 Today 到 Focus 再到 Review

用户打开应用时，不应该面对一张静态待办清单，而应该看到：

- **Today**：Agent 动态生成的今日承诺集合；
- **Focus**：当前最值得进入的单一行动及其上下文；
- **Review**：当天结束后由事件与 Journal 自动汇总的回顾草稿。

Today 解决“今天做什么”，Focus 解决“现在做什么”，Review 解决“今天发生了什么、它如何改变明天”。

### 2.4 任务管理与人生规划、研究、阅读、写作的闭环

Orbit 的闭环不是“任务完成率”，而是：

**愿景提供方向感 → directive 提供执行约束 → 项目承接阶段目标 → 任务驱动当下行动 → 阅读/研究提供支撑材料 → 写作/产出形成外部成果 → Journal/Review 提炼经验 → 反哺下一轮选择。**

任务管理因此是用户生活系统的编排器，而不是单独页面。

## 3. 核心设计

### 3.1 规划域对象结构

在任务管理域中，核心对象如下：

| 对象 | 作用 | 关键特性 |
|---|---|---|
| Vision | 人生方向与长期期望 | 用户手写、低频修改、Agent 只读 |
| Directive | 可操作的行为规则 | 可由 Agent 建议，但必须用户确认 |
| Project | 一段有明确产出的持续努力 | 是任务、研究、写作的汇聚点 |
| Milestone | 项目阶段性交付物 | 强调阶段完成定义，不是大任务别名 |
| Task | 单次工作块内可推进的行动 | 必须有完成口径、上下文和下一步 |
| Review | 对一段时间内事实的解释与决策 | 可以引用事件，但不替代事件真相 |
| Resource / Support Link | 支撑理解或执行的材料关系 | 把阅读、研究、写作接入任务流 |

设计原则：

- 愿景不是 KPI；它只提供方向感。
- Directive 不是愿景；它是执行规则。
- Milestone 保留独立语义，因为它承担阶段性交付与风险判断。
- Task 必须小到可被 Focus 消化。
- Review 是“解释层”，不能替代 `events` 里的事实层。

### 3.2 任务生命周期

任务不是从“创建”直接跳到“完成”，而是经过可被 Agent 编排的生命周期：

1. **Captured**：来自输入箱、阅读高亮、语音、Journal、研究结论或 Agent 建议。
2. **Clarifying**：Agent 尝试判断这是任务、项目、里程碑、资源还是仅需归档的想法。
3. **Ready**：已具备目标、完成口径、归属项目/里程碑、必要上下文。
4. **Scheduled**：被放入某个时间窗口、Today 列表或等待队列。
5. **Focused**：成为当前执行对象，系统加载 support links、前置材料、输出目标。
6. **Done / Blocked / Dropped**：完成、阻塞、取消三种结果都必须留下事件。
7. **Reviewed**：被日回顾、周回顾或项目复盘吸收，决定是否生成后续任务、规则调整或资源沉淀。

其中状态应以**事件驱动**表达，而不是只改一列状态值。状态列用于快速展示，真实历史由 `events` 记录。

### 3.3 Today / Focus / Review 机制

#### Today：动态工作集合

Today 不是用户手工维护的清单，而是 Agent 每次根据以下条件重算的**候选承诺集合**：

- 任务是否 Ready，且未被 `depends_on` 阻塞；
- 所属项目与里程碑的风险等级；
- 与 vision / directive 的相关性；
- 截止日期、停滞时长、最近 review 决策；
- 用户今日可用时间、精力窗口、上下文切换成本；
- 是否已经存在足够的 support resources 可直接开工。

Today 默认控制在少量承诺：

- 1 个主推进项；
- 1 个次重点项；
- 1 个维护/响应项；
- 适量缓冲。

#### Focus：当前唯一动作

Focus 是 Today 中被提升为“现在就做”的唯一行动。进入 Focus 时，Agent 自动加载：

- 任务目标与完成定义；
- 所属项目、里程碑上下文；
- 相关 resource/support links；
- 最近一次相关 review 的提醒；
- 可能的输出去向（文档、笔记、文章草稿、研究结论）。

Focus 视图的目标是降低启动成本，而不是增加信息噪声。

#### Review：事实压缩为下一步决策

Review 分三层：

- **日回顾**：压缩当天 `events`、Journal、Focus 结果，生成第二天候选行动；
- **周回顾**：判断项目进度、时间分配、输入→输出转化是否健康；
- **项目复盘**：当项目完成或停滞时，提炼可复用工作法、风险模式与衍生项目。

Review 必须产出决策，而不是只有总结文字。至少要回答：

- 哪些任务应继续、暂停、拆分或删除；
- 哪些资源值得升级为研究或写作素材；
- 下一件最该做的事是什么。

### 3.4 “下一件该做的事”的动态计算

Orbit 不维护一个固定的“Next Action”字段，而是实时计算。计算过程分四步：

1. **筛候选**：仅保留 Ready、未阻塞、上下文完整、所在项目有效的任务。
2. **算约束**：考虑 deadline、milestone 风险、directive、可用时间、能量匹配、等待外部输入等硬约束。
3. **算收益**：评估它是否推进关键路径、是否能把最近输入转成输出、是否能消除阻塞、是否能延续已有工作记忆。
4. **生成建议**：输出 1 个当前最优 Focus、2 个备选项，并说明理由与代价。

建议公式不是对用户的审计，而是帮助 Agent 在多目标之间做权衡。一个典型决策权重为：

- 关键路径推进度；
- 截止风险；
- 与愿景/规则的一致性；
- 输入转行动的时效；
- 上下文切换成本；
- 用户最近承诺与疲劳状态。

所以“下一件事”不是数据库里预先存好的答案，而是对当前事实的一次计算结果。

### 3.5 与资源、研究、写作的联动闭环

任务系统必须和知识对象形成双向流动：

1. **阅读 → 任务**：文章、高亮、网页、摘录可通过 `supports` / `inspires` / `source_of` 进入任务上下文。
2. **研究 → 任务**：研究空间与研究产物可以直接生成 milestone 或 task，也可以只作为 support resource。
3. **任务 → 写作**：任务完成后产生 note、document、draft、post 等输出对象，并通过 `output_of` 或 `derived_from` 建链。
4. **写作 → 回顾**：发布、归档、反馈等事实进入 `events`，为 Review 提供结果信号。
5. **回顾 → 新任务**：Review 不直接改写愿景，但可以建议新的 directive、项目或任务，等待确认后进入下一轮。

这意味着 Orbit 中的“任务完成”不是终点，而是知识网络和创作网络继续生长的起点。

## 4. Agent 行为与自动化机制

### 4.1 自动化规则

Agent 在任务管理中默认承担以下自动化责任：

1. **自动澄清**：所有捕获对象先做归类建议，区分任务、项目、里程碑、资源、笔记、归档项。
2. **自动拆分**：超过单次工作块上限的 task，自动建议拆分；高风险 milestone 自动建议补足前置任务。
3. **自动排程**：根据 Today 预算、已有会议、最近延迟记录、项目风险重排今日顺序。
4. **自动重排**：未完成任务默认不机械顺延，而是重新判断是否继续、拆小、降级、删除或等待。
5. **自动链接**：阅读高亮、研究产物、文档草稿与相关 task/project 建立 support links。
6. **自动回顾**：按日/周/项目节点生成回顾草稿，并给出下一步建议。
7. **自动发现阻塞**：识别依赖未满足、资源不足、长期拖延、愿景偏离、项目停滞。

### 4.2 用户确认点

Agent 不能把任务系统变成黑箱，因此以下节点必须显式让用户确认：

| 确认点 | 是否必须确认 | 原因 |
|---|---|---|
| 将模糊意图升级为正式项目 | 必须 | 会引入长期承诺 |
| 改写 vision | 绝对必须且只能用户编辑 | 愿景是主体性，不是 Agent 输出 |
| 新增/修改 directive | 必须 | 属于行为约束 |
| 跨项目优先级大调整 | 必须 | 影响近期承诺 |
| 里程碑 deadline 变更 | 必须 | 影响计划语义 |
| 删除/归档项目、批量删除任务 | 必须 | 不可逆或高风险 |
| 对外发布、发送消息、创建外部日程 | 必须 | 涉及外部世界 |

以下行为可在透明前提下自动执行：

- 建立高置信度 support links；
- 生成 Today 初稿；
- 未开始任务的低风险重排；
- 日回顾草稿与项目统计；
- 提醒可能的下一步行动。

### 4.3 Today / Focus / Review 的 Agent 分工

- **Planner Agent**：负责澄清、拆分、排程、重排、下一步计算。
- **Focus Agent**：负责进入执行态时的上下文压缩、材料加载、打断管理与完成判定辅助。
- **Review Agent**：负责从 Journal、events、产出对象中压缩事实，生成回顾与建议。

三者共享同一事实层，不共享可漂移的“私有记忆真相”。长期稳定背景只来自 vision、directive 和用户确认过的规则。

## 5. 数据模型 / 接口 / 能力边界

### 5.1 逻辑最小模型与物理落地

本专题采用“逻辑三件套 + 物理类型表”的兼容方案：

| 逻辑层 | 物理层 |
|---|---|
| Object | `visions`、`directives`、`projects`、`milestones`、`tasks`、`reviews`、`notes`、`documents`、`articles` 等类型表 + `object_index` |
| Link | `links` |
| Event | `events` |

`object_index` 负责统一检索和引用，`links` 负责跨类型关系，`events` 负责不可约事实。Agent 编排不需要知道每张类型表的全部细节，只需要知道：

- 对象是什么；
- 当前状态是什么；
- 和谁有关；
- 最近发生了什么。

### 5.2 关键关系设计

任务管理至少需要以下关系：

| relation | 含义 |
|---|---|
| `parent_of` | project → milestone → task 的层级 |
| `depends_on` | task / milestone 的依赖 |
| `supports` | resource / note / artifact 支撑 task / project |
| `source_of` | 某个资源直接触发某项任务 |
| `output_of` | note / document / draft / post 是某任务或项目的产出 |
| `reviews` | review 面向某个项目、周期或对象集合 |
| `relates_to` | 弱关联，用于发现网络而非驱动执行 |

其中 `supports` 是关键：它把“任务管理”与“阅读、研究、写作、资料库”接成一张网。

### 5.3 关键事件设计

事件类型至少覆盖：

- `captured`
- `clarified`
- `decomposed`
- `scheduled`
- `rescheduled`
- `focus_started`
- `focus_ended`
- `blocked`
- `completed`
- `dropped`
- `review_generated`
- `review_confirmed`
- `agent_suggested`
- `user_confirmed`
- `user_rejected`

规则是：**只把事实写入事件；解释写进 review；价值判断留给 Agent 在读取时即时生成。**

### 5.4 能力接口

对内对外统一暴露以下能力，而不是为每个页面硬编码独立逻辑：

- `task.capture`
- `task.clarify`
- `task.decompose`
- `task.schedule`
- `task.compute_next`
- `task.start_focus`
- `task.complete`
- `task.review_period`
- `task.confirm_change`
- `task.link_support`

这些能力既服务 Today/Focus/Review，也服务外部 Agent、MCP、自动化规则和未来多端同步。

### 5.5 边界

本专题明确不做以下事情：

- 不让 Agent 自动改写愿景；
- 不把任务分数化成对人的审计系统；
- 不把所有推理结果都固化成长期记忆；
- 不要求用户维护复杂标签、字段、看板才能获得“下一件事”；
- 不把 Review 写成无法回溯来源的主观散文。

## 7. 风险、边界与设计决策

### 7.1 为什么不走“万能任务表”

如果只用一张 `items` 表，短期很简洁，但 Orbit 需要把任务和阅读、研究、写作、日志、资源网络真正打通。纯万能表会削弱类型边界，最终把高价值语义塞进 JSON。这里选择：

- 逻辑上保持最小；
- 物理上保持类型化；
- 编排上保持统一。

### 7.2 为什么不让 Agent 全自动

用户不需要维护复杂系统，不等于用户失去主导权。Orbit 的目标是**减少维护负担**，不是**代替人生决策**。所以：

- 自动化负责准备选项；
- 用户负责确认承诺；
- 愿景、规则、优先级的大转向都必须保留人工确认。

### 7.3 为什么 Today 不能是静态列表

真正的“下一件事”会被新输入、项目风险、身体状态、刚完成的输出即时改变。静态 Today 只会迅速失真，最终重新变成用户维护系统。动态计算虽然复杂，但它把复杂度留给 Agent，而不是留给用户。

### 7.4 最终设计决策

本专题最终采用以下判断：

1. **愿景是罗盘，directive 是规则，二者严格分离。**
2. **任务管理的事实底座只有对象、关系、事件。**
3. **Today/Focus/Review 是同一条执行链，而不是三个孤立页面。**
4. **“下一件该做的事”必须动态计算，而不是手工维护。**
5. **任务系统必须与阅读、研究、写作、Journal 联动，否则不会形成闭环。**
6. **Agent 的价值在于降低系统维护成本，不在于制造更多管理动作。**
