# 12. 关系型对象数据库

## 1. 设计定位

Orbit Reboot 需要的不是图数据库，也不是一张“万能对象表”，而是一套**以关系型数据库为底座、为对象网络服务、又与文件系统优先架构协同**的对象数据库设计。这里的“对象数据库”不是在数据库层模拟 OO，而是把项目、任务、文章、高亮、笔记、研究空间、写作产物、AI 对话、日志等都纳入同一套可引用、可检索、可回放、可被 Agent 操作的关系模型。

本专题的最终裁决是：

1. **全局物理 schema 采用“每种对象类型一张表 + `object_index` + 统一 `links` + 统一 `events`”**。
2. **“极简统一 `items/links/events`”只保留为逻辑抽象与 Agent 访问视图，不作为全局物理存储策略。**
3. **数据库负责元数据、关系、事件、搜索投影与能力编排；长文本正文与二进制资产仍优先落在文件系统。**

因此，Orbit 的数据库不是用户内容唯一真相层，而是文件系统之上的**关系编译层 / 查询层 / 回放层**：它把分散在 `sources/`、`wiki/` 与 `.orbit/` 中的对象编译为统一对象网络，供 UI、搜索、Agent、MCP 与同步系统稳定消费。

## 2. 关键用户场景

### 2.1 统一查询：同一个搜索框查到任务、文章、笔记、研究与写作对象

用户搜索“Agent memory”，系统不应该分别查询任务表、文章表、研究表、文档表再在前端拼接，而应先从 `object_index` 与搜索投影中得到统一结果集，再按对象类型补全详情。这样搜索、最近更新、全局命令面板、对象选择器与 backlinks 面板才能共享同一条查询路径。

### 2.2 统一引用：任何对象都能稳定指向任何对象

一条高亮可以支撑一个任务；一个研究结论可以喂给写作文档；一个 AI 对话可以关联到文章、项目与研究空间；一个 Journal 条目可以回看某次任务完成事件。这里需要的不是专用关联表爆炸，而是**全局稳定对象引用**：所有关系都能通过统一对象标识建立、查询与回放。

### 2.3 统一 Agent CRUD：Agent 不需要记住 20 套表结构

Agent 在做澄清、建链、生成任务、沉淀研究、更新文档、生成回顾时，不应直接面向每张物理表写 SQL。它应只看见一套稳定能力：`object.query`、`object.read`、`object.write`、`link.write`、`event.append`、`search.query`。数据库内部再把这些统一能力路由到正确的类型表与索引投影。

### 2.4 统一日志与回放：能回答“发生了什么”，也能重建“现在为什么是这样”

用户需要在 Today、Journal、对象详情与调试界面里看到：某个任务为什么进入 Today、某条关系是谁建的、某份研究报告是由哪些来源演化而来、外部编辑器修改 Markdown 后系统怎样重编译对象状态。这要求 Orbit 不是只保留最终状态，而是保留统一事件层，并支持从事件重建投影。

### 2.5 文件系统优先：用户资产仍然可迁移、可外部编辑、可脱离 Orbit 存活

笔记、文档、愿景、研究报告、部分日志应优先保存在 `sources/` 或 `wiki/` 中。数据库不能反过来吞掉这些内容；它必须能在文件变更后重新索引、重建关系和搜索投影，并保持对象引用不漂移。

## 3. 核心设计

### 3.1 最终 schema 策略裁决

#### 推荐方案：类型表 + 全局索引 + 统一关系 + 统一事件

全局物理层采用如下结构：

- **对象类型表族**：`visions`、`directives`、`projects`、`milestones`、`tasks`、`articles`、`books`、`highlights`、`notes`、`research_spaces`、`research_artifacts`、`documents`、`ai_chats` 等。
- **全局索引表**：`object_index`。
- **统一关系表**：`links`。
- **统一事件表**：`events`。
- **统一搜索投影**：`object_search_fts`、`object_chunks_fts` 等。
- **统一访问层**：通过对象仓库与能力接口暴露给 UI / Agent / MCP。

这个方案吸收了两边的长处：

- 吸收“极简三件套”的**认知简洁性**：全局只关心对象、关系、事件。
- 保留“类型表”的**物理可维护性**：专属字段、强语义约束、可针对性索引、与文件系统的明确映射。

#### 不采用的方案：全局一张 `items` 表作为物理主表

`items/links/events` 很适合作为**规划域第一性抽象**，但不适合作为 Orbit 全局对象数据库的物理底座。原因有五个：

1. **类型语义会塌陷**：任务、文章、高亮、AI 对话、研究资料池、写作草稿、文件对象的字段差异极大；强行进 `items`，最终只能走“稀疏列 + 大量 nullable + JSON meta”。
2. **约束能力下降**：很多对象需要自己的外键、唯一约束、版本约束、锚点结构或文件映射规则；统一表会让这些约束退化到应用层。
3. **索引策略失真**：高亮需要按来源对象和锚点查；任务需要按状态、排程与层级查；研究对象需要按问题树、证据链查；统一表很难兼顾。
4. **文件系统协同更差**：文件类对象与数据库原生对象并不对称。统一表会把“文件对象”也压成同一种记录，反而模糊 `sources/` / `wiki/` / `.orbit/` 的边界。
5. **Agent 能力表面简化、内部复杂度转移**：看起来 CRUD 统一了，实际上所有类型特有校验、投影、回写、搜索更新都会堆到一层巨大的 `meta` 解释器里，长期更难维护。

因此，**`items/links/events` 只能作为访问抽象或 SQL View，不作为全局真实 schema**。它在 Orbit 中的正确位置是：

- 给 Agent 暴露一个极简、稳定的逻辑模型；
- 给任务规划等子域提供低认知负担视图；
- 作为跨类型查询的投影层，而不是存储真相层。

### 3.2 核心表族

#### A. 类型对象表族：保存“对象专属真相”

每个对象类型保留自己的强类型表，示例：

- 规划域：`visions`、`directives`、`projects`、`milestones`、`tasks`、`reviews`
- 输入域：`articles`、`books`、`highlights`、`notes`、`assets`
- 研究域：`research_spaces`、`research_questions`、`research_source_sets`、`research_artifacts`
- 输出域：`documents`、`drafts`、`posts`
- Agent 域：`ai_chats`、`tool_runs`、`agent_sessions`

设计原则：

1. **对象专属字段留在各自表里**，不进入万能 JSON。
2. **真正的结构性父子关系允许保留本地外键**，例如 `milestones.project_id`、`tasks.parent_id`、`highlight.source_object_id`、`document.file_id`。
3. **一切需要跨域统一消费的对象，都必须同步进入 `object_index`**。

这意味着 Orbit 的全局数据库不是“单表宇宙”，而是“类型表宇宙 + 统一对象入口”。

#### B. 文件注册表：连接 11 的文件系统优先架构

本文不把文件系统并进 `items`，而是要求 11 提供稳定的文件注册层（如 `file_index` / `file_registry`）。对象表通过 `source_file_id`、`content_hash`、`version_token` 与文件系统绑定：

- Markdown / 文档 / 报告：文件正文在 `sources/` 或 `wiki/`
- SQLite：保存对象 ID、状态、关系、索引、搜索、事件
- 外部编辑修改文件后：由文件编译管线增量更新对象表与 `object_index`

这样文件系统是真相层，关系库是编译层，而不是互相争夺主权。

### 3.3 全局索引层：`object_index`

`object_index` 是全局对象数据库的中心，不是冗余附属表。建议采用稳定 `object_uid`：

```sql
CREATE TABLE object_index (
  object_uid TEXT PRIMARY KEY,            -- 例如 task:tsk_123 / article:art_456
  object_type TEXT NOT NULL,
  object_id TEXT NOT NULL,
  canonical_table TEXT NOT NULL,
  layer TEXT NOT NULL,                    -- source | wiki | system
  source_file_id TEXT,                    -- 对接 11 的文件注册层
  title TEXT,
  summary TEXT,
  status TEXT,
  origin TEXT NOT NULL DEFAULT 'human',   -- human | ai | system
  visibility TEXT NOT NULL DEFAULT 'private',
  version_token TEXT NOT NULL,
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL,
  deleted_flg INTEGER NOT NULL DEFAULT 0,
  UNIQUE(object_type, object_id)
);
```

`object_index` 负责五件事：

1. **统一查询入口**：所有跨类型列表、对象选择器、最近更新、全局命令面板都先查这里。
2. **统一引用协议**：`links`、`events`、搜索、MCP、同步层都以 `object_uid` 为主引用，不再反复处理复合键。
3. **统一权限与来源标签**：`layer`、`origin`、`visibility` 让上层能分清用户对象、AI 对象、系统对象。
4. **统一版本与失效控制**：`version_token` 用于搜索投影、缓存失效、回放校验。
5. **统一对象解析**：Agent 拿到 `object_uid` 后可继续 hydrate 到具体类型表。

建议补充几个核心索引：

- `(object_type, updated_at DESC)`
- `(layer, object_type, deleted_flg, updated_at DESC)`
- `(origin, updated_at DESC)`
- `(source_file_id)`

### 3.4 关系层：统一 `links`，不用关系表爆炸

Orbit 最终应采用一张统一 `links` 表，而不是重新拆成 `content_links`、`entity_links`、`tag_links` 三套主表。三者如果都存在，也应只是**兼容视图或查询别名**，底层仍是一张统一关系表：

```sql
CREATE TABLE links (
  link_id TEXT PRIMARY KEY,
  source_uid TEXT NOT NULL REFERENCES object_index(object_uid),
  target_uid TEXT NOT NULL REFERENCES object_index(object_uid),
  relation_type TEXT NOT NULL,
  origin TEXT NOT NULL,                   -- human | ai | system
  context_json TEXT,
  weight REAL,
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL,
  deleted_flg INTEGER NOT NULL DEFAULT 0,
  UNIQUE(source_uid, target_uid, relation_type, origin)
);
```

关系层设计原则：

1. **跨域关系统一进 `links`**：`supports`、`references`、`derived_from`、`evidenced_by`、`aligned_to`、`tagged_with`、`reviews`、`output_of` 等都共用同一张表。
2. **`origin` 必须保留 human / ai / system 区分**，这是主体性与可解释性的基础。
3. **结构性强关系可同时保留本地外键，但仍要镜像到 `links`**，这样既能高性能查询，又能进入全局图谱。
4. **关系语义通过枚举治理，而不是临时拼字符串**；应有 `relation_catalog` 或代码级 registry 约束合法关系。

这让 Orbit 获得：

- 统一 backlinks
- 统一关系面板
- 统一对象推荐
- 统一 Agent 召回上下文
- 统一来源过滤（只看 human / 全部）

### 3.5 事件层：统一 `events`，用于日志、审计与回放

Orbit 不能只保留最终状态。所有对象写入、状态迁移、关系创建、文件重编译、Agent 建议确认都必须进入追加式事件层：

```sql
CREATE TABLE events (
  event_id TEXT PRIMARY KEY,
  stream_uid TEXT,                        -- 主要对象流，可为空
  event_type TEXT NOT NULL,
  actor_type TEXT NOT NULL,               -- user | agent | system
  actor_id TEXT,
  causation_id TEXT,
  correlation_id TEXT,
  payload_json TEXT NOT NULL,
  occurred_at TEXT NOT NULL,
  created_at TEXT NOT NULL
);
```

事件层承担四个职责：

1. **统一日志**：Journal、对象详情页、调试面板都能读这张表。
2. **统一回放**：搜索投影、Today 计算缓存、对象快照损坏后，都可以基于事件重建。
3. **统一解释来源**：为什么对象状态变了、为什么 Today 里出现这项任务、谁创建了这条 AI 关系，都能追出来。
4. **统一同步游标**：未来跨端同步、增量导出、冲突诊断，都需要事件 cursor。

关键设计决策：

- `events` 记录**事实**，不是大段解释；解释应产出为 review/note/document 等对象。
- 所有写入事务都遵循：**先写类型表 / 链接表当前状态，再追加事件，最后更新搜索投影**，并保持同事务或可靠 outbox。
- 若需要更强回放能力，可为关键对象补充 `snapshot_every_n_events` 策略，但快照只是优化，不是真相。

### 3.6 搜索层：独立投影，不把搜索逻辑塞回对象表

统一搜索不能直接在类型表和 `object_index` 上做大量 `LIKE`。Orbit 应明确区分：

1. **对象检索投影**：`object_search_fts`
   - 索引标题、摘要、别名、标签、对象类型、状态
   - 返回统一对象结果集
2. **正文块检索投影**：`object_chunks_fts`
   - 索引文章段落、笔记 block、文档片段、研究结论块
   - 命中后再映射回 `object_uid` 与具体 anchor / block

建议：

```sql
CREATE VIRTUAL TABLE object_search_fts USING fts5(
  object_uid UNINDEXED,
  title,
  summary,
  keywords,
  tokenize='unicode61'
);
```

搜索层只做投影，不做真相存储。其输入来自：

- 类型表结构化字段
- `object_index` 公共字段
- 文件解析器抽取的正文块
- `links` 派生的关键词与 backlinks 计数

这样统一搜索、命令面板、引用插入器、Agent 召回都能共享同一索引，而不需要每个模块各建一套搜索库。

### 3.7 Agent 访问层：给 Agent 的是统一能力，不是原始表

Orbit 必须提供一层稳定的对象访问协议，把复杂物理 schema 收束为少量能力：

- `object.query(filter)`：查 `object_index` + 搜索投影
- `object.read(objectUid)`：hydrate 具体对象详情
- `object.write(type, payload)`：路由到对应类型仓库并同步索引/事件
- `link.write(sourceUid, targetUid, relationType, meta)`
- `event.append(streamUid, eventType, payload)`
- `search.query(text, scope)`
- `replay.range(cursor)`

这里可以额外提供一个**Agent 逻辑视图**，把全局 schema 投影成极简三件套：

- `agent_items_v`
- `agent_links_v`
- `agent_events_v`

这样做的意义是：

1. **Agent 的认知负担保持极低**；
2. **全局数据库的物理设计仍保持强类型与高约束**；
3. **03 的最小任务模型可以继续成立，但不再绑架全局存储策略。**

## 4. Agent 行为与自动化机制

### 4.1 统一 CRUD 流程

Agent 对任何对象的创建与更新都必须走统一对象仓库，而不是直接写表：

1. 解析对象类型与意图。
2. 调用对应类型仓库做字段级校验。
3. 在同一事务中写入类型表。
4. 同步/刷新 `object_index`。
5. 如有关系变化，写入 `links`。
6. 追加 `events`。
7. 异步或事务后更新搜索投影。

### 4.2 Agent 自动建链规则

Agent 可以自动创建高置信度关系，但必须满足：

- 关系必须落到 `links.origin='ai'`
- 必须有来源说明或证据片段
- 不得绕过对象注册层直接写裸引用
- 高风险关系（跨项目承诺、愿景改写、对外发布）仍需用户确认

### 4.3 Agent 回放与解释能力

当用户问“为什么你建议我今天做这个？”时，Agent 应能组合：

- `object_index` 当前状态
- `links` 中的支撑材料 / 阻塞关系 / 愿景对齐关系
- `events` 中最近的完成、延期、回顾、建议确认记录

也就是说，Orbit 的可解释性不是靠 prompt 编故事，而是靠数据库里可追溯的对象、关系与事件。

## 5. 数据模型 / 接口 / 能力边界

### 5.1 推荐最小数据契约

| 层 | 必备表 / 投影 | 作用 |
|---|---|---|
| 类型对象层 | 各类型表 | 保存对象专属字段与约束 |
| 全局索引层 | `object_index` | 统一查询、统一引用、统一对象入口 |
| 关系层 | `links` | 统一 backlinks、支撑关系、标签关系、来源区分 |
| 事件层 | `events` | 统一日志、统一回放、统一同步 cursor |
| 搜索层 | `object_search_fts`、`object_chunks_fts` | 统一对象检索与正文片段检索 |
| 访问层 | object/link/event/search/replay 能力接口 | 统一 UI / Agent / MCP 调用面 |

### 5.2 统一引用协议

全局统一使用 `object_uid`，而不是让每个上层模块自己拼 `(type, id)`：

- `task:tsk_123`
- `article:art_456`
- `note:nte_789`
- `research_artifact:ra_001`

上层永远只传 `object_uid`；数据库内部再解析到 `object_type + object_id + canonical_table`。这样有三个好处：

1. MCP 与多端同步协议更稳定；
2. block provenance、wikilink、引用块、回链面板都能共享同一引用格式；
3. `links`、`events`、搜索结果不需要反复带两列主键。

### 5.3 与文件系统优先架构的协同

与 11 的协同原则必须写死：

1. **用户正文优先在文件中**：Markdown、长文、报告、对话沉淀、部分日志不以内嵌 SQLite 为唯一真相。
2. **数据库保存“编译结果”**：对象元数据、引用、关系、状态、搜索投影、事件索引。
3. **文件变更驱动重编译**：外部编辑器修改文件后，通过 `file_registry -> parser -> object repo -> object_index / links / search / events` 管线重建。
4. **数据库可重建**：在最坏情况下，至少能从 `sources/`、`wiki/`、`.orbit/` 的文件注册信息与事件导出重建主要对象投影。
5. **AI 只写允许层**：`sources/` 默认只读、`wiki/` 可受控写入、`.orbit/` 持有内部数据库与索引。

### 5.4 明确边界

本设计明确不做：

- 不采用图数据库作为全局主库；
- 不把所有对象压进一张物理 `items` 表；
- 不让 Agent / MCP 直接操作底层 SQLite 表；
- 不让搜索索引成为真相源；
- 不让文件系统与数据库重复竞争正文所有权。

## 6. 与其他专题的依赖关系

| 专题 | 依赖关系 |
|---|---|
| **03** | 03 的任务、项目、里程碑、review 必须作为强类型对象表接入本文 schema；同时 03 可继续保留“对象/关系/事件”的逻辑最小模型，但其物理落地必须服从 `type tables + object_index + links + events`。 |
| **09** | 09 作为统一对象图谱与关系语义层，依赖本文提供统一对象引用、统一回链、统一搜索结果与 block / 引用可追溯能力；反过来，09 产生的新对象也必须注册进 `object_index`。 |
| **11** | 11 的本地优先同步、加密传输与出境边界决定本文哪些字段可跨端复制、哪些索引只在本地重建、哪些对象必须分级加密。没有 11，关系库会退化成默认可读的中心主库。 |
| **13** | 13 是本文的文件真相层前提。本文数据库必须建立在 13 的 `sources/` / `wiki/` / `.orbit/` 三层边界、文件注册、内容哈希与增量编译机制之上。没有 13，关系库会重新吞回正文真相。 |
| **14** | 14 的 Agent 编排必须把本文当作统一状态面：对象读取、上下文召回、自动建链、结果沉淀、回放解释都通过对象访问层完成，而不是通过模块私有接口绕过数据库契约。 |
| **15** | 15 的 MCP / capability layer 只能暴露本文定义的受控能力接口，不能把 SQLite 直通给外部 Agent。所有外部写入都必须经过对象注册、关系治理、事件记录与权限边界。 |
| **16** | 16 若承接同步、审计、安全、集成或系统治理能力，则必须基于本文的 `object_uid`、`origin`、`version_token`、事件 cursor 与回放语义工作；否则跨端一致性、冲突诊断与权限审计都会失去稳定锚点。 |

## 7. 风险、边界与设计决策

### 7.1 为什么明确拒绝图数据库

Orbit 的核心问题不是“图查询不够表达”，而是“需要一套 boring、可索引、可迁移、可多端复用、可与文件系统协同的对象基础设施”。关系型数据库已经足够表达对象网络；真正困难的是对象注册、关系语义、事件回放与文件编译，而不是图遍历本身。

### 7.2 为什么明确拒绝全局物理 `items` 表

`items/links/events` 是一把很好的剃刀，但它适合砍掉认知冗余，不适合替代整个物理世界。Orbit 的全局架构需要：

- 类型边界
- 文件映射
- 专属索引
- 多端同步锚点
- 溯源与版本治理

这些都要求强类型对象表继续存在。**简化应发生在访问层，而不是把底层真相压扁。**

### 7.3 最终设计决策

1. **全局主库使用 SQLite 等关系型数据库，不使用图数据库。**
2. **全局物理 schema 采用“类型表 + `object_index` + 统一 `links` + 统一 `events`”。**
3. **`items/links/events` 仅作为逻辑视图、Agent 抽象或子域投影，不作为全局主表。**
4. **文件系统继续保存正文与资产，数据库负责编译后的关系、索引、搜索与事件。**
5. **所有统一查询、统一引用、统一 Agent CRUD、统一日志与回放，都以 `object_uid` 和统一访问层为中心实现。**
6. **所有跨域关系统一进入 `links`，所有事实变更统一进入 `events`，所有跨类型入口统一进入 `object_index`。**

一句话总结：**Orbit 的“对象数据库”本质上是一个建立在文件系统之上的关系型对象编译层，而不是图数据库，也不是一张万能表。**
