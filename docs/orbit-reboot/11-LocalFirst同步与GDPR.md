# 11. Local-First 同步与 GDPR

## 1. 设计定位

本专题定义 Orbit Reboot 的**数据主权底座**：用户的内容、关系、上下文与 Agent 产物首先存在于用户设备，本地才是真相；云端的角色是加密中转与变更通知，而不是默认可读的内容主库。

### 核心设计决策：面向个人用户简化同步

Orbit 面向的是**个人用户**，不是多人协作团队。个人用户的同步场景有三个根本特征：

1. **通常 1-2 台设备活跃**（桌面 + 手机），不存在"10 人同时编辑同一段文字"的场景。
2. **真正的冲突极少**——大部分"冲突"只是延迟同步（上午桌面改了，中午手机还没拉到），不是并发编辑。
3. **加密是生命线**——个人笔记、愿景、Journal 是最私密的数据，云端不可读是硬底线。

因此，**本方案明确不采用 CRDT**。CRDT 解决的是多人实时协同编辑问题，对个人用户场景来说引入的实现复杂度和 bug 风险远大于收益。取而代之的是三条极简同步通道 + 统一加密层。

### 核心目标

1. **本地真相层不可降级**：断网时所有核心读写、检索、编辑、Agent 内部编排都可继续。
2. **云端默认不可读用户内容**：服务端只保存加密后的 blob、设备元数据与变更通知。
3. **加密方案必须简单可靠**：密钥体系越简单，出错概率越低，用户丢数据的风险越低。
4. **GDPR 思想内建**：数据最小化、导出权、删除权、明示同意是架构约束。
5. **Agent/LLM/MCP 出境可见、可控、可撤销**。

## 2. 关键用户场景

### 2.1 新用户首次启动：只在本地创建工作区

用户首次启动 Orbit 时，系统先在本地创建工作区：`sources/`、`wiki/`、`.orbit/` 与本地密钥。即使用户尚未注册账号，也已经拥有完整可用的数据空间。注册账号的意义是开启跨端同步与远程加密备份，而不是"激活数据所有权"。

### 2.2 增加第二台设备：扫码配对

用户在桌面端已积累内容，随后在 iPhone 上登录。iPhone 通过扫码或输入恢复短语获取主密钥，然后从云端拉取加密对象到本地解密落盘。云端全程不接触明文。

### 2.3 离线写入：飞机上改文档、记 Journal

用户在无网环境下继续写笔记、做高亮、记日记，所有变更先写入本地，再进入 `outbox`。恢复联网后批量加密上传。UI 不等待网络。

### 2.4 冲突发生：桌面改标题，手机改正文

个人用户的真实冲突场景几乎只有一种："两端都改了，还没来得及同步"。处理策略极简：

- **结构化字段**（标题/状态/标签）：按时间戳取最新（LWW）
- **短文本内容**（笔记/高亮/批注）：整体 LWW，内容短不会丢信息
- **长文档正文**：3-way merge，无重叠自动合并，有重叠保留两版让用户选
- **关系/链接**：合并取并集
- **删除**：删除优先 + 7 天回收站

### 2.5 导出或彻底删除

用户可一键导出解密后的工作区（Markdown + 元数据 JSON），也可发起"删除全部云端数据"。

### 2.6 调用外部模型或 MCP：数据出境边界

当用户要求 Agent 调用云端 LLM 或外部 MCP 时，系统必须展示：会发送什么数据、发送给谁、是否被保留、是一次性还是长期授权。

## 3. 核心设计

### 3.1 三通道同步架构

Orbit 的所有数据同步通过三条独立通道完成，覆盖全部数据类型：

```
┌─────────────────────────────────────────────────┐
│                   Orbit 同步层                    │
├──────────────┬──────────────┬───────────────────┤
│   结构化对象   │   长文档内容   │    大文件 blob     │
│  对象级 LWW   │  3-way merge  │   内容寻址不可变    │
│  version++   │  保留祖先版本  │   hash 去重        │
├──────────────┴──────────────┴───────────────────┤
│              统一加密层                           │
│         masterKey → AES-256-GCM                 │
├─────────────────────────────────────────────────┤
│              云端（只存密文）                      │
│         对象桶 + blob 桶 + 通知队列               │
└─────────────────────────────────────────────────┘

不同步：FTS / vector / thumbnail / cache → 本地重建
```

#### 通道 1：结构化对象同步（覆盖 95% 的数据）

适用于：project / task / goal / research_space / highlight / note（短文本）/ links / events / settings / agent_session 等。

```ts
interface SyncMeta {
  objectId: string
  version: number        // 每次修改 +1
  updatedAt: number      // 时间戳，用于 LWW
  deletedAt?: number     // 软删除
  checksum: string       // 内容哈希，快速判断是否真的变了
}
```

同步流程：

1. 本地修改 → `version++` → 写入 `outbox`
2. 有网时 → 加密 → 上传
3. 另一端轮询或推送通知 → 拉取 → 解密
4. 比较 version：
   - 远程 > 本地 → 直接覆盖
   - 本地 > 远程 → 本地是最新（等上传）
   - 相同 version 但 checksum 不同 → 冲突

冲突策略：

| 场景 | 策略 | 说明 |
|------|------|------|
| 只有一端改了 | 直接覆盖 | 99% 的情况 |
| 结构化字段冲突 | LWW（按时间戳） | 个人用户不会纠结"哪个标题对" |
| 关系/链接冲突 | 合并取并集 | 两边加的链接都保留 |
| 删除冲突 | 删除优先 + 7 天回收站 | 简单且安全 |

#### 通道 2：长文档同步（覆盖 4% 的数据）

适用于：写作稿件、研究报告等用户可能在两端编辑不同段落的长 Markdown 文档。

采用 **git 式 3-way merge**，不用 CRDT：

```ts
interface DocSyncMeta {
  objectId: string
  version: number
  updatedAt: number
  baseVersion: number   // 本次修改基于哪个版本
  checksum: string
}
```

冲突处理：

1. 取 `baseVersion` 的内容作为 common ancestor
2. `diff3(ancestor, localDoc, remoteDoc)`
3. 无重叠 → 自动合并（桌面改第一段、手机改第三段，自动合入）
4. 有重叠 → 标记冲突段落，保留两版让用户选

个人场景下"有重叠冲突"可能**一年发生不到 5 次**。为这 5 次上 CRDT 引入的工程风险远大于收益。

#### 通道 3：大文件 blob 同步（覆盖 1% 的数据）

适用于：图片、PDF、音视频、EPUB 等二进制文件。

采用**内容寻址 + 不可变存储**：

```ts
// 大文件用 hash 寻址，同一内容只存一份
const hash = sha256(fileBuffer)
const encrypted = encrypt(masterKey, fileBuffer)
await blobStore.putIfAbsent(hash, encrypted)

// 对象里只存引用
interface AssetRef {
  blobHash: string      // 内容寻址
  fileName: string
  mimeType: string
  size: number
}
```

- 同一张图片被多个笔记引用，只存一份密文
- 文件内容不可变，不存在"合并冲突"
- 删除是引用计数归零后延迟清理

#### 不同步的数据

以下数据明确不参与跨端同步，本地重建：

- FTS 全文搜索索引
- 向量索引 / embedding
- 缩略图 / 渲染缓存
- TipTap 编辑器运行时 JSON
- Agent 临时推理缓存

### 3.2 各数据类型归属通道

| 数据类型 | 同步通道 | 冲突策略 |
|----------|---------|---------|
| project / task / goal / milestone | 通道 1（对象 LWW） | 按字段时间戳取最新 |
| highlight / 短笔记 / 批注 | 通道 1 | 整体 LWW |
| links（对象关系） | 通道 1 | 合并取并集 |
| events / action_log | 通道 1 | 只追加，天然无冲突 |
| settings / preferences | 通道 1 | LWW |
| agent_session / memory | 通道 1 | 追加或 LWW |
| 长文档（写作稿件/研究报告） | 通道 2（3-way merge） | 段落级自动合并，冲突保留两版 |
| 图片 / PDF / 音视频 | 通道 3（blob） | 不可变，无冲突 |
| FTS / vector / cache | 不同步 | 本地重建 |

### 3.3 加密方案：简单即安全

#### 为什么不用 6 层密钥体系

原方案提出 RRK / AMK / WK / LK / SOK / DK 六层密钥。对个人用户场景，这种复杂度的真正风险是**密钥丢失导致数据不可恢复**。密钥层级越多，用户理解成本越高，出错概率越大。

#### 简化密钥方案

```
用户口令 / 恢复短语
  → 派生主密钥（Argon2id）
      → 加密所有对象与 blob（AES-256-GCM）
```

核心加密逻辑：

```ts
// 派生主密钥
const masterKey = await argon2id(userPassphrase, salt, {
  memory: 65536,
  iterations: 3,
  parallelism: 1,
  hashLength: 32
})

// 加密对象
function encryptObject(masterKey: Uint8Array, payload: unknown): EncryptedEnvelope {
  const nonce = crypto.getRandomValues(new Uint8Array(12))
  const plaintext = new TextEncoder().encode(JSON.stringify(payload))
  const ciphertext = aesGcmEncrypt(masterKey, nonce, plaintext)
  return { nonce, ciphertext }
}

// 解密对象
function decryptObject(masterKey: Uint8Array, envelope: EncryptedEnvelope): unknown {
  const plaintext = aesGcmDecrypt(masterKey, envelope.nonce, envelope.ciphertext)
  return JSON.parse(new TextDecoder().decode(plaintext))
}
```

#### 设备间密钥传递

新设备获取主密钥只有两条路径：

1. **扫码配对**：已信任设备生成临时配对码，新设备扫码后通过端到端加密通道接收主密钥
2. **恢复短语**：用户输入注册时生成的 12-24 词恢复短语，派生出相同主密钥

```ts
// 恢复短语 → 主密钥（确定性派生，同一短语永远得到同一密钥）
const masterKey = await argon2id(recoveryPhrase, fixedSalt, params)
```

#### 安全边界

- **云端不持有主密钥**：云端只存密文，永远无法解密
- **传输加密**：全部同步接口强制 TLS 1.3
- **本地静态加密**：`.orbit/` 下敏感数据使用主密钥加密存储
- **密钥不进日志**：任何日志系统不记录密钥、恢复短语或明文口令

### 3.4 本地真相层

Orbit 的本地真相层由三部分组成（继承 13 号文档定义）：

1. **`sources/`**：用户原始内容真相层（笔记、文档、文章、附件）。AI 默认只读。
2. **`wiki/`**：AI 编译层（实体页、概念页、综述）。AI 可增量写入。
3. **`.orbit/`**：数据库、索引、同步队列、缓存、密钥。

同步真相不是单一文件或数据库，而是以下组合：

- 文件真相：Markdown、HTML、EPUB、图片、附件
- 结构真相：`object_index + links + events`
- 时序真相：追加式变更日志与 tombstone
- 安全真相：主密钥派生参数与设备信任状态

### 3.5 云端角色

云端只承担四类职责：

1. **加密 blob 存储**：保存加密后的对象与大文件
2. **变更通知**：通知其他设备"有新数据可拉取"
3. **设备目录**：设备 ID、最后同步时间、设备名
4. **合规执行**：处理删除工单与导出作业

云端**不负责**：

- 不做默认可读的用户内容检索
- 不作为 Agent 的明文语料库
- 不在生产支持中开放工程师查阅用户内容

### 3.6 三端同步策略

#### Electron（桌面端）

- 全量工作区节点，保有完整 `sources/ + wiki/ + .orbit/`
- 负责高强度索引、后台编译、批量导出
- 空闲时执行全量同步与备份校验

#### iOS

- 高频内容优先同步：Today、近期阅读、Journal、活跃项目
- 低频大资产按需拉取（PDF、音视频点击时才下载解密）
- 密钥存储走 Keychain / Secure Enclave

#### Web

- 按工作区分片订阅：最近项目、最近 Journal、当前研究空间
- 使用 OPFS / IndexedDB 保存本地副本
- 退出浏览器后可配置"保留离线副本"或"仅会话缓存"

### 3.7 离线写入流程

```
1. UI / Agent 修改对象
2. 写入本地 SQLite / 文件系统
3. 生成 change_record 放入本地 outbox
4. 后台有网时：加密 → 上传 → 清 outbox
5. 另一端收到通知 → 拉取 → 解密 → 按通道策略合并
```

UI 不等待网络确认。用户在飞机上写的笔记、做的高亮、记的 Journal 都立即可用。

## 4. GDPR 与隐私

### 4.1 数据最小化

- 账号层只保留邮箱/登录凭证、设备列表、订阅状态
- 默认不上传全文搜索索引、Embedding、原文日志到云端
- 同步服务只保存密文、游标与最少元数据

### 4.2 导出权

提供两种导出：

1. **用户可读导出**：解密后的 `sources/`、`wiki/`、附件、Markdown
2. **系统迁移导出**：对象 JSON、关系 JSONL、SQLite dump

导出在本地执行，不经过云端明文。

### 4.3 删除权

- 支持删除单对象、工作区、账号三个层级
- 对象删除产生 tombstone，7 天恢复窗口后永久清除
- 账号删除清除云端所有密文、设备令牌、统计映射

### 4.4 匿名化统计

- 默认关闭内容级遥测
- 仅采集启动耗时、同步成功率、崩溃类型等产品指标
- 日志中的 object id 使用单向散列
- 绝不把 Journal 正文、文章正文、Agent 提示全文写入分析平台

### 4.5 明示同意

- 同意按目的拆分：同步备份、崩溃报告、匿名分析、外部 LLM、外部 MCP
- 每项同意可回看、可撤销
- 高敏内容出境默认逐次确认

## 5. Agent / LLM / MCP 出境控制

当 Agent 需要调用云端 LLM 或外部 MCP 时：

### 5.1 出境前

- 系统展示"出境说明卡"：发送什么、发给谁、目的、是否保留
- 用户可选择：全文发送 / 仅摘要 / 仅结构化元数据 / 拒绝
- 高敏对象（Journal、愿景、健康等）默认不出境，需逐次授权

### 5.2 出境后

- 记录 `egress_record`：时间、目标供应商、发送范围、用户授权类型
- 用户可在控制面板查看历史出境记录
- 可随时撤销某个供应商的长期授权

### 5.3 外部返回

- 外部 LLM/MCP 返回的结果先进入本地候选层
- 经用户确认或 Agent 判断后才进入正式对象系统
- 外部返回不自动成为长期真相

## 6. 数据模型与接口

### 6.1 同步相关表

| 表 | 作用 |
|---|---|
| `sync_outbox` | 待上传的本地变更 |
| `sync_cursors` | 每设备同步进度 |
| `sync_conflicts` | 需要用户介入的冲突记录 |
| `devices` | 设备 ID、名称、平台、最后同步时间 |
| `tombstones` | 删除记录，含恢复窗口 |
| `egress_records` | 外部出境审计记录 |
| `consents` | 用户同意状态 |

### 6.2 核心接口

```ts
// 同步
sync.enqueue(changeRecord)         // 本地变更入队
sync.push()                        // 加密上传 outbox
sync.pull()                        // 拉取远端变更
sync.resolve(conflictId, choice)   // 解决冲突

// 加密
crypto.deriveMasterKey(passphrase)
crypto.encrypt(payload)
crypto.decrypt(envelope)

// 设备
device.pair(qrCode)                // 扫码配对
device.revoke(deviceId)            // 撤销设备

// 隐私
privacy.export(scope)              // 导出工作区
privacy.erase(scope)               // 删除数据
privacy.requestEgress(policy)      // 外部出境授权
privacy.getEgressHistory()         // 查看出境记录
```

## 7. 用户控制面板

统一的"同步与隐私"面板，至少包含：

1. **同步状态**：最近同步时间、离线队列大小、冲突待处理数
2. **设备管理**：设备列表、信任状态、远程撤权
3. **密钥与恢复**：恢复短语状态、导出恢复文件
4. **数据出境**：外部供应商列表、授权范围、历史记录
5. **导出与删除**：工作区导出、账号删除
6. **冲突处理**：冲突列表、版本对比、选择保留

## 8. 与其他专题的依赖关系

| 专题 | 依赖关系 |
|---|---|
| **08（Journal / 日志）** | Journal 默认属于高敏数据，出境需逐次授权。日志的追加式特性天然适合通道 1 同步。 |
| **12（关系型对象数据库）** | 12 提供 `object_index + links + events` 的物理结构，本文在其之上定义同步单元与冲突策略。 |
| **13（文件系统优先数据架构）** | 13 定义 `sources/ + wiki/ + .orbit/` 三层边界。本文的同步以对象为单位，不以文件系统目录为单位。 |
| **14（Agent 架构）** | Agent 不得绕过本文的加密层与出境控制直接访问云端明文。 |
| **15（MCP 能力层）** | 所有 MCP 工具必须声明是否出境、是否需要用户确认。外部 MCP 写操作必须可审计。 |

## 9. 风险与设计决策

### 9.1 为什么不用 CRDT

| 维度 | CRDT | 本方案 |
|------|------|--------|
| 解决的问题 | 多人实时协同编辑 | 个人多设备延迟同步 |
| 实现复杂度 | 极高，容易出 bug | 低，每个通道逻辑独立 |
| 冲突处理 | 自动但可能产生非预期合并 | 简单规则 + 极少数人工选择 |
| 存储开销 | 需要保留操作历史 | 只需 version + baseVersion |
| 适合场景 | Google Docs、Figma | 个人笔记、个人项目管理 |

**决策：个人用户场景下 CRDT 的复杂度代价远超收益，不采用。**

### 9.2 为什么简化密钥体系

6 层密钥（RRK/AMK/WK/LK/SOK/DK）的风险：

- 用户需要理解和管理更多密钥概念
- 密钥轮换逻辑更复杂，更容易出错
- 任何一层丢失都可能导致部分数据不可恢复

**决策：一个主密钥 + 恢复短语。简单即安全。未来如需多工作区或团队协作，再按需扩展密钥层级。**

### 9.3 其他决策

- **云端默认不可读**：增加同步与客服复杂度，但换来真正的数据主权。
- **长文档用 3-way merge 而非 CRDT**：git 已证明这套策略对文本合并足够好，个人场景下"有重叠冲突"的概率极低。
- **大文件走内容寻址**：不可变 blob + hash 去重，避免大文件反复上传。
- **派生索引不同步**：FTS、向量索引、缩略图在本地重建，减少同步体积和攻击面。

**一句话总结：三条同步通道（对象 LWW + 长文档 3-way merge + 大文件 blob）覆盖全部数据类型，一层加密保护全部内容，没有 CRDT，没有多层密钥，加密不打折。**
