# 13. 文件系统优先数据架构

## 1. 设计定位

这一专题定义 Orbit 新项目的数据根基：**用户资产优先落在文件系统，数据库承担元数据、关系、状态与索引，AI 只在明确边界内进行“编译”而不侵入用户真相层。** 它不是单纯讨论“文件放哪”，而是同时回答五个系统级问题：

1. 用户真正拥有、可带走、可用外部编辑器直接操作的内容是什么。
2. Orbit 的对象系统如何把 `sources/`、`wiki/`、`.orbit/` 三层边界统一起来。
3. 搜索、回链、对象图谱、Block 溯源如何建立在这套物理布局之上。
4. Agent 在什么地方可以自由写、什么地方必须只读、什么内容只能写索引不能写正文。
5. 多端同步、加密、编辑器与对象图谱如何共享同一套 ID、变更与编译契约。

4/1 的《Orbit 数据架构设计：文件系统优先 + 元数据数据库》给出了“**文件为真相、SQLite 做索引和增值层**”的核心方向，但物理目录仍是按类型扁平展开，AI 产物与用户内容边界不够硬。4/5 的《Orbit V2 数据架构完整设计方案》和《数据架构设计决策记录》进一步把架构推进到“**三层物理隔离 + LLM Wiki 编译模式 + human/ai 关系二分**”，但仍需要和对象系统、Block 编辑器、同步/加密专题做一次最终收口。

本专题的最终结论是：**采用 4/5 的三层边界作为总框架，保留 4/1 的文件真相、frontmatter、项目视图即查询、Block 注释锚点等成熟机制，并把它们升级为面向全局对象网络的统一规则。**

## 2. 关键用户场景

### 2.1 用户在外部编辑器里直接改文件，Orbit 仍能理解它

用户可以在 VS Code、Obsidian、Typora 甚至 Finder 中直接修改 `sources/` 下的 Markdown、重命名资料、替换附件。Orbit 启动或监听到变更后，必须能重新扫描文件、恢复 object 绑定、更新搜索索引、保留既有关系，而不是把“脱离 Orbit 的编辑”视为异常。

### 2.2 Agent 从原始材料持续编译出知识页，但不能篡改用户原文

用户导入文章、写下笔记、做了若干高亮后，Agent 可以自动生成实体页、概念页、对比分析、项目 dossier、晨报与研究综述；这些内容进入 `wiki/`，并通过对象图谱与来源材料连接。AI 能大量写 wiki，但**不能静默改写 `sources/` 中的正文**。

### 2.3 创作时插入一段素材，最后能一路回到原文

用户在 Block 编辑器里通过 `/from-note`、`/from-highlight`、拖拽资料卡片等方式插入内容。写作完成后，某一段文字要能追溯到来源 note/highlight/article 的具体 block 或 anchor；搜索与对象详情页也能反向看到“这段材料被哪些文档使用”。

### 2.4 多端同步与加密不破坏本地优先体验

桌面端、iOS、Web 都以本地存储为工作集。无论文件内容是否开启加密、某端是否离线、某些索引是否只在本地重建，用户都应感受到：文件可带走、关系不丢、索引可恢复、同步不要求把整份 SQLite 二进制当作云端真相。

## 3. 核心设计

### 3.1 4/1 与 4/5 的差异，以及最终合成方案

| 维度 | 4/1 方案 | 4/5 方案 | 最终采用 |
|---|---|---|---|
| 顶层目录 | 按 `notes/ documents/ research/ library/ assets/` 扁平分区 | `sources/ + wiki/ + .orbit/` 三层隔离 | **采用三层隔离**，但 `sources/` 内部继续按对象类型分子目录 |
| 用户/AI 内容边界 | 逻辑区分，物理混放 | 物理隔离，AI 只写 wiki | **采用物理隔离** |
| 文件真相原则 | 文件系统为真相，DB 为索引 | 保留，但更强调 Wiki 编译层 | **保留并强化** |
| 项目视图 | 查询视图，不建项目目录 | 查询视图 | **完全保留** |
| frontmatter / Block 注释 | 已定义 `orbit_id`、`orbit_type`、`<!-- block:... -->` | 继承但未细化 | **保留并扩展到所有 Block 容器** |
| AI 关系来源 | `manual/agent/auto/...` 较细 | `human/ai` 二分 | **采用 human/ai 二分** |
| AI 对话 | 研究空间局部对象 | `ai_chat` 全局一等公民 | **采用 ai_chat 一等公民** |
| 知识组织 | 仍偏“文件池” | 明确 LLM Wiki 持续编译 | **采用 Wiki 编译模式** |

因此最终方案不是简单二选一，而是：

1. **用 4/5 决定边界**：`sources/` 是用户真相层，`wiki/` 是 AI 编译层，`.orbit/` 是元数据/索引/状态层。
2. **用 4/1 决定文件契约**：Markdown frontmatter、项目视图即查询、Block ID 注释、文件系统优先写入策略全部保留。
3. **用对象系统做统一胶水**：所有文件对象、数据库原生对象、wiki 对象都进入统一 `object_index + typed tables + links + events`。
4. **用 4/5 的 LLM Wiki 模式约束 AI**：AI 主要通过编译写 wiki 和索引，不直接改用户原文。

这里有三个关键边界：

1. **`sources/` 只承载“用户应能带走的原始长期资产”**。即便是导入内容，也必须以 Markdown/原文件/附件 bundle 的形式落在这里。
2. **`wiki/` 只承载“AI 编译后的长期知识页”**。它是长期资产，但不是用户原始真相。
3. **`.orbit/` 只承载“Orbit 的内部增值能力”**：数据库、搜索索引、向量索引、同步状态、缓存、密钥封装、编译规则。这里的数据可以迁移，但其存在不应成为用户带走内容的前提。

### 3.3 对象系统如何统一三层边界

Orbit 仍采用“**类型表 + object_index + links + events**”的关系型对象系统，但把文件系统对象和数据库原生对象统一纳入同一张对象网络。

#### A. 文件对象

凡是 `sources/` 与 `wiki/` 中的长期文件对象，都必须有 `object_id`，并在 `object_index` 中有一行，关键字段包括：

- `object_id`
- `object_type`
- `storage_kind = file`
- `layer = source | wiki`
- `canonical_path`
- `bundle_path`
- `title`
- `status`
- `content_hash`
- `created_at / updated_at`
- `encryption_class`

#### B. 数据库原生对象

像 `project`、`action`、`highlight`、`ai_chat`、`research_space`、`event` 这类不以 Markdown 文件为真相的对象，仍是数据库主存，但同样进入 `object_index`：

- `storage_kind = db`
- `canonical_path = null`
- 可通过 `links` 连接到任意 file-backed object

#### C. Block 级对象引用

Block 不升级为独立文件对象，但在数据库中必须有稳定的二级定位：

- `block_ref = object_id + block_id`
- `block_index`：保存 block 文本摘要、顺序、hash、anchor
- `block_provenance`：保存 block 来源链路

这样对象图谱、全文搜索、写作溯源都不必把 Block 当成一级对象，却仍能把它当作稳定锚点。

### 3.4 文件与对象 ID 绑定规则

#### 3.4.1 全局对象 ID

所有长期对象统一采用 **ULID 风格的稳定字符串 ID**，例如：

- `note_01JZ...`
- `doc_01JZ...`
- `art_01JZ...`
- `wiki_entity_01JZ...`
- `proj_01JZ...`

规则如下：

1. **对象 ID 一经生成不因重命名、移动目录而变化。**
2. **文件路径不是主键，object_id 才是主键。**
3. `sources/` 与 `wiki/` 中每个可管理文件 bundle 都必须在 frontmatter 写入 `orbit_id` 与 `orbit_type`。
4. 数据库表的主键与 frontmatter `orbit_id` 保持一致，不再引入第二套 file_id。

#### 3.4.2 文件路径规则

1. Markdown 单文件对象直接以 `{object_id}.md` 命名，路径只承担人类可预测性和冲突最小化，不承担语义主键。
2. 含原始附件的对象一律使用 bundle 目录：`{object_id}/content.md`、`{object_id}/meta.md`、`{object_id}/source.ext`。
3. 任何外部重命名都通过 `file_index` 重新绑定到既有 `object_id`；若 frontmatter 被删，则通过旧 hash / 标题 / path lineage 做一次恢复，恢复失败才生成新对象并记录冲突。

#### 3.4.3 Block ID 规则

1. 每个可编辑容器（note/document/wiki artifact/research artifact）中的顶层 Block 都可获得 `block_id`。
2. `block_id` 只需在单一对象内唯一，数据库层用 `(object_id, block_id)` 形成全局唯一键。
3. Markdown 写法沿用 4/1 与 Block 编辑器方案：`<!-- block:blk_xxx -->`。
4. 外部编辑删除注释时，Orbit 打开文件后允许按文本 hash 进行 block 回附；回附失败时补发新 `block_id`，并把旧 provenance 标为 `detached`，而不是静默覆盖。

### 3.5 元数据索引策略

索引分三层，不把所有需求都塞进一张表：

#### 3.5.1 文件清单索引：`file_index`

职责是“**看见文件系统里有什么**”，而不是表达业务语义。它记录：

- `object_id / layer / canonical_path / bundle_path`
- `mtime / size / content_hash`
- `parse_status / parse_error`
- `frontmatter_snapshot`
- `deleted_flg`

它服务文件监听、外部修改检测、增量扫描与恢复。

#### 3.5.2 全局对象索引：`object_index`

职责是“**让任何对象都能被统一查到**”。它是全局搜索、引用选择器、对象详情页、Agent 检索的统一入口，记录对象标题、类型、状态、层级、更新时间、可见性、排序信号等。

#### 3.5.3 检索与溯源索引

1. **`fts_objects` / FTS5**：全文搜索标题、摘要、正文预览。
2. **`chunk_index`**：分块检索，服务长文 grounding 与语义召回。
3. **`vector_index`**：仅存放可重建 embedding/ANN 数据，默认设备本地，不作为跨端真相。
4. **`block_index`**：block 级定位与片段预览。
5. **`block_provenance`**：写作来源链。
6. **`links`（以及 `tag_links` / `entity_links` 等兼容视图）**：对象关系图谱统一由单一 `links` 真相表承载，并带 `origin = human | ai`。
7. **`compile_edges`**：记录某个 wiki 页由哪些 source object / source revision 编译而来。

索引策略的核心原则是：**清单索引解决“文件在哪”，对象索引解决“它是什么”，检索索引解决“怎么找到它”，溯源索引解决“它从哪来、被谁用过”。**

### 3.6 哪些内容存文件，哪些内容存数据库，哪些内容双写索引

| 内容类型 | 文件系统 | 数据库 | 说明 |
|---|---|---|---|
| 用户手写长文本（note/document/journal） | **真相存文件** | 索引/关系双写 | 正文不进业务表，只做预览与索引 |
| 导入原文（article/book/web clip） | **原文/原文件存文件** | 元数据、阅读状态、锚点、索引 | 原始 HTML/EPUB/PDF 必须保留 |
| AI 编译知识页（entity/concept/dossier/synthesis） | **真相存 wiki 文件** | 索引、来源链、关系、编译状态 | wiki 文件属于 AI 编译层长期产物 |
| highlight / selection anchor | 不单独落文件 | **真相存数据库** | 高频、小粒度、强状态性数据 |
| project / action / milestone / research_space / ai_chat | 不落正文文件 | **真相存数据库** | 结构化对象与流程状态 |
| tags / links / backlinks / object graph | 不落文件 | **真相存数据库** | 图谱是增值层，不应反向污染正文 |
| frontmatter 最小元数据（id/title/tags/time） | **写入文件** | **同步入库** | 这是必要双写，不是正文双写 |
| FTS / chunk / vector / OCR / thumbnail | 不写用户内容层 | **仅存 `.orbit/` 可重建索引** | 派生数据，默认不同步或按策略重建 |
| block provenance | Block 注释保留最小锚点 | **真相存数据库** | 文件只存 block_id，不存完整溯源图 |

最终原则可以概括为三句话：

1. **长期可携带内容存文件。**
2. **结构化关系与运行时状态存数据库。**
3. **搜索、向量、分块、预览等派生能力只做索引双写，不复制正文真相。**

### 3.7 AI 编译层写入规则

AI 编译层是这套架构最容易失控的地方，因此必须有硬约束：

#### A. AI 可以写什么

1. `wiki/` 下的新建、更新、归档：实体页、概念页、dossier、comparison、synthesis、brief、report。
2. `.orbit/` 下的派生索引：FTS、chunk、vector、compile cache、graph enrichment、AI tag/link 建议。
3. 数据库中的 `links` 关系记录、`block_provenance`、`compile_edges`、`event_log` 等增值记录，且 `origin='ai'`；如需兼容 `tag_links/entity_links/content_links`，也只能由 `links` 投影生成视图。

#### B. AI 不能写什么

1. **不能直接改 `sources/` 正文。**
2. 不能无提示改用户 frontmatter 中的主字段（标题、手动 tags、手写 body）。
3. 不能把未确认的外部抓取结果直接写成用户真相。
4. 不能把自己的临时回答直接当作长期知识，除非进入 wiki 或被用户显式提升。

#### C. AI 编译写入契约

每个 wiki 文件都必须在 frontmatter 声明：

- `orbit_id`
- `orbit_type`
- `compiled_from: [object_id...]`
- `source_revisions: [...]`
- `compiler: orbit-agent`
- `generated_at`
- `compile_kind: entity|concept|dossier|summary|comparison|brief|report`
- `confidence`（可选）

同时数据库写入：

- `compile_job`
- `compile_edges`
- `links(relation='mentions' | 'tagged_with' | 'derived_from' | 'supports' ...)`
- `event_log(compiled|recompiled|archived)`

如需兼容旧查询接口，可从 `links` 派生 `tag_links/entity_links/content_links` 视图，但它们不是持久化真相表。

#### D. AI 与人工共编的规则

`wiki/` 默认是机器管理层，但允许用户在受保护区域留下引导。推荐模式是：

- AI 管理主体结构
- 用户仅在 `user_notes` 或保留区写引导/批注
- 重新编译时 AI 只覆盖机器管理区，不覆盖用户保留区

这样既保留“AI 可自由维护 wiki”的优势，又避免用户少量批注被下一轮编译抹掉。

## 4. Agent 行为与自动化机制

### 4.1 文件摄入到知识编译的标准流水线

每次新增或修改 `sources/` 内容后，后台管线都按固定顺序执行：

1. **Scan**：文件监听器发现变更，更新 `file_index`。
2. **Parse**：解析 frontmatter、正文结构、block 注释、附件 bundle。
3. **Project**：把文件投影到 `object_index` 与类型表。
4. **Index**：更新 FTS、chunk、vector、block_index。
5. **Link**：抽取 wikilink、tag、实体候选，写入 `origin=human/ai` 关系。
6. **Compile**：如命中规则，增量更新 `wiki/` 页面与 `compile_edges`。
7. **Audit**：记录 `event_log`，必要时生成 inbox 建议或冲突提示。

Agent 不应绕过这条流水线直接写数据库，也不应先写关系、后补文件。**文件与对象索引必须由同一条内容投影管线产出。**

### 4.2 三类自动化任务

#### 4.2.1 Ingest

- 新文章进入 `sources/library/articles/`
- 新笔记写入 `sources/notes/`
- 新附件进入 `sources/assets/`

系统做的事：补齐对象索引、生成块索引、推荐链接、触发 wiki 增量编译。

#### 4.2.2 Reconcile

- 外部编辑器改了 frontmatter
- 用户重命名了 bundle
- 某端同步回来了远程文件版本

系统做的事：以 `object_id` 为主键重绑文件、修复 path lineage、更新 hash、标记 provenance 是否仍可回附。

#### 4.2.3 Compile / Lint

- 从 sources 编译 wiki 页
- 从对象图谱发现孤岛内容、过时摘要、断裂溯源
- 为 AI 建议关系打上 `origin='ai'`

Lint 的输出不应直接污染 `sources/`，而是进入 wiki、graph 或 inbox。

## 5. 数据模型 / 接口 / 能力边界

### 5.1 与同步的协同方式

本专题与同步架构的契约是：**同步的是逻辑对象与文件变更，不是整份 `.orbit/db/orbit.sqlite` 的二进制镜像。**

1. `sources/` 与 `wiki/` 作为文件对象集合参与同步，最小同步单元是“对象 bundle + revision”。
2. 数据库中的 `object_index/typed tables/links/events/block_provenance` 走逻辑复制或变更流同步。
3. `FTS/vector/chunk/cache` 默认是设备本地可重建索引，不作为云端 authoritative data。
4. 文件冲突优先按对象 revision 处理；数据库冲突优先按对象字段与事件流合并；两者都必须回到同一 `object_id`。

### 5.2 与加密的协同方式

文件系统优先并不意味着明文优先。最终策略是：

1. **加密对象仍保留同样的物理层级与 object_id。** 只是 bundle 内正文/原文件变为密文，`.orbit/crypto/` 保存密钥封装信息。
2. `object_index` 只保留最小可同步元数据：`object_id/type/layer/path_stub/encryption_class`；敏感标题、摘要、chunk、embedding 只在本地解锁后建立索引。
3. `wiki/` 的编译结果若来源于受限材料，默认继承同级安全域，不能降级为公开明文摘要。
4. AI 处理加密内容时，解密发生在本地能力层；云端 Agent 或外部 MCP 不直接接触明文原始材料，除非用户显式授权导出。

### 5.3 与编辑器的协同方式

1. Block 编辑器写入 `sources/`/`wiki/` 时，必须遵守 frontmatter + block 注释契约。
2. 编辑器保存流程是：**先写文件，再投影索引，再刷新对象图谱**；不能只改数据库缓存。
3. `/from-note`、`/from-highlight`、拖拽插入、AI 改写都必须写入 `block_provenance`，并在必要时创建 `compile_edges` 或 `links(relation='derived_from' | 'quotes' | 'supports')`。
4. 外部编辑器是一级公民，因此任何 Orbit 专属标记都必须“删掉后仍能读，只是丢部分增值能力”，不能把正文绑死在 Orbit 私有格式上。

### 5.4 与对象图谱的协同方式

对象图谱不是另一个存储层，而是这套架构的统一查询面：

- `sources/` 提供原始对象
- `wiki/` 提供编译对象
- 数据库原生对象提供流程对象
- `links`、`block_provenance`、`compile_edges` 把三者织成一张网

因此任何能力——搜索、回链、研究、写作引用、Agent grounding——都应优先走对象图谱查询，而不是直接遍历磁盘目录。

## 6. 与其他专题的依赖关系

### 6.1 对 09 的依赖：统一对象系统与关系层

本专题把文件层做成对象系统的底座，但**对象类型注册、`object_index` 主模型、统一 links 语义**必须由 09 定义。13 需要 09 提供：

1. `object_type` 枚举与类型表边界。
2. 统一 `links` 词表、兼容视图与 `events` 的最终语义。
3. 文件对象、数据库对象、AI 对象在图谱中的统一可见性规则。

反过来，09 必须接受 13 的约束：file-backed object 的主键就是 `orbit_id/object_id`，不能再额外发明一层与文件脱钩的“对象代理 ID”。

### 6.2 对 11 的依赖：同步与冲突解决

11 需要基于本专题定义的“三层边界 + 对象 bundle + 可重建索引”来设计同步协议。13 向 11 输出四条硬约束：

1. 同步不能把 `.orbit/indexes/` 当作真相上云。
2. 冲突处理必须围绕 `object_id + revision`，不是围绕文件路径。
3. `sources/` 与 `wiki/` 的权限与冲突策略不同：前者优先保护用户编辑，后者允许 AI 编译重建。
4. `block_provenance` 与 `compile_edges` 要能跨端收敛，而不是每端各自生成孤立溯源。

### 6.3 对 12 的依赖：加密、密钥与隐私边界

12 负责定义安全域、密钥轮换、共享授权、端侧解密与云侧零知识边界。13 对 12 的依赖点是：

1. 哪些对象类型默认可加密、哪些索引允许明文同步。
2. 加密 bundle 的封装格式与 `path_stub` 规则。
3. 加密对象的搜索策略：仅本地搜索，还是支持密文可检索能力。
4. AI 编译在加密内容上的执行位置与授权模型。

### 6.4 对 16 的依赖：Block 编辑器与创作链路

16 决定这套架构在编辑体验里是否真的成立。13 依赖 16 落实：

1. Block ID 的生成、恢复、序列化与反序列化。
2. 选区上下文、引用插入、AI 改写对 `block_provenance` 的写入。
3. 外部编辑后 block 回附与 detached 处理体验。
4. 写作、笔记、研究、wiki 四类容器共用同一套 block 契约。

## 7. 风险、边界与设计决策

1. **风险：数据库重新膨胀成正文真相。** 设计决策：正文只进文件；数据库只保留预览、索引、关系与状态。
2. **风险：AI 编译层侵入用户文件。** 设计决策：AI 默认只写 `wiki/` 和 `.orbit/` 派生层，绝不静默改 `sources/`。
3. **风险：外部编辑破坏 Orbit 私有标记。** 设计决策：frontmatter 与 block 注释都是“可失去但可恢复”的增值标记，而非强绑定格式。
4. **风险：同步把索引与缓存也当真相，造成冲突放大。** 设计决策：`FTS/vector/chunk/cache` 全部定义为可重建派生层。
5. **风险：加密后搜索与 AI 能力失效。** 设计决策：把解密、分块、embedding、编译严格收在端侧能力层，本地建立索引，云端只同步必要元数据与密文。
6. **最终设计裁决**：Orbit 的数据架构不是“文件系统对数据库”的二选一，而是“**文件系统承载真相、数据库承载图谱与能力、wiki 承载 AI 编译知识**”的三层协作体系。只有这样，Local-first、对象网络、Agent 编译、Block 溯源、同步与加密才能同时成立。
