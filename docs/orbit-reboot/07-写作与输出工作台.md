# 07. 写作与输出工作台

## 1. 设计定位

写作与输出工作台不是一个“对着聊天框说一句话，然后吐出一篇文章”的生成器，而是 Orbit 在“研究 → 组装 → 创作 → 润色 → 发布”链路上的成品车间。它服务的不是单次文本生成，而是让用户把应用内已经沉淀下来的研究结论、笔记、高亮、素材、项目上下文与历史表达习惯，持续转化为对外输出。

这里的“输出”也不应被限定为长文。文章、newsletter、社媒贴文、演讲稿、视频脚本、PRD、内部 memo，本质上都是同一条生产线上的不同终端形态：它们共享材料底座、共享 voice profile、共享引用与溯源能力，只是在目标受众、结构模板、长度约束、发布渠道与后续反馈上不同。

因此本专题的核心目标有三点：

1. **把写作建立在 Orbit 内部对象网络之上**，让输出天然继承项目、研究、阅读、笔记、Journal 的上下文。
2. **把 AI 放在产线节点，而不是聊天框中央**，让 Agent 负责材料推荐、结构组装、初稿生成、定向润色与渠道适配，但不抹掉证据链和用户主导权。
3. **把“文档编辑器”升级为“输出工作台”**，使编辑、引用、溯源、风格管理、变体生成、发布与反馈回流成为一个连续系统。

## 2. 关键用户场景

### 场景 A：基于研究空间写出一篇可引用、可发布的长文

用户在 05 研究工作台中已经围绕“Agent 产品的下一代交互范式”沉淀了问题树、来源池、结论卡与研究报告。进入写作工作台后，系统自动读取该研究空间的 `claim`、`artifact`、相关高亮与项目背景，先组装候选大纲，再由用户拖拽/确认材料，最终生成一篇可继续精修的博客文章。文中每个关键判断都能展开看到来源链，而不是生成后再倒查“这句话从哪来的”。

### 场景 B：把同一份材料底座扩展成多个输出形态

用户完成一篇深度文章后，不希望重新从零写 newsletter、社媒预告和演讲提纲。工作台应允许同一 `output_document` 派生多个 `output_variant`：例如从长文提炼出 5 段 newsletter 摘要、3 条社媒贴文、一个 10 分钟分享提纲。它们共享同一 evidence base 与 voice profile，但各自有独立的长度限制、结构模板和发布状态。

### 场景 C：从高亮、边注和碎片笔记快速拼出初稿

用户未必每次都先做完整研究。有时只是围绕若干高亮、阅读批注、闪念和项目日志，希望快速写一篇开发周报或想法短文。写作工作台可以直接消费 06 中的高亮、annotation note、synthesis note 与当天 Journal 片段，在材料面板里先做分组，再生成一个“可继续写”的结构化初稿，而不是只给一段不可控的整文输出。

### 场景 D：写作、研究与发布形成闭环

用户在写作中发现某个论点证据不足，可以一键回推到 05 创建研究缺口；发布后收到反馈、收藏数据或评论摘要，又能回流为新的研究问题、写作复盘与 voice profile 调整建议。写作不是终点，而是 Orbit 内部知识与外部世界交互的接口。

## 3. 核心设计

### 3.1 写作工作台是“输出产线”，不是单页编辑器

工作台采用三栏 + 阶段轨的结构：

- **左侧：输出列表与模板入口**
  - 当前项目下的文章、newsletter、社媒草稿、演讲稿、脚本、PRD
  - 模板入口：长文、短文、线程、播客提纲、视频脚本、演讲稿等
  - 同一主题下的派生版本树
- **中间：统一 Block 编辑器**
  - 基于 TipTap 的创作文档模式
  - 支持大纲视图、章节折叠、块级 AI 操作、专注模式
  - 任意 Block 都能挂载引用、溯源、改写状态与发布适配信息
- **右侧：材料面板与 Agent 面板**
  - 材料、研究结论、引用块、笔记、高亮、资产、历史输出、voice profile 建议
  - Agent 不驻留为持续聊天主界面，而是以“阶段动作卡 + 可回写产物”的形式参与
- **顶部/底部：产线阶段轨**
  - 研究完成度 → 组装完成度 → 初稿状态 → 润色状态 → 发布状态

这样设计的关键，是让用户始终知道自己处在“哪一阶段、用了哪些材料、下一步缺什么”，而不是被一串对话历史拖着走。

### 3.2 核心对象：从写作任务到发布记录的完整对象链

写作工作台的基本单位不是“一个编辑器页面”，而是一组稳定对象：

| 对象 | 作用 | 关键内容 |
|---|---|---|
| `writing_workspace` | 某个主题/项目下的输出容器 | 目标、受众、关联项目、默认 voice、默认材料板 |
| `output_brief` | 本次输出的任务说明 | 目标读者、输出类型、核心论点、限制条件、渠道目标 |
| `material_board` | 当前写作所使用的材料编排板 | 分组、优先级、采纳状态、引用候选、冲突材料 |
| `outline_version` | 大纲版本对象 | 章节树、证据缺口、预估长度、论点分布 |
| `output_document` | 主文稿 | Block 结构、章节、状态、主 voice profile |
| `output_variant` | 渠道变体 | newsletter / 社媒 / 演讲稿 / 脚本等派生版本 |
| `revision_pass` | 一轮润色/审校记录 | 目标、变更摘要、风险提示、修改来源 |
| `publish_job` | 一次发布或导出任务 | 平台、格式化规则、审批状态、发布时间 |
| `publish_record` | 发布后的外部记录 | URL、渠道 ID、反馈摘要、回流对象 |

其中 `output_document` 只是中间对象，不是全部系统。真正重要的是：**写作 brief、材料板、大纲、文稿、变体、发布记录彼此可追溯。**

### 3.3 材料面板：不是附件栏，而是可编排的证据与素材工作区

材料面板是写作工作台最关键的差异化设计。它不只是“把相关笔记列出来”，而是一个带状态和分组能力的 `material_board`。材料来源包括：

- 05 研究工作台的 `research_artifact`、`research_claim`、`research_source_item`
- 06 笔记与高亮系统中的 `highlight`、`annotation note`、`synthesis note`
- 项目文档、任务完成记录、Journal 片段、图片/PDF/表格等资产
- 历史已发布文章中的可复用段落、结构模板、开头/结尾模式

每个 `material_item` 至少要有以下字段：

- 来源对象 ID / 类型
- 标题与摘要
- 可引用片段或 block 范围
- 可信度 / 新鲜度 / 适用输出类型
- 与当前大纲节点的关系：`supports / contrasts / anecdote / data / quote`
- 当前状态：`candidate / pinned / adopted / consumed / excluded`
- 引用与溯源能力：能否直接拖拽成引用块、事实块、背景块

用户可在材料面板里先做“分组与排序”，再决定是否进入正文。Agent 也只是在这个面板上推荐材料、标记缺口、建议剔除冗余来源，而不是绕过材料板直接整文生成。

### 3.4 Voice Profile：把“口吻”对象化，而不是把风格埋在提示词里

Orbit 需要把用户表达风格沉淀为稳定对象 `voice_profile`，供不同输出类型复用。它不是一段抽象提示词，而是一组可解释、可校正、可组合的风格约束：

- **基础语气**：克制 / 犀利 / 温暖 / 教练式 / 学术式 / 叙事式
- **句法特征**：句长偏好、段落密度、列表倾向、转折频率、口语度
- **修辞习惯**：是否喜欢设问、比喻、总结句、金句、案例先行
- **受众适配**：面向技术读者、团队成员、泛用户、投资人、社媒关注者的不同表达层级
- **禁忌与边界**：避免鸡汤、避免夸张承诺、避免过度 AI 腔、避免套话
- **代表样本**：用户确认过的历史文章、newsletter、演讲稿片段
- **负样本**：用户明确说“不像我”的段落

Agent 学习 voice 的方式必须是**显式可见的**：

1. 从用户授权的历史输出中抽取候选模式。
2. 生成一份 voice summary，交由用户确认或修改。
3. 将其保存为可版本化的 `voice_profile`，而不是每次临时猜测。
4. 写作时按“主 voice + 渠道覆盖层”组合，例如：主 voice 为“冷静分析”，社媒变体覆盖层为“更短、更口语、更有 hook”。

这保证 Agent 学的是“表达偏好”，不是机械模仿某篇旧文，也避免把写作变成一次次重新调 prompt。

### 3.5 Citation / Provenance：从引用到段落谱系的完整追踪

写作工作台必须继承 Block 编辑器与 06 的溯源能力，把“这段话从哪来”落成一等能力。

系统至少维护两层对象：

1. **`citation_record`**：面向读者或导出格式的引用记录，描述某段内容引用了哪些来源。
2. **`block_provenance` / `lineage`**：面向系统内部的段落谱系，描述该 Block 是从哪条高亮、哪条笔记、哪个研究产物、哪次 AI 改写、哪次合并演化而来。

建议支持以下转化状态：

- `verbatim`：原文引用
- `edited`：人工修改后保留来源
- `ai_rewrite`：AI 改写
- `merged`：多源组装
- `distilled`：从多条材料归纳提炼

用户在编辑器中点击任一带标记的 Block，应能看到：来源对象、插入方式、修改历史、关联研究空间、引用锚点，以及“打开原文 / 打开研究结论 / 打开来源笔记”的入口。这样 Orbit 的写作优势不在“写得更快”，而在“写得可追溯、可复核、可再利用”。

### 3.6 研究工作台、Block 编辑器、发布能力的打通方式

这三者不是串行外接，而是同一条内容供应链的不同视图：

- **05 → 07**：研究工作台提供问题树、结论卡、证据池、结构化报告，写作工作台消费这些对象并把“未解问题”反写回研究缺口。
- **06 → 07**：高亮、边注、闪念、综合笔记进入材料面板，成为可拖拽的写作积木。
- **Block 编辑器 → 07**：创作文档模式负责块级结构、溯源、Slash 插入、AI 块操作与大纲管理。
- **07 → 发布系统**：每个输出可派生多个渠道变体，并附带平台格式规则、封面/摘要/CTA、发布时间与发布记录。

因此“写作工作台”不是孤立文档页，而是**研究工作台的下游组装层、发布系统的上游源头、对象网络中的中继节点**。

## 4. Agent 行为与自动化机制

### 4.1 Agent 以阶段编排参与，而不是以聊天框常驻

写作相关 Agent 由统一编排层调度，但在工作台中的表现形式应是阶段卡、建议卡、可接受/可拒绝的结构化操作。建议拆为以下角色：

- `Writing Orchestrator`：理解当前输出 brief，决定是否需要检索材料、补充 voice、调用子 Agent
- `Material Curator`：从 05/06/项目/历史输出中推荐与去重材料
- `Outline Planner`：围绕目标读者、论点和材料组装大纲
- `Draft Composer`：基于已确认大纲与材料生成章节草稿
- `Style Editor`：按 voice profile 执行润色、压缩、扩写、改口吻
- `Channel Adapter`：把主文稿改写为 newsletter、社媒、演讲稿、脚本等变体
- `Publishing Assistant`：执行预发布检查、平台适配、导出与发布记录回写

这些角色不一定都对应独立 UI，但在系统内部应是可委派、可审计的阶段性能力，而非一个万能写手。

### 4.2 Agent 参与的五个阶段

#### 阶段 1：研究 grounding 与材料推荐

默认顺序必须是“先内部材料，后外部补充；先显式 grounding，后生成”。Agent 先读取当前 `output_brief`、项目上下文、研究空间、笔记/高亮，再构造候选材料板，并明确指出：

- 哪些材料足以支撑现有大纲
- 哪些章节证据不足
- 哪些来源重复或过时
- 是否需要回到 05 发起研究补洞

#### 阶段 2：大纲组装

Agent 不应直接跳到全文起草，而应先给出 2-3 版结构方案，例如“论点先行”“故事先行”“框架先行”，并标明各自适用的受众和材料覆盖度。用户确认后，系统保存为 `outline_version`，后续所有生成都以该大纲为基础。

#### 阶段 3：初稿生成

初稿生成以章节为单位进行，而不是一次吐出整篇文档。每次生成必须绑定：当前大纲节点、所用材料、目标 voice、预期长度。生成结果直接写回 Block 编辑器，并为相关 Block 记录 provenance。这样后续改写与审校才能精确落到段落级，而不是整篇重来。

#### 阶段 4：润色与审校

润色不是“让文风更好看”这一种动作，而应拆成多轮 `revision_pass`：

- 结构收紧：删冗余、补过渡、重排顺序
- 论证校正：检查断言是否有依据，提示缺引证段落
- 口吻对齐：根据 voice profile 做风格统一
- 渠道适配：针对 newsletter / 社媒 / 演讲口播进行长度和节奏调节
- 风险检查：避免过度承诺、事实不清、出处缺失、标题党

#### 阶段 5：发布与回流

Agent 在发布前负责生成摘要、标题候选、封面文案、渠道预览、CTA 建议，并执行平台格式化。发布后，系统把 URL、发布时间、反馈摘要、渠道表现回流到 `publish_record`，供后续复盘与 voice profile 微调。

### 4.3 Agent 的边界

1. **不能绕过材料板直接发布整文**。
2. **不能把外部检索结果直接当作正式引用**；必须先进入 05 或材料板成为可审查对象。
3. **不能静默修改用户 voice profile**；所有风格学习都需要用户确认。
4. **不能把“写得像你”理解成机械复刻旧文**；Agent 学的是可迁移风格约束，不是复制文本。
5. **不能把写作退化成连续聊天历史**；关键对象必须沉淀为 brief、材料板、大纲、文稿、修订记录与发布记录。

## 5. 数据模型 / 接口 / 能力边界

### 5.1 关键数据模型

```ts
interface WritingWorkspace {
  id: string;
  projectId?: string;
  title: string;
  defaultVoiceProfileId?: string;
  primaryOutputKinds: Array<'article' | 'newsletter' | 'social_post' | 'talk' | 'script' | 'memo'>;
}

interface OutputBrief {
  id: string;
  workspaceId: string;
  outputKind: 'article' | 'newsletter' | 'social_post' | 'talk' | 'script' | 'memo';
  audience: string;
  objective: string;
  constraints: string[];
  channelTargets?: string[];
}

interface MaterialItem {
  id: string;
  boardId: string;
  sourceObjectType: string;
  sourceObjectId: string;
  role: 'supports' | 'contrasts' | 'quote' | 'data' | 'anecdote';
  status: 'candidate' | 'pinned' | 'adopted' | 'consumed' | 'excluded';
  citationAnchor?: Record<string, unknown>;
}

interface VoiceProfile {
  id: string;
  ownerId: string;
  toneSummary: string;
  styleRules: string[];
  antiPatterns: string[];
  sampleRefs: Array<{ objectType: string; objectId: string }>;
  version: number;
}

interface OutputDocument {
  id: string;
  briefId: string;
  outlineVersionId?: string;
  status:
    | 'collecting_materials'
    | 'assembling'
    | 'drafting'
    | 'revising'
    | 'ready_to_publish'
    | 'published'
    | 'superseded';
  filePath: string;
  voiceProfileId?: string;
}
```

除上述对象外，还需要：

- `citation_record`
- `block_provenance`
- `revision_pass`
- `output_variant`
- `publish_job`
- `publish_record`

### 5.2 关键关系语义

建议在统一 `links` 体系中支持：

- `output_document -> research_artifact : derived_from`
- `output_document -> highlight/note/asset : uses`
- `output_document -> voice_profile : styled_by`
- `output_variant -> output_document : adapts`
- `citation_record -> research_source_item/highlight/note : cites`
- `publish_record -> output_variant/output_document : published_from`
- `publish_record -> feedback/research_space : feeds_back_to`
- `revision_pass -> output_document : revises`

### 5.3 核心能力接口

| 接口 | 用途 |
|---|---|
| `writing_workspace.create/update` | 创建写作空间与默认设置 |
| `output_brief.create/refine` | 定义受众、目标、输出类型与限制 |
| `material_board.build/recommend/reorder` | 组装材料板、推荐来源、调整分组 |
| `outline.generate/compare/apply` | 生成和切换大纲版本 |
| `draft.composeSection` | 按章节生成或改写草稿 |
| `voice_profile.learn/simulate/apply` | 学习口吻、预览风格、应用到当前输出 |
| `citation.trace/insert` | 插入引用并查看证据链 |
| `revision.run/list/apply` | 执行不同目标的润色回合 |
| `output_variant.generate` | 生成 newsletter / 社媒 / 演讲稿 / 脚本等变体 |
| `publish.prepare/preview/run` | 预发布检查、平台预览、发布执行 |
| `publish.feedback.ingest` | 把反馈与表现数据回流 |

### 5.4 能力边界

- 写作工作台负责**面向读者的输出编排**，不替代 05 的研究对象管理。
- 它消费 06 的高亮与笔记，但不反向承担全文阅读批注职责。
- 它负责生成与发布准备，但**真正的外部渠道接入与权限控制**应由发布层统一治理。
- 它可以学习用户 voice，但**不能把用户所有私人文本都默认纳入学习语料**。

## 6. 与其他专题的依赖关系

| 依赖专题 | 关系说明 |
|---|---|
| **05 研究工作台** | 05 是写作的上游证据层。07 必须消费研究问题、结论卡、资料池、研究报告，并把写作中暴露出的证据缺口、反例需求、待验证论点回写到 05。没有 05，07 只能做无 grounding 的文案生成。 |
| **06 笔记与高亮系统** | 06 提供高亮、边注、闪念、综合笔记与 Selection Context，是写作材料面板最细颗粒度的输入来源。07 要继承 06 的锚点、摘录、片段级溯源能力。 |
| **09 对象网络与全局关系系统** | 07 中的文稿、变体、voice profile、引用记录、发布记录都必须进入统一对象网络，才能获得 backlinks、全局搜索、跨域召回与长期复用。09 是 07 可追溯、可组合的底座。 |
| **15 MCP 与开放能力层** | 07 的外部导出、平台格式化、封面生成、渠道发布、外部检索补料等能力需通过 15 的受控能力层暴露；但写作对象化、引用入库、发布记录沉淀仍由 Orbit 内部完成。 |
| **16 Agent 编排与记忆系统** | 07 依赖 16 提供 Writing Orchestrator、子 Agent 委派、上下文压缩、记忆注入、工具路由与安全网关，使材料推荐、大纲组装、分章节写作与润色能以多阶段方式执行。 |
| **17 发布与分发系统** | 07 的输出终点不是“保存文档”，而是进入 17 的渠道层完成平台适配、排版、定时、发布记录与失败重试。07 负责内容准备，17 负责真正的渠道执行。 |
| **18 反馈回流与效果分析系统** | 发布后的评论、收藏、打开率、读者反馈与用户手动复盘，需经 18 回流为新的研究问题、写作复盘、voice 调整建议与素材池增量。没有 18，07 仍是单向输出，不形成复利。 |

## 7. 风险、边界与设计决策

1. **明确拒绝“聊天即写作”**：聊天只是一种触发方式，不能替代材料板、大纲、文稿、修订和发布记录这些稳定对象。
2. **坚持“先组装、后生成”**：哪怕用户要求快速起草，也应至少先形成最小材料板与大纲，不直接跳过 grounding。
3. **Voice Profile 必须显式授权与可回滚**：风格学习是强能力，也是高风险能力；必须支持查看语料、删除样本、回滚版本。
4. **引用与改写要同时成立**：允许 AI 改写，但不能因此丢失来源；系统要保留“改写后仍源于哪些材料”的谱系。
5. **一个材料底座对应多种输出，而不是重复建稿**：长文、newsletter、社媒、演讲稿、脚本应作为派生对象管理，避免多份内容彼此漂移。
6. **发布前保留人工确认门槛**：Agent 可准备标题、摘要、封面文案、渠道版本，但不应在默认情况下自动对外发布。
7. **写作工作台是 Orbit 的外化界面**：它连接内部知识沉淀与外部表达世界，因此设计上必须同时满足“对内可追溯、对外可交付”。
