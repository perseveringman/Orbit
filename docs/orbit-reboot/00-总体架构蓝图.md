# 00. 总体架构蓝图

## 1. 为什么要重启

这次重启不是“给当前 Orbit 再叠一层 AI 能力”，而是承认旧项目的产品中心仍然是阅读器与知识工具，而新的目标已经发生了根本变化：要做的是一个**全面拥抱主动性 Agent 的智能助手应用**。在这个新应用中，人生规划、任务管理、阅读、研究、笔记、写作、日志、对象关联都不是并列的小模块，而是一整套由 Agent 编排的个人操作系统。

所以新项目要同时完成两件事：

1. **保留 Orbit 最有价值的东西**：输入、思考、行动是一条河流；本地优先；对象网络；深度阅读与深度工作。
2. **重写它的骨架**：从“一个阅读/知识产品 + AI 功能”重写为“一个 Agent-first 的人生与工作操作系统”。

这也是为什么本轮设计不再受旧项目的单端实现边界束缚，而是直接以完整理想态为目标：桌面端、iOS、Web 三端并存；对象与能力统一可编排；本地内容可携带；Agent 成为整个系统的默认操作者与解释层。

## 2. 产品核心主张

新项目的核心主张可以概括为一句话：

> **用 Agent 把愿景、规划、输入、研究、笔记、输出与回顾编织成一个连续系统，让人把精力重新放回“想成为什么样的人、正在推进什么、为什么这件事值得做”。**

围绕这个主张，系统有六个一级支柱：

1. **愿景与规划层**：愿景是罗盘，不是 KPI；规划是多时间尺度的持续收敛。
2. **执行编排层**：任务系统由 Agent 维护，用户主要做意图表达与关键确认。
3. **输入与理解层**：阅读、转写、翻译、摘录、笔记、高亮共同构成材料转化管线。
4. **研究与输出层**：研究空间与写作工作台形成“证据 → 洞察 → 产物”的产线。
5. **时间与记忆层**：日志、Journal、Review 负责把发生过的事变成下一轮判断的依据。
6. **对象与能力层**：所有能力都围绕统一对象网络与 capability interface 组织，并可被 Agent 和 MCP 消费。

## 3. 统一对象观

新项目的首要架构前提是：**功能不是中心，对象才是中心。**

系统里的愿景、direction、theme、goal、project、milestone、task、article、book、highlight、note、research_space、research_artifact、document、journal_entry、agent_session 等都应视为对象。对象之间通过统一关系网络连接，而不是散落在各自模块的局部表或 UI 状态里。

这样做有三个直接结果：

1. **任何内容都能关联到任何内容**：研究空间可以关联材料、任务、项目、写作产物；高亮可以流向研究、任务、写作与 Journal。
2. **Agent 不需要为每个模块学一套完全不同的世界观**：它只需要理解对象、关系、事件与能力接口。
3. **应用的核心体验从“页面跳转”变成“对象网络中的上下文切换”**：打开一个项目，不是只看到项目自身，而是看到它关联的任务、资料、研究、文档、日志与 Agent 建议。

因此，产品层的中心不再是“阅读器”“任务页”“笔记页”这些传统模块，而是下面这组对象家族：

| 对象家族 | 代表对象 | 作用 |
|---|---|---|
| 方向对象 | `vision` `direction` `theme` `goal` | 定义往哪里走 |
| 执行对象 | `project` `milestone` `task` `commitment` | 定义当前推进什么 |
| 输入对象 | `feed` `article` `book` `podcast` `video` `asset` | 定义读到了什么 |
| 思考对象 | `highlight` `note` `research_question` `research_claim` | 定义看到了什么、想到了什么 |
| 产出对象 | `research_artifact` `document` `post` `deck` | 定义留下了什么 |
| 记忆对象 | `journal_entry` `review` `agent_session` | 定义发生过什么、系统学到了什么 |

## 4. 体验架构：从人生方向到今日动作

从用户体验上看，这个系统是一条由高层方向持续收敛到当下行动的链条：

`Vision → Direction → Theme / Goal → Project / Commitment → Today / Focus → Journal / Review`

这条链不是静态规划树，而是持续由 Agent 重算的“多时间尺度对齐系统”：

1. 用户在首次进入时写下愿景。
2. Agent 帮用户将愿景翻译为年度方向、季度主题与阶段目标。
3. 项目、研究、阅读、写作围绕这些主题持续流动。
4. Today 与 Focus 负责把复杂系统压缩成“今天最值得做什么”。
5. Journal 与 Review 负责把事实再抬回高层，修正下一轮规划。

因此，用户并不需要维护一张永远更新的全局地图。用户只需要在关键节点做决定，而 Agent 负责把上层方向与下层事实接起来。

## 5. 系统分层架构

从实现角度，新项目应采用五层架构：

### 5.1 Experience Layer

面向用户的三端界面：

- `apps/desktop`：Electron 桌面容器
- `apps/web`：浏览器访问入口，最大化复用桌面工作台 UI
- `apps/mobile`：iOS 端，侧重捕捉、轻编辑、轻研究、轻回顾

桌面端与 Web 共享绝大部分工作台级 UI；iOS 端共享核心业务逻辑、对象协议、同步协议、设计 token 与部分 Headless 交互模型，但在编辑、阅读和系统集成上允许有平台差异。

### 5.2 Agent Runtime Layer

系统级 Orchestrator 负责接收用户意图、拉取上下文、调用能力、分派子 Agent、处理审批与压缩上下文。围绕它布置领域子 Agent：

- Planning Agent
- Task / Focus Agent
- Reading Agent
- Research Agent
- Writing Agent
- Journal / Review Agent
- Graph / Linking Agent

这一层借鉴 Hermes 的核心原则：稳定 system prompt、动态 turn context、tool registry、隔离 sub-agent、session lineage、异步学习回路、工具边界安全。

### 5.3 Capability Layer

应用内所有能力都先收敛成统一 capability interface，再通过两种路径暴露：

1. **内部路径**：给 UI 与内建 Agent 使用
2. **外部路径**：通过本地 MCP server 给外部 Agent / 自动化工具使用

这意味着 Orbit 不应让 UI、内部 Agent、外部 MCP 客户端分别调用三套不同 API，而要让它们共享同一能力边界，只是在权限、审计与交互方式上不同。

### 5.4 Domain Data Layer

领域数据采用**类型化对象表 + 全局对象索引 + 统一关系层 + 事件层**的模式：

- 类型表：保留每类对象的专属字段
- `object_index`：统一检索、统一引用、统一列表入口
- `links`：统一表达对象之间的语义关系
- `events`：记录真实发生过的事实

这样既避免“万物一张超级 items 表”的语义塌陷，也避免“每加一种对象就复制一套关联逻辑”的扩展灾难。

### 5.5 Storage & Sync Layer

数据底座采用**文件系统优先 + 元数据数据库 + Local-first 同步**：

- `sources/`：用户与外部内容的真相层
- `wiki/`：AI 编译、沉淀与再组织层
- `.orbit/`：数据库、索引、缓存、规则与内部状态

本地始终是主真相层；云端首先承担同步、备份与跨设备流转职责，而不是默认可读的内容主库。与云模型或外部 MCP 的交互，必须被视为一次可感知的数据出境行为。

## 6. 三条核心流

### 6.1 输入流

RSS、网页、社媒日报、书籍、播客、视频、文件、语音、网页剪藏等都先进入输入流。输入流负责：

- 发现与订阅
- 抓取与转写
- 翻译与解析
- 缓存原始内容
- 建立对象索引
- 允许片段化选择与高亮

### 6.2 思考流

高亮、边注、笔记、研究问题、研究会话、研究结论、Journal 手记构成思考流。思考流的核心不是“记录更多”，而是让片段持续流动：

- 从 Reader 进入 Note
- 从 Note 进入 Research
- 从 Research 进入 Task / Writing
- 从 Journal 与 Review 回到 Planning

### 6.3 输出流

研究空间与写作工作台共同构成输出流。研究负责证据与判断，写作负责面向读者与外部世界的表达，输出结果再回流到：

- 项目进展
- Journal 与 Review
- Agent 对用户长期方向的理解

## 7. 关键架构原则

1. **愿景与规则分离**：愿景是用户写的方向感，directive 才是行为约束。
2. **对象网络优先于模块边界**：所有模块都要能回到统一对象与关系层。
3. **文件是用户资产，数据库是系统智能层**：文件可带走，数据库提供组织与推理能力。
4. **Agent 深度融入，但必须有护栏**：任何高风险写入、发布、外部动作都需显式权限与审计。
5. **研究与写作是一条连续产线**：聊天不是终点，产物才是可复用资产。
6. **日志不是埋点附属品，而是系统记忆基础设施**：Journal、Review、planning 全都建立在事件之上。
7. **MCP 是能力总线，不是数据库直通车**：对外暴露能力，而不是裸露内部对象存储。
8. **三语与设计系统是地基，不是收尾项**：国际化、设计 token、换肤、可访问性要先进入架构。

## 8. 这套蓝图如何阅读

本总纲负责回答“这个新应用到底是什么、为什么这样分层、哪些原则不能动”。

建议阅读路径：

1. 先看 `01 / 02 / 03`，理解愿景、规划与执行的纵向主线。
2. 再看 `04 / 05 / 06 / 07 / 08 / 09`，理解输入、研究、笔记、写作、日志与对象网络的横向流动。
3. 最后看 `10` 到 `18`，理解平台、数据、同步、Agent runtime、MCP、编辑器、三语与设计系统的地基如何支撑整个产品。
