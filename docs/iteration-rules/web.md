# Web 迭代规则

## 职责边界

Web 迭代只解决 `apps/web` 的应用壳层问题：页面容器、浏览器端路由、状态装配、端内体验、浏览器能力接入。任何跨端共用逻辑、共享 UI、平台抽象、服务契约都不应在这里临时发明。

## 可改目录

- `apps/web/**`

## 禁止改目录

- `apps/desktop/**`
- `apps/ios/**`
- `apps/server/**`
- `packages/**`
- 根目录工作区配置

## 迭代节奏

- **接契约**：先读取 core/ui/platform/server 已冻结的接口，只接受已确认的依赖窗口。
- **做壳层**：在 `apps/web` 内完成路由、页面装配、状态接线、浏览器 API 集成。
- **做回归**：验证主要页面、导航、状态恢复、浏览器兼容性与错误提示。
- **做回传**：把缺失的共享能力整理成需求交给对应 packages owner，而不是在 Web 目录里私补。

## 提交前检查

- 确认没有把本应进入 package 的逻辑留在 `apps/web`。
- 确认新增依赖优先来自 workspace 包，不引入重复技术栈。
- 至少完成与改动相关的 build / typecheck / test / 关键链路手测之一或组合。
- 确认环境变量、路由注册、页面入口的变更已同步给接手人。

## 完成定义

- Web 端在既定共享契约上可运行或可演示关键流程。
- 浏览器专属逻辑全部收口在 app 层。
- 缺少的共享能力已向上游提明确诉求，不留私有复制代码。
- 下游能从交付说明直接继续联调。

## 与上下游协作方式

上游主要是 `packages-core`、`packages-ui`、`packages-platform`、`packages-server`；跨包冲突由 `monorepo-lead` 处理。Web 侧只消费公共入口，不引用共享包内部文件；如果发现 contract 不足，先提变更请求，等待上游冻结后再接入。
