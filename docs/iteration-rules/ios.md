# iOS 迭代规则

## 职责边界

iOS 迭代只处理 `apps/ios` 的移动宿主、原生桥接、权限、生命周期和移动端体验适配。iOS 目录的职责是消费共享能力并落地到移动平台，而不是定义新的共享协议。

## 可改目录

- `apps/ios/**`

## 禁止改目录

- `apps/web/**`
- `apps/desktop/**`
- `apps/server/**`
- `packages/**`
- 根目录工作区配置

## 迭代节奏

- **确认移动差异**：先识别权限、前后台切换、网络波动、手势导航等约束。
- **桥接接入**：在 `apps/ios` 内完成原生能力与共享层的桥接。
- **移动回归**：验证安装、启动、权限、前后台切换、关键业务流程。
- **回传能力缺口**：把需要上游补充的 contract、UI 组件或服务协议整理清楚。

## 提交前检查

- 确认原生桥和权限声明收口在 iOS 目录或对应平台包。
- 确认没有为了移动端需求去改其他 app 壳层。
- 至少完成一次移动关键路径验证；无法验证的项要明确列出。
- 确认共享依赖都通过公共入口接入，而不是内部文件。

## 完成定义

- iOS 宿主接入完整，关键生命周期和权限路径可说明。
- 移动端差异没有倒灌到 core / ui / server 共享层。
- 联调方可以基于交付说明继续推进。
- 未完成验证的风险和责任方已明确。

## 与上下游协作方式

上游主要是 `packages-core`、`packages-ui`、`packages-platform`，涉及接口或同步时还要和 `packages-server` 对齐。遇到必须抽象为共享能力的移动诉求，应先升级给相应 package owner，再等待冻结接口。
