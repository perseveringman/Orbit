# Desktop 迭代规则

## 职责边界

Desktop 迭代只处理 `apps/desktop` 的宿主层、桌面工作流和桌面端运行时装配，包括窗口、菜单、本地文件、桌面权限与进程通信等。共享能力应沉淀在 package 层，不能在桌面 app 中单独养一份长期分叉。

## 可改目录

- `apps/desktop/**`

## 禁止改目录

- `apps/web/**`
- `apps/ios/**`
- `apps/server/**`
- `packages/**`
- 根目录工作区配置

## 迭代节奏

- **界定宿主职责**：先明确哪些逻辑必须是桌面宿主，哪些应回到共享层。
- **宿主装配**：在 `apps/desktop` 内完成窗口、菜单、文件系统、IPC 或桥接接线。
- **端内回归**：验证启动、主流程、异常恢复、配置加载等关键桌面路径。
- **同步约束**：把桌面独有限制、配置项、权限要求回传给总控与下游。

## 提交前检查

- 确认桌面专属能力没有被硬塞到 core / ui 包。
- 确认没有直接依赖其他 app 的文件来复用逻辑。
- 完成最小可运行验证，例如启动应用、打开关键窗口、执行核心操作。
- 确认任何新增平台能力都已走 platform contract 或经 monorepo-lead 批准留在 app。

## 完成定义

- 桌面端关键宿主链路可运行，边界清楚。
- 宿主层代码与共享业务层分离，不靠复制共享逻辑推进。
- 桌面端需要的环境、配置、桥接方式有可执行说明。
- 已知限制和未验证风险已显式记录。

## 与上下游协作方式

上游主要是 `packages-core`、`packages-ui`、`packages-platform`，涉及服务接口时再与 `packages-server` 对齐。Desktop 侧应在 contract 稳定后接入；如果某项桌面特性会影响 workspace 或构建链路，必须升级给 `monorepo-lead`。
