# Server 迭代规则

## 职责边界

Server 迭代只处理 `apps/server` 的服务入口、配置、部署拼装、路由注册、中间件 wiring 和运行时装配。复用型服务能力、API 类型、数据库 schema 与基础设施包归共享层维护。

## 可改目录

- `apps/server/**`

## 禁止改目录

- `apps/web/**`
- `apps/desktop/**`
- `apps/ios/**`
- `packages/**`
- 根目录工作区配置

## 迭代节奏

- **读取共享协议**：先锁定 API 类型、schema、infra contract，再动服务入口。
- **装配入口**：在 `apps/server` 内完成配置、中间件、路由、依赖注入与启动逻辑。
- **运行验证**：至少做一次服务启动与关键接口 smoke 检查。
- **通知调用方**：同步环境变量、路由变化、错误码变化和联调窗口。

## 提交前检查

- 确认业务规则没有堆积在 app/server 壳层。
- 确认配置差异（本地/测试/生产）有明确落点。
- 确认 API 变化已同步到消费方，不让客户端靠报错反推。
- 确认未直接修改客户端 app 来硬配合服务变更。

## 完成定义

- 服务入口、部署拼装和运行时接线完整可解释。
- 关键接口 smoke 或等价验证已完成。
- 消费方接力信息充分，知道该按什么版本和配置联调。
- 回退方式和兼容窗口清楚。

## 与上下游协作方式

上游主要是 `packages-server` 和 `packages-core`；下游是 Web/Desktop/iOS 等调用方。Server 侧必须建立在共享协议已冻结的前提上，不能一边改入口一边改协议；如涉及多个消费端 breaking change，由 `monorepo-lead` 统一排期。
