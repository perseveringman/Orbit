# desktop-builder 职责卡

## 职责边界

负责 `apps/desktop` 的桌面宿主、窗口与菜单装配、桌面端权限/文件系统/本地能力接入，以及桌面用户流程的闭环验证。它只拥有桌面壳层，不拥有跨端领域逻辑、共享 UI 资产或平台 contract。

## 可改目录

- `apps/desktop/**`

## 禁止改目录

- `apps/web/**`
- `apps/ios/**`
- `apps/server/**`
- `packages/**`（共享层诉求交给对应 packages builder）
- 根目录治理配置

## 迭代节奏

- **阶段 1：确认宿主边界** —— 明确哪些能力必须留在桌面宿主，哪些应继续走共享包。
- **阶段 2：接线实现** —— 完成窗口生命周期、桌面 API、菜单/快捷键/本地文件等装配。
- **阶段 3：桌面回归** —— 验证启动、主要流程、崩溃恢复、配置读取等关键链路。
- **阶段 4：输出接力信息** —— 将桌面端专属约束、配置项和已知风险反馈给总控与下游。

## 提交前检查

- 确认桌面专属逻辑没有被塞进 core / ui / platform contract 之外的共享目录。
- 确认 Electron/Tauri 等宿主行为只在 `apps/desktop` 或平台包中出现。
- 完成桌面端最小可运行验证，如启动、主窗口打开、关键交互链路走通。
- 若依赖平台包变更，确认引用的是已冻结的 contract，而不是 app 内私有路径。

## 完成定义

- 桌面端专属能力可运行且边界清楚。
- 宿主层与共享业务层分离，没有长期临时桥接代码。
- 桌面构建/启动/关键流程验证已完成并可复述。
- 需要共享层配合的问题已沉淀为明确接口诉求。

## 与上下游协作方式

上游主要是 `packages-core-builder`、`packages-ui-builder`、`packages-platform-builder`，涉及服务通信时还要和 `packages-server-builder` 对齐协议。与上游协作时先拿到 contract，再补宿主接线；如果桌面特性会影响根构建链路，必须升级给 `monorepo-lead`。
