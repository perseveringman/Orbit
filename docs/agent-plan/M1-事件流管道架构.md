## 1. 事件类型体系设计

### 1.1 设计原则

事件体系遵循三条核心原则：

1. **Discriminated Union**：每个事件携带字面量 `type` 字段，TypeScript 编译器可在 `switch` 中自动窄化类型
2. **层级包含**：高层事件（`OrchestratorEvent`）是低层事件（`AgentEvent`、`CapabilityEvent`）的超集
3. **时间戳+因果链**：每个事件携带 `timestamp` 和 `runId`，可重建完整执行时间线

### 1.2 基础事件接口

所有事件共享一个基础形状，定义于 `events.ts`：

```typescript
/** 所有事件的公共字段 */
export interface BaseEvent {
  readonly type: string;
  readonly runId: string;
  readonly timestamp: string;
}
```

### 1.3 Capability 层事件（最内层）

Capability 层事件描述单个工具/能力调用的生命周期。由 `ToolRegistry.dispatch()` 或未来的 `CapabilityExecutor.run()` 产生。

```typescript
export interface CapabilityStartedEvent extends BaseEvent {
  readonly type: 'capability:started';
  readonly capabilityName: string;
  readonly args: Record<string, unknown>;
}

export interface CapabilityProgressEvent extends BaseEvent {
  readonly type: 'capability:progress';
  readonly capabilityName: string;
  readonly message: string;
  readonly progress?: number; // 0-1 可选进度百分比
}

export interface CapabilityCompletedEvent extends BaseEvent {
  readonly type: 'capability:completed';
  readonly capabilityName: string;
  readonly output: string;
  readonly durationMs: number;
}

export interface CapabilityErrorEvent extends BaseEvent {
  readonly type: 'capability:error';
  readonly capabilityName: string;
  readonly error: string;
  readonly durationMs: number;
}

export type CapabilityEvent =
  | CapabilityStartedEvent
  | CapabilityProgressEvent
  | CapabilityCompletedEvent
  | CapabilityErrorEvent;
```

**设计决策**：`CapabilityProgressEvent` 的 `progress` 字段为可选——大多数工具无法提供精确进度，但文件导入/批处理等长时间操作可以利用此字段驱动前端进度条。

### 1.4 Safety 层事件

Safety 事件由 `SafetyGate` 在检查能力调用时产生。它们不属于 Capability 层（因为发生在调用之前），而是作为独立事件注入 Agent 层事件流。

```typescript
export interface SafetyCheckPassedEvent extends BaseEvent {
  readonly type: 'safety:check-passed';
  readonly capabilityName: string;
  readonly tier: ApprovalPolicy;
}

export interface SafetyApprovalRequiredEvent extends BaseEvent {
  readonly type: 'safety:approval-required';
  readonly capabilityName: string;
  readonly approval: ApprovalRequest;
}

export interface SafetyBlockedEvent extends BaseEvent {
  readonly type: 'safety:blocked';
  readonly capabilityName: string;
  readonly reason: string;
  readonly threats: readonly string[];
}

export type SafetyEvent =
  | SafetyCheckPassedEvent
  | SafetyApprovalRequiredEvent
  | SafetyBlockedEvent;
```

### 1.5 Compression 层事件

上下文压缩是一个可能耗时的操作（涉及 LLM 调用生成摘要），需要通知前端。

```typescript
export interface CompressionStartedEvent extends BaseEvent {
  readonly type: 'compression:started';
  readonly originalTokens: number;
  readonly threshold: number;
}

export interface CompressionCompletedEvent extends BaseEvent {
  readonly type: 'compression:completed';
  readonly originalCount: number;
  readonly compressedCount: number;
  readonly tokensSaved: number;
}

export type CompressionEvent =
  | CompressionStartedEvent
  | CompressionCompletedEvent;
```

### 1.6 Agent 层事件（中间层）

Agent 层事件描述单个领域 Agent（如 Planning、Reading、Research）的执行生命周期。它是 Capability 事件的超集。

```typescript
export interface AgentStartedEvent extends BaseEvent {
  readonly type: 'agent:started';
  readonly domain: AgentDomain;
  readonly model: string;
}

export interface AgentReasoningEvent extends BaseEvent {
  readonly type: 'agent:reasoning';

}

export interface AgentStreamDeltaEvent extends BaseEvent {
  readonly type: 'agent:stream-delta';
  readonly domain: AgentDomain;
  readonly delta: string;
}

export interface AgentToolCallEvent extends BaseEvent {
  readonly type: 'agent:tool-call';
  readonly domain: AgentDomain;
  readonly toolName: string;
  readonly toolArgs: string;
}

export interface AgentIterationEvent extends BaseEvent {
  readonly type: 'agent:iteration';
  readonly domain: AgentDomain;
  readonly iteration: number;
  readonly maxIterations: number;
}

export interface AgentCompletedEvent extends BaseEvent {
  readonly type: 'agent:completed';
  readonly domain: AgentDomain;
  readonly responseMessage: AgentMessage;
  readonly totalTokenUsage: TokenUsage;
  readonly steps: readonly AgentStep[];
}

export interface AgentErrorEvent extends BaseEvent {
  readonly type: 'agent:error';
  readonly domain: AgentDomain;
  readonly error: string;
}

export type AgentEvent =
  | AgentStartedEvent
  | AgentReasoningEvent
  | AgentStreamDeltaEvent
  | AgentToolCallEvent
  | AgentIterationEvent
  | AgentCompletedEvent
  | AgentErrorEvent
  | CapabilityEvent
  | SafetyEvent
  | CompressionEvent;
```

**关键设计**：`AgentEvent` 的联合类型**包含**了 `CapabilityEvent`、`SafetyEvent` 和 `CompressionEvent`。这意味着 Agent 层的 `AsyncGenerator<AgentEvent>` 可以直接 `yield` 底层事件，消费者通过 `switch(event.type)` 统一处理所有层级的事件。

### 1.7 Orchestrator 层事件（最外层）

### 1.8 事件层级关系总览

```
OrchestratorEvent（最外层，前端订阅此类型）
├── orchestrator:started / routed / delegated / completed / error / cancelled
└── AgentEvent（中间层）
    ├── agent:started / reasoning / stream-delta / tool-call / iteration / completed / error
    ├── CapabilityEvent（最内层）
    │   └── capability:started / progress / completed / error
    ├── SafetyEvent
    │   └── safety:check-passed / approval-required / blocked
    └── CompressionEvent
        └── compression:started / completed
```

**事件总数**：6（Orchestrator）+ 7（Agent）+ 4（Capability）+ 3（Safety）+ 2（Compression）= **22 种事件类型**

## 2. AsyncGenerator 管道架构

### 2.1 核心设计：三层 yield 管道

Orbit 的事件流管道由三层 `AsyncGenerator` 嵌套构成，数据通过 `yield` 向上冒泡，通过 `yield*` 委托向下穿透：

```
Orchestrator.executeStream()         → AsyncGenerator<OrchestratorEvent>
  ├── yield orchestrator:started
  ├── yield orchestrator:routed
  ├── yield* agentLoop()             → AsyncGenerator<AgentEvent>
  │     ├── yield agent:started
  │     ├── [LLM call]
  │     ├── yield agent:reasoning
  │     ├── yield agent:tool-call
  │     ├── yield* dispatchCapability()  → AsyncGenerator<CapabilityEvent>
  │     │     ├── yield capability:started
  │     │     ├── yield capability:progress  (可选)
  │     │     └── yield capability:completed | capability:error
  │     ├── yield agent:iteration
  │     └── yield agent:completed
  └── yield orchestrator:completed
```

`yield*` 的语义是**完全委托**——将内层生成器的所有事件透传给外层消费者，同时内层生成器的 `return` 值作为 `yield*` 表达式的结果。这使得外层可以获取内层的汇总结果（如工具调用结果、Agent 最终消息），同时所有中间事件自动冒泡。

### 2.2 ExecutionContext：跨层共享上下文

为了在三层管道间传递共享状态（AbortSignal、运行 ID、累积 Token 计数），定义一个不可变的执行上下文：

```typescript
/** 跨层传递的执行上下文 */
export interface ExecutionContext {
  readonly runId: string;
  readonly sessionId: string;
  readonly surface: AgentSurface;
  readonly signal: AbortSignal;
}
```

此上下文在 `Orchestrator.executeStream()` 入口创建，向下传递给 `agentLoop()` 和 `dispatchCapability()`。每个函数只读使用，不修改上下文。

### 2.3 Orchestrator.executeStream() 完整实现

```typescript
class Orchestrator {
  // ... 现有字段不变 ...

  /**
   * 流式执行：返回 AsyncGenerator，逐步 yield 事件。
   * 这是新的核心入口，execute() 将基于此实现。
   */
  async *executeStream(
    input: OrchestratorInput,
    options?: { signal?: AbortSignal },
  ): AsyncGenerator<OrchestratorEvent, OrchestratorOutput> {
    const signal = options?.signal ?? new AbortController().signal;
    const runId = generateId('run');

    const ctx: ExecutionContext = {
      runId,
      sessionId: input.session.id,
      surface: input.session.surface,
      signal,
    };

    // ---- 阶段 1：启动 ----
    yield {
      type: 'orchestrator:started',
      runId,
      sessionId: input.session.id,
      surface: input.session.surface,
      timestamp: new Date().toISOString(),
    };

    throwIfAborted(signal);

    // ---- 阶段 2：意图路由 ----
    const domain = this.routeIntent(input.userMessage, input.session.surface);

    yield {
      type: 'orchestrator:routed',
      runId,
      sessionId: input.session.id,
      domain,
      reason: `Routed to ${domain} based on surface "${input.session.surface}" and message content`,
      timestamp: new Date().toISOString(),
    };

    throwIfAborted(signal);

    // ---- 阶段 3：上下文装配（可能触发压缩）----
    const contextMessages = yield* this.assembleContextStream(
      input.session, ctx,
    );

    const memoryBlock = this.memory.buildContextBlock(input.availableContext);

    // ---- 阶段 4：Agent 循环（核心管道委托）----
    const agentResult = yield* this.agentLoop(
      domain, contextMessages, memoryBlock, input, ctx,
    );

    throwIfAborted(signal);

    // ---- 阶段 5：构建输出 ----
    const output: OrchestratorOutput = {
      run: agentResult.run,
      responseMessage: agentResult.responseMessage,
      objectMutations: agentResult.objectMutations,
      pendingApprovals: agentResult.pendingApprovals,
    };

    yield {
      type: 'orchestrator:completed',
      runId,
      sessionId: input.session.id,
      output,
      timestamp: new Date().toISOString(),
    };

    return output;
  }
}
```

### 2.4 assembleContextStream() 实现

压缩过程现在也是流式的，产生 `CompressionEvent`：

```typescript
private async *assembleContextStream(
  session: AgentSession,
  ctx: ExecutionContext,
): AsyncGenerator<CompressionEvent, readonly AgentMessage[]> {
  let messages = [...session.messages];

  if (this.compressor.shouldCompress(messages, this.config.compressionThreshold)) {
    const estimatedTokens = messages.reduce(
      (sum, m) => sum + this.compressor.estimateTokens(m.content), 0,
    );

    yield {
      type: 'compression:started',
      runId: ctx.runId,
      originalTokens: estimatedTokens,
      threshold: this.config.compressionThreshold,
      timestamp: new Date().toISOString(),
    };

    const result = await this.compressor.compress(
      messages,
      this.config.compressionThreshold,
      async (text) => `Summary: ${text.slice(0, 200)}`,
    );
    messages = [...result.compressedMessages];

    yield {
      type: 'compression:completed',
      runId: ctx.runId,
      originalCount: result.originalCount,
      compressedCount: result.compressedCount,
      tokensSaved: result.estimatedTokensSaved,
      timestamp: new Date().toISOString(),
    };
  }

  return messages;
}
```

### 2.5 agentLoop() 实现

这是核心循环的流式版本，对应当前 `execute()` 中的 `while` 循环：

```typescript
interface AgentLoopResult {
  readonly run: AgentRun;
  readonly responseMessage: AgentMessage;
  readonly objectMutations: readonly string[];
  readonly pendingApprovals: readonly ApprovalRequest[];
}

private async *agentLoop(
  domain: AgentDomain,
  contextMessages: readonly AgentMessage[],
  memoryBlock: string,
  input: OrchestratorInput,
  ctx: ExecutionContext,
): AsyncGenerator<AgentEvent, AgentLoopResult> {
  const domainConfig = DOMAIN_AGENT_CONFIGS[domain];
  const availableTools = this.registry.getDefinitions({
    domain,
    surface: input.session.surface,
  });

  const steps: AgentStep[] = [];
  const pendingApprovals: ApprovalRequest[] = [];
  const objectMutations: string[] = [];
  const accUsage = { promptTokens: 0, completionTokens: 0, totalTokens: 0 };

  const model = domainConfig?.model ?? this.config.defaultModel;

  // 构建对话上下文
  const systemMsg: AgentMessage = {
    id: generateId('msg'),
    role: 'system',
    content:
      (domainConfig?.systemPrompt ?? '') +
      (memoryBlock ? `\n\n${memoryBlock}` : ''),
    timestamp: new Date().toISOString(),
  };

  const userMsg: AgentMessage = {
    id: generateId('msg'),
    role: 'user',
    content: input.userMessage,
    timestamp: new Date().toISOString(),
  };

  const conversation: AgentMessage[] = [
    systemMsg,
    ...contextMessages.filter((m) => m.role !== 'system'),
    userMsg,
  ];

  yield {
    type: 'agent:started',
    runId: ctx.runId,
    domain,
    model,
    timestamp: new Date().toISOString(),
  };

  let iterations = 0;
  let lastAssistantMessage: AgentMessage = userMsg;
  const maxIterations = domainConfig?.maxIterations ?? this.config.maxIterations;

  while (iterations < maxIterations) {
    throwIfAborted(ctx.signal);

    iterations++;

    yield {
      type: 'agent:iteration',
      runId: ctx.runId,
      domain,
      iteration: iterations,
      maxIterations,
      timestamp: new Date().toISOString(),
    };

    // LLM 调用
    const response = await this.llm.chatCompletion({
      model,
      messages: conversation,
      tools: availableTools.length > 0 ? availableTools : undefined,
    });

    accUsage.promptTokens += response.usage.promptTokens;
    accUsage.completionTokens += response.usage.completionTokens;
    accUsage.totalTokens += response.usage.totalTokens;

    const choice = response.choices[0];
    if (!choice) break;

    lastAssistantMessage = choice.message;
    conversation.push(lastAssistantMessage);

    // 推理事件
    if (choice.message.content) {
      steps.push({
        id: generateId('step'),
        runId: ctx.runId,
        kind: 'reasoning',
        content: choice.message.content,
        timestamp: new Date().toISOString(),
      });

      yield {
        type: 'agent:reasoning',
        runId: ctx.runId,
        domain,
        content: choice.message.content,
        tokenUsage: response.usage,
        timestamp: new Date().toISOString(),
      };
    }

    // 无工具调用 → 结束
    if (choice.finishReason !== 'tool_calls' || !choice.message.toolCalls?.length) {
      break;
    }

    // 处理工具调用
    for (const tc of choice.message.toolCalls) {
      throwIfAborted(ctx.signal);

      yield {
        type: 'agent:tool-call',
        runId: ctx.runId,
        domain,
        toolName: tc.name,
        toolArgs: tc.arguments,
        timestamp: new Date().toISOString(),
      };

      // Safety check（yield SafetyEvent）
      const toolDef = availableTools.find((t) => t.name === tc.name);
      if (toolDef) {
        const check = this.safety.checkCapability(toolDef, {
          surface: input.session.surface,
          scopeLimit: toolDef.scopeLimit,
        });

        if (!check.allowed && check.requiresApproval) {
          const approval: ApprovalRequest = {
            id: generateId('apr'),
            runId: ctx.runId,
            capabilityName: tc.name,
            riskLevel: toolDef.riskLevel,
            policy: check.tier,
            reason: check.reason ?? 'Requires approval',
            impactSummary: `Tool "${tc.name}" on ${input.session.surface}`,
            status: 'pending',
            createdAt: new Date().toISOString(),
          };
          pendingApprovals.push(approval);

          yield {
            type: 'safety:approval-required',
            runId: ctx.runId,
            capabilityName: tc.name,
            approval,
            timestamp: new Date().toISOString(),
          };

          steps.push({
            id: generateId('step'),
            runId: ctx.runId,
            kind: 'approval-wait',
            content: `Awaiting approval for ${tc.name}`,
            toolName: tc.name,
            approvalRequestId: approval.id,
            timestamp: new Date().toISOString(),
          });
          continue;
        }

        if (!check.allowed) {
          yield {
            type: 'safety:blocked',
            runId: ctx.runId,
            capabilityName: tc.name,
            reason: check.reason ?? 'Not allowed',
            threats: check.threats,
            timestamp: new Date().toISOString(),
          };

          const errorMsg: AgentMessage = {
            id: generateId('msg'),
            role: 'tool',
            content: `Error: ${check.reason ?? 'Not allowed'}`,
            toolCallId: tc.id,
            timestamp: new Date().toISOString(),
          };
          conversation.push(errorMsg);
          continue;
        }

        yield {
          type: 'safety:check-passed',
          runId: ctx.runId,
          capabilityName: tc.name,
          tier: check.tier,
          timestamp: new Date().toISOString(),
        };
      }

      // Dispatch capability（yield* 委托 CapabilityEvent）
      steps.push({
        id: generateId('step'),
        runId: ctx.runId,
        kind: 'tool-call',
        content: `Calling ${tc.name}`,
        toolName: tc.name,
        toolArgs: tc.arguments,
        timestamp: new Date().toISOString(),
      });

      let args: Record<string, unknown>;
      try {
        args = JSON.parse(tc.arguments) as Record<string, unknown>;
      } catch {
        args = {};
      }

      const result = yield* this.dispatchCapability(
        tc.name, args, ctx,
      );

      steps.push({
        id: generateId('step'),
        runId: ctx.runId,
        kind: 'tool-result',
        content: result.success ? result.output : (result.error ?? 'Unknown error'),
        toolName: tc.name,
        toolResult: result.output,
        timestamp: new Date().toISOString(),
      });

      const toolResultMsg: AgentMessage = {
        id: generateId('msg'),
        role: 'tool',
        content: result.success ? result.output : `Error: ${result.error}`,
        toolCallId: tc.id,
        timestamp: new Date().toISOString(),
      };
      conversation.push(toolResultMsg);
    }
  }

  const totalUsage: TokenUsage = { ...accUsage };

  yield {
    type: 'agent:completed',
    runId: ctx.runId,
    domain,
    responseMessage: lastAssistantMessage,
    totalTokenUsage: totalUsage,
    steps,
    timestamp: new Date().toISOString(),
  };

  const run: AgentRun = {
    id: ctx.runId,
    sessionId: ctx.sessionId,
    agentDomain: domain,
    model,
    status: pendingApprovals.length > 0 ? 'awaiting-approval' : 'completed',
    tokenUsage: totalUsage,
    steps,
    createdAt: new Date().toISOString(),
    completedAt: pendingApprovals.length > 0 ? undefined : new Date().toISOString(),
  };

  return { run, responseMessage: lastAssistantMessage, objectMutations, pendingApprovals };
}
```

### 2.6 dispatchCapability() 实现

将 `ToolRegistry.dispatch()` 包装为 `AsyncGenerator<CapabilityEvent, ToolResult>`：

```typescript
private async *dispatchCapability(
  name: string,
  args: Record<string, unknown>,
  ctx: ExecutionContext,
): AsyncGenerator<CapabilityEvent, ToolResult> {
  yield {
    type: 'capability:started',
    runId: ctx.runId,
    capabilityName: name,
    args,
    timestamp: new Date().toISOString(),
  };

  const start = Date.now();

  try {
    throwIfAborted(ctx.signal);

    const result = await this.registry.dispatch(name, args);
    const durationMs = Date.now() - start;

    if (result.success) {
      yield {
        type: 'capability:completed',
        runId: ctx.runId,
        capabilityName: name,
        output: result.output,
        durationMs,
        timestamp: new Date().toISOString(),
      };
    } else {
      yield {
        type: 'capability:error',
        runId: ctx.runId,
        capabilityName: name,
        error: result.error ?? 'Unknown error',
        durationMs,
        timestamp: new Date().toISOString(),
      };
    }

    return result;
  } catch (err: unknown) {
    const durationMs = Date.now() - start;
    const message = err instanceof Error ? err.message : String(err);

    yield {
      type: 'capability:error',
      runId: ctx.runId,
      capabilityName: name,
      error: message,
      durationMs,
      timestamp: new Date().toISOString(),
    };

    return {
      toolName: name,
      success: false,
      output: '',
      error: message,
      durationMs,
    };
  }
}
```

### 2.7 yield 与 yield* 的数据流方向

```
消费者（前端/审计系统）
    ↑ for await (const event of orchestrator.executeStream(input))
    │
    │  yield OrchestratorEvent       ← Orchestrator 自身事件
    │  yield* agentLoop()            ← 委托：Agent 层所有事件透传
    │    │
    │    │  yield AgentEvent          ← Agent 自身事件
    │    │  yield SafetyEvent         ← Safety 检查事件
    │    │  yield* dispatchCapability() ← 委托：Capability 层事件透传
    │    │    │
    │    │    │  yield CapabilityEvent  ← Capability 自身事件
    │    │    └──return ToolResult     ← 返回值（不是事件，被 yield* 捕获）
    │    │
    │    └──return AgentLoopResult    ← 返回值（不是事件，被 yield* 捕获）
    │
    └──return OrchestratorOutput     ← 最终返回值
```

**要点**：
- `yield` 向消费者推送单个事件
- `yield*` 将内层生成器的所有 `yield` 透传给消费者，同时捕获 `return` 值
- 消费者通过 `for await...of` 接收所有层级的事件，但不接收 `return` 值
- `return` 值仅在层间传递，用于汇总构建最终输出

### 2.8 背压控制策略

AsyncGenerator 自带天然的背压控制——`yield` 会暂停生成器直到消费者调用 `.next()`。这意味着：

1. **前端慢消费不会导致事件堆积**：如果前端处理 UI 渲染较慢，`yield` 自动暂停 Orchestrator
2. **LLM 调用不受影响**：LLM 调用是 `await`，不是 `yield`，所以 LLM 响应时间不受背压影响
3. **批量消费模式**：`execute()` 通过 `for await...of` 快速消费所有事件，仅保留最终结果

对于需要缓冲的场景（如 WebSocket 推送），在消费端增加缓冲队列即可：

```typescript
// 消费端缓冲示例（不在 agent-core 中实现，仅供参考）
async function bufferEvents(
  gen: AsyncGenerator<OrchestratorEvent>,
  bufferSize = 100,
): AsyncGenerator<OrchestratorEvent> {
  const buffer: OrchestratorEvent[] = [];
  for await (const event of gen) {
    buffer.push(event);
    if (buffer.length >= bufferSize) {
      for (const e of buffer.splice(0)) yield e;
    }
  }
  for (const e of buffer) yield e;
}
```

## 3. AbortController 取消传播

### 3.1 设计原则

取消传播使用 Web 标准的 `AbortController` / `AbortSignal`，而非自定义标志位。优势：

1. **标准 API**：Node.js 16+ 和所有现代浏览器原生支持
2. **可组合**：`AbortSignal.any()` 可组合多个信号源（Node 20+，Orbit 可 polyfill）
3. **Promise 集成**：`fetch(url, { signal })` 等异步 API 原生支持
4. **一次性语义**：信号一旦触发即不可逆，避免竞态条件

### 3.2 信号穿透三层管道

```typescript
// 顶层：用户触发取消
const controller = new AbortController();
const eventStream = orchestrator.executeStream(input, {
  signal: controller.signal,
});

// 用户点击取消按钮
cancelButton.onclick = () => controller.abort('User cancelled');

// 信号穿透路径：
// controller.signal
//   → ExecutionContext.signal（在 executeStream 入口创建 ctx）
//     → agentLoop(... ctx)（每次迭代前检查）
//       → dispatchCapability(... ctx)（调用工具前检查）
```

### 3.3 throwIfAborted 辅助函数

```typescript
/**
 * 检查 AbortSignal，若已取消则抛出 AbortError。
 * 放置于每个异步边界前。
 */
export function throwIfAborted(signal: AbortSignal): void {
  if (signal.aborted) {
    throw new DOMException(
      signal.reason ?? 'Operation aborted',
      'AbortError',
    );
  }
}
```

### 3.4 各层的取消检查点

| 层级 | 检查点位置 | 触发时机 |
|------|-----------|----------|
| Orchestrator | `executeStream()` 入口 | 路由前 |
| Orchestrator | `assembleContextStream()` 后 | 压缩后、Agent 循环前 |
| Orchestrator | `agentLoop()` 返回后 | 构建输出前 |
| Agent Loop | `while` 循环头部 | 每次迭代开始 |
| Agent Loop | 每个 `toolCall` 处理前 | 工具调用前 |
| Capability | `dispatchCapability()` 入口 | 实际 dispatch 前 |

### 3.5 取消时的错误处理与清理

```typescript
async *executeStream(
  input: OrchestratorInput,
  options?: { signal?: AbortSignal },
): AsyncGenerator<OrchestratorEvent, OrchestratorOutput> {
  // ... 正常流程 ...

  try {
    // 核心执行逻辑
    const agentResult = yield* this.agentLoop(domain, contextMessages, memoryBlock, input, ctx);
    // ...
  } catch (err: unknown) {
    if (err instanceof DOMException && err.name === 'AbortError') {
      yield {
        type: 'orchestrator:cancelled',
        runId,
        sessionId: input.session.id,
        reason: err.message,
        timestamp: new Date().toISOString(),
      };
      // 构建部分结果以保持谱系可追溯
      return this.buildPartialOutput(runId, input, 'cancelled');
    }
    // 非取消错误，正常抛出
    yield {
      type: 'orchestrator:error',
      runId,
      sessionId: input.session.id,
      error: err instanceof Error ? err.message : String(err),
      timestamp: new Date().toISOString(),
    };
    throw err;
  }
}
```

### 3.6 审批等待中的取消

当 Agent 循环遇到需要审批的工具调用时，当前设计会将审批请求加入 `pendingApprovals` 并 `continue`（跳过该工具调用）。在流式模式下，审批等待的处理方式是：

1. `yield` `safety:approval-required` 事件通知前端
2. 继续处理同一轮中的其他工具调用
3. 如果所有工具调用都需要审批，当前迭代无新的 `tool-result` 推入对话，LLM 在下次迭代中会感知到缺失的结果
4. 审批在当前 `executeStream()` 生命周期内**不阻塞等待**——run 以 `awaiting-approval` 状态完成
5. 审批通过后，外层调用新的 `executeStream()` 并通过 Session Lineage（`resumed_from_job`）恢复上下文

如果用户在审批等待期间取消：

```typescript
// 外层伪代码
const controller = new AbortController();
const stream = orchestrator.executeStream(input, { signal: controller.signal });

for await (const event of stream) {
  if (event.type === 'safety:approval-required') {
    showApprovalDialog(event.approval);
  }
  // 用户点取消 → controller.abort()
  // → 下一次 yield 前 throwIfAborted 抛出
  // → catch 块 yield orchestrator:cancelled
}
```

### 3.7 子 Agent 委托中的取消

`delegate()` 方法的流式版本将复用同一个 `AbortSignal`：

```typescript
async *delegateStream(
  parentRun: AgentRun,
  domain: AgentDomain,
  task: string,
  ctx: ExecutionContext,
): AsyncGenerator<OrchestratorEvent, AgentRun> {
  yield {
    type: 'orchestrator:delegated',
    runId: ctx.runId,
    sessionId: ctx.sessionId,
    parentRunId: parentRun.id,
    childDomain: domain,
    task,
    timestamp: new Date().toISOString(),
  };

  const delegateSession: AgentSession = {
    id: generateId('ses'),
    workspaceId: '',
    surface: 'global-chat',
    anchorObjectIds: [],
    lineage: [{ type: 'delegated_from', sourceId: parentRun.id }],
    status: 'active',
    messages: [],
    createdAt: new Date().toISOString(),
    updatedAt: new Date().toISOString(),
  };

  // 复用同一个 signal → 父级取消自动传播到子级

    { session: delegateSession, userMessage: task, availableContext: [] },
    { signal: ctx.signal },

}
```

取消信号**自动级联**：父级 `AbortController.abort()` → 同一个 `signal` 在子级 `agentLoop()` 的下一个 `throwIfAborted()` 检查点触发 → 子级抛出 `AbortError` → 父级 `yield*` 捕获异常 → 父级也以取消状态结束。

## 4. 向后兼容策略

### 4.1 保留 execute() 方法

`execute()` 成为 `executeStream()` 的同步包装器。它消费完整事件流，仅返回最终 `OrchestratorOutput`：

```typescript
/**
 * 同步式执行（向后兼容）。
 * 内部消费 executeStream()，丢弃中间事件，返回最终结果。
 */
async execute(input: OrchestratorInput): Promise<OrchestratorOutput> {
  const stream = this.executeStream(input);
  let lastOutput: OrchestratorOutput | undefined;

  for await (const event of stream) {
    // 从 orchestrator:completed 事件中提取最终输出
    if (event.type === 'orchestrator:completed') {
      lastOutput = event.output;
    }
  }

  // 如果流正常完成但没有 completed 事件（不应发生），
  // 则使用 generator 的 return 值
  if (!lastOutput) {
    throw new Error('executeStream completed without producing an output');
  }

  return lastOutput;
}
```

**兼容性保证**：
- `execute()` 的方法签名**完全不变**（`(input: OrchestratorInput) => Promise<OrchestratorOutput>`）
- 返回的 `OrchestratorOutput` 结构**完全不变**
- 行为语义**完全不变**（同步等待完成后返回结果）
- 现有所有测试**无需修改**即可通过

### 4.2 OrchestratorOutput 从事件流中聚合

事件流产生的数据最终通过两种方式聚合为 `OrchestratorOutput`：

1. **层间 return 值传递**：`dispatchCapability()` return `ToolResult` → `agentLoop()` return `AgentLoopResult`（包含 run、steps、approvals）→ `executeStream()` return `OrchestratorOutput`
2. **orchestrator:completed 事件**：在 `executeStream()` 的最后一个 `yield` 中包含完整的 `OrchestratorOutput`，供 `execute()` 包装器提取

```
事件流（yield）：               返回值链（return）：
orchestrator:started            │
orchestrator:routed             │
agent:started                   │
agent:iteration                 │
agent:reasoning                 │
agent:tool-call                 │
safety:check-passed             │
capability:started              │
capability:completed            │
  └─ return ToolResult ─────────┤
agent:completed                 │
  └─ return AgentLoopResult ────┤
orchestrator:completed ◀────────┤ output 嵌入事件
  └─ return OrchestratorOutput ─┘
```

### 4.3 delegate() 兼容

现有的 `delegate()` 方法同样改为基于 `delegateStream()` 的包装：

```typescript
async delegate(
  parentRun: AgentRun,
  domain: AgentDomain,
  task: string,
): Promise<AgentRun> {
  const ctx: ExecutionContext = {
    runId: parentRun.id,
    sessionId: parentRun.sessionId,
    surface: 'global-chat',
    signal: new AbortController().signal,
  };

  const stream = this.delegateStream(parentRun, domain, task, ctx);
  let lastRun: AgentRun | undefined;

  for await (const event of stream) {
    if (event.type === 'orchestrator:completed') {
      lastRun = event.output.run;
    }
  }

  if (!lastRun) {
    throw new Error('delegateStream completed without producing a run');
  }

  return lastRun;
}
```
