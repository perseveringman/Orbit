## 0. 现状分析与问题清单

当前 `memory-manager.ts`（171行）存在七个关键缺陷：

| # | 问题 | 影响 | 优先级 |
|---|------|------|--------|
| P1 | 仅 `InMemoryMemoryStore`（Map-based），无持久化 | 重启丢失所有记忆 | P0 |

## 1. 存储后端设计

### 1.1 SQLiteMemoryStore 总体架构

新增文件：`packages/agent-core/src/sqlite-memory-store.ts`

SQLiteMemoryStore 实现 `MemoryStore` 接口，通过 `better-sqlite3`（Node.js 原生 SQLite 绑定）与本地数据库交互。选择 `better-sqlite3` 而非 `sql.js` 的原因：同步 API 简化事务管理、WAL 模式性能优秀、FTS5 原生支持。

```typescript
// packages/agent-core/src/sqlite-memory-store.ts

```


**中英文分词策略**：FTS5 的 `unicode61` tokenizer 对英文分词效果好，但对中文需要额外处理。我们采用混合策略：

1. **写入时预处理**：在 `store()` 方法中，对中文内容按字符间插入空格（`segmentChinese()` 辅助函数），使 `unicode61` 可以按字/词粒度索引。这是 SQLite FTS5 处理 CJK 文本的标准工程实践。
2. **查询时同步处理**：`fullTextSearch()` 对查询字符串做同样的预处理。
3. **未来升级路径**：当引入 `jieba-wasm` 等分词库后，可替换 `segmentChinese()` 为真正的分词器，无需变更表结构。

```typescript
// CJK 文本预处理：在连续 CJK 字符间插入空格
function segmentCJK(text: string): string {
  return text.replace(
    /([\u4e00-\u9fff\u3400-\u4dbf])/g,
    ' $1 '
  ).replace(/\s+/g, ' ').trim();
}
```

### 1.3 WAL 模式与并发读写

数据库初始化时执行以下 PRAGMA 配置：

```typescript
private initPragmas(): void {
  this.db.pragma('journal_mode = WAL');       // Write-Ahead Logging
  this.db.pragma('synchronous = NORMAL');     // 平衡性能与安全
  this.db.pragma('cache_size = -8000');       // 8MB 页缓存
  this.db.pragma('busy_timeout = 5000');      // 写冲突等待 5 秒
  this.db.pragma('foreign_keys = ON');
  this.db.pragma('temp_store = MEMORY');      // 临时表在内存
}
```

**并发模型**：WAL 模式下 SQLite 支持"多读单写"——多个读事务可以与一个写事务并发执行。在 Orbit 的单进程模型中（Electron 主进程 / Server 单实例），这足以覆盖：
- Orchestrator 的 `assembleContext()` 读取记忆（读事务）
- `storeFromTurn()` 写入新记忆（写事务）
- `fullTextSearch()` 全文检索（读事务）

如果未来多进程场景出现写冲突，`busy_timeout = 5000` 提供 5 秒的自动重试窗口。

## 2. 六层记忆的具体实现

### 2.1 L0 Turn —— 纯内存推理缓冲

**生命周期**：单回合创建，回合结束时清除。

**设计**：L0 是 Orchestrator 在一次 `execute()` 调用中的临时工作区。它不写入 SQLite，仅存在于内存中。

```typescript
// packages/agent-core/src/layers/l0-turn-buffer.ts

export class L0TurnBuffer {
  private entries: MemoryEntry[] = [];

  push(content: string, sourceRef?: { objectId?: string }): MemoryEntry { ... }
  getAll(): readonly MemoryEntry[] { return [...this.entries]; }
  clear(): void { this.entries = []; }
  estimateTokens(): number {
    return this.entries.reduce((sum, e) => sum + Math.ceil(e.content.length / 4), 0);
  }
}
```

**与 Orchestrator 的集成点**：

```typescript
// orchestrator.ts execute() 内部
const turnBuffer = new L0TurnBuffer();
// ... 工具调用过程中，中间推理结果存入 turnBuffer
turnBuffer.push(`Tool ${tc.name} returned: ${result.output.slice(0, 200)}`);
// 回合结束
turnBuffer.clear();
```

**接口合约**：
- `push(content)` → 立即写入内存数组，返回 MemoryEntry（id 带 `l0_` 前缀）
- `getAll()` → 返回只读数组
- `clear()` → 在 Orchestrator 完成 `execute()` 后调用
- `estimateTokens()` → 用于 token 预算计算

### 2.2 L1 Session —— 会话级工作记忆

**生命周期**：会话开始到会话结束/压缩。

**设计**：L1 采用"内存热缓存 + 可选 SQLite 持久化"双模式。活跃会话的记忆驻留内存，会话暂停或结束时持久化到 SQLite。

```typescript
// packages/agent-core/src/layers/l1-session-cache.ts

export interface L1SessionCacheConfig {
  readonly maxEntriesPerSession: number;   // 默认 50
  readonly maxTokensPerSession: number;    // 默认 8000
  readonly persistOnPause: boolean;        // 默认 true
}

export class L1SessionCache {
  private activeSessions = new Map<string, MemoryEntry[]>();
  private readonly store: MemoryStore;     // SQLite 后端
  private readonly config: L1SessionCacheConfig;

  constructor(store: MemoryStore, config?: Partial<L1SessionCacheConfig>) { ... }

  // 向当前会话追加记忆
  async append(sessionId: string, content: string, confidence?: number): Promise<MemoryEntry> { ... }

  // 获取会话的全部工作记忆
  getSessionMemories(sessionId: string): readonly MemoryEntry[] { ... }

  // 会话暂停时持久化到 SQLite
  async persistSession(sessionId: string): Promise<void> { ... }

  // 会话恢复时从 SQLite 加载到内存
  async loadSession(sessionId: string): Promise<void> { ... }

  // 热数据管理：当条目超限时，淘汰最旧的低置信度条目
  private evict(sessionId: string): void { ... }
}
```

**热数据管理策略**：
1. 当 `entries.length > maxEntriesPerSession` 时，按 `confidence ASC, createdAt ASC` 排序，淘汰前 20%
2. 当 `estimateTokens(entries) > maxTokensPerSession` 时，触发微压缩（合并相邻的低置信度条目）
3. 被淘汰的条目如果 `confidence >= 0.7`，降级写入 L5-archive 而不是直接丢弃

### 2.3 L2 Object Memory —— 对象层结构化事实

**核心理念**：L2 记忆不是 Agent 创造的，是从 Orbit 对象网络中提取的结构化事实。每条事实都可追溯到具体的对象 ID 和字段。

#### 2.3.1 每个对象类型的"关键事实"定义

#### 2.3.2 关系事实提取

通过 `@orbit/object-graph` 的 `ObjectGraphIndex` 提取对象间关系：

```typescript
function extractRelationFacts(
  objectKey: string,
  graph: ObjectGraphIndex,
): ObjectFact[] {
  const facts: ObjectFact[] = [];
  const outgoing = graph.outgoingByKey[objectKey] ?? [];
  const incoming = graph.incomingByKey[objectKey] ?? [];

  for (const edge of outgoing) {
    const targetNode = graph.nodesByKey[toObjectGraphKey(edge.to)];
    const targetLabel = targetNode ? getDisplayName(targetNode) : edge.to.id;
    facts.push({
      factType: 'relation',
      text: `${edge.type} → ${edge.to.kind}:"${targetLabel}"`,
      confidence: 1.0,
    });
  }

  for (const edge of incoming) {
    const sourceNode = graph.nodesByKey[toObjectGraphKey(edge.from)];
    const sourceLabel = sourceNode ? getDisplayName(sourceNode) : edge.from.id;
    facts.push({
      factType: 'relation',
      text: `${edge.from.kind}:"${sourceLabel}" ${edge.type} → 此对象`,
      confidence: 1.0,
    });
  }

  return facts;
}
```

#### 2.3.3 实时拉取 vs 缓存策略

```
请求 L2 记忆 (objectId)
       │
       ▼
  objects_cache 中是否有命中？
       │
   ┌───┴───┐
   │ YES   │ NO
   ▼       ▼
  检查 TTL   从对象层拉取 DomainObject
   │           │
 ┌─┴─┐        提取事实（FACT_EXTRACTORS）
 │有效│过期       │
 ▼   ▼        计算 source_hash
返回  从对象层      │
缓存  重新拉取     写入 objects_cache
      │           │
      ▼           ▼
    更新缓存    返回事实列表
```

实现要点：

```typescript
export class L2ObjectMemory {
  private readonly store: SQLiteMemoryStore;
  private readonly objectResolver: ObjectResolver; // 从对象数据库获取 DomainObject
  private readonly graphResolver: GraphResolver;   // 从对象图获取关联

  async getObjectFacts(
    objectId: string,
    kind: OrbitObjectKind,
  ): Promise<readonly ObjectFact[]> {
    const key = `${kind}:${objectId}`;

    // 1. 检查缓存
    const cached = this.store.getObjectCache(key);
    if (cached && !this.isStale(cached)) {
      return JSON.parse(cached.facts_json) as ObjectFact[];
    }

    // 2. 从对象层拉取
    const object = await this.objectResolver.resolve(kind, objectId);
    if (!object) return [];

    // 3. 提取事实
    const facts = FACT_EXTRACTORS[kind](object);

    // 4. 提取关系事实
    const graph = await this.graphResolver.getLocalGraph(objectId);
    if (graph) {
      facts.push(...extractRelationFacts(key, graph));
    }

    // 5. 写入缓存
    const hash = this.computeHash(object);
    this.store.setObjectCache(key, kind, objectId, facts, hash);

    return facts;
  }

  private isStale(cached: ObjectCacheRow): boolean {
    const age = Date.now() - new Date(cached.cached_at).getTime();
    return age > cached.ttl_seconds * 1000;
  }
}
```

**接口依赖**：`ObjectResolver` 和 `GraphResolver` 是抽象接口，由上层（server-core 或 platform 层）注入具体实现。这保持了 `agent-core` 对数据来源的无感知。

```typescript
export interface ObjectResolver {

}

export interface GraphResolver {
  getLocalGraph(anchorId: string, depth?: number): Promise<ObjectGraphIndex | null>;

```

### 2.4 L3 User Longterm —— 用户长期记忆

**核心理念**：L3 存储用户亲手确认的偏好、愿景、规则。来源是用户显式输入，不是 Agent 推测。

#### 2.4.1 文件存储格式

L3 采用 JSON 文件持久化（而非 SQLite），因为：
- 用户可能希望直接编辑/查看自己的长期记忆
- 数据量小（通常 < 100 条），不需要索引
- 便于跨设备同步（文件粒度同步比 SQLite 行粒度更简单）

存储路径：`{workspaceDir}/.orbit/user-memory.json`

```typescript
// 文件格式
interface UserLongtermFile {
  readonly version: 1;
  readonly updatedAt: string;
  readonly entries: readonly UserLongtermEntry[];
}

interface UserLongtermEntry {
  readonly id: string;
  readonly category: 'preference' | 'vision' | 'rule' | 'context';
  readonly content: string;
  readonly confirmedByUser: boolean;    // 必须为 true 才注入上下文
  readonly confidence: number;          // 用户确认的 = 1.0，Agent 建议的 = 0.5
  readonly createdAt: string;
  readonly updatedAt: string;
  readonly source: 'user-input' | 'agent-suggested' | 'vision-summary';
}
```

#### 2.4.2 加载时机

```
应用启动 / 工作区切换
       │
       ▼
  读取 user-memory.json
       │
       ▼
  过滤 confirmedByUser === true 的条目
       │
       ▼
  按 category 分组，生成冻结快照
       │
       ▼
  注入 L3 层记忆（整个会话期间不变）
```

**冻结快照机制**（借鉴 Hermes 的 `_system_prompt_snapshot`）：会话开始时加载一次 L3 内容，在整个会话期间保持稳定。即使用户在其他界面修改了 user-memory.json，当前会话的 L3 也不会实时变化——只在下次会话开始时刷新。这避免了会话内记忆不一致。

## 3. 智能回忆引擎 (RecallEngine)

### 3.1 多层联合检索策略

`RecallEngine` 替代当前 `MemoryManager.recallForTurn()` 的简单全量搜索，实现分层优先级的联合检索：

```typescript
// packages/agent-core/src/recall-engine.ts

export interface RecallRequest {
  readonly query: string;
  readonly sessionId: string;
  readonly anchorObjectIds: readonly string[];  // 当前锚定对象
  readonly surface: AgentSurface;
  readonly tokenBudget: number;                 // 总 token 预算
}

export interface RecallResult {
  readonly entries: readonly ScoredMemoryEntry[];
  readonly tokenUsage: number;
  readonly layerBreakdown: Record<MemoryLayer, number>;  // 每层使用的 token 数
}

export interface ScoredMemoryEntry extends MemoryEntry {
  readonly score: number;           // 综合评分 [0, 1]
  readonly scoreBreakdown: {
    readonly ftsRank: number;       // FTS5 相关度
    readonly timeDecay: number;     // 时间衰减分
    readonly objectAffinity: number;// 对象亲和度
    readonly layerBoost: number;    // 层级加权
  };
}

export class RecallEngine {
  private readonly store: SQLiteMemoryStore;
  private readonly l2: L2ObjectMemory;
  private readonly l3: L3UserLongterm;
  private readonly l4: L4ProceduralMemory;
  private readonly tokenCalculator: TokenCalculator;

  async recall(request: RecallRequest): Promise<RecallResult> {
    const budget = new TokenBudgetAllocator(request.tokenBudget);

    // 阶段 1：固定注入（L3 用户长期记忆 — 冻结上下文）
    const l3Entries = this.l3.getSnapshot()
      .map(e => this.toScoredEntry(e, { layerBoost: 1.0 }));
    budget.allocate('L3-user-longterm', l3Entries);

    // 阶段 2：对象事实（L2 — 基于锚定对象）
    const l2Entries: ScoredMemoryEntry[] = [];
    for (const objId of request.anchorObjectIds) {
      const facts = await this.l2.getObjectFacts(objId, 'project'); // kind 从上下文推断
      l2Entries.push(...facts.map(f => this.factToScoredEntry(f, objId)));
    }
    budget.allocate('L2-object', l2Entries);

    // 阶段 3：会话记忆（L1 — 当前会话）
    const l1Entries = await this.store.recall({
      scope: 'L1-session',
      query: request.query,
      sessionId: request.sessionId,
      maxEntries: 20,
    });
    budget.allocate('L1-session', l1Entries.map(e =>
      this.toScoredEntry(e, { layerBoost: 0.8 })
    ));

    // 阶段 4：历史归档（L5 — 跨会话搜索）
    if (budget.remaining() > 500) {
      const l5Entries = await this.store.fullTextSearch(request.query, {
        layer: 'L5-archive',
        maxResults: 5,
      });
      budget.allocate('L5-archive', l5Entries.map(e =>
        this.toScoredEntry(e.entry, {
          ftsRank: e.score,
          layerBoost: 0.5,
        })
      ));
    }

    // 阶段 5：工作法模板（L4 — 如果有匹配的）
    if (budget.remaining() > 300) {
      const domain = this.inferDomain(request);
      const templates = await this.l4.findMatchingTemplates(domain, request.query, 2);
      budget.allocate('L4-procedural', templates.map(t =>
        this.toScoredEntry(t.entry, { layerBoost: 0.6 })
      ));
    }

    return budget.finalize();
  }
}
```

### 3.2 相关性评分算法

综合评分公式：

```
score = w_fts * ftsRank + w_time * timeDecay + w_obj * objectAffinity + w_layer * layerBoost
```

其中：

```typescript
const SCORING_WEIGHTS = {
  fts: 0.35,      // FTS5 全文相关度
  time: 0.20,     // 时间衰减
  object: 0.25,   // 对象亲和度
  layer: 0.20,    // 层级加权
} as const;
```

#### FTS5 Rank（`ftsRank`）

使用 FTS5 内置的 BM25 排名函数：

```sql
SELECT m.*, rank AS fts_rank
FROM memories_fts
JOIN memories m ON memories_fts.rowid = m.rowid
WHERE memories_fts MATCH ?
  AND (?1 IS NULL OR layer = ?1)
ORDER BY rank
LIMIT ?2;
```

`rank` 值归一化到 [0, 1]：`normalizedRank = 1 / (1 + Math.abs(rank))`

#### 时间衰减（`timeDecay`）

指数衰减函数，半衰期可配置：

```typescript
function computeTimeDecay(createdAt: string, halfLifeHours: number = 168): number {
  const ageHours = (Date.now() - new Date(createdAt).getTime()) / (1000 * 60 * 60);
  return Math.pow(0.5, ageHours / halfLifeHours);
}
```

默认半衰期 168 小时（7 天）：7 天前的记忆衰减到 50% 权重。

#### 对象亲和度（`objectAffinity`）

当记忆的 `sourceObjectId` 与当前锚定对象匹配时获得高亲和度：

```typescript
function computeObjectAffinity(
  entry: MemoryEntry,
  anchorObjectIds: readonly string[],
  graph?: ObjectGraphIndex,
): number {
  if (!entry.sourceObjectId) return 0;

  // 直接匹配 → 1.0
  if (anchorObjectIds.includes(entry.sourceObjectId)) return 1.0;

  // 一跳关联 → 0.6
  if (graph) {
    for (const anchorId of anchorObjectIds) {
      const connected = listConnectedTargets(graph, { kind: 'project', id: anchorId });
      if (connected.some(ref => ref.id === entry.sourceObjectId)) return 0.6;
    }
  }

  // 同一会话 → 0.3
  return 0;
}
```

#### 层级加权（`layerBoost`）

不同层级的记忆有不同的默认优先级加权：

### 3.3 Token 预算分配

#### 分层配额

```typescript

  readonly layerQuotas: Record<MemoryLayer, number>; // 每层最大占比

  totalBudget: 4000,
  layerQuotas: {
    'L0-turn': 0.05,       // 200 tokens  — 当前回合缓冲，极少
    'L1-session': 0.25,    // 1000 tokens — 会话上下文
    'L2-object': 0.25,     // 1000 tokens — 对象事实
    'L3-user-longterm': 0.20, // 800 tokens — 用户长期记忆

    'L5-archive': 0.15,    // 600 tokens  — 历史归档
  },
```

`TokenBudgetAllocator` 实现：

```typescript
export class TokenBudgetAllocator {
  private readonly budget: number;
  private readonly layerQuotas: Record<MemoryLayer, number>;
  private allocated = new Map<MemoryLayer, ScoredMemoryEntry[]>();
  private usedTokens = 0;

  constructor(budget: number, quotas?: Partial<Record<MemoryLayer, number>>) {
    this.budget = budget;
    this.layerQuotas = { ...DEFAULT_BUDGET_CONFIG.layerQuotas, ...quotas };
  }

  allocate(layer: MemoryLayer, entries: readonly ScoredMemoryEntry[]): void {
    const maxForLayer = Math.floor(this.budget * this.layerQuotas[layer]);
    const remaining = this.budget - this.usedTokens;
    const available = Math.min(maxForLayer, remaining);

    // 按 score 降序排列，贪心填充到配额
    const sorted = [...entries].sort((a, b) => b.score - a.score);
    const selected: ScoredMemoryEntry[] = [];
    let layerTokens = 0;

    for (const entry of sorted) {
      const tokens = this.estimateTokens(entry);
      if (layerTokens + tokens > available) continue;
      selected.push(entry);
      layerTokens += tokens;
    }

    this.allocated.set(layer, selected);
    this.usedTokens += layerTokens;
  }

  remaining(): number {
    return this.budget - this.usedTokens;
  }

  finalize(): RecallResult {
    const allEntries: ScoredMemoryEntry[] = [];
    const breakdown: Record<string, number> = {};

    for (const [layer, entries] of this.allocated) {
      allEntries.push(...entries);
      breakdown[layer] = entries.reduce(
        (sum, e) => sum + this.estimateTokens(e), 0
      );
    }

    return {
      entries: allEntries.sort((a, b) => b.score - a.score),
      tokenUsage: this.usedTokens,
      layerBreakdown: breakdown as Record<MemoryLayer, number>,
    };
  }

  private estimateTokens(entry: ScoredMemoryEntry): number {
    return Math.ceil(entry.content.length / 4);
  }
}
```

### 3.4 混合 Token 计算

借鉴 Claude Code 的 `tokenCountWithEstimation()` 策略：

```typescript
// packages/agent-core/src/token-calculator.ts

export class TokenCalculator {
  // 最后一次 API 返回的精确 usage
  private lastKnownUsage: { messageIndex: number; totalTokens: number } | null = null;

  /**
   * 混合 token 计算：
   * - 对已有 API usage 数据的消息，使用精确值
   * - 对之后的消息，使用 chars/4 粗估
   * - 两者相加得出总量
   */
  estimateTotal(messages: readonly AgentMessage[]): number {
    if (!this.lastKnownUsage) {
      // 无 API 数据，全部粗估
      return messages.reduce(
        (sum, m) => sum + Math.ceil(m.content.length / 4),
        0,
      );
    }

    const { messageIndex, totalTokens } = this.lastKnownUsage;
    // 精确部分
    let total = totalTokens;

    // 粗估部分：messageIndex 之后的消息
    for (let i = messageIndex + 1; i < messages.length; i++) {
      total += Math.ceil(messages[i].content.length / 4);
    }

    return total;
  }

  /**
   * 从 API 响应更新精确 token 数据
   */
  updateFromAPIResponse(
    messageIndex: number,
    usage: { promptTokens: number; totalTokens: number },
  ): void {
    this.lastKnownUsage = {
      messageIndex,
      totalTokens: usage.promptTokens,
    };
  }

  /**
   * 粗估单条文本的 token 数
   */
  estimate(text: string): number {
    return Math.ceil(text.length / 4);
  }
}
```

### 4.2 分层注入顺序

注入顺序遵循"确定性优先、时效性次之"原则：

```
┌─────────────────────────────────────────────┐
│ 1. L3 User Longterm  (冻结上下文，最高优先)    │  ← 用户确认的偏好/愿景/规则
│    confidence ≥ 0.9, 会话内不变                │
├─────────────────────────────────────────────┤
│ 2. L2 Object Facts   (对象事实，确定性高)      │  ← 从对象层实时拉取
│    confidence 1.0, 基于锚定对象               │
├─────────────────────────────────────────────┤
│ 3. L1 Session        (会话工作记忆)           │  ← 当前会话积累
│    confidence 0.7-1.0, 按时间排序             │
├─────────────────────────────────────────────┤
│ 4. L4 Procedural     (工作法模板，参考性)      │  ← 匹配的执行模板

├─────────────────────────────────────────────┤

│    confidence 0.4-0.8, 经轻量模型摘要          │
└─────────────────────────────────────────────┘

  L0 Turn 不参与注入——它是回合内的临时缓冲，直接在 Orchestrator 内部使用。
```

注入在 Orchestrator 的 `assembleContext()` 中执行：

```typescript
// orchestrator.ts 修改
async assembleContext(session: AgentSession): Promise<readonly AgentMessage[]> {
  let messages = [...session.messages];

  // 1. 通过 RecallEngine 获取分层记忆
  const recallResult = await this.recallEngine.recall({
    query: this.extractLatestUserQuery(messages),
    sessionId: session.id,
    anchorObjectIds: session.anchorObjectIds,
    surface: session.surface,
    tokenBudget: this.config.memoryTokenBudget ?? 4000,
  });

  // 2. 构建分层注入 XML
  const injectionBlocks = recallResult.entries.map(e => ({
    layer: e.layer,
    source: e.sourceObjectId ?? e.sourceSessionId ?? 'system',
    confidence: e.confidence,
    content: e.content,
  }));
  const memoryXML = this.memoryInjector.buildInjectionXML(injectionBlocks);

  // 3. 压缩会话消息（如果需要）
  if (this.compressor.shouldCompress(messages, this.config.compressionThreshold)) {
    const result = await this.compressor.compress(
      messages,
      this.config.compressionThreshold,
      this.createSummarizer(),
    );
    messages = [...result.compressedMessages];
  }

  // 4. 注入记忆到 system 消息
  // memoryXML 附加到 system prompt 末尾
  return messages;
}
```

### 4.3 注入安全（sanitizeContext 升级）

当前 `sanitizeContext()` 仅剥离 `<memory-context>` 标签。升级为全面的注入防护：

```typescript
export class MemoryInjector {
  // ... buildInjectionXML 等

  /**
   * 升级版 sanitizeContext：
   * 1. 剥离所有 memory 相关 XML 标签（防伪造注入）
   * 2. 检测并清除嵌套的 memory 标签
   * 3. 转义属性中的特殊字符
   */
  sanitizeUserInput(text: string): string {
    return text
      // 移除所有 memory-context 和 memory 标签
      .replace(/<\/?memory-context\b[^>]*>/gi, '')
      .replace(/<\/?memory\b[^>]*>/gi, '')
      // 移除嵌套尝试（多层编码）
      .replace(/&lt;\/?memory/gi, '')
      // 移除 CDATA 包裹的注入尝试
      .replace(/<!\[CDATA\[.*?memory.*?\]\]>/gis, '');
  }

  /**
   * 验证记忆内容不包含注入攻击
   */
  validateMemoryContent(content: string): boolean {
    const INJECTION_PATTERNS = [
      /<memory\b/i,
      /<memory-context\b/i,
      /\bignore\s+(previous|above)\s+instructions\b/i,
      /\byou\s+are\s+now\b/i,
      /\bsystem\s*:\s*/i,
    ];
    return !INJECTION_PATTERNS.some(p => p.test(content));
  }

  private escapeAttr(value: string): string {
    return value
      .replace(/&/g, '&amp;')
      .replace(/"/g, '&quot;')
      .replace(/</g, '&lt;')
      .replace(/>/g, '&gt;');
  }
}
```

**安全原则**：
- 用户输入经过 `sanitizeUserInput()` 后才能进入任何记忆层
- 所有记忆内容在注入前经过 `validateMemoryContent()` 检查
- XML 属性值经过 `escapeAttr()` 转义
- SafetyGate 的 `scanContext()` 在记忆注入后对完整 system prompt 做最终检查

## 5. 从 InMemoryMemoryStore → SQLiteMemoryStore 的迁移

### 5.1 接口兼容性

`SQLiteMemoryStore` 严格实现 `MemoryStore` 接口，无需修改接口定义：

```typescript
export interface MemoryStore {
  recall(query: MemoryRecallQuery): Promise<readonly MemoryEntry[]>;
  store(entry: Omit<MemoryEntry, 'id' | 'createdAt'>): Promise<MemoryEntry>;
  remove(id: string): Promise<void>;
  getByLayer(layer: MemoryLayer): Promise<readonly MemoryEntry[]>;
}
```

`SQLiteMemoryStore` 额外暴露扩展方法（`fullTextSearch`, `getBySession`, `vacuum` 等），但这些通过类型窄化访问，不影响接口契约。

`MemoryManager` 构造函数签名不变：`constructor(store: MemoryStore)`。调用方只需替换注入的 store 实例即可切换后端。

### 5.2 渐进式切换策略

分三阶段迁移，确保每个阶段都可独立测试和回滚：

#### Phase 1：并行运行（可观测）

```typescript
// packages/agent-core/src/dual-memory-store.ts

export class DualMemoryStore implements MemoryStore {
  private readonly primary: MemoryStore;    // InMemoryMemoryStore（生产路径）
  private readonly shadow: MemoryStore;     // SQLiteMemoryStore（影子写入）

  async store(entry: Omit<MemoryEntry, 'id' | 'createdAt'>): Promise<MemoryEntry> {
    const result = await this.primary.store(entry);
    // 影子写入 SQLite，不影响主路径
    this.shadow.store(entry).catch(err =>
      console.warn('[DualMemoryStore] shadow write failed:', err)
    );
    return result;
  }

  async recall(query: MemoryRecallQuery): Promise<readonly MemoryEntry[]> {
    const [primaryResult, shadowResult] = await Promise.allSettled([
      this.primary.recall(query),
      this.shadow.recall(query),
    ]);
    // 对比结果差异（记录到日志，不影响返回值）
    if (primaryResult.status === 'fulfilled' && shadowResult.status === 'fulfilled') {
      this.compareResults(primaryResult.value, shadowResult.value, query);
    }
    return primaryResult.status === 'fulfilled' ? primaryResult.value : [];
  }
}
```

#### Phase 2：SQLite 为主，InMemory 降级

```typescript
// 配置驱动切换

  ? new SQLiteMemoryStore({ dbPath: config.memoryDbPath })
  : new InMemoryMemoryStore();

const memoryManager = new MemoryManager(memoryStore);
```

#### Phase 3：移除 InMemory（仅测试保留）

`InMemoryMemoryStore` 保留在代码库中，作为测试和轻量场景的默认实现。`SQLiteMemoryStore` 成为生产默认值。

```typescript
// packages/agent-core/tests/memory-store-contract.test.ts

import { describe, expect, it, beforeEach } from 'vitest';
import type { MemoryStore, MemoryRecallQuery } from '../src/memory-manager.js';
import type { MemoryEntry, MemoryLayer } from '../src/types.js';

/**
 * Abstract contract tests for MemoryStore implementations.

export function runMemoryStoreContractTests(

  factory: () => MemoryStore | Promise<MemoryStore>,
  cleanup?: () => void | Promise<void>,

  describe(`MemoryStore contract: ${name}`, () => {
    let store: MemoryStore;

    beforeEach(async () => {
      store = await factory();

    afterEach(async () => {
      await cleanup?.();
    });

    // -- store & recall --

    it('should store and recall an entry', async () => {
      const entry = await store.store({
        layer: 'L1-session',
        content: 'test memory content',

      expect(entry.createdAt).toBeTruthy();

      expect(entry.layer).toBe('L1-session');

        scope: 'L1-session',
        query: 'test',

      expect(recalled.some(e => e.id === entry.id)).toBe(true);
    });

      await store.store({ layer: 'L2-object', content: 'object data', confidence: 1.0 });

      const sessionOnly = await store.recall({ scope: 'L1-session', query: '' });
      expect(sessionOnly.every(e => e.layer === 'L1-session')).toBe(true);

      const all = await store.recall({ scope: 'all', query: '' });
      expect(all.length).toBeGreaterThanOrEqual(2);
    });

    it('should recall with sessionId filter', async () => {
      await store.store({
        layer: 'L1-session', content: 'session A', confidence: 1.0,
        sourceSessionId: 'ses-a',

      await store.store({
        layer: 'L1-session', content: 'session B', confidence: 1.0,
        sourceSessionId: 'ses-b',

        scope: 'L1-session', query: '', sessionId: 'ses-a',
      });

    });

        layer: 'L2-object', content: 'obj A data', confidence: 1.0,
        sourceObjectId: 'obj-a',
      });
      await store.store({
        layer: 'L2-object', content: 'obj B data', confidence: 1.0,
        sourceObjectId: 'obj-b',

      const result = await store.recall({
        scope: 'L2-object', query: '', objectId: 'obj-a',

      expect(result.every(e => e.sourceObjectId === 'obj-a')).toBe(true);
    });

    it('should respect maxEntries limit', async () => {

          layer: 'L1-session', content: `entry ${i}`, confidence: 1.0,

        scope: 'L1-session', query: '', maxEntries: 5,

      expect(result.length).toBeLessThanOrEqual(5);

        layer: 'L1-session', content: 'to remove', confidence: 1.0,

    });

      await store.store({ layer: 'L3-user-longterm', content: 'pref 2', confidence: 1.0 });
      await store.store({ layer: 'L4-procedural', content: 'proc 1', confidence: 1.0 });

      const l3 = await store.getByLayer('L3-user-longterm');
      expect(l3.length).toBe(2);
      expect(l3.every(e => e.layer === 'L3-user-longterm')).toBe(true);
    });

    // -- query text matching --

    it('should match entries by query text', async () => {
      await store.store({
        layer: 'L1-session', content: 'The quick brown fox', confidence: 1.0,
      });
      await store.store({
        layer: 'L1-session', content: 'A lazy dog', confidence: 1.0,
      });

      const result = await store.recall({ scope: 'all', query: 'fox' });
      expect(result.some(e => e.content.includes('fox'))).toBe(true);
    });

    // -- confidence range --

    it('should store confidence in valid range', async () => {
      const entry = await store.store({
        layer: 'L1-session', content: 'test', confidence: 0.5,
      });
      expect(entry.confidence).toBeGreaterThanOrEqual(0);
      expect(entry.confidence).toBeLessThanOrEqual(1);
    });

    // -- all six layers --

    it('should support all six memory layers', async () => {

        'L0-turn', 'L1-session', 'L2-object',
        'L3-user-longterm', 'L4-procedural', 'L5-archive',

      for (const layer of layers) {
        const entry = await store.store({
          layer, content: `${layer} content`, confidence: 1.0,
        });
        expect(entry.layer).toBe(layer);
      }
      const all = await store.recall({ scope: 'all', query: '' });
      expect(all.length).toBeGreaterThanOrEqual(6);
    });
  });
}
```
