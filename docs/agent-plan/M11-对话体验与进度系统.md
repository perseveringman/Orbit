## 一、目标与问题陈述

### 1.1 现状问题

当前 `@orbit/feature-workbench` 的 Agent 聊天界面存在六个体验缺陷：

1. **无流式渲染**：LLM 回复在完整生成后才一次性呈现，用户在等待期间只看到静态 "思考中" 文本，无法感知 Agent 是否仍在运行。
2. **无工具执行可视化**：工具调用在后台静默执行，用户无法知道 Agent 正在调用什么工具、传递了什么参数。
3. **无进度指示**：多轮迭代场景（如 Research Agent 的 15 轮检索）中，用户无法知道当前处于第几轮、距完成还有多远。
4. **无成本透明度**：每条消息消耗了多少 token、花费了多少钱，用户无从得知。
5. **错误反馈原始**：API 报错以 raw JSON 形式抛出，缺少用户友好的错误描述和恢复建议。
6. **无交互通道**：Agent 执行中途无法向用户提问或请求确认，只能盲目猜测意图。

### 1.2 设计目标

打造用户友好的 Agent 对话体验，让用户在执行过程中**始终了解 Agent 在做什么、进展如何、成本多少**，同时在出错时有清晰的反馈和恢复选项。

核心度量指标：

| 指标 | 当前 | 目标 |
|------|------|------|
| 首字节渲染延迟 | >3s（完整响应） | <200ms（流式首 token） |
| 工具执行透明度 | 0%（无任何可视化） | 100%（全部工具可见） |
| 错误恢复率 | 手动刷新/重试 | 80% 自助恢复 |
| 进度感知度 | "思考中" 唯一状态 | 6 阶段粒度 |

## 二、流式对话体验

### 2.1 设计原则

流式渲染必须满足三项约束：

1. **增量渲染**：每个 `text-delta` 事件到达时立即渲染，不等待完整段落
2. **结构感知**：Markdown 块级元素（标题、列表、代码块、表格）需要特殊的渐进式渲染策略
3. **回退安全**：流式中断时，已渲染内容保持完整可读

### 2.2 打字机效果

#### 事件流 → 渲染管道

```
M1 EventStream               StreamingRenderer              DOM
─────────────────────────────────────────────────────────────────
text-delta ──────────────────► appendText(chunk)  ──────────► 逐字显示
                               │
                               ├─ 检测 Markdown 块边界
                               │   ├─ ``` → 进入代码块模式
                               │   ├─ |   → 进入表格模式
                               │   ├─ #   → 标题渲染
                               │   └─ -/* → 列表渲染
                               │
                               └─ 更新光标位置  ──────────► 光标闪烁 █
```

```typescript
export interface IterationProgress {
  readonly currentIteration: number;
  readonly maxIterations: number;
  readonly totalTokensUsed: number;
  readonly estimatedCost: number;
  readonly urgency: ProgressUrgency;
}

export const PROGRESS_URGENCY_LEVELS = [
  'normal',   // 绿色，< 70% 使用率
  'warning',  // 黄色，70-85%
  'danger',   // 橙色，85-95%
  'critical', // 红色，> 95%
] as const;
export type ProgressUrgency = (typeof PROGRESS_URGENCY_LEVELS)[number];

/** 根据当前/最大值计算紧急程度 */
export function computeUrgency(current: number, max: number): ProgressUrgency {
  const ratio = current / max;
  if (ratio > 0.95) return 'critical';
  if (ratio > 0.85) return 'danger';
  if (ratio > 0.70) return 'warning';
  return 'normal';
}

## 附录 B：设计决策记录

### 为什么不使用现成的 Markdown 流式渲染库?

现有库（如 `react-markdown`）设计为一次性渲染完整 Markdown 字符串，不支持增量追加。逐字 delta 场景下，每次 delta 都 re-render 整个文档会导致严重的性能问题（频繁 DOM 重排）。因此需要自建增量渲染引擎，仅更新变化的 DOM 节点。

### 为什么工具预览使用静态映射而非动态生成?

静态映射（`TOOL_PREVIEW_MAP`）的优势：

1. **可控**：每个工具的预览格式经过人工打磨，确保可读性
2. **性能**：无需解析工具 schema，O(1) 查找
3. **i18n 友好**：模板字符串可轻松替换为多语言版本
4. **fallback**：未知工具使用通用模板 `🔧 {toolName}({args})` 自动降级

### 为什么 Clarify 默认超时 5 分钟?

参考 GitHub Copilot CLI 和 Cursor 的实践：

- 太短（<1 分钟）：用户可能临时离开，回来发现 Agent 已用默认值执行完毕
- 太长（>10 分钟）：占用 Agent 执行槽，影响并发任务
- 5 分钟是一个平衡点，且可在 `ClarifyTimeoutConfig` 中自定义
