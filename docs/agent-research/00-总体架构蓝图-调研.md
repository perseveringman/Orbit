# 00. 总体架构蓝图 — Agent 调研报告

> **调研目标**：结合 Hermes Agent 与 Claude Code 的架构实践，为 Orbit 总体架构蓝图提供可落地的启发与建议。
>
> **参考文档**：Orbit 设计方案 00-总体架构蓝图、Hermes/Claude Code 综合对比分析、架构与核心循环深度分析

---

## 1. Orbit 设计方案摘要

Orbit 的重启不是在旧有阅读器上叠加 AI 功能，而是从根基上重新定义产品——构建一个 **Agent-first 的人生与工作操作系统**。其核心主张是：用 Agent 把愿景、规划、输入、研究、笔记、输出与回顾编织成一个连续系统。

设计方案确立了以下几个关键架构决策：

**统一对象观**：系统以对象而非功能模块为中心。愿景、任务、文章、笔记、研究空间等一切皆为对象，通过统一关系网络互联。这意味着 Agent 只需理解"对象 + 关系 + 事件 + 能力接口"这一套世界观，而不用为每个模块学习独立的数据模型。

**五层系统架构**：Experience Layer（三端 UI）→ Agent Runtime Layer（Orchestrator + 领域子 Agent）→ Capability Layer（统一能力接口 + MCP）→ Domain Data Layer（类型化对象表 + 全局索引 + 关系层 + 事件层）→ Storage & Sync Layer（文件系统优先 + 本地优先同步）。

**三条核心流**：输入流（RSS、书籍、播客等 → 解析 → 索引 → 高亮）、思考流（高亮 → 笔记 → 研究 → 任务/写作）、输出流（研究空间 + 写作工作台 → 产物 → 回流至规划）。

**Agent Runtime 设计**：Orchestrator 统管意图解析、上下文拉取、能力调用、子 Agent 分派与审批流。下设 Planning、Task/Focus、Reading、Research、Writing、Journal/Review、Graph/Linking 七个领域子 Agent。

**八条架构原则**：其中最核心的包括——对象网络优先于模块边界、文件是用户资产而数据库是系统智能层、Agent 深度融入但必须有护栏、MCP 是能力总线而非数据库直通车。

## 2. Hermes 的可借鉴之处

### 2.1 多模型兼容的 Provider 抽象层

Hermes 通过 `auxiliary_client.py` 的工厂模式实现了对 OpenAI、Anthropic、Google、Mistral、Ollama、OpenRouter 等十余家 LLM 提供商的统一适配。其核心思路是：所有提供商统一到 OpenAI 兼容接口，非兼容的（如 Anthropic）通过 Adapter 桥接。

**对 Orbit 的启发**：Orbit 设计方案虽然明确了 Agent Runtime Layer，但未详细讨论 LLM 提供商策略。作为一个面向个人用户的长期工具，Orbit 不应绑定单一模型厂商。建议在 Agent Runtime 层引入类似 Hermes 的 Provider 抽象——定义统一的 `ModelClient` 接口，针对不同提供商实现适配器。这不仅让用户可以根据成本、隐私偏好自由选择模型（例如隐私敏感场景用本地 Ollama，性能要求高的场景用 Claude/GPT），还能在某个提供商宕机时自动回退。Hermes 的 `error_classifier.py` 实现了 API 错误分类和故障转移机制，这种鲁棒性设计对于一个"人生操作系统"级别的应用来说是必需的。

### 2.2 生命周期钩子系统

Hermes 定义了 10 个精确的生命周期钩子（`pre_tool_call`、`post_tool_call`、`pre_llm_call`、`post_llm_call`、`on_session_start`、`on_session_end` 等），允许插件在这些节点注入自定义行为。插件通过三种路径发现：用户插件目录、项目插件目录、pip 入口点。

**对 Orbit 的启发**：Orbit 的 Capability Layer 设计了"内部路径"和"外部路径"双轨暴露，但缺少对 Agent 执行生命周期的钩子定义。建议在 Agent Runtime 层借鉴 Hermes 的钩子模式，在以下关键节点开放扩展：

- **意图解析后**：可以注入用户自定义规则（如"工作时间不安排个人任务"）
- **能力调用前/后**：可以注入审计日志、成本追踪、隐私过滤
- **子 Agent 分派时**：可以注入上下文预加载、优先级覆盖
- **会话结束时**：可以触发 Journal 自动记录、记忆沉淀

这些钩子既服务于 Orbit 内部的跨领域关注点（审计、隐私、成本控制），也为未来的用户自动化脚本提供扩展点。

### 2.3 记忆系统的可插拔架构

Hermes 的记忆系统是其最突出的亮点之一：通过 `MemoryProvider` 抽象基类定义了统一接口（`initialize`、`system_prompt_block`、`prefetch`、`sync_turn`、`handle_tool_call`、`shutdown`），内置实现之外还支持 Honcho、Mem0、ChromaDB 等 8 种外部记忆插件。

**对 Orbit 的启发**：Orbit 的"时间与记忆层"（Journal、Review、Agent Session）是系统六大支柱之一，但设计方案对记忆如何被 Agent 消费的机制描述不足。Hermes 的实践表明，记忆系统需要解决几个关键问题：

1. **记忆的注入时机**：Hermes 在每次 API 调用前通过 `prefetch(query)` 动态拉取相关记忆注入上下文。Orbit 也应该在 Orchestrator 构建 turn context 时，从 Journal/Review/事件层中动态拉取与当前意图相关的历史记忆。
2. **记忆的沉淀机制**：Hermes 的 `sync_turn(user, assistant)` 在每轮对话后自动同步记忆。Orbit 可以在每次 Agent 会话结束时，自动将关键决策、用户偏好变化、任务状态转换写入事件层。
3. **记忆的层次结构**：短期记忆（当前会话）、中期记忆（近期 Journal）、长期记忆（用户愿景/偏好/规律），不同层的存储、检索和衰减策略应当不同。

### 2.4 迭代预算与安全边界

Hermes 的 `IterationBudget` 是一个线程安全的迭代计数器，控制 Agent 循环的最大迭代次数，同时支持 `refund()`（退还预算）以应对非消耗性操作。此外，Hermes 使用 106 条正则模式进行注入防护，并实现了完整的 SSRF IP 范围阻断。

**对 Orbit 的启发**：Orbit 的第四条架构原则"Agent 深度融入，但必须有护栏"需要具体的技术落地。建议：

- 每个 Agent 会话设置 token 预算和迭代预算双重限制
- 高风险操作（涉及外部 API 调用、文件删除、数据出境）使用分级审批
- 对 MCP 外部路径的调用实施更严格的预算与审计，防止外部 Agent 过度消费系统资源

## 3. Claude Code 的可借鉴之处

### 3.1 异步生成器管道架构

Claude Code 的核心架构创新是将整个 Agent 循环构建为**异步生成器管道**：`QueryEngine.submitMessage() → query() → queryLoop() → runTools()`，每一层通过 `yield` 传递事件流。这种设计带来三个关键优势：天然的流式响应（用户在 Agent 思考过程中就能看到中间结果）、背压控制（下游消费速度自动调节上游生产速度）、层级解耦（每层只关注自己的职责）。

**对 Orbit 的启发**：Orbit 作为 TypeScript monorepo，天然适合采用 Claude Code 的异步生成器模式。具体来说：

- **Orchestrator 的核心循环**应设计为 `async function* orchestrate(intent: UserIntent): AsyncGenerator<AgentEvent>`，将意图解析、上下文拉取、能力调用、子 Agent 分派的每一步都作为事件 yield 出来。
- **UI 层**可以通过 `for await (const event of orchestrate(intent))` 实时渲染 Agent 的思考过程、工具调用结果、子 Agent 进展，而不是等待整个流程结束后才返回结果。
- **三端复用**：这种生成器管道模式天然支持跨端复用——桌面端渲染完整的流式过程，iOS 端可以只渲染最终结果或关键节点，Web 端通过 SSE/WebSocket 传递事件流。

这与 Orbit 五层架构中 Agent Runtime Layer 的定位完美契合：Runtime 产出事件流，Experience Layer 消费事件流。

### 3.2 四层编排层次与关注点分离

Claude Code 的编排是精心分层的：QueryEngine（SDK 层，面向外部集成）→ query（生命周期管理层）→ queryLoop（核心循环层）→ runTools（工具编排层，含并发/串行分区）。每一层有明确的单一职责。

**对 Orbit 的启发**：Orbit 的 Agent Runtime 设计目前只提到了 Orchestrator + 领域子 Agent 的两级结构，建议细化为四层：

1. **Intent Layer**：接收用户意图，解析为结构化的 AgentRequest（对应 QueryEngine 层）
2. **Orchestration Layer**：管理会话生命周期、上下文构建、子 Agent 路由、审批流（对应 query 层）
3. **Execution Layer**：核心 Agent 循环——调用模型、处理响应、分派工具（对应 queryLoop 层）
4. **Capability Dispatch Layer**：工具编排、并发决策、权限检查（对应 runTools 层）

这种四层结构的好处在于：Intent Layer 可以同时服务 UI 直接输入和 MCP 外部调用；Orchestration Layer 可以独立处理跨子 Agent 的协调逻辑；Execution Layer 可以针对不同领域 Agent 有不同的循环策略（如 Research Agent 需要多轮深入搜索，Task Agent 可能一轮就够）；Capability Dispatch Layer 统一处理内部/外部能力的调用与权限。

### 3.3 MCP 作为标准化扩展协议

Claude Code 将 MCP（Model Context Protocol）作为核心扩展协议，支持 stdio/SSE/HTTP 三种传输方式，将外部 MCP 工具统一包装为内部 Tool 接口。MCP 工具的 Annotations 被映射为行为策略（只读 vs 写入 vs 危险操作），实现了声明式的安全管控。

**对 Orbit 的启发**：Orbit 的第七条架构原则"MCP 是能力总线，不是数据库直通车"已经明确了方向，但 Claude Code 的实践提供了具体的落地参考：

- **内外统一**：Claude Code 通过 `MCPTool` 包装器让 MCP 外部工具和内部工具共享同一个 Tool 接口。Orbit 应该让 Capability Layer 的统一 capability interface 同时支持：内部 Agent 直接调用、UI 组件调用、以及通过 MCP 暴露给外部。三者共享同一套能力定义、权限模型和审计日志。
- **Annotations 驱动的安全策略**：MCP 工具通过声明式注解标记自己的行为类型（read-only / destructive），系统据此自动应用不同的审批策略。Orbit 的每个 capability 也应该声明自己的副作用级别，Agent Runtime 据此决定是否需要用户确认。
- **传输层灵活性**：本地场景用 stdio（低延迟），远程/跨设备场景用 HTTP（跨网络）。这对 Orbit 的三端同步尤为重要——桌面端的 Agent 可以通过 stdio 调用本地能力，iOS 端可以通过 HTTP 调用桌面端暴露的 MCP 服务。

### 3.4 深度模块化与组件化

Claude Code 的代码组织体现了极致的关注点分离：每个工具一个独立目录（内含实现、测试、类型定义）、100+ React 组件、80+ Hooks、清晰的六层架构。其状态管理使用 Zustand Store + React Context，依赖注入通过 ToolUseContext 对象传递。

**对 Orbit 的启发**：Orbit 作为三端应用（Desktop/Web/iOS），代码复用和模块化是架构成败的关键。建议：

- **工具/能力目录化**：每个 capability 一个目录，包含定义（schema）、实现（handler）、测试、权限声明。这样无论是内部 Agent 调用、UI 绑定还是 MCP 暴露，都引用同一份能力定义。
- **领域逻辑与 UI 分离**：核心业务逻辑（对象 CRUD、关系管理、Agent 编排）作为纯 TypeScript 包，三端共享；UI 层各端独立实现但共享设计 token 和交互模型。
- **状态管理统一**：参考 Claude Code 的 Zustand 实践，在 Orbit 的 Domain Data Layer 之上建立响应式的状态层，让 UI 组件可以声明式地订阅对象和关系的变化。

### 3.5 编译时特性标记

Claude Code 使用 `feature()` 函数实现编译时特性门控（如 `feature('ABLATION_BASELINE')`、`feature('DUMP_SYSTEM_PROMPT')`），这些标记在编译时解析，零运行时开销，不存在的特性路径会被 tree-shaking 移除。

**对 Orbit 的启发**：Orbit 涵盖六大支柱（愿景/执行/输入/研究/记忆/对象），功能体量庞大。编译时特性标记可以用于：

- **渐进式发布**：按支柱逐步交付功能，未启用的支柱代码不进入最终包
- **平台差异化**：iOS 端编译时排除桌面端专有功能（如本地文件系统深度集成），减小包体积
- **A/B 实验**：测试不同的 Agent 编排策略或 UI 交互模式

## 4. 不适用于 Orbit 的设计

### 4.1 Hermes 的单体核心类设计

Hermes 的 `AIAgent` 类是一个约 7500 行的单体文件，构造函数接受 53 个参数。这种"一个类统治一切"的设计虽然在初期迭代速度快、调试直接，但对 Orbit 来说是一个反面教材。Orbit 拥有七个领域子 Agent、六大对象家族、三端 UI，如果将编排逻辑集中在一个巨型 Orchestrator 中，将迅速陷入不可维护的状态。Orbit 需要的是 Claude Code 那种分层模块化的编排架构。

### 4.2 Hermes 的同步主循环 + 异步桥接

Hermes 的核心循环是同步的 `while` 循环，为了适配异步工具和 Gateway，不得不引入复杂的线程池 + 事件循环桥接（每线程维护一个持久事件循环）。这种架构增加了大量偶发复杂性。Orbit 作为 TypeScript 项目，运行在 Node.js 的原生异步事件循环之上，应该从一开始就采用全异步模型（async/await + async generator），完全没有必要引入同步/异步桥接层。

### 4.3 Hermes 的多平台 Gateway 层

Hermes 的 Gateway 层支持 14+ 消息平台（WhatsApp、Telegram、Discord 等），这是其作为"通用 AI 基础设施"的关键卖点。但 Orbit 是一个**个人操作系统**，其交互界面是自有的三端应用（Desktop/Web/iOS），不需要也不应该去适配第三方消息平台。Orbit 的"多端"是同一个应用的不同形态，而非不同的分发渠道。Gateway 的复杂性对 Orbit 没有价值。

### 4.4 Claude Code 的 Anthropic 深度绑定

Claude Code 专为 Claude 模型优化，利用 prompt caching、extended thinking 等 Anthropic 独家特性，甚至在 API 调用层直接使用 Anthropic 原生 SDK。这种深度绑定换来了极致的单模型体验，但对 Orbit 来说风险过高。Orbit 是用户的"人生操作系统"，需要长期稳定运行，不能因为某个 LLM 提供商调价、降质或停服而受到致命影响。Orbit 应该在能力层做模型无关的抽象，允许用户按场景选择模型。

### 4.5 Claude Code 的终端 UI 投入

Claude Code 自研了基于 React + Ink + Yoga 布局引擎的终端 UI 系统，实现了完整的 Vim 模式和虚拟 DOM 渲染。这种投入对于一个终端优先的开发者工具是合理的，但 Orbit 的主要交互场景是 GUI（Electron 桌面端、Web 端、iOS 原生端），终端 UI 不在产品范围内。Orbit 的 UI 投入应该集中在三端 GUI 的设计系统和共享组件上。

### 4.6 Hermes 的 RL 训练闭环

Hermes 内置了强化学习训练工具，支持数据生成与 RL 闭环，这是其"自我进化"愿景的体现。但 Orbit 不是一个 AI 研究平台，它的 Agent 进化应该通过记忆沉淀、规则学习和用户反馈来实现，而不是通过 RL 训练。RL 训练需要大量的计算资源和数据工程，这远超 Orbit 作为个人应用的合理复杂度边界。

## 5. 具体建议与行动项

### 建议一：设计 Agent Runtime 的四层编排架构

**目标**：将当前"Orchestrator + 子 Agent"的两级结构细化为四层，确保关注点分离和三端复用。

**具体方案**：

```
Intent Layer        → 意图解析、多端入口统一、MCP 请求适配
Orchestration Layer → 会话管理、子 Agent 路由、审批流、上下文构建
Execution Layer     → 核心 Agent 循环（async generator 管道）、模型调用
Capability Layer    → 能力分派、并发/串行决策、权限检查、MCP 桥接
```

每层定义清晰的输入/输出接口（TypeScript interface），层间通过 AsyncGenerator<AgentEvent> 流式传递事件。Orchestration Layer 额外维护会话状态和跨子 Agent 的协调上下文。这套分层应作为 `packages/agent-runtime` 独立包实现，不依赖任何 UI 框架，三端共享。

### 建议二：实现模型无关的 Provider 抽象与回退机制

**目标**：确保 Orbit 不被锁定在任何一家 LLM 提供商，同时为不同场景匹配最优模型。

**具体方案**：

定义 `ModelClient` 接口（`chat`、`stream`、`embed` 三个核心方法），针对 OpenAI、Anthropic、Google、Ollama 等实现适配器。在此之上建立路由层：

- **场景路由**：Research Agent 使用高推理能力模型，Task/Focus Agent 使用快速响应模型，隐私敏感内容使用本地 Ollama
- **故障回退**：主提供商超时/报错时自动切换到备选提供商（参考 Hermes 的 `error_classifier` + 回退链）
- **成本感知**：追踪每次调用的 token 消耗和成本，允许用户设置月度预算上限

这套抽象应作为 `packages/model-client` 独立包实现。

### 建议三：将 Capability Layer 设计为声明式注册 + Annotations 驱动的安全策略

**目标**：统一内部 Agent、UI、MCP 外部调用三条路径的能力定义和安全管控。

**具体方案**：

每个 capability 使用声明式 schema 定义（借鉴 Claude Code 的 TypeBox 强类型）：

```typescript
defineCapability({
  name: 'task.create',
  description: '创建新任务',
  input: TaskCreateInput,    // TypeBox schema
  output: TaskCreateOutput,
  annotations: {
    sideEffect: 'write',     // read-only | write | destructive
    dataScope: 'local',      // local | cloud | external
    requiresApproval: false,
  },
  handler: async (input, ctx) => { ... }
})
```

Agent Runtime 根据 `annotations` 自动决定：是否需要用户确认（`destructive` 操作必须确认）、是否记录审计日志（`write` 以上必须记录）、是否允许 MCP 外部调用（`external` 数据范围需要额外授权）。这套机制直接落地了第四条架构原则"Agent 深度融入，但必须有护栏"和第七条"MCP 是能力总线"。

### 建议四：建立三级记忆架构，定义记忆的注入与沉淀协议

**目标**：让 Agent 能够有效利用用户的历史数据，使"时间与记忆层"真正成为 Agent 决策的依据。

**具体方案**：

- **短期记忆**（Session Memory）：当前会话的对话历史、中间结果、已执行的工具调用。存储在内存中，会话结束后归档。
- **中期记忆**（Working Memory）：近期 Journal 条目、最近的 Review 结论、进行中项目的关键状态。存储在 Domain Data Layer 的事件表中，Agent 每次启动时预加载。
- **长期记忆**（Persistent Memory）：用户愿景、长期偏好、行为规律、能力使用模式。存储在专门的记忆索引中，通过语义搜索检索。

定义统一的记忆接口（参考 Hermes 的 `MemoryProvider`）：`prefetch(context)` 在每次模型调用前注入相关记忆；`commit(events)` 在会话结束时沉淀新记忆；`decay()` 定期衰减不再相关的短/中期记忆。

### 建议五：在 Agent Runtime 中引入生命周期钩子，支持审计、成本追踪与未来扩展

**目标**：在不修改核心循环的情况下，允许横切关注点（审计、成本、隐私、调试）以插件方式接入。

**具体方案**：

定义 8 个核心钩子：

| 钩子 | 时机 | 用途 |
|------|------|------|
| `onIntentParsed` | 意图解析完成后 | 记录用户意图日志、应用自定义规则 |
| `onContextBuilt` | 上下文构建完成后 | 隐私过滤、上下文审计 |
| `beforeModelCall` | 模型调用前 | 成本预估、预算检查 |
| `afterModelCall` | 模型响应后 | 成本记录、质量追踪 |
| `beforeCapabilityCall` | 能力调用前 | 权限验证、审计日志 |
| `afterCapabilityCall` | 能力执行后 | 结果验证、事件写入 |
| `onSessionEnd` | 会话结束时 | 记忆沉淀、Journal 自动记录 |
| `onError` | 任何错误发生时 | 错误分类、恢复策略 |

钩子注册为有序数组，支持同步和异步钩子，支持中断链（钩子返回 `abort` 可取消后续执行）。初始阶段由系统内建钩子实现审计和成本追踪；中期可开放给高级用户编写自定义自动化。

## 6. 总结

Hermes 和 Claude Code 代表了 AI Agent 工程化的两条路径：Hermes 走的是"最大覆盖面"——多模型、多平台、多后端，用广度换灵活性；Claude Code 走的是"最深单点突破"——专注终端开发者体验，用深度换极致。

Orbit 的定位决定了它既不应走 Hermes 的"无限扩展"路线（个人操作系统不需要 14 个消息平台），也不应走 Claude Code 的"单一绑定"路线（人生工具不能押注在一家 LLM 上）。Orbit 需要走一条**中间路线**：

- **从 Hermes 借鉴灵活性**：多模型 Provider 抽象、可插拔记忆系统、生命周期钩子、迭代预算与安全边界。这些设计保证了 Orbit 在模型生态快速变化的环境中的适应力和长期生命力。
- **从 Claude Code 借鉴工程品质**：异步生成器管道、四层编排架构、声明式能力定义、MCP 标准化协议、深度模块化与组件化。这些设计保证了 Orbit 在三端应用、多领域 Agent、丰富对象类型的复杂性下仍能保持代码质量和可维护性。
- **排除两者的极端选择**：不要 Hermes 的单体核心类和同步桥接，不要 Claude Code 的单模型绑定和终端 UI 投入，不要 Hermes 的 RL 训练和多平台 Gateway。

最终，Orbit 的总体架构应该是：**一个以异步生成器管道为骨架、以统一对象网络为血液、以声明式能力接口为肌肉、以多层记忆系统为神经、以模型无关抽象为心脏的 Agent-first 个人操作系统**。五条具体建议（四层编排、Provider 抽象、声明式能力、三级记忆、生命周期钩子）为这一蓝图提供了可落地的第一步。
