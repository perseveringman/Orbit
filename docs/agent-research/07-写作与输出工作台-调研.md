# 07. 写作与输出工作台 — Hermes / Claude Code 调研报告

> **调研范围**：结合 Hermes Agent 与 Claude Code 的架构分析，聚焦**输出渲染、编辑工具、流式生成**三大维度，为 Orbit「写作与输出工作台」提供设计参考。

---

## 1. Orbit 设计方案摘要

Orbit 07 方案将写作工作台定位为一条完整的"输出产线"，而非单一文档编辑器。其核心设计理念包含三个层面：

**第一，写作建立在对象网络之上。** 工作台消费的不是裸文本，而是来自 05 研究工作台的 `research_artifact`/`research_claim`、06 笔记系统的 `highlight`/`annotation note`/`synthesis note`，以及项目文档、Journal 片段等结构化对象。每一段输出内容都有可追溯的材料来源，形成从研究到写作的证据链。

**第二，AI 以阶段编排参与而非聊天常驻。** 方案明确拒绝"聊天即写作"的模式，将 Agent 拆分为 Writing Orchestrator、Material Curator、Outline Planner、Draft Composer、Style Editor、Channel Adapter、Publishing Assistant 七个角色，分五个阶段（研究 grounding → 大纲组装 → 初稿生成 → 润色审校 → 发布回流）参与。关键对象包括 `output_brief`、`material_board`、`outline_version`、`output_document`、`output_variant`、`revision_pass`、`publish_record` 等。

**第三，编辑器升级为输出工作台。** 采用三栏布局：左侧输出列表与模板入口、中间基于 TipTap 的 Block 编辑器（支持大纲视图、章节折叠、块级 AI 操作）、右侧材料面板与 Agent 面板。顶部/底部配备产线阶段轨，让用户随时知道自己处于哪个阶段。

方案还引入了 Voice Profile 对象化（将写作风格沉淀为可版本化的风格约束集）、Citation/Provenance 段落谱系追踪（记录每个 Block 从高亮、笔记到 AI 改写的完整演化路径）、以及同一材料底座派生多种输出变体（长文、newsletter、社媒、演讲稿）等差异化能力。

## 2. Hermes 可借鉴之处

### 2.1 多后端渲染与多渠道输出架构

Hermes 的 Gateway 架构支持 16+ 消息平台（Telegram、Discord、Slack、WhatsApp、企业微信、飞书等），其核心抽象是 `BasePlatformAdapter`——同一份内容通过不同适配器输出到不同渠道。这一设计哲学与 Orbit「同一材料底座 → 多种输出变体」的理念高度吻合。

Orbit 可以借鉴这一**平台适配器模式**来设计 Channel Adapter 层：为每种发布渠道（Medium、Substack、微信公众号、Twitter/X、LinkedIn 等）实现独立的格式化适配器，共享同一份 `output_document` 数据源，只在最终渲染阶段做渠道特有的格式转换。具体来说，Hermes 的 `PlatformConfig` 包含消息长度限制、富文本支持程度、媒体附件规格等平台元数据，Orbit 的 `output_variant` 也应在对象层面携带类似的渠道约束信息（字数上限、标题规范、CTA 格式、封面尺寸等），使 Agent 在生成变体时能够精确遵守渠道规则。

### 2.2 流式输出的行缓冲与盒子渲染模式

Hermes 的流式输出采用"逐行缓冲 + 实时盒子渲染"方案：`_stream_delta()` 回调接收 LLM 的 token 增量，通过行缓冲机制避免不完整行的渲染，并用响应盒子（`╭─ ⚕ Hermes ───╮ ... ╰───╯`）和推理盒子（独立的 `┌─ Reasoning ─┐ ... └───┘`）在视觉上区分不同类型的输出流。

这种**分区盒子渲染**的思路对 Orbit 的 Block 编辑器中的流式初稿生成很有启发。当 Draft Composer 按章节生成草稿时，每个章节的生成过程可以用类似的"生成中盒子"包裹，内部展示实时 token 流，外部显示章节标题与进度。生成完成后，盒子"落定"为正式的 Block 结构，挂载 provenance 信息。这比整篇文档一次性吐出要可控得多——用户可以在某个章节生成中途暂停、调整大纲、或替换材料后重新生成该章节。

### 2.3 工具结果的三层预算系统

Hermes 的工具结果预算系统（`BudgetConfig`）实现了 pinned → tool_overrides → registry default 的三层优先级，对不同类型的工具输出设置不同的大小阈值。这种分层预算思路可以移植到 Orbit 的材料面板设计中。

在写作场景下，不同来源的材料天然有不同的"展示预算"：研究结论卡需要完整展示核心论点；高亮和引用块只需显示关键片段加上下文锚点；图片和表格资产需要缩略图预览加元数据。Orbit 的 `material_board` 可以为每种 `sourceObjectType` 配置默认的展示预算和预览策略，在材料面板中实现"按需展开"的渐进式信息暴露，而非一视同仁地铺陈所有材料全文。

### 2.4 开放的技能市场生态与模板化扩展

Hermes 的 Skills Hub 生态（支持 official/github/clawhub/lobehub 四个来源）和 SKILL.md 格式（YAML 前置信息 + Markdown 指令）提供了一种轻量级的能力扩展模式。Orbit 的写作模板系统可以参考这一设计——将文章模板、newsletter 模板、社媒模板、演讲稿模板定义为类似 SKILL.md 的可分享格式，包含结构规范、长度约束、受众定义、voice 覆盖层等元数据。用户可以创建、分享、安装不同的输出模板，Agent 在大纲组装和初稿生成阶段根据所选模板的约束执行。

### 2.5 XML 标签抑制与推理过程可视化

Hermes 在流式输出中实现了对 `<REASONING_SCRATCHPAD>`、`<think>`、`<reasoning>` 等 XML 标签的抑制，并通过 `show_reasoning=True` 开关将推理过程路由到独立显示盒子。这对 Orbit 写作 Agent 的透明性设计有直接参考价值。

当 Draft Composer 或 Style Editor 工作时，其内部推理过程（"为什么选择这个论证顺序""为什么删除这段冗余""为什么调整了这个语气"）应该以可折叠的方式暴露给用户，而非完全隐藏在 prompt 内部。这与 Orbit 设计方案中"Agent 学习 voice 的方式必须是显式可见的"原则一致。

## 3. Claude Code 可借鉴之处

### 3.1 AsyncIterator 流式管道与 React 状态驱动渲染

Claude Code 构建了一套完整的流式响应管道：`Stream<T>` 异步迭代器 → React 状态更新 → 虚拟 DOM diff → 最小化终端重绘。其中 `StreamingToolExecutor` 通过 `ToolStatus`（queued → executing → completed → yielded）状态机追踪每个工具的执行进度，并利用 `isConcurrencySafe` 声明实现并发安全工具的并行执行。

这一架构对 Orbit 写作工作台的**多阶段并行生成**极具启发。在写作场景中，Material Curator 推荐材料、Outline Planner 组装大纲、Draft Composer 生成章节这三个阶段虽然有逻辑依赖，但在同一阶段内部可以高度并行：例如多个章节可以并发生成，多个材料源可以并发检索。Orbit 可以借鉴 Claude Code 的 `StreamingToolExecutor` 模式，为写作 Agent 的各个子任务设计类似的并发调度器，每个子任务通过声明式的 `isConcurrencySafe` 属性决定是否可以并行执行。

更具体地说，Claude Code 的 `follow-scroll` 特性（内容流入时视口自动跟踪到底部，用户上翻时暂停）直接适用于 Orbit Block 编辑器中的流式初稿生成场景。当 Agent 正在生成第 3 章时，用户应该既能看到实时生成进度，又能自由回翻审阅已生成的第 1、2 章。

### 3.2 FileEditTool 的精确行级编辑与冲突检测

Claude Code 的 `FileEditTool` 实现了精确的行级编辑能力——不是整文替换，而是在文件的特定位置做精准的增删改，并内置冲突检测机制。每个工具还通过 `renderToolUseMessage` 和 `renderToolResultMessage` 方法自带 React 渲染逻辑，使编辑操作在 UI 层有清晰的视觉反馈。

这直接映射到 Orbit 写作工作台中 Agent 对文稿的**块级操作**需求。当 Style Editor 执行润色时，它不应该返回整篇改写后的文档，而应该精确到 Block 级别：哪个段落被压缩了、哪个转折被优化了、哪个引用被补充了。每次块级操作都应生成一条可审计的 `revision_pass` 记录。Claude Code 的 `isDestructive` 声明也值得借鉴——对于大段删改等破坏性操作，系统应自动触发确认流程，而非静默执行。

### 3.3 声明式工具行为特征与权限系统

Claude Code 为每个工具定义了 `isConcurrencySafe`、`isReadOnly`、`isDestructive` 等行为特征声明，并通过声明式规则引擎 + ML 分类器实现权限判定。这种"工具自描述其行为边界"的模式非常适合 Orbit 的 Agent 行为治理。

在 Orbit 写作场景中，不同 Agent 操作有不同的风险等级：Material Curator 推荐材料是只读操作（低风险）；Draft Composer 生成章节是写入操作但有 undo（中风险）；Style Editor 批量改写是破坏性操作（高风险）；Publishing Assistant 对外发布是不可逆操作（最高风险）。Orbit 可以为写作相关的每个 Agent 动作定义类似的行为特征声明，并据此决定是否需要用户确认、是否可以并发执行、以及失败后的回滚策略。

### 3.4 Markdown 渲染的 LRU 缓存与快速路径优化

Claude Code 的 Markdown 组件实现了一套精巧的缓存优化：先用正则快速判断文本是否包含 Markdown 语法（`MD_SYNTAX_RE`），不包含则跳过解析直接返回；包含则通过 `hashContent` 生成缓存键，命中 LRU 缓存则直接返回已解析的 token 树，未命中才调用 `marked.lexer` 解析并缓存结果。

Orbit 的 Block 编辑器在渲染用户已有内容（笔记、高亮、研究结论）和 Agent 生成内容时，都需要频繁的 Markdown/富文本解析。特别是在材料面板中，同一份材料可能在不同写作任务中被反复引用和渲染。借鉴 Claude Code 的缓存策略，Orbit 可以为材料预览和 Block 渲染实现类似的快速路径 + LRU 缓存机制，显著提升工作台的渲染性能。

### 3.5 SDK 模式与结构化 JSON 流

Claude Code 的 `--output-format stream-json` 和 `--input-format stream-json` 模式使其可以被其他应用程序化调用，输出结构化的 JSON 事件流。这种设计使得 Claude Code 不仅是一个终端应用，更是一个可嵌入的 Agent SDK。

Orbit 的写作工作台同样需要这种双重身份：它既是用户直接操作的 GUI 工作台，也需要为外部系统（MCP 客户端、Shortcuts/自动化脚本、第三方集成）暴露结构化的写作能力接口。方案中定义的 `draft.composeSection`、`revision.run`、`output_variant.generate` 等接口，应同时支持 GUI 内调用和结构化 API 调用，输出格式统一为可解析的事件流，而不仅仅是渲染后的富文本。

## 4. 不适用的设计

### 4.1 Hermes 的单体 Agent 类模式

Hermes 将几乎所有逻辑集中在一个 ~7500 行的 `Agent` 类中，通过 53 个构造函数参数和回调钩子扩展。这种"一个类统治一切"的架构虽然在灵活性上有优势，但对 Orbit 写作工作台这种明确需要多阶段、多角色协作的场景并不适用。Orbit 的方案已经正确地将 Agent 拆分为七个角色，应坚持管道化、模块化的架构，而非集中式的单体。

### 4.2 Claude Code 的 Vim 状态机

Claude Code 投入大量工程实现了完整的 Vim 模式状态机（支持 operator-motion 组合如 `d2w`、`ci(` 等）。这对终端开发者是核心价值，但 Orbit 的写作工作台面向知识工作者而非程序员，其编辑体验应基于 TipTap 的所见即所得模式，而非终端键盘驱动模式。Vim 的投入对 Orbit 没有直接价值。

### 4.3 Hermes 的 RL 训练闭环

Hermes 内置了 10+ 个 RL 训练工具，支持数据生成与强化学习闭环。这属于 AI 基础设施层的能力，与 Orbit 作为面向用户的知识写作产品定位不同。Orbit 的 Agent 能力进化应通过 Voice Profile 学习、用户反馈回流、发布效果分析等产品层面的闭环实现，而非底层模型训练。

### 4.4 Hermes 的概率采样工具集分发

Hermes 的 `toolset_distributions` 系统通过概率采样决定每次会话启用哪些工具集，主要服务于 RL 训练数据多样性。写作工作台需要的是确定性的阶段化工具编排——在材料推荐阶段启用检索工具，在草稿生成阶段启用写作工具，在发布阶段启用渠道适配工具——而非随机采样。

### 4.5 Claude Code 对 Anthropic API 的深度绑定

Claude Code 深度利用了 Anthropic 的 prompt caching、extended thinking 等专有特性。Orbit 作为面向通用场景的产品，不应锁定单一 LLM 提供商。写作 Agent 的模型层应保持抽象，允许用户选择不同的 LLM（OpenAI、Anthropic、本地模型等），这一点更应参考 Hermes 的多模型兼容架构。

## 5. 具体建议

### 建议一：为 Block 编辑器设计「流式章节生成器」，借鉴双项目的流式架构

**问题**：Orbit 方案要求"初稿生成以章节为单位进行"，但未详细设计章节级流式生成的 UX 和技术方案。

**建议**：综合 Hermes 的行缓冲盒子渲染和 Claude Code 的 AsyncIterator + React 状态驱动模式，在 TipTap Block 编辑器中实现「流式章节生成器」。技术层面，每个章节的生成启动一个独立的 `Stream<BlockDelta>` 异步迭代器，输出 Block 粒度的增量更新（新段落、新引用块、新数据表格等），编辑器通过 TipTap 的 transaction 机制实时接收并渲染。UX 层面，参考 Hermes 的盒子模式，每个正在生成的章节用可视化的"生成中"边框包裹，显示当前使用的材料来源、预估进度、以及"暂停/重新生成/跳过"操作按钮。生成完成后，边框消退，Block 落定为正式内容并自动挂载 `block_provenance`。多个无依赖关系的章节可以借鉴 Claude Code 的 `StreamingToolExecutor` 并发机制进行并行生成。

### 建议二：为 Agent 操作实现「声明式行为特征 + 分级确认」机制

**问题**：方案列出了 Agent 的五条行为边界，但缺乏系统化的执行保障机制。

**建议**：借鉴 Claude Code 的工具行为声明模式，为 Orbit 写作 Agent 的每个操作定义标准化的行为特征接口：

```typescript
interface WritingAgentAction {
  name: string;
  phase: 'grounding' | 'outlining' | 'drafting' | 'revising' | 'publishing';
  isReadOnly: boolean;        // Material Curator 推荐 = true
  isDestructive: boolean;     // 批量改写 = true, 单段润色 = false
  isReversible: boolean;      // Block 编辑 = true, 外部发布 = false
  isConcurrencySafe: boolean; // 多章节并行生成 = true
  requiresConfirmation: 'never' | 'first_time' | 'always';
  affectedObjects: string[];  // 受影响的对象 ID 列表
}
```

系统根据这些声明自动决定：是否需要弹出确认对话框、是否生成 undo 快照、是否可以与其他操作并行、以及失败后的回滚策略。这比逐条硬编码边界规则更可维护、更可扩展。

### 建议三：材料面板实现「渐进式展示 + 分类型预览预算」

**问题**：方案定义了 `material_board` 的丰富字段结构，但未详细设计材料在面板中的渲染策略。

**建议**：借鉴 Hermes 的三层预算系统和 Claude Code 的大结果磁盘持久化 + 预览方案，为材料面板实现分类型的渲染预算：

| 材料类型 | 默认展示 | 展开展示 | 预览策略 |
|---------|---------|---------|---------|
| `research_claim` | 核心论点 + 可信度 | 完整论证链 + 支撑来源 | 全文摘要 |
| `highlight` | 高亮片段 + 来源标题 | 上下文段落 + 批注 | 锚点定位 |
| `synthesis_note` | 综合摘要（前 200 字） | 完整笔记 | LRU 缓存渲染 |
| `data_asset` | 表格/图表缩略图 | 交互式数据视图 | 懒加载 |
| `historical_output` | 标题 + 复用片段 | 完整段落 + 结构模板 | 按需加载 |

配合 Claude Code 的 Markdown LRU 缓存优化，对频繁访问的材料预览结果做缓存，避免在拖拽排序、分组切换等高频操作时的重复渲染开销。

### 建议四：渠道变体生成采用「适配器模式 + 约束元数据」架构

**问题**：方案提出了 `output_variant` 概念，但 Channel Adapter 的具体架构未深入。

**建议**：综合 Hermes 的 `BasePlatformAdapter` 多平台适配模式和 Claude Code 的声明式工具构建模式（`buildTool` 工厂函数 + 行为特征声明），为 Orbit 设计渠道适配器体系：

```typescript
interface ChannelAdapter {
  channel: 'newsletter' | 'social_twitter' | 'social_linkedin' | 'talk' | 'script' | 'wechat';
  constraints: {
    maxLength: number;
    structureTemplate: string;       // 渠道特定的结构模板
    voiceOverlay: Partial<VoiceProfile>;  // 渠道特定的 voice 覆盖层
    mediaRequirements: MediaSpec;
    ctaFormat?: string;
  };
  transform(doc: OutputDocument, materials: MaterialBoard): OutputVariant;
  preview(variant: OutputVariant): RenderedPreview;
  validate(variant: OutputVariant): ValidationResult[];
}
```

每个适配器是独立模块，可以通过 MCP 协议动态注册新渠道（参考 Hermes 的 MCP 工具动态发现机制和 Claude Code 的 MCP annotations → 行为声明映射），使第三方开发者能为新平台贡献适配器而无需修改核心代码。

### 建议五：Voice Profile 学习采用「样本驱动 + 显式确认 + 版本快照」模式

**问题**：方案详细定义了 Voice Profile 的字段和学习原则，但未明确技术实现路径。

**建议**：综合 Hermes 技能系统的渐进式信息暴露（`SkillReadinessStatus`：available → setup_needed → unsupported）和 Claude Code 的 5 层 CLAUDE.md 配置叠加机制，为 Voice Profile 学习设计三阶段流程：

1. **样本采集阶段**：用户授权历史输出作为学习语料，系统展示"即将分析哪些文本"的透明清单（类似 Hermes 技能安装前的隔离扫描），用户可以逐篇确认或排除。
2. **特征提取阶段**：Agent 从授权样本中抽取风格特征，生成结构化的 voice summary（基础语气、句法特征、修辞习惯、禁忌项），以可编辑的表单而非自由文本形式呈现给用户确认。
3. **版本固化阶段**：确认后的 Voice Profile 保存为带版本号的快照（参考 Claude Code 的配置叠加机制），支持"主 voice + 渠道覆盖层"的组合，支持回滚到任意历史版本，支持查看每个版本的学习语料来源。

运行时，Voice Profile 作为 prompt 层级的配置注入（类似 CLAUDE.md 的叠加方式），Style Editor 在每次润色时都引用明确的 profile 版本，确保输出风格的可复现性。

---

## 6. 总结

Hermes Agent 和 Claude Code 虽然都不是写作工具，但它们在**输出渲染、编辑工具、流式生成**三个维度的工程实践对 Orbit 写作工作台有丰富的启发。

从 Hermes 中，Orbit 应重点借鉴其**多渠道适配器模式**（Gateway + BasePlatformAdapter）、**流式行缓冲与盒子渲染**（分区可视化不同类型的生成流）、**三层工具结果预算系统**（为不同类型材料设置差异化的展示预算）、以及**开放技能市场的模板化扩展思路**（将写作模板做成可分享的标准格式）。Hermes 的核心优势在于开放性和广度，这对 Orbit 渠道变体生成和模板生态有直接价值。

从 Claude Code 中，Orbit 应重点借鉴其**AsyncIterator 流式管道 + React 状态驱动渲染**（构建章节级流式生成器）、**FileEditTool 的精确行级编辑与冲突检测**（实现 Block 级 Agent 操作与 revision 追踪）、**声明式工具行为特征**（为每个 Agent 操作标注只读/破坏性/可并发等属性）、**Markdown 渲染的 LRU 缓存优化**（提升材料面板渲染性能）、以及 **SDK 模式的结构化 JSON 流**（使写作能力可被外部系统程序化调用）。Claude Code 的核心优势在于工程深度和精细度，这对 Orbit 编辑器内的 Agent 交互体验至关重要。

两个项目共同验证的一个重要原则是：**AI 生成的输出必须是结构化的、可追踪的、可操作的，而非一次性的文本 blob**。Hermes 通过工具结果类型化和预算控制实现这一点，Claude Code 通过工具行为声明和类型化错误分类实现这一点。Orbit 应将这一原则贯穿到写作工作台的每个环节——从材料面板的结构化预览，到 Block 编辑器的带 provenance 的增量更新，到渠道变体的约束化生成，到发布记录的结构化回流。

最终，Orbit 写作工作台的技术差异化不在于"AI 写得更快"，而在于**"AI 写得可追溯、可审计、可组合"**——每一段输出都知道它从哪来、经过了什么变换、为什么是这种风格、适配了什么渠道，并且这一切都沉淀为稳定的对象而非消散在聊天历史中。
