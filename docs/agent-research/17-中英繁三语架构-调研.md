# 17-中英繁三语架构 调研报告

> **调研基础**：Hermes Agent（Python，767+ .py 文件）与 Claude Code（TypeScript，1884+ .ts/.tsx 文件）的架构与设计深度对比分析
>
> **对应设计方案**：`docs/orbit-reboot/17-中英繁三语架构.md`

---

## 1. Orbit 设计方案摘要

Orbit 的三语架构（`zh-CN`、`en-US`、`zh-TW`）不是传统意义上的"事后补语言包"，而是一套从系统边界起就内建的基础设施。方案明确将语言能力拆分为五个层次：产品界面语言、内容对象语言、阅读翻译语言、Agent 输出语言、检索与排版语言。核心设计理念在于**显式区分各层语义**——`app_locale`、`content_locale`、`agent_output_locale`、`translation_target_locale`、`search_locale` 各司其职，而不是用一个 `locale` 字段混写全部语义。

方案的关键架构决策包括：

- **Locale 体系拆分**：UI locale 不代表内容语言，内容语言不等于翻译目标，Agent 输出语言独立于 UI locale。
- **共享主干 + 平台适配层**：三端（Electron / iOS / Web）共享 `i18n-core`、`i18n-resources`、`termbase`、`linguistics`、`agent-language` 等包，平台只做 adapter 覆写。
- **术语表治理**：通过独立的 `termbase` 系统统一管理概念 ID、首选词、禁用译法和别名，同时服务 UI 翻译校验、Agent 生成约束和搜索扩展。
- **内容语言检测双层识别**：对象级（文章/笔记主语言）+ 片段级（段落/block/高亮 quote）。
- **Reader 翻译层**：原文只读，译文作为可版本化的派生层保存，支持多目标语言并存。
- **Agent 语言策略**：内部控制面语言稳定（system prompt、schema 不随语言切换），表面层输出按 `agent_output_locale` 本地化。
- **三层搜索索引**：原文倒排、归一化语言索引（简繁/全半角/别名归一）、语义 embedding 索引并存。

这是一个极为完整的语言架构设计，覆盖了从 UI 渲染到 Agent 编排、从内容管理到检索排版的全链路。

## 2. Hermes 可借鉴之处

### 2.1 多平台消息网关与 Locale 适配模式

Hermes 最独特的架构之一是其 **Gateway 多平台适配层**。它通过一个 `BasePlatformAdapter` 抽象类，统一适配了 16+ 消息平台（Telegram、Discord、Slack、WhatsApp、企业微信、钉钉、飞书等）。每个平台有各自的消息格式、字符编码、文本长度限制和 Markdown 子集支持。这种"一个核心引擎 + 多个平台适配器"的模式，与 Orbit 方案中"**共享主干 + 平台适配层**"的三端语言资源组织理念高度契合。

对 Orbit 的启发在于：正如 Hermes 的命令注册表作为单一事实来源（Single Source of Truth）同时驱动 CLI 处理器、Gateway 分发器、帮助系统和 Telegram/Slack 菜单，Orbit 的 `i18n-resources` 包也应当成为三端（Desktop/iOS/Web）的唯一翻译真相源，各平台只做必要的格式适配（如 iOS 的 `Localizable.strings` 应由编译管线从共享资源自动生成），而不是独立维护翻译文案。

### 2.2 数据驱动的皮肤引擎与 Design Token 思路

Hermes 的 YAML 皮肤系统（`SkinConfig`）将颜色、品牌、图标、表情等视觉元素全部抽象为数据驱动的配置。用户可以在 `~/.hermes/skins/` 目录创建任意 YAML 皮肤文件，系统通过 `load_skin()` 按优先级加载——先用户自定义，后内置回退。皮肤中定义了 30+ 样式类，所有 TUI 渲染都从皮肤配置继承颜色。

这一模式对 Orbit 的 **locale-aware design token** 设计有直接参考价值。Orbit 方案 3.8 中要求将日期格式、字体回退链、行高字重、截断换行策略等视觉差异"上收到 design token 与 formatter 层"。Hermes 的皮肤引擎证明了一点：将 UI 表现层的差异（颜色、布局、品牌元素）抽象为声明式数据结构并通过配置文件管理，可以有效避免各端各处"临时打补丁"。Orbit 可以将 locale-specific 的排版参数（如中文行高系数、英文断词策略、按钮最小宽度）纳入类似的 token 配置体系，由共享包统一分发到三端。

### 2.3 集中式命令注册表模式与术语表治理的呼应

Hermes 的 `COMMAND_REGISTRY` 将 70+ 命令集中定义在一个数据结构中（包含 name、description、category、aliases、cli_only/gateway_only 标记），然后由 CLI 处理器、Gateway 分发器、帮助系统和自动补全系统分别消费。这种"一处定义、多处消费"的模式，天然适配 Orbit 的术语表治理需求。

Orbit 的 `termbase` 需要一个概念 ID 同时服务 UI 翻译校验、Agent 生成约束和搜索扩展。Hermes 的命令注册表提供了一个具体的工程模式参考：**用一个冻结的数据类（`@dataclass(frozen=True)`）定义核心实体，附带丰富的元数据字段，然后让各消费方（翻译校验管线、Agent prompt builder、搜索扩展引擎）各取所需。** 这比为每个消费场景各自维护一份术语数据可靠得多。

### 2.4 插件化记忆系统与语言检测/翻译提供者的可插拔设计

Hermes 的记忆系统支持 8 个外部提供者插件（Honcho、Mem0、ChromaDB 等），通过 `MemoryProvider` 抽象基类实现统一接口。类似地，其 TTS 支持 5 个提供者、STT 支持 3 个提供者，每个都通过配置切换。

Orbit 方案明确声明"本专题不负责决定哪家翻译模型或分词器厂商"，只定义能力接口与治理边界。这与 Hermes 的多提供者策略不谋而合。Orbit 的 `language.detect()`、`translation.generate()` 等能力接口可以参照 Hermes 的提供者模式，定义统一的 `LanguageDetectionProvider` 和 `TranslationProvider` 抽象，允许底层实现在不同的翻译引擎（模型 API、本地分词器、MCP 外部工具）之间灵活切换，而上层消费方无需感知具体实现。

### 2.5 多渠道一致性经验：Gateway 如何在异构平台保持体验一致

Hermes Gateway 面对的挑战是：同一条 Agent 回复需要在 Telegram（支持 Markdown）、Slack（支持 Block Kit）、WhatsApp（纯文本 + 有限格式）、钉钉（自有消息格式）等完全不同的渲染环境中保持语义一致。它通过 `BasePlatformAdapter` 的 `format_response()` 方法在各平台间做适配。

这与 Orbit 面临的跨端语言一致性挑战类似：同一条 UI 文案需要在 Electron（Web 渲染）、iOS（原生 UIKit/SwiftUI）和 Web（浏览器）三个完全不同的渲染引擎中保持翻译一致和视觉协调。Hermes 的经验表明，**核心文案（包括 Agent 回复模板）必须在平台适配之前就定义好语言版本**，平台适配只处理格式降级（如 Markdown → 纯文本），不处理翻译逻辑。

## 3. Claude Code 可借鉴之处

### 3.1 多层叠加配置与语言策略优先级

Claude Code 最具启发性的设计之一是其 **5 层 CLAUDE.md 配置叠加机制**：本地项目 → 用户级 → 团队级 → 企业级 → 全局默认。每一层可以覆写或补充上一层的设置。这种多层叠加、逐级覆写的模式，与 Orbit 方案中 Agent 语言策略的优先级链高度对齐：

> `用户显式指令 > 当前空间语言策略 > 当前对象语言 > 用户长期偏好 > app_locale`

Claude Code 的实现方式是：每个配置源都被解析为同一类型的设置对象，运行时通过合并函数按优先级叠加。这为 Orbit 的 `LanguageContext` 生成提供了直接参考——可以将语言策略的各层源（用户指令、空间策略、对象语言、长期偏好、app_locale）解析为统一的 `LanguagePolicy` 对象，运行时按优先级合并，最终输出确定的 `LanguageContext`。这种声明式的策略合并，比命令式的 if-else 判断链更易维护和测试。

### 3.2 语义化主题系统与 locale-aware Design Token

Claude Code 的 `ThemeProvider` 使用语义化颜色令牌（`success`、`error`、`diffAdded`、`permission`），而非具体色值名。同时支持亮色/暗色/跟随系统三种模式，通过 React Context 向下传递主题数据。其设计系统组件库（`ThemedBox`、`ThemedText`、`StatusIcon` 等 16 个基础组件）都从主题 Context 消费语义令牌。

对 Orbit 而言，这种语义化 token + Context 分发的模式可以直接扩展为 **locale-aware design token**。具体来说：在 Claude Code 的 `Theme` 类型上增加 locale 维度的 token（如 `fontFamily.primary`、`lineHeight.body`、`truncation.strategy`、`buttonMinWidth`），使得同一个组件在不同 locale 下自动获取适当的排版参数。三端共享的 design token 包可以导出 locale-indexed 的 token map，各端的 ThemeProvider/DesignTokenProvider 负责注入当前 locale 对应的具体值。

### 3.3 异步生成器管道与 Agent 语言守卫的集成点

Claude Code 的核心数据流采用 async generator 管道架构：`QueryEngine.submitMessage()` → `query()` → `queryLoop()` → `runTools()`，每一层通过 `yield` 传递事件。这种管道模式天然提供了**拦截与变换节点**——可以在管道的任意层级插入中间件，而不需要修改上下游。

对 Orbit 的 Agent 语言守卫（方案 4.3）而言，这是一个理想的架构参考。当 Agent 输出偏离 `agent_output_locale` 策略时，Orbit 需要做后处理守卫（检测语言、轻量重写、保留引用原文、记录决策日志）。如果 Orbit 的 Agent 编排层采用类似的 generator 管道模式，语言守卫可以作为管道中的一个 `yield*` 层优雅注入：

```
orchestrator → languageGuard(yield* agentOutput) → userPresentation
```

每个 yield 出的事件都经过语言检测和潜在的本地化重写，而不需要在 Agent 核心逻辑中硬编码语言处理。这与方案中"子 Agent 内部用稳定控制语言执行，交回 orchestrator 时附带 `response_locale` 元数据"的设计完美契合。

### 3.4 特性门控（Feature Flag）与语言能力的渐进发布

Claude Code 大量使用 `feature()` 函数和 GrowthBook 特性开关来控制功能的渐进发布。语音功能就是一个典型案例——多重门控（`hasVoiceAuth()` + `isVoiceGrowthBookEnabled()`）确保只在条件完全满足时启用。

Orbit 的三语架构涉及多个复杂子系统（语言检测、术语表、Reader 翻译层、简繁归一搜索、Agent 语言策略），不可能一次性全部就绪。Claude Code 的特性门控模式提供了一个务实的发布策略参考：可以对每个语言子能力独立设置 feature flag（如 `TRILINGUAL_SEARCH`、`READER_TRANSLATION`、`AGENT_LANGUAGE_GUARD`），按准备度逐步开启，而不是等所有子系统完备后才做大爆炸式发布。

### 3.5 模块化工具目录与语言包的 namespace 组织

Claude Code 的每个工具都有独立的目录结构（`tools/BashTool/`、`tools/FileEditTool/` 等），目录内包含实现、测试和类型定义。这种"目录即模块"的组织方式使得每个工具的边界清晰、可独立测试。

Orbit 方案要求翻译资源按"领域 namespace"组织（`common`、`reader`、`search`、`agent`、`settings`、`billing`），而不是按平台复制。Claude Code 的目录组织模式可以直接借鉴：每个 namespace 一个目录，目录内包含三语 JSON 文件、该 namespace 的术语表子集、以及自动校验脚本。这样每个 namespace 可以独立维护、独立校验、独立测试翻译完整性，而不是靠一个巨大的 JSON 文件承载所有翻译。

### 3.6 Zustand 状态管理与 locale 状态的响应式分发

Claude Code 使用 Zustand store + React Context 管理应用状态，组件通过 `useStore()` hook 自动订阅状态变化。当状态更新时，只有依赖该状态切片的组件重新渲染。

Orbit 需要在三端响应式地分发 locale 相关状态（`app_locale` 变更、`agent_output_locale` 切换、术语表更新）。Zustand 的细粒度订阅模式可以避免 locale 切换时的全局重渲染——只有消费特定 locale 切片的组件（如日期 formatter、翻译组件、Agent 输出面板）需要响应变化。这比传统的 Context value 变更触发全树重渲染要高效得多。

## 4. 不适用的设计

### 4.1 Hermes 的单体 Agent 类模式不适合语言系统

Hermes 的 `AIAgent` 类是一个约 7500 行的单体文件，53 个构造函数参数，所有核心逻辑集中在一个类中。这种"一个类统治一切"的模式虽然在快速迭代阶段有效，但对于 Orbit 的语言系统而言是反模式。语言系统涉及 locale 解析、术语表查询、语言检测、翻译生成、搜索扩展、Agent 语言策略等多个正交关注点，如果集中到一个类中会导致修改任何一个子能力都影响全局。Orbit 方案已经正确地将语言能力拆分为 `i18n-core`、`termbase`、`linguistics`、`agent-language` 等独立包。

### 4.2 Claude Code 的 Anthropic 深度绑定策略不适用

Claude Code 的 prompt caching、extended thinking 等特性高度依赖 Anthropic 专有 API。其 Agent 语言行为（如 CLAUDE.md 的解析与应用）也是 Anthropic 模型特定的。Orbit 的 Agent 语言策略需要兼容多种 LLM 提供者（按方案 3.6，system prompt 稳定、tool registry 不随语言改 schema），因此不应依赖任何单一模型的专有语言能力，而应在编排层（orchestrator）独立实现语言策略逻辑。

### 4.3 Hermes 的正则匹配安全模式不适合术语校验

Hermes 使用约 106 条正则模式进行安全命令检测。虽然正则在命令安全检测场景有效，但对于 Orbit 的术语表校验（检查新文案是否违背首选术语、Agent 输出是否使用禁用译法）而言过于脆弱。中文语境下的术语匹配需要考虑分词边界、简繁变体、同义词映射等语言学因素，纯正则无法胜任。Orbit 应采用基于分词器 + 术语表索引的匹配方案。

### 4.4 两个项目都缺乏原生 i18n 支持——其经验是间接的

需要明确指出，无论 Hermes 还是 Claude Code，其产品 UI 本身都没有实现多语言国际化。Hermes 的 CLI 界面是英文的；Claude Code 的所有交互文案也是英文的。因此，它们在 i18n 资源管理、翻译工作流、locale-aware 排版等 Orbit 核心需求上没有直接的实现可参考。本调研报告的借鉴更多来自其**架构模式**（多层配置、管道拦截、提供者抽象、数据驱动 token）在语言系统场景的迁移应用，而非现成的 i18n 代码。

### 4.5 Hermes 的 Gateway 多平台覆盖策略过于发散

Hermes 支持 16+ 消息平台的策略虽然令人印象深刻，但其"广度优先"的哲学与 Orbit"三端深度一致"的语言目标有冲突。Orbit 不需要适配 Telegram、Discord 等外部平台的消息格式，而是需要在自有的 Electron、iOS、Web 三端之间保证翻译完整性和排版一致性。Hermes 的平台适配经验可以参考其"核心内容 + 平台格式降级"的分层思路，但具体的多平台架构实现不适合照搬。

## 5. 具体建议

### 建议一：采用 Claude Code 式多层策略合并实现 LanguageContext 生成

**问题**：Orbit 方案 4.1 要求编排层在每次 Agent 执行前生成 `LanguageContext`，涉及 app locale、空间策略、对象语言、用户要求、翻译目标、术语表 domain 共 6 个输入源。如果用命令式 if-else 实现优先级判断，代码将迅速变得复杂且难以测试。

**建议**：参照 Claude Code 的 5 层 CLAUDE.md 配置叠加机制，将 `LanguageContext` 生成设计为**声明式策略合并管线**。每个语言策略源（用户指令、空间配置、对象元数据、用户长期偏好、全局默认）都产出一个 `Partial<LanguagePolicy>` 对象，运行时按优先级从高到低 merge。合并函数只需处理"有值则覆写、无值则穿透"的简单逻辑。这使得每一层策略源可以独立测试，新增策略源时只需插入合并链而不修改已有逻辑。

```typescript
// 伪代码示例
const policy = mergeLanguagePolicies([
  parseUserExplicitInstruction(userMessage),  // 最高优先
  getWorkspaceLanguagePolicy(workspaceId),
  detectObjectLanguage(currentObject),
  getUserLongTermPreference(userId),
  getAppLocaleDefault(),                       // 最低优先
]);
const context: LanguageContext = resolveContext(policy);
```

### 建议二：将术语表建模为 Hermes 式冻结注册表 + 多消费方投影

**问题**：Orbit 的 `termbase` 需要同时服务三个截然不同的消费方——UI 翻译校验、Agent prompt 注入、搜索查询扩展。三者对术语数据的查询模式、响应格式和性能要求各不相同。

**建议**：参照 Hermes 的 `COMMAND_REGISTRY`（一个冻结数据类驱动 CLI、Gateway、帮助系统、补全系统等多个消费方），将 `termbase` 设计为**单一权威数据源 + 多视图投影**。核心数据存储为包含 `concept_id`、`domain`、`preferred_terms`、`aliases`、`forbidden_terms`、`definition` 等完整字段的不可变记录集合。在此基础上，为每个消费方构建专用视图：

- **UI 校验视图**：按 namespace 索引，支持快速查找"某个 key 的翻译是否使用了 forbidden_terms"。
- **Agent prompt 视图**：按 domain 过滤，输出可直接嵌入 system prompt 的术语表片段（首选词 + 使用说明）。
- **搜索扩展视图**：按同义词图谱组织，支持从任何 alias/variant 查找到 canonical term 及其所有简繁映射。

视图从核心数据自动派生，由编译管线（或运行时缓存）生成，避免人工同步三份数据。

### 建议三：在 Agent 编排管道中植入语言守卫中间件

**问题**：Orbit 方案 4.3 要求 Agent 输出语言守卫能检测偏离、轻量重写、保留引用原文、记录决策日志。这些步骤如果硬编码在 Agent 核心逻辑中，会严重耦合语言处理与推理逻辑。

**建议**：参照 Claude Code 的 async generator 管道模式（`query → queryLoop → runTools`，每层 yield 事件），将语言守卫实现为编排管道中的一个**透明中间件层**。具体做法：

1. Agent 内部执行使用稳定的控制面语言（英文 system prompt + 英文 tool schema），产出带有 `response_locale` 元数据的原始输出。
2. 语言守卫中间件接收原始输出，执行：(a) 检测输出语言是否符合 `agent_output_locale`；(b) 若不符合，调用轻量本地化重写（保留代码、引用、专名）；(c) 将语言决策（检测结果、是否重写、术语命中情况）写入 event log。
3. 经过守卫的输出继续流向下游的 UI 渲染层。

这种管道式设计使得语言守卫可以独立开关（通过 feature flag，参照 Claude Code 的 GrowthBook 模式）、独立测试、独立迭代，不影响 Agent 核心推理逻辑。

### 建议四：以 Hermes 的提供者抽象模式设计语言检测与翻译的可插拔后端

**问题**：Orbit 方案明确声明不绑定具体翻译模型或分词器厂商，但需要为 `language.detect()`、`translation.generate()`、`search.expandMultilingualQuery()` 等接口提供实际实现。

**建议**：参照 Hermes 的多提供者架构（STT 3 个提供者、TTS 5 个提供者、记忆 8 个插件，全部通过配置切换），为 Orbit 的语言能力接口设计统一的 **Provider 抽象层**：

```typescript
interface LanguageDetectionProvider {
  detect(input: string | ContentSegment): Promise<LanguageDetectionResult>;
  readonly supportedScripts: Set<string>;
  readonly confidence: 'high' | 'medium' | 'low';
}

interface TranslationProvider {
  translate(request: TranslationRequest): Promise<TranslationResult>;
  readonly supportedPairs: LocalePair[];
  readonly mode: 'machine' | 'glossary-aware';
}
```

每个接口可以有多个实现（本地 CLD3 检测器 vs LLM 检测器、MCP 外部翻译工具 vs 内置模型翻译），通过配置或运行时策略选择。关键原则是：**provider 只产出候选结果，最终的对象化、缓存、权限审计与用户呈现仍由 Orbit 内部语言层控制。**

### 建议五：利用 Claude Code 的 feature flag + 渐进发布策略管理三语子系统上线

**问题**：Orbit 的三语架构涉及至少 7 个子系统（locale 解析、术语表、语言检测、翻译层、Agent 语言策略、搜索扩展、排版 token），它们之间有依赖但成熟度不同。如果等全部完成再发布，MVP 周期将极长。

**建议**：参照 Claude Code 的 `feature()` 门控 + GrowthBook 远程控制模式，为每个语言子能力设置独立的 feature flag。建议的渐进发布路径：

1. **Phase 1**（最小可用）：`app_locale` 切换 + 共享 UI 翻译资源 + 基础日期/数字 formatter → flag: `TRILINGUAL_UI`
2. **Phase 2**（内容感知）：对象级语言检测 + `content_locale` 元数据写入 → flag: `CONTENT_LANGUAGE_DETECTION`
3. **Phase 3**（阅读增强）：Reader 翻译层 + 术语表基础版本 → flag: `READER_TRANSLATION`
4. **Phase 4**（搜索升级）：简繁归一索引 + 术语别名扩展 + 多语言搜索 → flag: `TRILINGUAL_SEARCH`
5. **Phase 5**（Agent 语言）：`LanguageContext` 生成 + Agent 输出语言守卫 → flag: `AGENT_LANGUAGE_POLICY`

每个 Phase 可以独立开发、独立测试、独立上线。flag 关闭时系统降级到上一阶段的行为，避免半成品语言功能影响用户体验。

---

## 6. 总结

Hermes Agent 和 Claude Code 虽然都不是多语言产品，但它们在各自领域展现出的架构模式——Hermes 的"一处定义、多处消费"注册表模式、多提供者可插拔架构、数据驱动的皮肤/token 系统，以及 Claude Code 的多层配置叠加、async generator 管道拦截、语义化设计系统、feature flag 渐进发布——都可以被有效迁移到 Orbit 的三语架构中。

核心启示可以归纳为三条原则：

**第一，语言策略应当是声明式的、可合并的，而非命令式的。** Claude Code 的 5 层 CLAUDE.md 叠加机制证明，将配置/策略建模为多层数据源 + 合并函数，远比 if-else 判断链更健壮。Orbit 的 `LanguageContext` 生成应当遵循同样的模式。

**第二，语言能力的实现应当是可插拔的、接口化的。** Hermes 的多提供者架构证明，将具体实现（翻译引擎、分词器、语言检测器）藏在抽象接口之后，通过配置切换，既保持了灵活性，又避免了供应商锁定。Orbit 方案已经正确定义了能力接口，关键是在实现层坚持 provider 抽象。

**第三，复杂系统应当支持渐进式发布。** Claude Code 的 feature flag 体系证明，即使是深度集成的功能（如语音交互），也可以通过多重门控实现可控发布。Orbit 的三语架构涉及五层语言语义和七个以上子系统，一次性完美交付几乎不可能，渐进式发布是务实的工程选择。

最终，Orbit 的三语架构之所以不同于传统 i18n，在于它将语言能力提升到了与对象网络和 Agent 编排同等的基础设施层级。Hermes 和 Claude Code 提供的不是 i18n 代码模板，而是在"如何将一个横切关注点优雅地融入复杂系统架构"这一更高层级问题上的工程经验。这些经验——声明式策略合并、管道式中间件拦截、数据驱动的 token 配置、可插拔的提供者抽象、渐进式的能力发布——构成了 Orbit 落地三语架构的工程方法论基础。
