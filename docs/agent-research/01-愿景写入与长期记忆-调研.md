# 01. 愿景写入与长期记忆 —— 基于 Hermes Agent 与 Claude Code 的调研报告

> **调研目标**：从 Hermes Agent 和 Claude Code 两个成熟 AI Agent 项目的记忆系统与上下文管理实践中，提取对 Orbit「愿景写入与长期记忆」设计方案的可借鉴点、不适用之处，以及具体的工程行动建议。

---

## 1. Orbit 设计方案摘要

Orbit 的「愿景写入与长期记忆」方案围绕一个核心理念展开：**愿景是用户亲手写下的方向感与身份感，而非系统用来审计的 KPI 结构**。整个设计遵循四条硬原则——用户首写与最终确认、文本对象而非指标对象、愿景进入上下文但不等于记忆主表、愿景位于分层顶端。

在对象模型层面，方案采用「Markdown 原文 + 关系型元数据 + 追加式版本」三层结构。愿景（Vision）作为一等对象，拥有独立的版本线（VisionVersion），每个版本保留用户原始 Markdown 文本和面向 Agent 的压缩摘要（`summaryForAgent`）。愿景通过 `links` 表与 directive、project、journal 等对象建立 `derived_from`、`aligned_to`、`reflects_on` 等语义关联。

Agent 读取愿景的路径被精心设计为三层：

- **system prompt 层**：仅注入使用规则（如"不得基于愿景输出审计式评分"），不放原文；
- **memory context 层**：注入当前生效版本的摘要、directive 列表、最近 reaffirm 时间等受控字段；
- **journal/review 层**：回顾场景下可读取原文与历史版本。

提醒策略采用「温和邀请」而非「偏离警报」——仅在选择、创建、回顾、主动求助四类节点触发，且话术规避审计感。隐私方面，原文默认本地存储，云端推理仅上传摘要，完整原文上传需显式授权。

这一方案的核心挑战在于：如何让一段**低频变化、高度个人化**的文本，以恰当的粒度和姿态，持久地影响一个**高频运行、上下文有限**的 AI Agent 的行为？这正是 Hermes 和 Claude Code 的记忆系统可以提供参考的地方。

## 2. Hermes 的可借鉴之处

### 2.1 双态记忆模型：冻结快照与实时数据的分离

Hermes 的 `MemoryStore` 设计了一个精巧的双态机制：`_system_prompt_snapshot` 是会话开始时的冻结快照，用于系统提示词注入且不再变化；而 `memory_entries` / `user_entries` 则是实时数据，可以被工具调用修改。这种"冻结层 + 活跃层"的分离，直接映射到 Orbit 愿景的读取需求：

- **冻结层** → Orbit 的 `summaryForAgent`：会话开始时加载一次的愿景摘要，在整个会话期间保持稳定，不因对话过程中的任何操作而被篡改或漂移。这确保了 Agent 对用户长期方向的理解在一次对话中是一致的。
- **活跃层** → Orbit 的 review/journal 场景：回顾时可以读取完整原文、历史版本、最新反思，这些信息是动态加载的，按需获取。

Orbit 可以借鉴这一模式，在 `vision.buildAgentContext(taskContext)` 的实现中明确区分：planning/general_chat 场景使用冻结摘要，review 场景动态加载原文。这避免了"摘要在会话中被 Agent 自己的理解逐步改写"的风险——Orbit 方案中已经识别了这一风险（"Agent 摘要逐渐取代原文"），Hermes 的工程实践提供了具体的解法。

### 2.2 结构化记忆格式：MEMORY.md 与 USER.md 的分类

Hermes 将记忆分为两个文件——`MEMORY.md`（Agent 的观察与笔记，2200 字符上限）和 `USER.md`（Agent 对用户的认知，1375 字符上限），并使用 `§` 分隔符支持多条结构化条目。

这种分类思路对 Orbit 的启发在于：愿景摘要注入 memory context 时，不应是一段混沌的文本，而应该是有明确边界的结构化块。Orbit 方案中已经定义了注入内容（标题、摘要、directive 列表、reaffirm 时间、相关链接），但可以进一步借鉴 Hermes 的做法——为每个注入块设置**字符上限**。例如：

- 愿景摘要：≤ 500 字符
- Directive 列表：≤ 800 字符（5 条 × 160 字符）
- 关联上下文：≤ 300 字符

这不仅控制了 token 消耗，更重要的是，它强制 `summaryForAgent` 的生成过程必须做到真正的压缩，而不是把原文略作删减就塞进去。

### 2.3 内容安全扫描：记忆写入的防护

Hermes 对所有写入记忆的内容执行威胁模式扫描（提示词注入、角色劫持、API 密钥泄露等）。虽然 Orbit 的愿景由用户亲手写下，看似不需要这类防护，但仍然需要考虑：

- 用户写的愿景文本在被送入云端模型时，是否可能被第三方在 prompt 中利用？
- `summaryForAgent` 的生成过程（Agent 整理表达）是否可能引入不安全内容？

Orbit 至少应在 `summaryForAgent` 生成环节加入基础的内容校验，确保摘要不包含可能被误解为系统指令的文本。

### 2.4 SQLite + FTS5：跨会话记忆检索

Hermes 的跨会话搜索能力（FTS5 全文搜索 + Gemini Flash 生成摘要）是一个重要参考。Orbit 的愿景虽然是低频变化的对象，但与愿景相关的 journal 片段、项目进展、review 快照是高频产生的。当 Agent 在回顾场景中需要"读取最近 7/30 天与该愿景相关的 journal 片段"时，需要高效的跨时间检索能力。

Hermes 的 FTS5 方案比简单的文件扫描更适合这类需求。Orbit 的数据库已经设计了 `journal_entries`、`review_snapshots`、`links` 等表，可以在此基础上引入全文索引，支持按愿景 ID + 时间范围的高效查询。

### 2.5 会话谱系（Session Lineage）：压缩后的上下文连续性

Hermes 在上下文压缩发生时创建新会话，并通过 `parent_session_id` 链接旧会话，形成压缩链。这保证了即使对话被压缩，仍然可以追溯到原始上下文。

Orbit 的愿景版本化（追加式版本 + `visionVersionId` 引用）本质上就是一种"语义层的谱系追踪"。可以进一步借鉴 Hermes 的做法：当 Agent 在压缩对话中引用了愿景摘要，压缩后的摘要应该保留该引用的版本 ID，而不是让引用在压缩过程中丢失。

## 3. Claude Code 的可借鉴之处

### 3.1 多层配置叠加：CLAUDE.md 的 5 层体系

Claude Code 最值得 Orbit 借鉴的设计是其 **5 层 CLAUDE.md 配置叠加机制**：

1. Managed CLAUDE.md（策略/规则）
2. Managed rules（`.claude/rules/*.md`）
3. User 全局（`~/.claude/CLAUDE.md`）
4. User rules（`~/.claude/rules/*.md`）
5. 项目级和本地级文件

这种多层叠加模式直接映射到 Orbit 的分层体系（愿景 → directive → 项目 → 计划）。Orbit 可以参考这一设计，为 Agent 的 memory context 构建一个类似的分层注入机制：

- **L1 层（全局）**：愿景摘要 + 核心 directive → 始终注入，类似 Claude Code 的 User 全局文件
- **L2 层（领域）**：当前活跃项目的上下文 → 按工作目录/项目自动切换，类似 Claude Code 的项目级文件
- **L3 层（会话）**：当前任务相关的 journal 片段、review 快照 → 按需加载，类似 Claude Code 的本地级文件

每一层有独立的 token 预算和优先级，当总预算紧张时，从 L3 开始裁剪，L1 始终保留。这确保了愿景作为"最高层用户对象"在上下文中的优先级。

### 3.2 记忆类型标记与自动提取

Claude Code 定义了 4 种记忆类型（`user`、`feedback`、`project`、`reference`），并通过 frontmatter 标注每个记忆文件的类型和描述。更重要的是，它实现了**自动记忆提取**——基于对话中可见消息的计数来决定何时触发。

这对 Orbit 的启发在于：当用户在 journal 中反复提到某个方向性主题（例如"我最近特别想做对外输出"），系统可以自动识别这一模式，并在回顾时将其与愿景关联呈现，而不需要用户手动建立链接。具体来说：

- 可以为 journal 条目引入类型标注（`reflection`、`progress`、`concern`、`aspiration`），类似 Claude Code 的记忆类型
- 在日/周回顾时，Agent 可以自动扫描近期 journal 中的 `aspiration` 类条目，与当前愿景进行语义匹配，作为"温和提醒"的内容来源

这比 Orbit 方案中设计的固定话术模板更加自然——提醒内容来自用户自己最近写下的话，而不是系统对愿景的机械引用。

### 3.3 上下文压缩后的恢复预算

Claude Code 在压缩后为关键资源设置了恢复预算：最多恢复 5 个文件、50K token 总预算、单文件上限 5K token、技能恢复 25K token。这意味着压缩不是简单的"砍掉旧内容"，而是"砍掉后主动恢复最重要的上下文"。

这对 Orbit 极为关键。当一次长对话被压缩时，愿景摘要应该被视为"必须恢复"的首要内容。Orbit 可以设计一个类似的恢复机制：

- 压缩后，首先重新注入当前愿景摘要（优先级最高）
- 然后注入与当前任务相关的 directive（优先级次之）
- 最后在预算内恢复最近的项目上下文

这确保了即使对话发生了压缩，Agent 对用户长期方向的感知不会丢失。

### 3.4 结构化摘要模板与压缩策略

Claude Code 的摘要生成使用 9 个结构化维度（Primary Request and Intent → Optional Next Step），包含对用户原始消息的逐字引用（verbatim quotes）要求。

Orbit 的 `summaryForAgent` 生成可以借鉴这种结构化方法，定义一个明确的摘要模板：

```
## 愿景核心
[用户原话中最核心的 1-2 句，逐字引用]

## 方向关键词
[从愿景中提取的 3-5 个方向标签]

## 当前阶段重点
[如果愿景有时间尺度标注，提取当前阶段的侧重点]

## 用户语气
[愿景的情感基调：探索型 / 坚定型 / 转型型 等]
```

这个模板确保摘要不会丢失用户的原始表达和情感色彩，同时保持了足够的压缩度。

### 3.5 项目引导（Onboarding）的渐进式设计

Claude Code 的 onboarding 流程值得关注：只有 2 步（workspace setup + CLAUDE.md 创建），展示 4 次后自动消失，完成后不再提示。这种"最小摩擦"的设计理念与 Orbit 愿景写入的 onboarding 目标一致。

Orbit 的 6 步 onboarding 流程可以借鉴 Claude Code 的渐进式思路：不是一次性要求用户完成所有步骤，而是只强调"写下第一句话"这一个核心动作，其余步骤（语气标签、提醒姿态、directive 候选）可以在后续使用中逐步引导完成。Claude Code 的"展示 N 次后自动消失"机制也可以应用于 Orbit 的提醒姿态设置——如果用户多次跳过，就默认为 `silent` 并停止提示。

## 4. 不适用于 Orbit 的设计

### 4.1 Hermes 的开放插件式记忆生态

Hermes 支持 8+ 外部记忆提供商（Honcho、Mem0、ChromaDB 等），通过抽象基类 `MemoryProvider` 实现插件化。这种设计追求的是**最大化灵活性**，让不同用户选择不同的记忆后端。

Orbit 不应采用这种模式。愿景是 Orbit 最核心、最敏感的用户数据，其存储和读取路径必须是**确定的、可审计的、本地优先的**。引入多个外部记忆提供商会：

- 增加数据泄露面——每个提供商都是一个潜在的隐私出口
- 引入一致性问题——不同提供商的记忆可能产生冲突的愿景理解
- 破坏"愿景不可被 Agent 改写"的硬原则——外部记忆系统可能在 Orbit 不知情的情况下修改或覆盖愿景摘要

Orbit 应该坚持自建的"Markdown 原文 + 关系型元数据"方案，不提供外部记忆插件接口。

### 4.2 Claude Code 的团队记忆同步

Claude Code 的团队记忆同步（server wins per-key、增量上传、30 秒超时）是为多人协作的代码项目设计的。Orbit 是个人知识管理系统，愿景是极其私密的个人数据，不存在"团队共享愿景"的场景。

因此，Claude Code 的 TeamMem 类型、服务端同步 API、冲突解决策略等设计对 Orbit 不适用。Orbit 的愿景同步（如果需要的话）应该走端到端加密的个人设备同步路径，而不是服务端中心化同步。

### 4.3 Hermes 的概率采样工具分布

Hermes 的 `toolset_distributions` 允许通过概率分布来决定哪些工具可用。这是为通用 Agent 平台设计的，用于在大量工具中做注意力管理。

Orbit 的愿景系统不需要这种随机性。愿景相关的 API（`vision.getActive()`、`vision.buildAgentContext()`、`vision.getVersionHistory()`）应该始终确定性地可用，不存在"有时候 Agent 能看到愿景，有时候看不到"的场景。

### 4.4 Claude Code 的 ML 安全分类器

Claude Code 使用机器学习分类器来判断操作的安全性（比正则更智能），并且有"拒绝率追踪降级"机制。这是为高频、多样化的代码操作设计的。

Orbit 的愿景操作是低频、高确定性的（创建、更新、读取、归档），不需要 ML 分类器来判断安全性。简单的规则引擎（如"Agent 不可直接创建/覆盖愿景版本"）就足够了。引入 ML 分类器反而会增加不确定性——用户可能不理解为什么某次愿景操作被拒绝。

### 4.5 两者的训练数据/RL 闭环

Hermes 内置了 RL 训练工具和训练数据压缩器（TrajectoryCompressor），用于 Agent 的自我进化。Claude Code 虽然没有显式的 RL 系统，但通过 OTel 遥测收集大量使用数据。

Orbit 不应将愿景数据用于任何形式的模型训练或统计分析——这一点在方案中已经明确（"愿景文本不得进入产品分析、推荐训练、跨用户统计画像"）。这里需要强调的是，这不仅是隐私原则，更是产品哲学：一旦愿景被用于训练，系统就有动机让用户写出"对训练更有价值"的愿景，而不是"对用户自己最真实"的愿景。

## 5. 具体建议与行动项

### 建议一：实现「冻结摘要 + 动态原文」的双态读取机制

**来源**：Hermes 的 `_system_prompt_snapshot` 冻结快照设计

**具体做法**：

1. 在 `vision.buildAgentContext(taskContext)` 中，为 `planning`、`project_creation`、`general_chat` 场景返回会话开始时冻结的 `summaryForAgent` 快照，而非实时查询数据库。
2. 仅在 `review` 场景和用户明确要求时，动态加载愿景原文与历史版本。
3. 冻结快照带有 `visionVersionId` 标记，确保摘要与原文版本的对应关系不会在会话过程中断裂。

**预期收益**：消除"Agent 在长对话中逐步改写自己对愿景的理解"的风险；降低每轮 API 调用的延迟（无需每次查询数据库）。

### 建议二：为 memory context 注入设置分层 token 预算

**来源**：Claude Code 的 5 层配置叠加 + 压缩后恢复预算机制

**具体做法**：

1. 定义三层注入预算：
   - L1（愿景摘要 + 核心 directive）：≤ 1500 tokens，始终注入
   - L2（当前项目上下文）：≤ 2000 tokens，按活跃项目注入
   - L3（相关 journal/review 片段）：≤ 3000 tokens，仅在回顾场景注入
2. 当上下文压缩发生时，L1 层内容必须作为首要恢复项被重新注入。
3. 总预算通过 `taskContext` 动态调整：general_chat 只用 L1，review 用 L1+L2+L3。

**预期收益**：确保愿景在 Agent 上下文中的优先级；避免愿景相关信息在 token 紧张时被不当裁剪。

### 建议三：为 journal 条目引入类型标注，实现自动化的愿景关联

**来源**：Claude Code 的 4 类记忆类型（user/feedback/project/reference）+ 自动记忆提取

**具体做法**：

1. 在 `journal_entries` 表中增加 `type` 字段：`reflection`（反思）、`progress`（进展）、`aspiration`（愿望/方向）、`concern`（困惑/担忧）。
2. Agent 在用户写 journal 时自动标注类型（用户可修改）。
3. 在周回顾时，自动扫描近期 `aspiration` 和 `reflection` 类条目，与当前愿景进行语义匹配。
4. 匹配结果作为"温和提醒"的内容来源——用用户自己最近的话来呼应愿景，而不是机械引用愿景原文。

**预期收益**：让提醒更加自然和有温度（"你上周三写过'最近特别想做对外输出'"比"你的愿景中提到'把复杂问题讲清楚'"更有说服力）；建立愿景与日常记录之间的活生生的连接。

### 建议四：为 `summaryForAgent` 设计结构化生成模板

**来源**：Claude Code 的 9 维结构化摘要 + Hermes 的 8 节压缩模板

**具体做法**：

1. 定义 `summaryForAgent` 的生成模板：
   ```
   [核心引用] 用户原话中最能代表方向感的 1-2 句（逐字保留）
   [方向标签] 3-5 个关键词（如：创造、表达、连接、研究）
   [阶段侧重] 当前时间尺度下的重点（如有）
   [语气基调] 探索型 / 坚定型 / 转型型
   ```
2. 模板强制要求保留用户原话引用，防止摘要完全抽象化。
3. 设置字符上限（建议 500 字符 / 约 125 tokens），确保摘要不膨胀。
4. 每次版本更新时重新生成，且生成结果必须经用户确认。

**预期收益**：确保摘要质量的一致性；保留用户的原始表达和情感色彩；防止摘要漂移。

### 建议五：在 onboarding 中采用「一个核心动作 + 渐进补全」策略

**来源**：Claude Code 的 2 步引导 + 展示 N 次后自动消失

**具体做法**：

1. 将 6 步 onboarding 重构为 1 核心 + 5 渐进：
   - **核心步骤（不可跳过）**：定位说明 + 写下第一句愿景（哪怕只有一句话）
   - **渐进步骤（可跳过，后续引导）**：Agent 整理表达、语气标签、提醒姿态、directive 候选、项目挂靠
2. 渐进步骤在后续使用中适时引导（例如：第一次创建项目时引导设置 directive，第一次周回顾时引导设置提醒姿态）。
3. 每个渐进步骤引导最多出现 3 次，之后使用默认值并停止提示。

**预期收益**：降低首次使用的认知负担；让 onboarding 从"一次性任务"变成"在使用中逐步完善"；提高愿景首写的完成率。

## 6. 总结

通过对 Hermes Agent 和 Claude Code 记忆系统的深度分析，我们可以看到两种截然不同但各有所长的设计哲学：

**Hermes** 提供了**开放性和结构化**的启发——它的双态记忆模型（冻结快照 vs 实时数据）、SQLite + FTS5 的跨会话检索、结构化字符上限管理，以及内容安全扫描机制，都直接回应了 Orbit 愿景系统中"如何让一段低频文本稳定地影响高频 Agent"的核心问题。Hermes 的工程实践证明，将"不变的背景知识"和"动态的会话信息"在架构层面显式分离，是保证记忆一致性的有效手段。

**Claude Code** 提供了**精细化和体验优先**的启发——它的 5 层配置叠加机制、压缩后恢复预算、自动记忆提取、结构化摘要模板，以及渐进式 onboarding 流程，为 Orbit 提供了关于"如何管理不同优先级信息的 token 竞争"和"如何让用户在最小摩擦下完成高质量的愿景写入"的具体工程方案。

同样重要的是识别不适用的设计：Hermes 的开放插件式记忆生态和 Claude Code 的团队同步机制，都与 Orbit"个人知识管理、本地优先、用户主权"的产品定位相矛盾。Orbit 不需要追求记忆系统的灵活性和协作性，而应追求**确定性、可审计性和最小暴露面**。

最终，五条行动建议构成了一个完整的实施路径：从底层的读取机制（双态读取）和预算管理（分层 token 预算），到中层的数据模型增强（journal 类型标注）和生成质量控制（结构化摘要模板），再到上层的用户体验优化（渐进式 onboarding）。这些建议彼此独立但互相增强，可以按优先级逐步实施。

核心原则始终不变：**愿景是用户写的、给自己的方向感。系统的一切设计——记忆、压缩、注入、提醒——都是为了在恰当的时刻、以恰当的方式，帮用户想起自己想成为什么样的人。**
