# 05-研究工作台 调研：Hermes Agent 与 Claude Code 对 Orbit 研究工作台的启发

> **调研范围**：工具编排、多Agent协作、知识检索三大维度
>
> **调研来源**：Hermes Agent 架构分析、Claude Code 架构分析、Orbit 设计方案 05-研究工作台
>
> **输出目标**：为 Orbit 研究工作台的实现提供可落地的技术参考

---

## 1. Orbit 设计方案摘要

Orbit 研究工作台的核心定位是：**围绕问题、来源、分析与产物长期演进的工作空间**。它不是带上下文的聊天框，而是一个可持续数天、数周甚至数月被反复进入、补充、校验、导出的研究容器。

设计方案将研究拆为四层架构：

1. **问题层**：用户要回答的主问题、子问题、假设与判断标准，构成一棵可追踪的问题树。
2. **来源层**：内部阅读材料、笔记、外部检索结果构成的资料池，区分已采纳来源、候选来源、待验证来源和派生来源。
3. **推理层**：Agent 的研究会话，包括证据整理、知识缺口识别与交叉验证过程。每次会话绑定一份资料池快照，确保可审计性。
4. **产物层**：摘要卡、对比矩阵、综述报告、结论 memo、文章提纲、PPT 大纲等可导出成果。

核心设计原则包括：

- **会话只是过程，研究空间才是容器；回答只是中间态，产物才是可复用资产。**
- **会话、资料池、产物三者解耦，但可追溯。**
- **先内部后外部、先 grounding 后生成**的 Agent 行为规范。
- **知识缺口作为稳定对象**，按事实、比较、论证、时效、反例五维分类。
- **MCP 作为外部传感器与执行器**，不直接成为内部知识真相。

研究空间对象模型涵盖 `research_space`、`research_question`、`research_source_set`、`research_source_item`、`research_session`、`research_gap`、`research_claim`、`research_artifact` 八个核心对象，通过统一的 `links` 体系与项目、Journal、写作工作台、对象图谱互联互通。

该方案的技术挑战集中在三个方面：**（1）如何编排复杂的多步研究工具链？（2）如何在多 Agent 协作中保持研究过程的可追溯性？（3）如何在内部知识检索与外部信息获取之间建立有效的分层机制？** 以下从 Hermes 和 Claude Code 两个项目中寻找启发。

## 2. Hermes 可借鉴之处

### 2.1 工具概率采样分布——适配研究场景的工具集编排

Hermes 实现了一套独特的**概率采样分布系统**（`toolset_distributions.py`），为不同场景预定义了工具集的启用概率。例如 `"research"` 分布会以 90% 的概率启用 Web 工具、70% 启用浏览器工具、50% 启用视觉工具。这个设计对 Orbit 研究工作台有直接启发意义：

研究工作台的每次 `research_session` 都需要声明"执行策略"——仅内部检索、可外部检索、需要多轮交叉验证。**可以将 Hermes 的概率分布思想转化为 Orbit 的"研究执行策略模板"**：根据研究问题的类型（事实查证、竞品分析、趋势研判）自动选择不同的工具组合，而非每次都手动配置。

此外，Hermes 的白名单/黑名单模式和动态 Schema 重建机制也值得参考。研究工作台可以根据资料池当前状态（来源充足还是知识缺口多）动态调整可用工具：当资料池已有足够内部来源时，收窄外部检索工具的可用范围；当识别到大量知识缺口时，自动激活更多外部获取工具。

### 2.2 多后端执行环境——支撑多样化的研究数据获取

Hermes 的 Terminal 工具支持 6 种执行后端（Local、Docker、Modal、SSH、Daytona、Singularity），通过统一的 `BaseEnvironment` + `ProcessHandle` 协议抽象实现了后端无关的命令执行。这个架构模式对研究工作台的外部数据获取层有重要参考价值。

研究工作台需要从多种外部渠道获取信息：网页搜索、学术数据库、行业 API、财报数据等。参照 Hermes 的做法，**可以设计统一的 `ResearchDataSource` 抽象协议**，让不同的数据获取后端（Web 检索、学术 API、内部知识库查询）实现相同的接口。这样 `research_session.run` 的执行逻辑只需面向统一协议编程，而不必关心具体的数据来源实现。

### 2.3 Mixture-of-Agents 多模型聚合——交叉验证研究结论

Hermes 的 MoA（Mixture of Agents）机制是一个独特的架构亮点：调用 Claude、Gemini、GPT、DeepSeek 四个前沿模型，各自独立回答后由聚合器综合结论。最低要求一个成功即可返回结果。

这个模式完美契合 Orbit 研究工作台中**交叉验证**的需求。设计方案中提到需要"多轮交叉验证"的执行策略，MoA 模式可以直接适配为一种"研究验证工具"：当研究结论涉及关键决策时，用多个 LLM 独立分析同一份证据集，然后让聚合器标记共识与分歧点。这比单个模型的置信度声明更有实际价值。

### 2.4 完善的批处理与 Cron 调度——支持长期研究自动推进

Hermes 拥有完整的批处理运行器（`batch_runner.py`）和 Cron 调度系统（`cron/scheduler.py`），支持 once / interval / cron 三种执行模式，含断点续传的 checkpoint 机制。

研究工作台的核心特征之一是"长期性"——一个研究主题可能持续数周。借鉴 Hermes 的调度能力，可以实现研究空间的**自动化维护**：

- 定期检查资料池中的时效缺口来源（来源过旧需更新验证）
- 对已有的知识缺口尝试自动补充外部检索
- 在新的内部内容（文章、笔记）入库时自动匹配现有研究空间的知识需求

### 2.5 开放技能市场生态——研究工具的社区化扩展

Hermes 构建了完整的技能市场（Skills Hub），包含发现、安装、管理全流程，支持 4 种来源（official、github、clawhub、lobehub），拥有三级信任模型（builtin / trusted / community）。

对研究工作台而言，不同的研究领域需要不同的专业工具（学术搜索、专利检索、财务分析、市场调研），不可能全部内置。参照 Hermes 的技能市场模式，**可以设计"研究工具包"（Research Toolkit）的扩展机制**，允许社区贡献面向特定研究领域的数据获取和分析工具，通过信任分级和沙箱隔离确保安全性。

## 3. Claude Code 可借鉴之处

### 3.1 Coordinator 四阶段工作流——研究会话的结构化编排

Claude Code 的 Coordinator 模式定义了一套清晰的四阶段工作流：**Research → Synthesis → Implementation → Verification**。Coordinator 通过约 370 行的系统提示词定义协调行为，核心原则包括：

- Worker 完成后主动通知，Coordinator 不主动查询
- 不在 Worker 间建立直接通信
- 并行能力是核心武器：只读任务自由并发，写入任务串行

这个模式对研究工作台的 `research_session.run` 流程有直接的映射价值：

1. **Research 阶段**：并行派出多个子 Agent 分别检索内部知识库、外部数据源、历史研究产物
2. **Synthesis 阶段**：主 Agent 读取所有子 Agent 的发现，编制 grounding bundle，识别知识缺口
3. **Implementation 阶段**：基于 grounding bundle 生成初步分析、结论草稿、产物框架
4. **Verification 阶段**：交叉校验结论的证据链完整性，标记未充分支撑的 claim

这比 Orbit 当前方案中未明确定义的"研究执行流程"更具工程可操作性。**建议 Orbit 将 Coordinator 模式适配为 ResearchOrchestrator**，在每次研究会话中强制执行这四个阶段。

### 3.2 LLM 驱动编排 vs 代码驱动编排——研究场景的最优策略

Hermes 是**代码驱动编排**（ThreadPoolExecutor + 路径检测），Claude Code 是 **LLM 驱动编排**（Coordinator 提示词语义调度）。对研究工作台来说，两种模式应结合使用：

- **确定性流程用代码驱动**：source grounding 的检索顺序（先内部后外部）、资料池的去重与摘要、知识缺口的分类规则——这些逻辑不应交给 LLM 决策，而应由代码保障执行顺序和完整性。
- **探索性推理用 LLM 驱动**：问题分解、子问题优先级判断、证据链的论证强度评估、知识缺口与已有来源的关联发现——这些需要语义理解能力的环节，适合 LLM 驱动。

Claude Code 的声明式工具行为特征（`isConcurrencySafe`、`isReadOnly`、`isDestructive`）在研究场景中也极有价值。可以为研究工具声明类似属性：`isRetrievalOnly`（纯检索、无副作用）、`modifiesSourcePool`（会修改资料池）、`requiresUserReview`（结果需人工审查），从而让编排器自动决定哪些操作可以并发执行。

### 3.3 多态任务类型系统——研究活动的精细建模

Claude Code 拥有 7 种具体任务类型（`LocalShellTaskState`、`LocalAgentTaskState`、`RemoteAgentTaskState` 等），每种任务有独立的状态扩展和生命周期管理，共享统一的 5 态基础（pending → running → completed/failed/killed）。

研究工作台可以借鉴这种**多态任务体系**，为不同类型的研究活动定义专门的任务子类型：

- `InternalRetrievalTask`：内部知识库检索，只读、可高并发
- `ExternalSearchTask`：外部检索，受 MCP 边界约束、需审计
- `EvidenceAnalysisTask`：证据分析与 claim 提取，需 LLM 推理
- `GapIdentificationTask`：知识缺口识别，依赖前序任务结果
- `ArtifactGenerationTask`：产物生成，需资料池快照绑定

每种任务类型继承统一的 `ResearchTaskBase`（含 ID、状态、开始/结束时间、资料池快照引用），但扩展各自的特定字段。这使得研究过程不仅对象化（如设计方案已定义的 8 个核心对象），其**执行过程也被对象化管理**。

### 3.4 双向消息通信与语义寻址——研究 Agent 间的协作

Claude Code 的 `SendMessageTool` 支持 Agent 间的双向通信，通过语义寻址（Agent 名称而非位置索引）定位目标，支持广播模式（`"*"`），消息格式包含结构化类型（shutdown_request、plan_approval_response）。

相比 Hermes 的单向父子通信，Claude Code 的模式更适合研究工作台中的**多 Agent 协作场景**。例如：

- 内部检索 Agent 发现了一篇高度相关的历史研究产物，可以直接通知外部检索 Agent 调整搜索策略
- 知识缺口识别 Agent 发现新缺口时，广播通知所有检索 Agent 补充查找
- 证据分析 Agent 完成分析后，通过结构化消息通知 Orchestrator 更新问题状态

**建议 Orbit 采用 Claude Code 的语义寻址 + 结构化消息模式**，为研究 Agent 定义专门的消息类型（`source_found`、`gap_identified`、`claim_proposed`、`evidence_conflict`），让 Agent 间的协作具有领域语义，而不只是通用文本传递。

### 3.5 延迟加载 + 工具搜索——大规模研究工具的按需激活

Claude Code 实现了**延迟加载（Deferred Loading）+ ToolSearchTool** 机制：MCP 工具默认延迟加载，只有在 LLM 通过 ToolSearchTool 搜索到时才激活。这避免了每次会话都加载全部工具导致的上下文膨胀。

研究工作台未来可能接入大量专业研究工具（学术 API、行业数据库、各种分析工具），不可能全部预加载到 Agent 上下文中。**借鉴 Claude Code 的延迟加载策略**，可以只预加载基础的内部检索和资料池管理工具，将外部专业工具标记为延迟加载，由研究 Agent 在分析知识缺口后按需搜索并激活所需工具。

### 3.6 磁盘持久化 + 预览的大结果处理——适配研究产物

Claude Code 对大型工具结果采用"磁盘持久化 + 预览"策略：完整结果写入磁盘，4KB 预览送入 LLM 上下文。这对研究工作台尤其重要——研究产物（综述报告、对比矩阵）往往远超 LLM 上下文窗口限制。

Orbit 的设计方案已经明确了"文件系统优先 + 元数据数据库"的存储分层原则，Claude Code 的实现模式正好提供了一个经过验证的工程范例。建议研究工作台的产物生成流程也采用类似策略：产物全文写入文件系统的统一内容层，LLM 上下文中只保留结构化摘要和关键片段引用。

## 4. 不适用的设计

### 4.1 Hermes 的多平台 Gateway 系统

Hermes 拥有 14 个消息平台适配器（Discord、Telegram、飞书、钉钉等），这是其"把 Agent 送到用户所在平台"的产品策略体现。**Orbit 研究工作台不需要这种多渠道分发能力**——研究是一个需要深度沉浸的活动，应在 Orbit 内部的统一界面中进行，而非散布到各种聊天平台。研究产物的分享可通过导出功能实现，不需要在 IM 平台上实时交互。

### 4.2 Hermes 的 RL 训练与自我进化机制

Hermes 内置了完整的强化学习训练工具链（基于 Atropos），支持环境发现、LoRA 微调、W&B 实验追踪。**这对 Orbit 当前阶段不适用**。研究工作台的首要任务是建立正确的研究流程框架和对象模型，而非训练自定义模型。未来如果需要优化研究 Agent 的 grounding 策略或 claim 提取质量，可以考虑在 Orbit 外部独立进行模型微调，而非将训练能力嵌入产品。

### 4.3 Claude Code 的 Vim 模式与终端渲染引擎

Claude Code 投入了大量工程资源构建自研的 Ink + Yoga 终端 UI 引擎和完整的 Vim 状态机。这是针对开发者终端体验的极致优化，**与 Orbit 作为 GUI 应用的定位不符**。Orbit 应使用标准的 Web/Desktop 前端技术栈来构建研究工作台的 UI。

### 4.4 Claude Code 的 Anthropic 深度绑定

Claude Code 专为 Anthropic 的 Claude 模型优化，利用了 Prompt Caching、Extended Thinking 等独家特性。**Orbit 不应将自身绑定到单一模型提供商**。Hermes 的多模型兼容策略更符合 Orbit 的长期利益——研究 Agent 应能在不同 LLM 之间切换，甚至利用 MoA 模式获得多模型聚合的优势。

### 4.5 Hermes 的单体 Agent 类架构

Hermes 的核心是一个 7500 行的单体 `Agent` 类，所有逻辑集中在一处。这种架构在早期迭代快，但随着复杂度增长会导致维护困难。**Orbit 应避免这种模式**，采用 Claude Code 式的管道模块化架构，将研究工作台的各个职责（问题管理、资料池管理、会话编排、产物生成）清晰分离。

## 5. 具体建议

### 建议一：实现 ResearchOrchestrator，采用四阶段编排模型

参照 Claude Code 的 Coordinator 模式，为研究工作台设计 `ResearchOrchestrator`，在每次 `research_session.run` 中强制执行四阶段流程：

```
Retrieval → Synthesis → Analysis → Verification
```

- **Retrieval**：并行派出内部检索 Agent 和外部检索 Agent（如策略允许），收集候选来源。此阶段用代码驱动编排，保障"先内部后外部"的顺序约束。
- **Synthesis**：主 Agent 汇总所有检索结果，编制 grounding bundle，识别知识缺口，更新资料池快照。此阶段混合使用代码逻辑（去重、分类）和 LLM 推理（摘要、相关性判断）。
- **Analysis**：基于 grounding bundle 进行推理，提取 `research_claim`，生成产物草稿。此阶段以 LLM 驱动为主。
- **Verification**：校验每个 claim 的证据链完整性，标记未充分支撑的结论，输出最终的知识缺口列表。可利用 MoA 模式进行多模型交叉验证。

每个阶段的输入输出都应是结构化对象（而非自由文本），确保研究过程可审计、可重放。

### 建议二：设计声明式研究工具特征系统

借鉴 Claude Code 的工具行为声明（`isConcurrencySafe`、`isReadOnly`、`isDestructive`），为研究工具定义领域专属的行为特征：

| 特征 | 含义 | 编排影响 |
|------|------|---------|
| `isRetrievalOnly` | 纯检索，无副作用 | 可自由并发 |
| `modifiesSourcePool` | 会修改资料池 | 需串行执行 |
| `requiresExternalAccess` | 需要外部网络 | 受 MCP 边界约束 |
| `requiresUserReview` | 结果需人工确认 | 不可完全自动化 |
| `affectsObjectGraph` | 会创建/修改对象关系 | 需事务保护 |

编排器根据这些声明自动决定执行策略：多个 `isRetrievalOnly` 的工具可以并发执行，`modifiesSourcePool` 的工具必须等待前序操作完成，`requiresUserReview` 的结果进入候选状态而非直接采纳。

### 建议三：实现分层知识检索管道（先内后外、渐进扩展）

结合 Hermes 的概率采样分布和 Claude Code 的延迟加载思想，设计研究工作台的三层检索管道：

1. **第一层（始终激活）**：Orbit 内部检索——文章、笔记、高亮、历史研究产物、对象图谱。使用设计方案中已定义的 `research_source.select/import` 接口。
2. **第二层（按需激活）**：通用外部检索——Web 搜索、页面抓取。当第一层结果不足以回答研究问题，或存在明确的知识缺口时，由 Orchestrator 自动激活。
3. **第三层（显式请求）**：专业外部检索——学术数据库、行业 API、付费数据源。仅在用户显式授权或 Orchestrator 识别到特定类型缺口时激活。

每一层的检索结果都经过统一的"候选来源 → 正式来源"转化流程：先成为 `research_source_item`（候选），经过去重、摘要、可信度标注后才可被采纳进入 grounding bundle。

### 建议四：为研究 Agent 间通信定义领域消息协议

参照 Claude Code 的结构化消息类型，定义研究工作台专用的 Agent 间消息协议：

```typescript
type ResearchMessage =
  | { type: 'source_found', sourceId: string, relevance: number, summary: string }
  | { type: 'gap_identified', gapType: GapCategory, description: string, priority: number }
  | { type: 'claim_proposed', claim: string, evidenceIds: string[], confidence: number }
  | { type: 'evidence_conflict', claimId: string, conflictingSources: string[] }
  | { type: 'retrieval_strategy_update', adjustedQuery: string, reason: string }
  | { type: 'phase_complete', phase: ResearchPhase, summary: string }
```

这样 Agent 间的协作不再是无结构的文本传递，而是具有明确领域语义的结构化事件流。Orchestrator 可以基于这些消息类型做出确定性的编排决策（如收到 `gap_identified` 后自动触发对应层级的检索任务）。

### 建议五：研究会话的上下文压缩与长期记忆分离

综合 Hermes 的双系统压缩（trajectory_compressor + LLM 摘要）和 Claude Code 的自动触发压缩机制，设计研究会话的上下文管理策略：

- **会话内**：采用 Claude Code 式的自动压缩——当会话上下文接近窗口限制时，自动将早期对话压缩为结构化摘要，但保留所有中间产出的 `research_claim`、`research_gap`、`research_source_item` 的引用。
- **会话间**：利用研究空间的对象模型实现"无损长期记忆"——压缩掉的不是研究本身，而是对话过程。问题树状态、资料池版本、已识别缺口、已提出结论全部作为独立对象持久化，新会话可基于这些对象快速恢复上下文。
- **跨空间**：将经过验证的 `research_claim`（有充分证据支撑的结论）提升为全局可检索的知识条目，可被其他研究空间引用，避免重复研究。

这样就实现了设计方案中"聊天历史可压缩、可归档，但问题树、资料池、缺口和产物必须对象化保存"的要求，同时有了成熟项目的工程实现参考。

## 6. 总结

Hermes Agent 和 Claude Code 代表了 AI Agent 领域两种互补的架构哲学，对 Orbit 研究工作台的设计各有独特启发：

**Hermes 的价值在于"广度与开放性"**：多后端执行环境为研究数据获取层提供了统一抽象的设计范式；MoA 多模型聚合为交叉验证提供了现成方案；概率采样分布为场景化工具选择提供了灵感；批处理与 Cron 调度为长期研究的自动化维护提供了基础能力；开放技能市场为研究工具的社区化扩展提供了生态模板。

**Claude Code 的价值在于"深度与精细性"**：Coordinator 四阶段工作流为研究会话的结构化编排提供了直接可映射的模式；声明式工具行为特征为研究工具的并发控制提供了优雅的抽象；多态任务类型系统为研究活动的精细建模提供了工程范例；双向语义寻址通信为研究 Agent 间协作提供了成熟的协议设计；延迟加载与工具搜索为大规模研究工具的按需激活提供了可行策略。

最终建议 Orbit 采取**"Claude Code 的架构精细度 + Hermes 的开放性与多样性"**的融合路线：

- 编排模型参照 Claude Code 的 Coordinator，但增加研究领域的四阶段语义
- 工具系统采用 Claude Code 的声明式特征，但结合 Hermes 的场景化工具集分布
- 多 Agent 协作采用 Claude Code 的双向结构化通信，但定义研究领域专属的消息类型
- 知识检索采用分层渐进策略，代码驱动保障确定性流程，LLM 驱动处理探索性推理
- 上下文管理结合两者的压缩策略，利用研究空间的对象模型实现会话可压缩但知识不丢失

研究工作台的核心技术挑战不在于任何单一能力，而在于将检索、编排、推理、沉淀四个环节无缝串联成一个**可追溯、可重复、可长期演进**的工作流。Hermes 和 Claude Code 分别从不同角度验证了这些环节的工程可行性，为 Orbit 的实现提供了坚实的技术基础。
