# 06. 笔记与高亮系统 — Hermes / Claude Code 对比调研

> **调研目标**：从 Hermes Agent 和 Claude Code 的内容管理、记忆系统、编辑器交互设计中提取对 Orbit「笔记与高亮系统」有价值的参考，指导实现一套统一捕捉、可追溯、可流动的笔记与高亮架构。

---

## 1. Orbit 设计方案摘要

Orbit 06 号方案「笔记与高亮系统」的核心定位不是阅读器附属面板，而是将「输入 → 理解 → 连接 → 产出」串联起来的**材料转化层**。方案围绕四个原则展开：统一捕捉入口、统一笔记对象、可追溯、可流动。

**统一捕捉单元 SelectionContext**：所有可选中文本的界面（Reader、笔记编辑器、创作文档、研究结果页、Journal、Agent 回答面板）共享同一个运行时对象，携带 `surfaceType`、`sourceObjectType`、`selectedText`、`anchor` 等字段。后续「高亮」「边注」「继续问 Agent」「加入笔记」只是对同一个 SelectionContext 执行不同动作。

**高亮锚点模型**：采用四层结构——对象层定位（source_object_type + source_object_id）、结构层定位（paragraph_index / epub_cfi / page+rects / block_id）、文本指纹层（quote + prefix + suffix + text_hash）、状态层（active / fuzzy / detached）。这套复合锚点确保内容更新、重排、跨端同步后仍能尽量回到原位。

**统一笔记对象 Note**：闪念（fleeting）、阅读批注（annotation）、研究笔记（research）、日志手记（journal）、沉淀观点（synthesis）不再拆为独立模块，而是同一类 Note 对象通过 `note_kind` 与 `maturity`（inbox → linked → synthesized → reusable）语义区分。所有 Note 以 Markdown 文件为真相来源，结构化索引进入 object_index。

**上下文提问交互**：选中文本后可直接向 Agent 发问，顶部展示 Context Card（选中文本、来源、位置、已有关联），Agent 回答后用户可选择仅保留在会话、保存为 Note、加入研究空间、创建任务或插入文档。提问不自动制造高亮污染——只有明确保存动作才产生持久对象。

**Block 溯源（BlockProvenance）**：从「文档专用」推广为「任意 Block 容器可用」，记录 `transformation`（verbatim / edited / ai_rewrite / merged）与 `lineageRoot`，实现「文章片段 → 高亮 → 批注 Note → 研究笔记 Block → 创作文档 Block」的完整追溯链路。

**Agent 行为**：围绕已选材料提供上下文问答、关系建议、聚类提炼、转化建议四类增强，默认轻自动化，自动建链优先于自动建任务，任何 AI 生成内容必须保留来源。

## 2. Hermes 可借鉴之处

### 2.1 SQLite + FTS5 全文搜索：为高亮与笔记提供强检索基底

Hermes 的对话历史基于 SQLite WAL 模式 + FTS5 全文搜索虚拟表，支持跨会话消息检索和 LLM 摘要生成。这一设计对 Orbit 的高亮与笔记系统有直接启发意义。

Orbit 06 方案明确了「高亮与 Anchor 结构化保存在数据库」和「对象索引与关系进入 object_index + links」的分工。Hermes 的 FTS5 实践验证了这条路径的可行性：通过在 SQLite 中为高亮的 `quote_text`、笔记正文摘要、关系标签等字段创建 FTS5 虚拟表，Orbit 可以在本地获得接近即时的全文检索能力，无需引入外部搜索引擎。Hermes 通过触发器自动同步 FTS 索引的做法尤其值得采纳——当高亮或笔记被创建、修改、删除时，索引自动更新，避免手动维护的复杂性。

此外，Hermes 的「会话谱系」（parent_session_id 链接）与 Orbit 06 方案的 `lineageRootType / lineageRootId` 在设计理念上高度一致——两者都试图在数据层面维护信息的来源链条。Hermes 在上下文压缩时创建新会话并保留父子链接的模式，可以启发 Orbit 在笔记成熟度升级（inbox → linked → synthesized）时保留转化历史。

### 2.2 双态记忆设计：冻结快照与实时数据分离

Hermes 的 MemoryStore 采用「冻结快照 + 实时数据」双态设计：`_system_prompt_snapshot` 在系统提示词注入时保持不变，而 `memory_entries` / `user_entries` 可被工具调用实时修改。这种分离策略解决了「读取一致性」与「写入及时性」之间的矛盾。

对 Orbit 06 方案来说，这一思路可以应用于 Agent 的上下文召回机制。当用户选中一段文本并向 Agent 提问时，Agent 需要补充该高亮已有的笔记、相关项目、研究空间与历史提问。如果这些关联数据在 Agent 对话过程中被用户同时修改（例如在另一个面板中添加了新笔记），系统需要决定是使用「提问时的冻结快照」还是「实时最新数据」。Hermes 的双态模式提供了一个清晰的参考范式——对于 Agent 的系统级上下文采用冻结快照保证推理一致性，对于用户主动触发的查询采用实时数据保证信息完整性。

### 2.3 内容安全扫描：笔记写入时的防注入检查

Hermes 对所有写入记忆的内容执行威胁模式扫描（提示词注入、角色劫持、系统提示词覆盖、SSH 后门、API 密钥泄露等）。Orbit 的笔记系统虽然主要面向个人用户，但考虑到 Agent 可以自动生成笔记内容（`created_by: 'agent'`）以及笔记会流向任务系统和写作系统，对 Agent 生成的笔记内容做基础安全校验仍然有必要，特别是在防止 Agent 输出中无意间包含敏感信息（如 API Key 片段）被持久化到笔记文件中的场景。

### 2.4 跨会话搜索 + LLM 摘要：「材料发现」的高阶模式

Hermes 的 `session_search` 工具不仅执行 FTS5 检索，还将原始检索结果通过 Gemini Flash 生成摘要。这种「检索 + 理解」的组合模式可以直接移植到 Orbit 的笔记发现场景中。

设想用户积累了数百条高亮和笔记后，需要为一个新研究项目找到相关材料。单纯的关键词搜索往往不够——用户可能用不同的词汇描述同一概念。Hermes 的做法是先用 FTS5 宽泛检索，再用 LLM 理解用户的真实意图并精选最相关的结果。Orbit 可以在「研究空间材料推荐」和「Agent 关系建议」功能中采用类似的两阶段检索范式。

### 2.5 多格式项目配置兼容：启发笔记导入策略

Hermes 兼容 4 种项目配置格式（hermes.md / agents.md / claude.md / cursorrules），通过优先级链实现「首个命中即停止」的发现逻辑。这种多格式兼容策略对 Orbit 的「笔记导入」场景有参考价值——用户可能从 Readwise、Hypothesis、Kindle、Apple Books 等不同来源导入高亮和笔记。Orbit 应设计一套统一的导入管线，将不同来源的格式转化为标准的 `HighlightObject` + `NoteObject`，同时通过 `created_by: 'import'` 与 `origin` 字段保留来源信息。

### 2.6 上下文补全语法：@context 模式的交互启发

Hermes 的 SlashCommandCompleter 支持丰富的上下文补全语法（`@diff`、`@staged`、`@file:path`、`@folder:path`、`@git:5`、`@url:...`），用户可以在输入时直接引用各种上下文资源。这种设计可以启发 Orbit 编辑器中的引用交互——在笔记或创作文档中输入 `@highlight:` 可以搜索并插入高亮引用，`@note:` 可以链接已有笔记，`@research:` 可以引用研究空间中的材料。这比单纯的拖拽或菜单操作更快捷，尤其在键盘密集型写作场景中。

## 3. Claude Code 可借鉴之处

### 3.1 React + Ink 组件化 UI 架构：为统一选择菜单提供范式

Claude Code 基于 React + Ink + Yoga 构建了完整的终端 UI 引擎，实现了组件化、声明式渲染、Flexbox 布局等现代前端能力。虽然 Orbit 是桌面/Web 应用而非终端工具，但 Claude Code 的组件化思路对 06 方案的「统一选择菜单」设计有直接参考价值。

Orbit 06 要求所有可选中文本界面共享同一套浮动菜单（高亮、添加边注、继续问 Agent、保存到笔记、创建任务）。Claude Code 证明了即使在终端这样受限的环境中，组件化架构也能支撑复杂的交互状态管理。Orbit 可以将浮动菜单设计为一个独立的 React 组件（如 `<SelectionActionMenu>`），接收 `SelectionContext` 作为 props，根据 `surfaceType` 动态调整可用动作。这个组件在 Reader、编辑器、Journal 等不同面板中复用，确保交互一致性。

### 3.2 多层 CLAUDE.md 配置叠加：启发笔记的多层上下文组装

Claude Code 的 5 层记忆文件体系（Managed → Managed Rules → User Global → User Rules → Project/Local）是其最精细的设计之一。所有层的配置不是互斥的，而是叠加合并的。每层有不同的语义标签（Project、Local、TeamMem、AutoMem、User），系统能区分哪些内容是项目级共享的、哪些是个人私有的、哪些是自动生成的。

这一分层叠加思路对 Orbit 的 Agent 上下文组装极具参考价值。当 Agent 处理用户的「继续问 Agent」请求时，需要组装多层上下文：选中文本本身（最内层）→ 所在段落/Block 的周围内容 → 源对象级别的摘要 → 该对象已有的高亮和笔记 → 用户当前聚焦的研究空间 → 全局知识库中的相关材料。参考 Claude Code 的做法，每层上下文应有明确的优先级和 token 预算，在上下文窗口有限时优雅降级——先保留选中文本和直接关联，再逐步加入更外层的背景信息。

### 3.3 自动压缩与恢复预算：笔记摘要与成熟度升级的参考

Claude Code 的上下文压缩系统有两个值得关注的细节：

1. **9 维结构化摘要模板**：压缩不是简单截断，而是从 Primary Request、Key Technical Concepts、Files and Code Sections、Errors and Fixes 等 9 个维度提取结构化摘要。Orbit 可以参考这种结构化压缩方式来设计笔记的自动摘要——当用户积累大量零散高亮和批注后，Agent 可以按照预定义的维度（核心论点、关键证据、开放问题、行动建议）生成结构化的 synthesis note。

2. **压缩后恢复预算**（50K token 用于恢复最多 5 个关键文件）：这种「先压缩，再选择性恢复」的模式对 Orbit 的笔记成熟度升级（inbox → linked → synthesized → reusable）有启发。当一批零散笔记被合成为一篇 synthesis note 时，系统不应丢弃原始素材，而是保留它们的引用关系，在用户需要深入追溯时按需展开。

### 3.4 混合 Token 计算：Agent 上下文精确预算管理

Claude Code 的 token 计算采用「API 精确值 + 粗估混合」策略——从最后一条有 API usage 的消息向前搜索，对后续消息使用粗估。这比 Hermes 的纯粗估（chars/4）更精确。

对 Orbit 06 方案来说，Agent 的上下文提问功能需要精确管理 token 预算：选中文本、Context Card 信息、已有关联数据、对话历史、系统提示词等都需要在有限窗口内平衡。Claude Code 的混合计算策略可以直接采用——对 API 已返回的对话轮次使用精确值，对新注入的上下文材料使用快速估算，在两者之间取得精度与性能的平衡。

### 3.5 4 种记忆类型（user / feedback / project / reference）：笔记分类的参考

Claude Code 将记忆分为 `user`（用户偏好）、`feedback`（交互反馈）、`project`（项目知识）、`reference`（参考资料）四种类型，每种有不同的生命周期和使用方式。这种类型化记忆设计与 Orbit 06 的 `note_kind` 分类有异曲同工之处，但 Claude Code 增加了「反馈」维度——记录 Agent 与用户交互中的修正和偏好。

Orbit 可以借鉴这一思路：当用户修改 Agent 建议的笔记关系、拒绝自动生成的 synthesis、手动调整 maturity 等级时，这些「反馈信号」不应被丢弃，而应被记录下来，用于优化 Agent 后续的关系建议和聚类提炼行为。这不需要新增 note_kind，但可以在 event_log 中增加 feedback 事件类型。

### 3.6 流式工具执行状态追踪：高亮操作的即时反馈

Claude Code 的 `StreamingToolExecutor` 支持 `queued → executing → completed → yielded` 的工具状态追踪，并通过 React 状态驱动实时更新 UI。在 Orbit 中，当用户选中文本并执行「继续问 Agent」操作时，涉及多个异步步骤：解析选区、创建 SelectionContext、构建 Agent 上下文、发起 LLM 调用、流式返回结果。Claude Code 的状态追踪模式可以被借鉴为一个统一的操作状态管理器，让用户在浮动菜单中清晰看到每步操作的进展，而不是面对一个无差别的 loading 状态。

### 3.7 Vim 模式状态机：编辑器快捷键体系的参考

Claude Code 实现了完整的 Vim 模式有限状态机（INSERT / NORMAL / operator-motion 组合），这对 Orbit 的编辑器快捷键设计有间接参考价值。06 方案强调「在编辑器中选中文本时，出现与 Reader 一样的浮动菜单」。在键盘密集型场景中，快捷键比鼠标菜单更高效。参考 Claude Code 的做法，Orbit 可以为高频操作设计键盘快捷键——如 `Cmd+Shift+H` 快速高亮、`Cmd+Shift+N` 快速创建边注、`Cmd+Shift+A` 向 Agent 提问，避免用户频繁在键盘和鼠标之间切换。

## 4. 不适用的设计

### 4.1 Hermes 的多平台消息网关（16+ 平台适配）

Hermes 通过 Gateway 架构将 Agent 能力分发到 Telegram、Discord、WhatsApp 等 16+ 消息平台。这种「把 Agent 送到用户所在的每个平台」的策略对 Orbit 不适用。Orbit 的笔记与高亮系统是深度嵌入桌面应用内部的交互层，它依赖精确的文本选区、DOM 锚点、Block 编辑器内核等能力，这些在消息平台中无法实现。Orbit 应专注于 Electron 桌面端和未来的 Web 端，在这两个平台上做到极致的选区和锚点体验。

### 4.2 Claude Code 的终端 ANSI 渲染引擎

Claude Code 投入巨大工程量构建了终端中的 React 应用框架（Ink + Yoga 布局），这是因为其运行在终端这个高度受限的环境中。Orbit 作为 Electron 桌面应用，天然拥有完整的 Web 渲染能力（CSS Flexbox/Grid、DOM API、Canvas/SVG），无需效仿终端 UI 的实现路径。不过，Claude Code 在受限环境中追求极致体验的工程精神值得学习。

### 4.3 Hermes 的多 LLM 提供商适配（10 级回退探测）

Hermes 的上下文窗口管理支持 10 级回退策略以兼容几乎所有 LLM 提供商。Orbit 的 Agent 能力层（15 号方案）已经规划了 MCP 能力接口，LLM 提供商的适配是 15 号方案的职责，06 号方案不应关心底层模型差异。笔记与高亮系统应通过标准化的能力接口（如 `context.ask(context, prompt)`）与 Agent 交互，将模型选择和适配的复杂性隔离在外。

### 4.4 Claude Code 的企业级团队记忆同步 API

Claude Code 的 TeamMem 类型记忆支持服务端 Pull/Push 同步、哈希差异比对、单文件 250KB 上限等企业级特性。Orbit 当前定位为个人知识管理工具，所有笔记以本地 Markdown 文件为真相来源。团队协作和云端同步是未来阶段的规划，当前不需要引入服务端同步的复杂性。

### 4.5 Hermes 的 RL 训练闭环

Hermes 内置了强化学习训练工具和 TrajectoryCompressor，支持 Agent 自我进化。Orbit 的 Agent 增强是辅助性的（关系建议、聚类提炼），不涉及模型训练。Agent 行为的优化应通过更好的提示词工程和用户反馈信号调节，而非 RL 训练循环。

## 5. 具体建议

### 建议一：基于 SQLite FTS5 构建高亮与笔记的统一检索层

**来源启发**：Hermes 的 SQLite + FTS5 对话历史 + 跨会话搜索

**具体方案**：

在 Orbit 的本地数据库中为 `highlight` 和 `note` 对象创建 FTS5 虚拟表。高亮的 `quote_text`、笔记的标题和正文摘要（前 500 字符）、关系标签、来源对象标题等字段应被索引。使用 SQLite 触发器在对象的 CRUD 操作时自动同步 FTS 索引。

在此基础上，实现两阶段检索管线：第一阶段用 FTS5 进行宽泛的文本匹配（可配合 BM25 排序），第二阶段可选地调用 Agent 对检索结果进行语义理解和精选。这个检索层将同时服务于：全局搜索、Agent 的上下文召回（补充相关高亮和笔记）、研究空间的材料推荐、Journal 的「今天我思考了什么」回顾。

```sql
CREATE VIRTUAL TABLE highlight_fts USING fts5(
  quote_text, source_title, tags,
  content='highlights', content_rowid='rowid'
);

CREATE VIRTUAL TABLE note_fts USING fts5(
  title, body_excerpt, tags, linked_objects,
  content='notes', content_rowid='rowid'
);
```

### 建议二：为 Agent 上下文提问设计分层上下文组装器（Context Assembler）

**来源启发**：Claude Code 的 5 层 CLAUDE.md 叠加 + 分级告警 + 恢复预算

**具体方案**：

设计一个 `ContextAssembler` 模块，负责在用户触发「继续问 Agent」时组装多层上下文。每层有独立的 token 预算，总预算根据模型上下文窗口动态调整：

```ts
interface ContextAssemblerConfig {
  layers: [
    { name: 'selection',     budget: 2000,  required: true  },  // 选中文本 + 前后文
    { name: 'source_object', budget: 3000,  required: true  },  // 源对象摘要
    { name: 'annotations',   budget: 2000,  required: false },  // 已有高亮和笔记
    { name: 'research',      budget: 3000,  required: false },  // 当前研究空间
    { name: 'knowledge',     budget: 5000,  required: false },  // 全局相关材料
    { name: 'history',       budget: 2000,  required: false },  // 对话历史
  ];
  totalBudget: number;  // 从模型能力查询获取
}
```

当总 token 超过预算时，从最外层（history → knowledge → research）逐层缩减，保证核心层（selection + source_object）始终完整。参考 Claude Code 的混合 token 计算策略，对 API 已返回的对话历史使用精确值，对新注入的材料使用快速估算。

### 建议三：实现「操作即反馈」的选区动作状态管理器

**来源启发**：Claude Code 的 StreamingToolExecutor 状态追踪 + React 状态驱动 UI

**具体方案**：

为 SelectionContext 的每个动作设计一个状态机，确保用户在执行操作时获得即时、精确的反馈：

```ts
type SelectionActionState =
  | { phase: 'idle' }
  | { phase: 'capturing', context: SelectionContext }
  | { phase: 'executing', action: ActionType, progress: number }
  | { phase: 'awaiting_agent', streamTokens: string }
  | { phase: 'completed', result: ActionResult }
  | { phase: 'error', error: Error, retryable: boolean };
```

在「继续问 Agent」场景中，状态流转为：`idle → capturing（解析选区、创建 Context）→ executing（构建 Agent 上下文）→ awaiting_agent（流式接收 LLM 响应）→ completed（展示结果和后续操作选项）`。每个阶段在 UI 上有对应的视觉反馈（选区高亮闪烁、Context Card 构建动画、流式文字渲染、操作按钮出现），参考 Claude Code 的 follow-scroll 智能滚动确保用户始终看到最新进展。

### 建议四：引入 @引用 语法实现键盘优先的笔记链接体验

**来源启发**：Hermes 的 @context 补全语法（@diff、@staged、@file:path）+ Claude Code 的快捷键体系

**具体方案**：

在 Orbit 的 TipTap Block 编辑器中实现 @ 触发的引用补全系统：

- `@h:` → 搜索并插入高亮引用（展示 quote_text 预览 + 来源标题）
- `@n:` → 链接已有笔记（展示标题 + note_kind 标签）
- `@r:` → 引用研究空间材料
- `@a:` → 从 Agent 历史回答中提取片段

输入触发后弹出模糊搜索面板（参考 Claude Code 的 `FuzzyPicker` 组件），使用 FTS5 后端实时检索。选择后自动创建 `block_provenance` 记录，写入 transformation 为 `verbatim` 或 `edited`。

同时为高频操作分配键盘快捷键：
- `Cmd+Shift+H`：将当前选区创建为高亮
- `Cmd+Shift+N`：为当前选区创建边注
- `Cmd+Shift+Q`：以当前选区向 Agent 提问
- `Cmd+Shift+S`：将当前选区/笔记加入当前研究空间

### 建议五：设计基于事件流的 Agent 反馈学习机制

**来源启发**：Claude Code 的 feedback 记忆类型 + 自动记忆提取 + Hermes 的内容安全扫描

**具体方案**：

在 event_log 中增加 `agent_feedback` 事件类型，记录用户对 Agent 行为的隐式和显式反馈：

```ts
interface AgentFeedbackEvent {
  eventType: 'agent_feedback';
  feedbackKind:
    | 'relation_accepted'     // 用户采纳了 Agent 建议的关系
    | 'relation_rejected'     // 用户拒绝了关系建议
    | 'synthesis_edited'      // 用户修改了 Agent 生成的 synthesis note
    | 'maturity_overridden'   // 用户手动调整了 maturity 等级
    | 'highlight_promoted'    // 用户将 Agent 建议提升为高亮
    | 'action_dismissed';     // 用户忽略了自动任务建议
  sourceAgentAction: string;
  timestamp: string;
  context?: Record<string, unknown>;
}
```

这些反馈事件积累后，可用于在 Agent 的系统提示词中注入偏好指引（如「用户倾向于手动控制 maturity 升级，减少自动 synthesis 建议的频率」），参考 Claude Code 的自动记忆提取机制在消息累积到阈值时触发反馈模式分析。

同时，参考 Hermes 的内容安全扫描，对 Agent 生成并即将持久化为 Note 的内容执行基础安全检查——至少包括：个人隐私信息检测（邮箱、电话、身份证号）、API 密钥/Token 模式检测、避免在笔记中无意间保存敏感信息。

## 6. 总结

通过分析 Hermes Agent 和 Claude Code 两个项目在内容管理、记忆系统与编辑器交互方面的实现，我们可以为 Orbit 06 号方案提炼出几个关键认知：

**第一，检索能力是笔记系统的隐形骨架**。Orbit 06 方案正确地将高亮和笔记定义为对象网络中的一等公民，但方案更多聚焦于对象关系（links）层面的连接，对「如何高效检索」着墨较少。Hermes 的 FTS5 实践提醒我们：当用户积累数百上千条高亮和笔记后，检索质量直接决定了「可流动」原则能否落地。没有强检索层，Agent 的关系建议和上下文召回将失去材料基础。

**第二，上下文组装的精细度决定 Agent 的辅助质量**。Claude Code 的 5 层配置叠加、分级告警、恢复预算等设计都在解决同一个问题——如何在有限的上下文窗口中放入最有价值的信息。Orbit 的「继续问 Agent」功能将面对更复杂的上下文组装挑战（选中文本、来源对象、已有标注、研究空间、全局知识），必须设计显式的分层预算机制，而不是将所有可用信息简单堆砌。

**第三，即时反馈是流畅交互的关键**。Claude Code 对工具执行状态的精细追踪和流式渲染，证明了在 AI 交互场景中「让用户知道系统在做什么」的重要性。Orbit 的选区动作涉及多步异步流程（解析选区 → 构建上下文 → LLM 推理 → 结果保存），每步都需要视觉反馈，否则用户会在等待中失去信心。

**第四，双态一致性值得早期设计**。Hermes 的冻结快照 vs 实时数据分离、Claude Code 的压缩后恢复预算，都在解决「数据在 Agent 推理期间被修改」的一致性问题。Orbit 作为一个多面板、多视图的应用，高亮和笔记可能同时被 Reader、编辑器、Agent 多个模块读写。在架构早期就引入类似 Hermes 双态模式的一致性策略，比后期补丁式修复要经济得多。

**最终建议优先级**：建议一（FTS5 检索层）和建议二（分层上下文组装器）应在核心架构中优先实现，它们直接影响 06 方案最核心的两个交互场景——材料发现与 Agent 提问。建议三（状态管理器）和建议四（@引用语法）应在编辑器交互实现阶段落地。建议五（反馈学习）可以在系统稳定运行后逐步引入。
