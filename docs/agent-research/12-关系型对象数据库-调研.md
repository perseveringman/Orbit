# 12. 关系型对象数据库 — Hermes/Claude Code 调研报告

> **调研目标**：从 Hermes Agent 和 Claude Code 两个成熟 AI Agent 项目中，提取数据存储、查询、对象模型、持久化方面的经验，为 Orbit「关系型对象数据库」设计提供借鉴与警示。

---

## 1. Orbit 设计方案摘要

Orbit 的关系型对象数据库（方案 12）定位为**文件系统之上的关系编译层**，而非独立的内容真相源。其核心架构决策是：

- **物理 schema**：采用「每种对象类型一张表 + 全局 `object_index` + 统一 `links` + 统一 `events`」的分层策略，拒绝图数据库，也拒绝全局单表 `items`。
- **全局索引**：`object_index` 以 `object_uid`（如 `task:tsk_123`）为全局统一引用协议，承担跨类型查询、引用解析、权限标注、版本控制五大职责。
- **统一关系**：一张 `links` 表承载所有跨域关系（`supports`、`references`、`derived_from` 等），通过 `origin` 字段区分 human/ai/system 来源。
- **统一事件**：追加式 `events` 表记录所有写入、状态迁移、关系创建等事实，支持日志、回放、同步与可解释性。
- **搜索投影**：独立的 FTS5 投影层（`object_search_fts`、`object_chunks_fts`），不污染对象表。
- **Agent 访问层**：对外暴露 `object.query`/`read`/`write`、`link.write`、`event.append`、`search.query` 等统一能力，底层复杂 schema 对 Agent 不可见。
- **文件系统协同**：用户正文（Markdown、长文、报告）优先存储在 `sources/`/`wiki/` 文件系统中，数据库只保存编译后的元数据、关系、索引和事件。

设计的核心理念可以概括为：**简化发生在访问层，强类型保留在存储层；文件系统是真相层，数据库是编译层。**

## 2. Hermes 可借鉴之处

### 2.1 SQLite + FTS5 全文搜索的成熟实践

Hermes 的对话历史存储是 Orbit 数据库设计最直接的参考样本。其采用 SQLite WAL 模式 + FTS5 虚拟表的组合，已经验证了以下关键点：

- **FTS5 触发器同步模式**：Hermes 通过 `INSERT/DELETE/UPDATE` 触发器（`hermes_state.py` lines 93-112）将主表变更自动同步到 FTS5 虚拟表，实现零延迟的全文索引更新。Orbit 的 `object_search_fts` 可以直接复用这一模式——当类型表或 `object_index` 变更时，触发器自动刷新搜索投影，而非依赖应用层异步任务。
- **WAL 并发模型**：Hermes 使用 WAL（Write-Ahead Logging）模式实现并发读 + 单写入（line 157），这恰好匹配 Orbit 的使用场景——UI 多处并发读（搜索、backlinks 面板、Today 列表），但写入串行（用户操作或 Agent 执行）。Orbit 应在数据库初始化时显式启用 WAL 模式。
- **跨会话 FTS 搜索**：Hermes 的 `session_search` 工具通过 FTS5 搜索历史会话，然后用 LLM 生成摘要（`session_search_tool.py` lines 247-300）。这个"搜索 + LLM 重排/摘要"的模式，可以启发 Orbit 的 Agent 召回流程：先通过 `object_search_fts` 粗筛，再由 Agent 根据当前任务上下文做语义过滤。

### 2.2 会话谱系与事件溯源思路

Hermes 的 Session 管理中有一个精巧的设计：**会话谱系（Session Lineage）**。当上下文压缩发生时，系统创建新会话并通过 `parent_session_id` 链接旧会话（`hermes_state.py` lines 355-383），形成压缩链。这与 Orbit `events` 表的 `causation_id`/`correlation_id` 设计高度呼应。

具体启发：

- **事件因果链**：Hermes 的 `parent_session_id` 链可以类比为 Orbit `events` 表中 `causation_id` 的简化版。Orbit 应确保 `causation_id` 在实践中被严格填充——每当一个事件由另一个事件触发（如"Agent 创建关系"由"用户确认建议"触发），因果链必须完整。
- **压缩即新会话**：Hermes 的"压缩后开新会话"模式暗示 Orbit 在处理大规模事件回放时，可以引入 **snapshot 检查点** 机制——每 N 个事件或按时间窗口生成快照，避免从创世事件重建状态。方案 12 已提到 `snapshot_every_n_events` 策略，Hermes 的实践证明这在真实系统中确实需要。

### 2.3 结构化记忆的双态设计

Hermes 的 MemoryStore 采用**双态记忆设计**：`_system_prompt_snapshot`（冻结快照，用于系统提示词注入）和实时数据（被工具调用修改）。这个分离模式对 Orbit 的 Agent 访问层有直接参考价值：

- Orbit 的 Agent 逻辑视图（`agent_items_v`、`agent_links_v`、`agent_events_v`）在一次 Agent 交互会话中应该是**时点一致的快照**，而非实时变化的活数据。否则 Agent 在多步推理过程中可能读到自己上一步刚写入的数据，导致循环引用或决策偏移。
- 建议 Orbit 的 Agent 访问层实现为"会话级读快照 + 事务性写入"模式：Agent 读取时看到的是会话开始时的一致性快照，写入在事务提交后才对下一轮可见。

### 2.4 多提供商 Usage 规范化

Hermes 的 `CanonicalUsage` 数据结构（`usage_pricing.py` lines 420-550）将三种不同 API 格式（Anthropic、Codex、OpenAI）统一为标准格式。这对 Orbit 的 `events` 表设计有启发：

- `events.payload_json` 的内部结构应该有**规范化的 envelope**，而不是每种事件类型各自定义 JSON 结构。建议定义一套 `EventPayload` 类型族，确保 Agent 写入的事件、用户操作事件、系统事件都遵循统一的解析契约。

### 2.5 工具结果的三层防御策略

Hermes 对工具输出实施了三层防御（`tool_result_storage.py`）：单工具上限→大结果持久化→每轮聚合预算。这提醒 Orbit 在 Agent 写入路径上也需要类似防护：

- Agent 通过 `object.write` 写入的对象应有字段级大小限制（如 `summary` 不超过 2000 字符）。
- 批量建链（`link.write`）应有单次操作数量上限。
- 这些限制应在对象仓库层强制执行，而非依赖 Agent 自觉。

## 3. Claude Code 可借鉴之处

### 3.1 分层存储策略：内联 vs 外部引用

Claude Code 的历史管理中有一个值得关注的设计：**大文本的分层存储**（`history.ts` lines 355-409）。小文本内联存储在 JSONL 条目中，大文本取哈希后外部存储，条目中只保留 `contentHash` 引用。

这个模式直接适用于 Orbit 的类型表与文件系统协同：

- Orbit 的类型表（如 `articles`、`documents`）应只保存**元数据和摘要**，正文通过 `source_file_id` 引用文件系统中的原始内容。这与 Claude Code 的"大内容取哈希外部存储"思路完全一致。
- `object_index.summary` 字段应有明确的长度上限，超出部分只在文件系统中保留完整版本。这样搜索投影和列表渲染都能快速读取，深度阅读时再按需 hydrate。
- 建议 Orbit 引入 **content_hash** 机制：当文件内容变更时，通过哈希比对决定是否需要重编译对象状态，避免不必要的全量重建。Claude Code 的哈希外部存储证明了这个模式在生产环境中是可行的。

### 3.2 版本检测与缓存失效

Claude Code 的 `memoize` 模式贯穿整个代码库——从系统上下文（`getSystemContext`）到用户上下文（`getUserContext`），所有昂贵计算都只执行一次。结合其 `version_token` 类似的缓存机制，对 Orbit 的 `object_index.version_token` 设计有重要启发：

- Orbit 的 `version_token` 不应只是一个递增计数器，而应该是**内容感知的**（如基于关键字段的哈希）。这样当文件被外部编辑器修改但内容实际未变时，可以跳过重编译。
- 搜索投影的更新应基于 `version_token` 比对，而非盲目重建。Claude Code 的 memoize 模式证明："只在数据真正变化时才重算"是性能优化的核心原则。

### 3.3 多层配置叠加模型

Claude Code 的 5 层 CLAUDE.md 配置叠加（Managed → User Rules → Project → Local → TeamMem）是一个经过验证的多层配置融合方案。对 Orbit 的 `object_index.layer`（source | wiki | system）和权限模型有直接参考：

- Orbit 的对象可见性与可操作性应支持**层级覆盖**：系统级对象提供默认行为，项目级对象可覆盖，用户级对象拥有最终决定权。
- `object_index.layer` 的三级分层（source/wiki/system）可以进一步细化为优先级链：当同一 `object_uid` 在多个层中有不同状态时，应有明确的合并策略（如 user 覆盖 system，local 覆盖 remote）。
- 这对未来多端同步场景（方案 11/16）尤其关键：同步冲突的解决策略可以借鉴 Claude Code 的"server wins per-key"语义。

### 3.4 压缩后的资源恢复预算

Claude Code 在上下文压缩后有一个精细的资源恢复机制：5 个文件、50K token 总预算、每文件 5K token 上限、技能 25K token 预算（`compact.ts` lines 122-130）。这个"压缩 + 恢复"的模式对 Orbit 的事件回放设计有启发：

- 当 Orbit 需要从 `events` 表重建对象状态时（如搜索投影损坏、Today 缓存失效），不应一次性回放所有事件，而应设置**回放预算**：优先恢复最近 N 天的事件，或优先恢复特定对象类型的事件。
- 对 Agent 的回放请求（`replay.range(cursor)`）也应有 token/条数预算控制，避免 Agent 请求过大范围的事件流。

### 3.5 四级告警与自动触发机制

Claude Code 的上下文窗口管理有四级告警（warning → error → autoCompact → blocking），每级都有独立的 token 缓冲常量。这个渐进式告警模式可以迁移到 Orbit 的数据库容量管理：

- `events` 表会持续增长。Orbit 应有类似的四级策略：正常运行 → 告警（超过 N 万条事件建议归档）→ 自动归档（将旧事件移入冷存储）→ 硬限制（拒绝写入直到归档完成）。
- `object_search_fts` 的索引大小也应有监控和自动维护机制。

### 3.6 结构化摘要的九维分析框架

Claude Code 的压缩摘要使用 9 个结构化维度（Primary Request、Key Technical Concepts、Files and Code Sections、Errors and Fixes、Problem Solving、All User Messages、Pending Tasks、Current Work、Optional Next Step）。这对 Orbit 的 Agent 回放与解释能力有参考价值：

- 当 Agent 需要回答"为什么建议我今天做这个？"时，应按结构化维度组织回答：目标对齐（关联的 vision/directive）、进度依据（最近完成的任务）、阻塞关系（links 中的 blocks/depends_on）、时间因素（events 中的排程记录）。
- 建议 Orbit 定义一套**标准解释模板**，Agent 在生成解释时必须填充这些维度，而非自由发挥。

## 4. 不适用的设计

### 4.1 JSONL 文件作为对话存储（Claude Code）

Claude Code 使用 JSONL 文件 + 内存缓冲存储对话历史，每个会话对应一个 `.claude/{sessionId}.jsonl` 文件。这个方案**不适用于 Orbit**，原因是：

- Orbit 的对象网络需要**跨类型关联查询**（如"查找与这篇文章相关的所有任务"），JSONL 文件无法高效支持这类查询。
- Orbit 需要**事务性写入**（同一事务中写入类型表 + 更新 object_index + 写入 links + 追加 events），JSONL 无法提供事务保证。
- Orbit 的 `events` 表需要支持**游标分页和范围查询**，这在 SQLite 中是原生能力，但在 JSONL 中需要全量扫描。

Claude Code 选择 JSONL 的原因是其场景简单——线性对话流，无跨实体关系，不需要复杂查询。Orbit 的需求远超这个范围。

### 4.2 插件化记忆提供商（Hermes）

Hermes 支持 8+ 外部记忆提供商（Honcho、Mem0、ChromaDB 等），每个插件可以独立存储和检索记忆。这个**过度开放的插件模型不适合 Orbit**：

- Orbit 的对象数据库是整个系统的**核心基础设施**，不能容忍数据被分散到多个外部提供商中。
- 所有对象、关系、事件必须通过统一的 `object_index + links + events` 管理，否则 backlinks、搜索、回放都会出现数据孤岛。
- Orbit 应只有一个数据库实例（SQLite），而非多个可插拔的存储后端。

### 4.3 Hermes 的单体 Agent 类模式

Hermes 将几乎所有逻辑集中在一个 ~7500 行的 `AIAgent` 类中，包括数据库操作、工具调度、上下文管理。这种**单体架构不适合 Orbit 的对象仓库层**：

- Orbit 的每种对象类型应有独立的仓库（Repository）实现，负责该类型的字段校验、文件映射、索引同步。
- 统一访问层（`object.query`/`read`/`write`）应是路由层，而非单体实现。
- Claude Code 的"每工具一个目录"的模块化策略更适合 Orbit：每种对象类型一个仓库模块，各自负责自己的校验和持久化逻辑。

### 4.4 Claude Code 的纯内存状态管理

Claude Code 使用 Zustand Store + React Context 进行状态管理，会话状态主要驻留在内存中，只在需要时写入项目配置文件。这**不适合 Orbit**：

- Orbit 的对象状态必须**持久化到数据库**，不能依赖进程内存。用户可能在使用 Orbit 的同时用外部编辑器修改文件，系统需要在重启后完整恢复状态。
- Orbit 的多端同步（方案 11/16）要求所有状态变更都有持久化的事件记录，纯内存模型无法满足。

### 4.5 粗估 Token 计算（Hermes）

Hermes 使用 `chars / 4` 的粗估法计算 Token 数（`model_metadata.py` lines 1015-1048）。这种粗估方式**不应直接用于 Orbit 的搜索投影和事件存储**——Orbit 的字段长度限制和存储预算应基于精确的字符数或字节数，而非近似的 Token 估算。

## 5. 具体建议

### 建议一：采用 SQLite WAL + FTS5 触发器同步模式

**来源**：Hermes 的 `hermes_state.py`（FTS5 虚拟表 + 触发器自动同步）

**具体做法**：

```sql
-- 初始化时启用 WAL 模式
PRAGMA journal_mode=WAL;

-- object_search_fts 通过触发器与 object_index 自动同步
CREATE TRIGGER object_index_ai AFTER INSERT ON object_index BEGIN
  INSERT INTO object_search_fts(object_uid, title, summary, keywords)
  VALUES (new.object_uid, new.title, new.summary, '');
END;

CREATE TRIGGER object_index_au AFTER UPDATE ON object_index BEGIN
  DELETE FROM object_search_fts WHERE object_uid = old.object_uid;
  INSERT INTO object_search_fts(object_uid, title, summary, keywords)
  VALUES (new.object_uid, new.title, new.summary, '');
END;

CREATE TRIGGER object_index_ad AFTER DELETE ON object_index BEGIN
  DELETE FROM object_search_fts WHERE object_uid = old.object_uid;
END;
```

**理由**：Hermes 已在生产中验证这一模式——FTS5 触发器同步可靠、零延迟、无需应用层调度。Orbit 的搜索投影层可以直接复用，避免自建异步索引任务的复杂性。

### 建议二：引入 content_hash 实现增量编译

**来源**：Claude Code 的大文本哈希引用模式（`history.ts` lines 355-409）

**具体做法**：

- 在 `object_index` 中增加 `content_hash TEXT` 字段（或复用已有的 `version_token`，使其基于内容哈希生成）。
- 文件编译管线在检测到文件变更时，先比对 `content_hash`：若哈希未变，跳过重编译；若哈希已变，执行增量更新。
- 对于文件系统驱动的对象（`source_file_id IS NOT NULL`），`content_hash` 基于文件内容计算；对于数据库原生对象，`content_hash` 基于关键字段计算。

**理由**：外部编辑器可能触发多次文件保存事件（如自动保存），但内容可能未实际变化。content_hash 避免不必要的重编译和索引重建，降低系统负载。Claude Code 的哈希外部存储证明了这个模式的可行性。

### 建议三：Agent 访问层实现"会话级读快照 + 事务性写入"

**来源**：Hermes MemoryStore 的双态设计（冻结快照 vs 实时数据）

**具体做法**：

- Agent 进入一次交互会话时，对象访问层创建一个**读快照**（可通过 SQLite 的 `BEGIN IMMEDIATE` 或 `SAVEPOINT` 实现）。
- Agent 的所有 `object.query` / `object.read` 操作在快照上执行，保证多步推理过程中数据一致。
- Agent 的所有 `object.write` / `link.write` / `event.append` 操作写入暂存区，在会话结束或用户确认时统一提交。
- 高置信度写入（如自动建链）可以立即提交，但对当前会话的读快照不可见，避免自引用。

**理由**：Hermes 的冻结快照模式证明了"读写分离"在 Agent 场景中的必要性。如果 Agent 在多步推理过程中能读到自己上一步的写入，可能导致决策循环或数据不一致。

### 建议四：事件表引入分级归档与回放预算

**来源**：Claude Code 的四级告警机制 + 压缩后资源恢复预算

**具体做法**：

- **分级归档**：
  - 热区（最近 30 天事件）：保留在主 `events` 表中，支持快速查询。
  - 温区（30-180 天）：移入 `events_archive` 表，仅按需加载。
  - 冷区（180 天以上）：压缩导出为 JSONL 文件，保留在 `.orbit/archives/` 中。
- **回放预算**：
  - Agent 的 `replay.range(cursor)` 请求默认返回最多 500 条事件。
  - UI 的事件时间线默认加载最近 100 条，支持分页加载更早事件。
  - 快照点（每 1000 条事件或每周一次）允许跳过历史事件直接加载近期状态。

**理由**：Claude Code 的渐进式容量管理证明了"不设限的追加式存储最终需要治理"。Orbit 的 `events` 表会随使用时间持续增长，没有归档策略将导致查询性能退化。

### 建议五：定义标准化 EventPayload 类型族

**来源**：Hermes 的 `CanonicalUsage` 规范化 + Claude Code 的 9 维摘要框架

**具体做法**：

```typescript
// 所有事件 payload 的基础 envelope
interface EventPayload {
  version: 1;
  changes?: FieldChange[];      // 字段级变更记录
  context?: string;             // 操作上下文描述（可选）
  evidence?: ObjectUid[];       // 关联的证据对象（可选）
}

interface FieldChange {
  field: string;
  old_value?: unknown;
  new_value?: unknown;
}

// 事件类型枚举
type EventType =
  | 'object.created' | 'object.updated' | 'object.deleted'
  | 'link.created'   | 'link.deleted'
  | 'status.changed' | 'file.recompiled'
  | 'agent.suggested' | 'user.confirmed' | 'user.rejected';
```

**理由**：Hermes 的 `CanonicalUsage` 证明了"多来源数据统一为标准格式"的价值。Orbit 的 `events.payload_json` 如果没有标准化的 envelope，不同事件类型的 JSON 结构会逐渐分化，导致回放引擎和 UI 解析层需要处理大量特殊情况。提前定义类型族可以从源头避免这个问题。

## 6. 总结

通过对 Hermes Agent 和 Claude Code 在数据存储与持久化方面的深入分析，可以得出以下核心结论：

**Orbit 的设计方向是正确的。** 方案 12 选择 SQLite 关系型数据库作为底座、类型表保留强语义、统一索引层提供跨类型入口、文件系统保留正文真相——这套架构在两个参考项目中都得到了间接验证。Hermes 使用 SQLite + FTS5 管理对话和会话，证明了 SQLite 作为本地优先 AI Agent 数据基础设施的可靠性；Claude Code 使用 JSONL + 内存的轻量方案，则从反面证明了在需要复杂关系查询的场景中，SQLite 是更好的选择。

**两个项目最大的差异在于"是否需要关系"。** Hermes 和 Claude Code 处理的都是**线性对话流**——消息按时间排列，会话之间通过 ID 链接，不存在复杂的对象网络。Orbit 的核心挑战恰恰是对象之间的**多维关系**（任务支撑文章、研究派生结论、高亮关联笔记），这是两个参考项目都未面对的问题。因此，Orbit 的统一 `links` 表和 `object_uid` 引用协议是超越参考项目的原创设计，没有现成方案可以直接复制，需要在实现中谨慎迭代。

**最值得借鉴的不是具体实现，而是设计原则。** 从两个项目中提炼的通用原则包括：
- **分层存储**：大内容外部存储、元数据内联——Claude Code 的哈希引用模式。
- **读写分离**：Agent 读快照、写暂存——Hermes 的冻结快照模式。
- **渐进式治理**：告警→自动→硬限制——Claude Code 的四级策略。
- **触发器同步**：搜索投影通过触发器自动更新——Hermes 的 FTS5 实践。
- **标准化 envelope**：所有事件/数据遵循统一格式——Hermes 的 CanonicalUsage。

这些原则的共同特点是：**它们解决的不是"如何存储"的问题，而是"如何在存储增长后仍然可控"的问题。** Orbit 的关系型对象数据库在设计之初就应该把可治理性、可回放性和可增量编译性作为核心约束，而非事后补丁。两个参考项目——一个因为架构简单而不需要这些（Claude Code），一个因为架构复杂而不得不补上这些（Hermes）——都在不同角度印证了这一点。
