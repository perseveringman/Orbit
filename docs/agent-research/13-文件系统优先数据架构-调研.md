# 13. 文件系统优先数据架构 —— Hermes / Claude Code 调研报告

> **调研对象**：Hermes Agent（Python）、Claude Code（TypeScript）
> **分析维度**：文件系统操作、数据持久化、目录结构设计、AI 与用户内容边界
> **对应设计方案**：`docs/orbit-reboot/13-文件系统优先数据架构.md`

---

## 1. Orbit 设计方案摘要

Orbit 的数据架构核心主张是 **"文件系统承载真相、数据库承载图谱与能力、wiki 承载 AI 编译知识"** 的三层协作体系。方案回答五个系统级问题：用户可带走的内容是什么、对象系统如何跨三层统一、搜索与溯源如何建立在物理布局上、AI 读写边界在哪里、多端同步与加密如何与本地优先共存。

**三层物理隔离** 是架构的骨架：

- **`sources/`**：用户真相层。用户手写笔记、导入文章、日记、附件等所有"用户应能带走"的原始长期资产都落在这里。AI 对此层只读。
- **`wiki/`**：AI 编译层。Agent 从原始材料自动生成的实体页、概念页、dossier、synthesis 等知识产物。AI 可增量读写。
- **`.orbit/`**：系统层。SQLite 数据库、FTS/向量/分块索引、同步状态、密钥封装、缓存与编译规则。这里的数据可迁移但不应成为用户带走内容的前提。

方案采用 **统一对象系统**（`object_index + typed tables + links + events`）把三层文件与数据库原生对象（project、action、highlight、ai_chat 等）纳入同一张对象网络。文件对象通过 frontmatter 中的 `orbit_id` 与 `orbit_type` 与数据库主键对齐，Block 级对象通过 `(object_id, block_id)` 二级定位实现稳定锚点而非独立一级对象。

索引分为四层：文件清单索引（`file_index`，文件在哪）、全局对象索引（`object_index`，它是什么）、检索索引（FTS/chunk/vector，怎么找到它）、溯源索引（`block_provenance` / `compile_edges`，它从哪来）。

AI 编译写入有硬约束：只能写 `wiki/` 与 `.orbit/` 派生层，绝不静默改写 `sources/` 正文；所有 wiki 文件 frontmatter 必须声明来源链（`compiled_from`、`source_revisions`）；关系记录区分 `origin = human | ai`。

## 2. Hermes 可借鉴之处

### 2.1 SQLite + FTS5 全文搜索的成熟实践

Hermes 以 SQLite（WAL 模式）作为核心持久化引擎，并通过 FTS5 虚拟表实现跨会话消息全文搜索。这与 Orbit 方案中 `.orbit/db/orbit.sqlite` + `fts_objects` 的设计高度一致，证明了 **"SQLite 做索引与增值层"** 这条路径在真实 Agent 系统中是可行且成熟的。

具体而言，Hermes 的 `messages_fts` 虚拟表通过触发器自动与主表同步，这正是 Orbit 需要为 `file_index → object_index → fts_objects` 建立的自动投影管线的参考原型。Hermes 还在 `hermes_state.py` 中实现了完整的会话谱系（`parent_session_id` 链），这对 Orbit 的 `block_provenance` 和 `compile_edges` 溯源链设计有直接参考价值——当内容经过压缩、编译、迁移等变换时，通过 parent 链保持可追溯性。

### 2.2 文件级记忆存储与双态设计

Hermes 的内置记忆系统采用文件存储（`MEMORY.md` / `USER.md`），并实现了 **双态记忆设计**：`_system_prompt_snapshot` 是冻结快照用于系统提示词注入（不变），而 `memory_entries` / `user_entries` 是实时数据被工具调用修改。这种"冻结 vs 实时"的分离策略，对 Orbit 的 frontmatter 双写机制有启发——frontmatter 中的最小元数据是"冻结在文件中的真相快照"，而数据库中的索引是"实时可查询的增值层"。

更重要的是，Hermes 对记忆写入实施了 **内容安全扫描**（检测提示词注入、角色劫持、SSH 后门、API 密钥泄露等），这为 Orbit 的 AI 编译写入提供了一个安全模式参考：AI 写入 `wiki/` 之前，也应对生成内容做安全审查，防止注入攻击通过编译链路传播。

### 2.3 多后端执行隔离与文件系统安全

Hermes 实现了 6 种执行后端隔离（Docker/SSH/Modal/Daytona/Singularity/Local），其中 Docker 后端的安全加固（`cap-drop ALL`、`no-new-privileges`、`pids-limit`、`nosuid tmpfs`）和本地后端的环境变量剥离（阻止 100+ API 密钥变量）展示了如何在文件系统操作中建立安全边界。

这对 Orbit 的 AI 编译隔离有直接启发：当 Agent 执行编译任务（从 `sources/` 读取、向 `wiki/` 写入）时，可以借鉴 Hermes 的 Docker 隔离模式，将编译过程限制在只能读 `sources/` + 读写 `wiki/` + 读写 `.orbit/` 的权限沙箱中，从架构层面强制执行"AI 不能写 sources"的硬约束。

### 2.4 凭证文件注册与符号链接防护

Hermes 的 `credential_files.py` 实现了严格的路径安全：拒绝绝对路径、阻止路径遍历（`..` 序列）、解析符号链接后验证包含关系、防止符号链接逃逸。这对 Orbit 的 `sources/` 保护至关重要——外部编辑器或恶意插件可能通过符号链接试图绕过三层隔离边界，Orbit 需要类似的路径验证机制来确保 `wiki/` 中的操作不会通过 symlink 穿透到 `sources/`。

### 2.5 106 条危险模式的防御性检测

Hermes 定义了 106 个正则模式来检测危险操作（如 `rm -rf /`、`DROP TABLE`、`DELETE FROM` 无 WHERE、fork bomb、curl pipe to shell 等）。虽然 Orbit 的场景不完全是终端命令执行，但在 AI 编译层写入时，类似的模式检测可以用于：检测 AI 是否试图在 wiki frontmatter 中注入恶意脚本、检测编译输出是否包含可执行代码、防止 AI 通过 `compile_edges` 或 `links` 注入循环引用导致系统卡死。

## 3. Claude Code 可借鉴之处

### 3.1 文件系统为核心的记忆与配置体系

Claude Code 的记忆系统以 **文件系统为一等公民**，这与 Orbit 的"文件系统优先"哲学高度共鸣。具体体现在：

- **CLAUDE.md 五层叠加**：Managed（策略设置）→ User 全局（`~/.claude/`）→ Project（仓库内）→ Local（用户私有）→ Rules 目录。每一层用 `.md` 文件承载，通过 frontmatter 标注类型（Project / Local / TeamMem / AutoMem / User）。
- **记忆文件扫描**：`memoryScan.ts` 递归扫描 `.md` 文件，读取 frontmatter 元数据，按修改时间排序，有文件数上限。
- **四种记忆类型**：`user`、`feedback`、`project`、`reference`，每种类型存为独立 `.md` 文件。

这种"配置/记忆/规则都是 Markdown 文件 + frontmatter"的模式，正是 Orbit `sources/` 和 `wiki/` 的核心设计范式。Claude Code 证明了这种模式在大规模工程实践中是可行的，并且提供了具体的实现参考：frontmatter 标注类型、递归扫描策略、分层优先级合并。

### 3.2 声明式路径权限与分层安全模型

Claude Code 在 `filesystem.ts` 中实现了精细的 **声明式文件系统权限检查**，层次从高到低：Deny 规则优先 → 内部可编辑路径 → 安全验证 → 工作目录检查 → 内部可读路径 → 沙箱写入白名单 → Allow 规则。

这种分层权限模型对 Orbit 的三层隔离实施有极高参考价值：

| Claude Code 的层 | Orbit 可对应的语义 |
|---|---|
| Deny 规则优先 | AI 对 `sources/` 的写入无条件 deny |
| 内部可编辑路径 | AI 对 `wiki/` 的合规写入 allow |
| 安全验证 | frontmatter 完整性、`compile_edges` 合法性检查 |
| 工作目录检查 | 操作必须在 `$ORBIT_HOME/` 范围内 |
| 沙箱写入白名单 | `.orbit/` 下的索引与缓存可写 |

Claude Code 还实现了 `isDangerousRemovalPath()` 检测（阻止删除根目录、家目录、驱动器根等），Orbit 可以扩展为"阻止 AI 删除 `sources/` 下的任何文件"。

### 3.3 JSONL + 内存缓冲的轻量会话持久化

Claude Code 的对话历史采用 **JSONL 文件 + 内存缓冲** 方案，大文本按哈希外部存储，小文本内联。Transcript 保存在 `.claude/{sessionId}.jsonl`。这种"项目目录内、按会话 ID 命名、JSONL 格式"的方案对 Orbit 的 `ai_chat` 持久化有参考意义：AI 对话可以作为 `.orbit/chats/{session_id}.jsonl` 存储，兼具可读性与追加效率，同时通过 `object_index` 纳入对象图谱。

Claude Code 的大文本分流策略（小文本内联、大文本取哈希后外部存储）也对 Orbit 的 Block 存储有启发：block 正文预览可以内联在 `block_index` 中，而完整内容通过 content_hash 引用原文件。

### 3.4 自动压缩触发与资源恢复预算

Claude Code 实现了精细的上下文压缩机制：自动触发阈值（窗口大小 - 13K 缓冲）、图片/文档预剥离、PTL 重试截断、9 维结构化摘要。关键的是，压缩后有 **资源恢复预算**（`POST_COMPACT_TOKEN_BUDGET = 50,000`、`MAX_FILES_TO_RESTORE = 5`），确保压缩后 Agent 仍能访问关键上下文。

这对 Orbit 的 AI 编译系统有重要启发：当 Agent 执行长链条编译任务（从多个 sources 编译一份 wiki synthesis）时，如果中间上下文溢出，需要类似的"编译上下文恢复"机制——保留最关键的 source 摘要和已完成的编译进度，而不是从零开始。

### 3.5 团队记忆同步 API 与冲突语义

Claude Code 的团队记忆同步实现了 **server-wins-per-key** 的拉取语义和 **增量 hash 差异上传** 的推送语义，文件删除不传播。这种"按 key 覆盖、增量上传、删除不同步"的策略，与 Orbit 的跨端同步需求高度匹配：

- `sources/` 的同步应该更保守（冲突时保护用户编辑，不自动覆盖）；
- `wiki/` 的同步可以更激进（AI 编译的内容，server-wins 或重新编译都可接受）；
- `object_index` / `links` 的同步走增量 hash 差异；
- 索引删除不需要跨端同步（各端本地重建）。

### 3.6 六种权限模式的渐进式信任

Claude Code 提供了从 `plan`（最保守）到 `bypassPermissions`（最激进）的 6 种权限模式，以及 `allow / deny / ask` 三态行为。这种渐进式信任模型对 Orbit 的 AI 行为控制有启发意义：

- **Plan 模式**：AI 只能生成编译计划但不执行写入，用户审阅后才落盘到 `wiki/`。
- **Default 模式**：AI 写 `wiki/` 需要用户确认，写 `.orbit/` 索引自动执行。
- **Auto 模式**：AI 自由写 `wiki/` 和索引，但仍不能碰 `sources/`。
- **Full Auto**：AI 可以在 `sources/` 中执行有限操作（如补齐缺失的 frontmatter）。

这比 Orbit 方案中"AI 只读 sources / 可写 wiki"的二元划分更细腻，允许不同用户根据信任级别选择不同的 Agent 自治度。

## 4. 不适用的设计

### 4.1 Hermes 的单体 Agent 类与中心辐射式架构

Hermes 将几乎所有逻辑集中在一个 ~7500 行的 `Agent` 类中，构造函数接受 53 个参数。这种"一个类统治一切"的模式与 Orbit 的三层物理隔离哲学相矛盾。Orbit 需要的是清晰分层的模块化架构——文件监听器、解析器、对象投影器、索引器、编译器、审计器各司其职，而不是一个全能 Agent 在所有层上自由读写。

### 4.2 Hermes 的多消息平台网关

Hermes 的 14+ 消息平台网关（WhatsApp、Telegram、Discord 等）虽然展示了极高的覆盖广度，但 Orbit 是一个本地优先的知识管理工具，不需要消息平台接入。网关层的设计复杂度（异步审批队列、FIFO 阻塞、跨平台消息格式转换）对 Orbit 没有参考价值。

### 4.3 Claude Code 的 Anthropic 深度绑定

Claude Code 大量利用 Anthropic 独有特性（prompt caching、extended thinking、1M context window beta header、Statsig 功能门控、远程管理设置）。Orbit 应该是模型无关的，不能把 AI 编译能力绑定在单一 LLM 提供商上。特别是 Statsig 远程熔断开关这种依赖外部 SaaS 的安全机制，与 Orbit 的本地优先理念完全冲突。

### 4.4 Claude Code 的 React/Ink 终端 UI 引擎

Claude Code 自研了基于 React/Ink 的终端渲染引擎（Yoga 布局 + 虚拟 DOM），这是为了在终端中打造类 IDE 体验。Orbit 的编辑器场景（Block 编辑器、富文本编辑）不在终端中运行，这套终端 UI 技术栈没有迁移价值。

### 4.5 Hermes 的训练数据压缩与 RL 闭环

Hermes 内置的 `TrajectoryCompressor`（离线压缩完成对话为训练数据）和 RL 训练工具属于 AI 模型训练基础设施，与 Orbit 的"文件系统优先数据架构"完全不相关。Orbit 的 AI 是消费者而非训练对象。

### 4.6 两者的对话历史格式不直接适用

Hermes 的 SQLite 消息表和 Claude Code 的 JSONL transcript 都是面向"对话历史"设计的，而 Orbit 的核心数据是"用户长期知识资产"。对话历史是线性时序流，知识资产是图谱网络。Orbit 的 `ai_chat` 可以参考 JSONL 格式，但 `sources/` 和 `wiki/` 的持久化不能照搬对话存储模式。

## 5. 具体建议

### 建议一：借鉴 Claude Code 的声明式路径权限模型，为三层隔离实现 hard enforcement

**问题**：Orbit 方案定义了"AI 不能写 sources"的规则，但缺乏强制执行机制。仅靠 Agent 提示词约束是软约束，不足以保障用户真相层安全。

**建议**：在 Orbit 的文件系统操作层实现类似 Claude Code `filesystem.ts` 的声明式路径权限引擎：

```typescript
// 伪代码示意
type LayerPolicy = {
  sources: { ai_read: 'allow', ai_write: 'deny', ai_delete: 'deny' },
  wiki:    { ai_read: 'allow', ai_write: 'allow', ai_delete: 'allow' },
  orbit:   { ai_read: 'allow', ai_write: 'allow', ai_delete: 'deny' },
}
```

所有文件 I/O 操作经过统一的权限网关，在操作执行前检查目标路径所属层级并应用策略。路径解析时参考 Hermes 的符号链接防护，先 `realpath()` 再判断层级，防止 symlink 穿透。

### 建议二：将 Hermes 的 FTS5 自动同步触发器模式应用到 Orbit 的索引管线

**问题**：Orbit 方案定义了 `file_index → object_index → fts_objects` 的多层索引，但没有明确索引间的自动同步机制。

**建议**：参考 Hermes 的 FTS5 触发器模式，在 SQLite 层为 `object_index` 建立自动触发的 FTS 同步：

```sql
-- 参考 Hermes 的 messages_fts 触发器
CREATE TRIGGER object_fts_insert AFTER INSERT ON object_index BEGIN
  INSERT INTO fts_objects(rowid, title, summary, body_preview)
  VALUES (new.rowid, new.title, new.summary, new.body_preview);
END;
```

同时将 `file_index` 到 `object_index` 的投影做成事务性管线（Scan → Parse → Project 在同一个 SQLite 事务中完成），确保"看见文件"和"认识对象"是原子操作，避免中间态导致搜索结果不一致。

### 建议三：引入 Claude Code 的渐进式 AI 自治模型，替代硬二元划分

**问题**：Orbit 方案中"AI 只读 sources / 可写 wiki"是一个二元划分，但实际场景中存在灰色地带（如 AI 帮用户补齐 frontmatter、AI 建议修改用户笔记中的 broken link）。

**建议**：参考 Claude Code 的 6 种权限模式，为 Orbit 设计 3-4 级 AI 自治度：

| 级别 | sources/ | wiki/ | .orbit/ | 用例 |
|---|---|---|---|---|
| **观察模式** | 只读 | 只读 | 只读 | 纯分析，不做任何写入 |
| **编译模式**（默认） | 只读 | 读写 | 读写 | 标准 AI 编译流程 |
| **辅助模式** | 有限写入（仅 frontmatter 补全） | 读写 | 读写 | AI 帮用户维护元数据 |
| **协作模式** | 读写（需逐次确认） | 读写 | 读写 | AI 与用户深度协作创作 |

用户可以在项目级或全局级设置默认自治度，类似 Claude Code 的 `PermissionMode` 配置。

### 建议四：采用 Claude Code 的"大内容哈希外部存储"策略优化 Block 索引

**问题**：Orbit 的 `block_index` 需要存储 block 文本摘要、顺序、hash、anchor，但完整 block 内容不应复制到数据库中。

**建议**：参考 Claude Code 对话历史中的大文本分流策略——小文本内联存储，大文本取 `content_hash` 后外部引用：

```typescript
// block_index 中
if (blockContent.length <= MAX_INLINE_PREVIEW) {
  // 内联存储预览文本
  blockRecord.preview = blockContent
} else {
  // 只存 hash，完整内容在文件中通过 (object_id, block_id) 定位
  blockRecord.preview = blockContent.substring(0, MAX_INLINE_PREVIEW)
  blockRecord.content_hash = hashContent(blockContent)
}
```

这既保证了 `block_index` 的查询效率（预览文本可直接用于搜索结果展示），又避免了正文真相的双写冗余。

### 建议五：参考 Hermes 的内容安全扫描，为 AI 编译写入建立防注入机制

**问题**：Orbit 的 `wiki/` 是 AI 可自由写入的层，但如果 AI 生成的内容包含恶意注入（如在 frontmatter 中注入可执行代码、在 Markdown 中嵌入恶意链接、在 `compiled_from` 中构造循环引用），可能导致安全问题。

**建议**：参考 Hermes 记忆系统的安全扫描模式，在 AI 写入 `wiki/` 的管线中加入内容验证层：

1. **Frontmatter 校验**：`orbit_id` 格式合法、`compiled_from` 中的 object_id 在 `object_index` 中存在、`compile_kind` 在允许列表内。
2. **内容扫描**：检测注入模式（`<script>`、`javascript:` 协议、异常大量的 `[[wikilink]]` 可能暗示循环引用攻击）。
3. **图谱完整性**：写入 `compile_edges` 前检测是否会形成环路或超出深度限制。
4. **速率限制**：单次编译任务写入的文件数量和总大小设上限，防止 AI 失控导致磁盘填满。

## 6. 总结

Hermes 和 Claude Code 虽然都不是"文件系统优先的知识管理工具"，但它们在文件系统操作、数据持久化、安全边界三个维度上积累的工程经验，对 Orbit 方案 13 有切实的启发价值。

**从 Hermes 学到的核心经验** 是防御性工程：SQLite + FTS5 的成熟索引模式、文件级记忆的双态设计、符号链接防护与路径安全、多后端执行隔离、106 条危险模式检测。这些是"把文件系统当核心数据层"时不得不面对的安全与可靠性挑战，Hermes 提供了经过实战验证的解决方案。

**从 Claude Code 学到的核心经验** 是精细化工程：声明式路径权限的分层检查、Markdown 文件作为配置/记忆/规则的统一载体、大内容哈希外部存储、渐进式信任模型、团队记忆同步的冲突语义。这些是"把文件系统优先做到生产级品质"时需要的精细化设计，Claude Code 展示了如何在工程实现层面把"文件系统为核心"这个理念落地。

两个项目最大的共同启示是：**文件系统操作必须有一个统一的权限网关层。** 无论是 Hermes 的环境变量剥离 + 凭证文件注册，还是 Claude Code 的声明式路径规则 + 沙箱写入白名单，它们都不是把文件 I/O 散落在各处自由调用，而是通过集中化的安全层来管控所有文件操作。Orbit 的三层隔离（sources / wiki / .orbit）要真正成立，这个统一权限网关层是不可或缺的基础设施。

同时需要清醒认识到，两个项目的文件系统操作都是面向"工具执行"场景（Agent 执行命令、编辑代码），而 Orbit 面向的是"知识资产管理"场景（长期内容的创建、组织、检索、编译、同步）。前者的文件操作是短暂的、事务性的，后者的文件操作是持久的、演化的。因此 Orbit 不能照搬它们的存储格式和生命周期管理，而应该取其安全机制与索引策略的精华，融入自己"文件承载真相、数据库承载图谱、wiki 承载编译知识"的三层体系之中。
