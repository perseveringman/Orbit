# 02-人生规划系统：基于 Hermes Agent 与 Claude Code 的调研分析

> **调研目标**：从 Hermes Agent 和 Claude Code 两个成熟 AI Agent 系统中，提炼在任务管理、目标追踪、多层级编排方面对 Orbit「人生规划系统」设计有启发的架构模式与实现策略。
>
> **输入文档**：`02-人生规划系统.md`（Orbit 设计方案）、`00-comprehensive-summary.md`（全景对比）、`06-multi-agent-task-orchestration.md`（编排专项分析）

---

## 1. Orbit 设计方案摘要

Orbit 的人生规划系统定位为一个 **Agent 驱动的多时间尺度对齐系统**。其核心判断有三：愿景是罗盘而非清单；规划是连续推导而非静态层级图；规划的价值在于降低维护成本。

系统采用六层规划体系——五年愿景（vision）、年度方向（direction）、季度主题（theme）、月度目标（goal）、周承诺（commitment）、日计划（today_plan）——通过"逐层约束、逐层具体化"的方式，将人生方向感连续收敛为当天的工作节奏。这六层不是刚性父子树，而是一条持续的推导链：`vision → direction → theme/goal → commitment → today_plan`。

Agent 在其中扮演五个角色：**抽取**规划信号、**推导**下层计划、**重排**因现实变化而过时的承诺、**补桥**断裂的研究-写作-项目关系、**回顾**驱动下一轮校准。系统通过"规划差值"而非固定提醒器触发建议，强调"温和提醒，不做审计"的交互风格。

最关键的设计护栏是决策权的三层区分：用户独占决定权（vision、direction 等高层对象）、Agent 可提议但需用户确认（theme 调整、项目重排等）、Agent 可在约束内自动完成（日计划生成、回顾起草、关系补全等）。这确保了 Agent 可以压缩复杂性，但绝不替用户定义人生方向。

规划系统还与项目（03）、写作（08）、日志（09）等专题形成闭环：Journal 提供事实 → review 解释事实 → planning 根据解释调整 → 新一轮执行产生新事实。

## 2. Hermes 可借鉴之处

### 2.1 完善的批处理与 Cron 调度系统——对"规划节奏"的技术支撑

Orbit 设计了从五年到日的分层回顾节奏：半年一次愿景轻校准、季度交界复看方向、双周主题检查、月初目标设定、周承诺重排、日计划物化。这套节奏需要一个可靠的调度基础设施。

Hermes 在这方面有显著优势。其 Cron 调度系统支持三种模式——单次执行（once）、间隔重复（interval，如 `5m`、`1h`）和 Cron 表达式（如 `0 9 * * *`）。Gateway 每 60 秒 tick 一次调度器，用文件锁防止并发执行，输出按 `{job_id}/{timestamp}.md` 保存。这种调度系统可以直接映射到 Orbit 的规划节奏需求：

- **日回顾**：每天晚间触发 `planning.generate_review('day')`，自动聚合 Journal 事实
- **周承诺重排**：每周一早晨触发 `planning.materialize_commitments()`
- **双周主题检查**：每隔两周触发 `planning.suggest_reprioritization('quarter')`
- **季度回顾**：Cron 表达式触发深度回顾生成

此外，Hermes 的批处理运行器（batch_runner.py）支持 checkpoint 断点续传和故障恢复。对于 Orbit 来说，月末或季末的回顾生成可能需要遍历大量 Journal 条目、项目事件和研究材料，这种批处理能力可以保障回顾质量不因数据量大而中断。

**对 Orbit 的启发**：规划系统应内建一套轻量但可靠的调度层，支持从分钟级到季度级的触发周期，并具备 checkpoint 能力以应对长时间运行的回顾任务。

### 2.2 多层委托与深度控制——对"连续推导链"的编排参考

Orbit 的核心推导链 `vision → direction → theme → goal → commitment → today_plan` 本质上是一个多层级的 Agent 任务。每一层需要不同的上下文、不同的工具集、不同的推理深度。

Hermes 的 delegate_tool 提供了有价值的参考模式：子 Agent 通过 `_build_child_agent()` 构建，获得完全隔离的上下文——独立的会话历史、受限的工具集、聚焦的系统提示词。其最大嵌套深度为 2，最大并发为 3，通过 `DELEGATE_BLOCKED_TOOLS` 阻止子 Agent 执行不安全操作（如再次委托、记忆写入等）。

这种"层次化委托 + 上下文隔离 + 深度限制"模式可以映射到 Orbit 的推导链：

- **推导 Agent**：从 vision 推导 direction 时，需要完整的愿景上下文但不需要任务工具
- **具体化 Agent**：从 theme 推导 goal 时，需要项目和研究材料上下文
- **日计划 Agent**：生成 today_plan 时，需要日历、精力模式、阻塞关系等实时数据

每层 Agent 拥有不同的上下文窗口和工具权限，避免了将所有规划信息塞入单一 Agent 的上下文膨胀问题。

**对 Orbit 的启发**：推导链的每一步应当是一个独立的"推导任务"，拥有隔离的上下文和明确的工具边界，而非一个万能 Agent 一口气完成六层推导。

### 2.3 工具级并行安全检测——对"多信号抽取"的保障

Agent 在抽取规划信号时，需要同时分析 Journal、项目进展、研究材料、任务状态等多个数据源。Hermes 的工具级并行机制实现了精细的安全检测：

- 将工具分为 `_PARALLEL_SAFE_TOOLS`（只读工具，如 `read_file`、`web_search`）和 `_NEVER_PARALLEL_TOOLS`
- 路径作用域工具（`read_file`、`write_file`）需要检测路径是否重叠
- 最多 8 个工具并发执行

这种机制对 Orbit 的"规划差值"检测尤为重要：当系统需要判断用户是否偏离主题时，它需要同时读取 Journal 时间轴、项目活跃状态、研究积累量、写作输出频率等多个维度的数据。只读操作可以安全并行，但写入操作（如更新规划对象状态）必须串行化。

**对 Orbit 的启发**：规划信号抽取应支持多数据源并行读取，但规划对象的状态更新必须通过严格的写入控制保障一致性。

### 2.4 多平台网关与统一消息抽象——对"规划触达"的渠道参考

Hermes 的 14 平台网关系统（WhatsApp、Telegram、Slack、飞书等）和统一的 `MessageEvent` 抽象，对 Orbit 的"规划建议触达"有直接参考价值。Orbit 的设计中，Agent 需要在"规划差值"出现时主动提醒用户。这些提醒不应被局限在单一界面内——用户可能在手机上、在聊天工具中、在邮件里。

Hermes 的 `MessageType` 枚举（TEXT、COMMAND、DOCUMENT 等）和统一消息路由模式，可以让 Orbit 将规划建议通过用户偏好的渠道推送，而不是强制用户打开应用查看。

**对 Orbit 的启发**：规划建议的推送应抽象出统一的消息接口，支持未来扩展到多渠道通知。

## 3. Claude Code 可借鉴之处

### 3.1 多态任务类型系统——对"规划对象生命周期"的类型安全参考

Orbit 定义了 6 种规划对象（vision、direction、theme、goal、commitment、review），每种都有 `horizon`、`status`、`decision_mode` 等字段，以及 `suggested → confirmed → reframed → rescheduled → dropped → reviewed` 等事件类型。这套对象体系需要类型安全的实现。

Claude Code 的 7 种多态 TaskState 子类型提供了极佳的参考：所有任务共享 `TaskStateBase`（id、type、status、description、startTime 等），每种具体类型在此基础上扩展独特字段。例如 `LocalAgentTaskState` 增加了 `agentId`、`progress`、`pendingMessages`，而 `InProcessTeammateTaskState` 增加了 `identity`、`awaitingPlanApproval`、`permissionMode`。

这种模式可以直接映射到 Orbit 的规划对象：

```
PlanningObjectBase = { id, type, horizon, title, body, status, time_window, decision_mode, ... }
VisionState = PlanningObjectBase & { type: 'vision', narrative_text, ... }  // 无截止日期、无完成率
GoalState = PlanningObjectBase & { type: 'goal', success_signals, implemented_by, ... }
CommitmentState = PlanningObjectBase & { type: 'commitment', max_count: 5, week_start, ... }
```

Claude Code 的统一 5 态生命周期（pending → running → completed / failed / killed）也可以参考。Orbit 的 `draft → active → paused → done → archived` 状态流与之类似，但增加了 `paused` 和 `archived` 这两个对人生规划更重要的状态。

**对 Orbit 的启发**：规划对象应采用"共享基类 + 类型特化"的多态模式，用 TypeScript discriminated union 或类似机制保障类型安全。

### 3.2 Coordinator 四阶段工作流——对"推导-重排-回顾"的编排范式

Claude Code 的 Coordinator 模式定义了一个清晰的四阶段工作流：Research → Synthesis → Implementation → Verification。这个模式与 Orbit 的规划循环有惊人的结构相似性：

| Claude Code 阶段 | Orbit 规划循环对应 |
|---|---|
| Research（并行调研） | 抽取：从 Journal、项目、研究中并行提取规划信号 |
| Synthesis（综合判断） | 推导/重排：将多信号综合为规划建议 |
| Implementation（执行变更） | 确认与写入：用户确认后更新规划对象 |
| Verification（验证结果） | 回顾：检验规划变更是否真的改善了方向对齐 |

更重要的是 Coordinator 的编排哲学——"并行是你的超能力"：只读任务自由并行，写入任务按文件串行，验证可以与不同区域的实现并行。这种原则翻译到 Orbit 就是：信号抽取可以并行（同时分析 Journal、项目、研究），但规划对象更新必须串行（避免 theme 和 goal 的并发修改冲突），回顾可以与下一轮信号抽取并行。

Coordinator 的另一个关键设计是"Worker 完成后主动通知，Coordinator 不主动查询"。这与 Orbit 的"规划差值触发"机制一致——不是 Agent 轮询检查用户行为，而是当 Journal 写入、项目状态变更等事件发生时，自然触发规划系统重新评估。

**对 Orbit 的启发**：规划循环应采用事件驱动的 Coordinator 模式，而非轮询模式。Agent 订阅 Journal、项目、研究等系统的事件流，在事件触发时执行"抽取 → 综合 → 建议 → 验证"四阶段工作流。

### 3.3 双向通信与语义寻址——对"规划对象关系网络"的通信参考

Orbit 设计了丰富的关系类型（`derived_from`、`focuses_on`、`supports`、`implemented_by`、`informed_by`、`reflected_in`），需要跨模块的双向通信来维持这些关系的一致性。

Claude Code 的 `SendMessageTool` 提供了一个优雅的模式：通过语义寻址（agent_name 而非 task_index），支持点对点和广播（`"*"`），传输层支持内存队列、文件邮箱和 WebSocket 三种方式。结构化消息类型（`shutdown_request`、`plan_approval_response`）确保了通信的类型安全。

映射到 Orbit：当项目系统（03）中一个任务完成时，它应该能通过语义寻址向规划系统发送 `{ type: 'progress_update', goal_id: 'xxx', signal: 'milestone_reached' }` 消息；规划系统收到后，可以广播给所有相关的回顾 Agent 和日计划 Agent。这比通过数据库轮询关系变化更实时、更可靠。

**对 Orbit 的启发**：规划系统与其他模块之间应建立基于语义寻址的事件消息通信，支持点对点通知和广播，避免轮询式的关系同步。

### 3.4 决策审批机制与权限模式——对"三层决策权"的实现参考

Orbit 的核心护栏是三层决策权区分。Claude Code 提供了两个直接可参考的机制：

**6 级权限模式**（plan → full auto）：从最严格的"只做规划不执行"到完全自动化，用户可以按需选择。这与 Orbit 的 `user_written → agent_suggested → user_confirmed` 三级决策非常吻合。

**InProcessTeammateTaskState 的 `awaitingPlanApproval` 字段**：当 Agent 生成计划后，任务进入等待审批状态，必须收到 `plan_approval_response` 消息后才继续执行。这个模式可以直接用于 Orbit 的"Agent 可提议、需用户确认"场景——当 Agent 建议调整季度主题时，建议对象进入 `awaiting_approval` 状态，只有用户明确确认后才写入。

**对 Orbit 的启发**：规划对象应内建审批状态机。Agent 生成的高层建议（direction、theme 调整）必须经过显式的审批流程，而低层自动操作（日计划生成、关系补全）可以在约束内直接执行并保留撤销记录。

### 3.5 5 层配置叠加——对"规划上下文层级"的配置参考

Claude Code 的 5 层 CLAUDE.md 配置叠加（本地 → 项目 → 用户 → 团队 → 企业）是一种"近处的配置覆盖远处的默认"的模式。这与 Orbit 的规划层级有结构上的相似性：

- 五年愿景 = 最底层默认（最稳定、最少改变）
- 年度方向 = 项目级配置
- 季度主题 = 工作区级配置（当前聚焦）
- 周承诺 / 日计划 = 本地配置（最频繁变化，优先级最高）

这种叠加模式意味着：当 Agent 生成日计划时，它不需要重新解析五年愿景全文——它只需要读取已经收敛好的周承诺上下文，再结合当天的实时信息。这大幅降低了每次推导所需的上下文窗口。

**对 Orbit 的启发**：规划上下文应采用分层缓存策略。高层规划（vision、direction）变更极少，可以编译为稳定的上下文摘要；低层规划（commitment、today_plan）频繁变化，需要实时计算。Agent 在不同时间尺度的推导任务中，按需加载对应层级的缓存上下文，而非每次都从 vision 开始全链推导。

## 4. 不适用的设计

### 4.1 Hermes 的 RL 训练闭环

Hermes 内置了完整的强化学习训练工具链（基于 Atropos，集成 W&B，支持 LoRA 微调）。这对于一个 AI 基础设施平台有意义，但对 Orbit 的人生规划系统不适用。原因有二：第一，Orbit 的规划系统不需要训练自己的模型，它依赖的是通用 LLM 的推理能力；第二，用用户的人生数据做 RL 训练存在严重的隐私和伦理风险，与 Orbit "Agent 不替用户定义人生"的核心原则冲突。

### 4.2 Hermes 的 Mixture-of-Agents 多模型聚合

MoA 机制（同时调用 Claude、Gemini、GPT-4、DeepSeek，再用聚合器综合结果）在需要"多角度参考意见"的场景有价值，但对规划系统的日常推导来说过于昂贵且延迟过高。规划推导需要的是上下文理解深度和一致性，而非多模型的分歧与聚合。在特殊场景（如年度方向的深度反思）中或许可以选择性使用，但不应成为常态机制。

### 4.3 Claude Code 的终端渲染引擎

Claude Code 自研了基于 Ink + Yoga 的终端 UI 引擎，实现了完整的 Vim 模式和虚拟 DOM。这对开发者工具体验极致有意义，但 Orbit 的人生规划系统面向更广泛的用户群体，其交互界面不应局限于终端。规划系统的 UI 应该是跨端的、可视化的，更接近传统的 Web/App 交互范式。

### 4.4 Claude Code 的企业级管控层

Claude Code 的远程管理策略、OAuth 企业认证、团队记忆同步等企业级功能，对于 Orbit 这个面向个人用户的知识管理系统来说复杂度过高。Orbit 的规划系统是为单个用户设计的"个人人生操作系统"，不需要团队权限分层和企业合规审计。

### 4.5 Hermes 的 6 种执行后端

Docker、Modal、SSH、Daytona、Singularity 等多种执行环境，服务于"一个 Agent 处处运行"的基础设施愿景。但 Orbit 的规划系统不需要在 HPC 集群上运行 Singularity 容器或在 Modal 上执行 Serverless 函数。规划推导是纯文本推理任务，本地执行即可满足需求。

## 5. 具体建议

### 建议一：构建事件驱动的分层调度器

**问题**：Orbit 的规划节奏覆盖从日到年的六个时间尺度，需要可靠的调度基础设施。

**建议**：参考 Hermes 的 Cron 调度系统，为 Orbit 构建一个分层调度器。调度器支持两种触发模式——定时触发（Cron 表达式，用于固定节奏如周一生成周承诺、月末生成回顾）和事件触发（参考 Claude Code 的 Coordinator 事件通知模式，当 Journal 写入、项目状态变更、研究积累达到阈值时自动触发规划差值检测）。

调度器应内置 checkpoint 能力：季度回顾可能需要分析三个月的 Journal 和项目数据，如果中途因 LLM API 限流失败，应能从断点恢复而非重新开始。每次调度触发的结果保存为结构化日志，便于后续回顾分析。

### 建议二：实现推导链的隔离上下文编排

**问题**：六层推导链如果在单一 Agent 上下文中完成，将导致上下文窗口膨胀和推理质量下降。

**建议**：借鉴 Hermes 的 delegate_tool 隔离模式和 Claude Code 的 Coordinator 四阶段工作流，将推导链拆分为独立的推导任务。每个推导任务拥有隔离的上下文——推导 direction 时加载 vision 全文和年度事实摘要；推导 today_plan 时只加载本周承诺、当天日历和最近 Journal。

参考 Claude Code 的分层配置缓存策略，高层规划对象（vision、direction）编译为稳定的上下文摘要，只在变更时重新生成；低层规划对象实时计算。这样，日计划 Agent 的上下文窗口始终可控，不会因为五年愿景的长文本而浪费 token 预算。

推导任务之间通过结构化消息通信（参考 Claude Code 的 SendMessageTool），上层推导的输出作为下层推导的输入约束，而非直接共享完整上下文。

### 建议三：为规划对象实现多态类型系统与审批状态机

**问题**：6 种规划对象和 3 层决策权需要类型安全的实现和明确的审批流程。

**建议**：参考 Claude Code 的 discriminated union TaskState 模式，为规划对象实现"共享基类 + 类型特化"的多态体系。`PlanningObjectBase` 定义通用字段（id、type、horizon、status、decision_mode 等），每种具体类型扩展自身特有字段（如 vision 的 narrative_text 不带截止日期，goal 的 success_signals 等）。

在此之上，参考 Claude Code 的 `awaitingPlanApproval` 模式，为规划对象增加审批状态机。当 Agent 建议调整 theme 或 direction 时，建议以 `{ status: 'awaiting_approval', suggested_by: 'agent', confidence: 0.8 }` 的形态存在，只有收到用户的 `approval_response` 后才转为 `active`。低层对象（today_plan、关系补全）则跳过审批直接生效，但保留完整的变更日志支持撤销。

### 建议四：设计规划差值检测引擎

**问题**：Orbit 的主动建议不靠固定提醒器，而靠"规划差值"触发，需要一套持续运行的偏差检测机制。

**建议**：结合 Hermes 的工具级并行读取能力和 Claude Code 的事件通知机制，设计一个轻量的规划差值检测引擎。引擎订阅来自 Journal（09）、项目（03）、研究和写作（08）等系统的事件流，当事件积累到一定阈值时，触发并行的多维度信号抽取：

- **方向稀释检测**：并行读取 theme 对应项目的最近两周事件，计算推进频率
- **项目漂移检测**：扫描活跃任务的上层归属关系，识别无 theme/goal 挂靠的孤立任务
- **研究悬空检测**：统计研究材料积累量与下游产出（文章、项目决策）的比率
- **节奏失真检测**：对比周承诺数量与实际完成数量的长期趋势

所有检测为只读操作，可以安全并行（参考 Hermes 的 `_PARALLEL_SAFE_TOOLS` 分类）。只有检测结果超过阈值时，才生成建议推送给用户，避免"洞察轰炸"。

### 建议五：建立跨模块语义寻址的消息总线

**问题**：规划系统不是孤立模块，需要与项目、写作、日志、研究等系统实时联动。

**建议**：参考 Claude Code 的 `SendMessageTool` 语义寻址和多传输层设计，为 Orbit 构建一个轻量的内部消息总线。各模块通过语义地址互相通信：

- 项目系统发送 `{ to: 'planning', type: 'task_completed', goal_id: '...', signal: '...' }`
- 规划系统发送 `{ to: 'journal', type: 'review_generated', period: 'week', content: '...' }`
- 研究系统发送 `{ to: 'planning', type: 'research_threshold', theme_id: '...', count: 15 }`

消息总线支持点对点和广播（`{ to: '*', type: 'direction_changed' }`），传输层初期用内存队列，后续可扩展为持久化消息队列。这种设计使得规划系统的"补桥"功能（发现研究-写作-项目间的断裂并建议关联）可以通过订阅多模块事件流自然实现，而非周期性全量扫描。

## 6. 总结

Hermes Agent 和 Claude Code 虽然都不是"人生规划系统"，但它们在任务管理、多层级编排和 Agent 协作方面积累的工程经验，对 Orbit 的设计有三层启发：

**基础设施层**：Hermes 的 Cron 调度系统和批处理 checkpoint 机制，为 Orbit 的多时间尺度规划节奏提供了成熟的技术模式。规划系统需要的不是花哨的定时器，而是能从日级到季度级可靠触发、支持故障恢复的调度基础设施。

**编排层**：Claude Code 的 Coordinator 四阶段工作流（Research → Synthesis → Implementation → Verification）和 Hermes 的隔离上下文委托模式，共同指向了 Orbit 推导链应该采用的架构——不是一个全知 Agent 从 vision 到 today_plan 一口气推导，而是多个专精 Agent 在隔离上下文中逐层收敛，通过结构化消息传递约束。

**治理层**：Claude Code 的多级权限模式和审批状态机，为 Orbit "三层决策权"的实现提供了直接参考。规划系统最大的风险不是技术失败，而是 Agent 逐渐替代用户做出人生决定。类型安全的审批流程（而非仅靠约定和文档）是防止这种滑坡的技术护栏。

最终，Orbit 的人生规划系统应当成为一个**事件驱动、分层编排、类型安全、尊重用户主权**的连续对齐系统。它从 Hermes 借鉴调度与并行的工程能力，从 Claude Code 借鉴编排与治理的架构智慧，但其核心灵魂——"Agent 可以压缩复杂性，但不能替用户定义人生方向"——是 Orbit 独有的设计哲学，也是整个系统最不可妥协的底线。
