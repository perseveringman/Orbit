# 18-设计系统与换肤 调研报告

> **调研目标**：结合 Hermes Agent 与 Claude Code 两个成熟 AI Agent 项目在设计系统、主题管理、组件库和样式架构方面的实践，为 Orbit 的「设计系统与换肤」方案提供参考和建议。
>
> **参考文档**：Orbit 设计方案 `18-设计系统与换肤.md`；Hermes/Claude Code 综合分析 `00-comprehensive-summary.md`；CLI/UX 交互分析 `05-cli-ux-interaction.md`

---

## 1. Orbit 设计方案摘要

Orbit 的设计系统方案参照 Capacities 构建，核心涵盖以下几个方面：

### 1.1 布局系统

采用全屏三栏弹性布局（Three-Panel Layout）：左侧边栏固定 288px、中间主内容区自适应（`flex: 1`）、右侧面板固定 431px。各区域通过语义化 CSS 变量控制背景与边框，形成清晰的层次关系。

### 1.2 设计令牌（Design Tokens）

这是方案的核心亮点之一。采用 **OKLCH 色彩空间** 定义了一套完整的语义化令牌系统：

- **背景色阶层**：`--bg-back` → `--bg-base` → `--bg-front` → `--bg-el`，形成从深到浅的 4 级背景层次
- **文本色阶层**：`--text-primary` → `--text-secondary` → `--text-subtle`，3 级文字层次
- **边框与交互色**：`--border-base`、`--border-el-hover` 覆盖静态与悬停状态
- **对象类型专属色**：为 Project（薄荷绿）、Quote（腮红红）、Atomic note（奶油黄）、Daily Note（蓝紫色）等类型定义了恒定不变的专属色板，不随明暗主题切换

### 1.3 排版与间距

- 字体以 Inter 为主、Courier New 为代码体
- 基准字号 15px，覆盖 11px—48px 的完整阶层
- 间距系统基于 4px 基准
- 圆角分 3 级：8px（基础）、12px（中型面板）、16px（大型模态）

### 1.4 组件规范

定义了按钮、对象卡片、下拉菜单/视图切换器、模态框（Command Palette）、详情编辑器 5 类核心组件的视觉规范，包括尺寸、颜色令牌引用、悬停动效等。

### 1.5 换肤机制

- **CSS 变量驱动**：所有颜色均通过 CSS Custom Properties 定义
- **主题切换**：通过 `<html data-theme="light|dark">` 属性切换
- **持久化**：使用 `localStorage` 保存用户偏好
- **技术兼容**：设计兼容 Tailwind CSS v4 机制

**总体评价**：Orbit 的设计方案在令牌化层面做得非常扎实，OKLCH 色彩空间的选择具有前瞻性（感知均匀性优于 HSL），语义化命名也保证了可维护性。但在组件封装策略、主题扩展能力和设计系统工程化方面，还有进一步深化的空间。

## 2. Hermes Agent 可借鉴之处

Hermes Agent 虽然是 Python CLI 工具而非 Web 应用，但其在皮肤系统和样式架构上的设计哲学对 Orbit 有明确的启发价值。

### 2.1 数据驱动的皮肤引擎（Skin Engine）

Hermes 实现了一个完整的 YAML 驱动皮肤系统，这是其在设计系统方面最值得 Orbit 借鉴的理念。其 `SkinConfig` 数据类包含 `colors`（颜色映射）、`spinner`（加载动画）、`branding`（品牌标识）、`tool_prefix`（工具前缀符号）和 `tool_emojis`（工具图标）等维度。

**对 Orbit 的启发**：

- **令牌文件外置化**：Orbit 当前方案中 CSS 变量写在全局 CSS 的 `:root` 下，这对于仅支持明暗两套主题是足够的。但如果未来需要支持更多主题（如"护眼模式"、"高对比度模式"、品牌定制主题），可以借鉴 Hermes 将令牌定义从代码中抽离到独立配置文件的思路。例如，将每套主题的令牌定义为独立的 JSON/YAML 文件，运行时动态加载。
- **皮肤加载优先级**：Hermes 的 `load_skin()` 实现了"用户自定义 → 内置"的回退链。Orbit 可以考虑类似机制：先读取用户自定义主题配置，未定义的令牌回退到默认主题值，实现渐进式主题覆盖。
- **品牌维度的可定制性**：Hermes 不仅允许自定义颜色，还支持品牌名称、图标、表情等非色彩维度的定制。对于 Orbit 这种知识管理工具，可以考虑让对象类型的图标、标签等也纳入设计令牌管理。

### 2.2 功能性颜色命名（Functional Color Naming）

Hermes 的颜色令牌命名偏功能性：`banner_border`、`prompt`、`ui_error` 等，直接绑定到具体 UI 元素。这种命名方式的优势在于直观性——开发者可以立即知道某个令牌用在哪里。

**对 Orbit 的启发**：Orbit 当前的令牌命名采用语义化分层策略（`--bg-back`、`--text-primary`），这比 Hermes 的功能性命名更具系统性和可扩展性。不过，对于一些高度专用的场景——例如对象类型专属色——可以考虑引入功能性命名作为补充层（例如 `--color-project-bg`、`--color-quote-text`），形成"语义层 + 功能层"的双层令牌结构。这在 Orbit 方案中实际上已有雏形（对象类型专属色部分），可以进一步明确为架构模式。

### 2.3 多层样式继承

Hermes 的 TUI 定义了 30+ 样式类，且所有颜色从皮肤配置继承。这种"配置层 → 样式类 → 组件"的三层分离让主题切换时只需替换配置层，中间的样式类和组件层完全不变。

**对 Orbit 的启发**：Orbit 方案中 CSS 变量已实现"令牌层 → 组件层"的分离。建议进一步明确三层架构：

1. **原始层（Primitive Tokens）**：OKLCH 原始值定义（如 `--oklch-primary-l: 0.389`）
2. **语义层（Semantic Tokens）**：语义化映射（如 `--bg-base: oklch(var(--oklch-bg-base-l) ...)`）
3. **组件层（Component Tokens）**：组件级别映射（如 `--button-primary-bg: var(--bg-button-primary)`）

这种三层架构在换肤时只需替换原始层，语义层和组件层通过变量引用自动级联更新。

### 2.4 颜色降级策略

Hermes 实现了 `should_use_color()` 函数，检测终端环境并优雅降级。虽然 Orbit 是 Web 应用不存在终端限制，但这种"环境感知 + 优雅降级"的设计思想值得借鉴——例如检测用户系统是否开启了"减少动画"偏好（`prefers-reduced-motion`）、是否启用了高对比度模式（`prefers-contrast`），并自动调整设计令牌。

## 3. Claude Code 可借鉴之处

Claude Code 在设计系统工程化方面的实践更加精致，与 Orbit 作为 Web 应用的技术栈也更为贴近。

### 3.1 语义化主题类型系统（Semantic Theme Type System）

Claude Code 的 `Theme` 类型定义是一个教科书级的设计令牌类型化方案。它将颜色分为品牌色（`claude`、`claudeShimmer`）、语义色（`success`、`error`、`warning`）、功能色（`diffAdded`、`diffRemoved`）和 Agent 专用色（`red_FOR_SUBAGENTS_ONLY`）四大类，每个令牌都有 TypeScript 类型约束。

**对 Orbit 的启发**：

- **类型安全的令牌系统**：Orbit 使用 TypeScript，应该为设计令牌建立完整的类型定义。可以创建 `DesignTokens` 接口，将所有 CSS 变量名映射为 TypeScript 类型，确保组件引用令牌时有类型检查和自动补全。这比纯 CSS 变量方案更安全——拼写错误、缺失令牌都能在编译期被发现。
- **Agent/实体专用色的管理模式**：Claude Code 为每个 sub-agent 分配独立颜色的做法，与 Orbit 为每个对象类型分配专属色的需求高度一致。Claude Code 使用 `_FOR_SUBAGENTS_ONLY` 后缀约束使用范围的做法虽然简单粗暴，但传递了一个重要的设计原则——**某些令牌应该有明确的使用范围限制**。Orbit 可以通过 TypeScript 工具类型或 ESLint 规则实现更优雅的令牌访问控制。

### 3.2 ThemeProvider 与系统主题跟随

Claude Code 的 `ThemeProvider` 实现了三种模式：亮色、暗色、跟随系统。它使用 React Context 提供主题状态，支持"预览"功能（`previewTheme`），允许用户在确认切换前预览新主题效果。

**对 Orbit 的启发**：

- **跟随系统主题**：Orbit 方案中仅提到了明暗手动切换和 `localStorage` 持久化。建议增加 `prefers-color-scheme` 媒体查询支持，实现"跟随系统"第三选项。实现方式可以是：在 `localStorage` 中存储 `"system"` | `"light"` | `"dark"` 三态，当值为 `"system"` 时监听 `matchMedia('(prefers-color-scheme: dark)')` 变化事件。
- **主题预览能力**：Claude Code 的预览机制允许用户在不提交的情况下试看主题效果。这对 Orbit 有直接参考价值——在设置面板中，当用户悬停或点击某个主题选项时，应用立即预览效果，用户确认后才持久化。技术实现上，可以维护一个临时的 `previewTheme` 状态覆盖到 CSS 变量上。
- **React Context 集成**：Orbit 当前方案通过 `data-theme` 属性切换主题。建议同时在 React 层建立 `ThemeContext`，让组件可以通过 `useTheme()` Hook 获取当前主题名和切换方法，而不仅仅依赖 CSS 选择器。这为后续实现条件渲染（如在不同主题下使用不同的图标或组件变体）打下基础。

### 3.3 设计系统组件库

Claude Code 构建了包含 16 个组件的设计系统库（`src/components/design-system/`），涵盖 `Byline`、`Dialog`、`Divider`、`FuzzyPicker`、`KeyboardShortcutHint`、`ListItem`、`LoadingState`、`Pane`、`ProgressBar`、`Ratchet`、`StatusIcon`、`Tabs`、`ThemedBox`、`ThemedText`、`ThemeProvider`、`color.ts`。这些组件全部与主题系统深度集成，通过 `ThemedText` 和 `ThemedBox` 两个基础组件确保所有文本和容器都自动使用当前主题颜色。

**对 Orbit 的启发**：

- **"ThemedBox/ThemedText"模式**：这是一个非常精巧的设计。Orbit 可以创建类似的 `<ThemedBox>` 和 `<ThemedText>` 基础组件，它们自动从 ThemeContext 中读取对应的 CSS 变量，确保任何使用这些组件的地方都天然支持主题切换。这比让每个业务组件自行引用 CSS 变量更统一、更不容易出错。
- **语义化输出风格（Output Styles）**：Claude Code 的 `OUTPUT_STYLE_CONFIG` 将输出分为 `default`、`Explanatory`、`Learning` 三种风格。这提示我们：设计系统不仅仅是视觉层面的，也可以包含"内容呈现策略"。Orbit 作为知识管理工具，可以考虑为不同的阅读场景（如速览模式、深度阅读模式、演示模式）定义不同的排版和间距令牌组合。
- **`color.ts` 集中管理**：Claude Code 将颜色逻辑集中在 `color.ts` 中。Orbit 也应该创建一个集中的令牌管理模块，而非将 CSS 变量分散在多个样式文件中。这个模块可以导出令牌常量、主题切换工具函数、以及用于 JS-in-CSS 场景的颜色值。

### 3.4 Flexbox 布局引擎与组件化渲染

Claude Code 基于 Yoga 布局引擎实现了 CSS Flexbox 风格的布局系统，配合 React 组件化渲染。虽然 Orbit 作为 Web 应用天然具备 CSS Flexbox 能力，但 Claude Code 的架构思路仍有借鉴价值：**将布局定义从样式中分离出来，作为独立的"布局令牌"管理**。Orbit 的三栏布局中 288px、431px 等固定值，以及 24px 内边距、58px 顶栏高度等布局常量，都应该纳入设计令牌体系，而非硬编码在各组件的 CSS 中。

## 4. 不适用的设计

### 4.1 Hermes 的 YAML 皮肤文件系统

Hermes 通过 YAML 文件定义皮肤，用户可以在文件系统中创建自定义皮肤。这种方式适合 CLI 工具（开发者习惯编辑配置文件），但对 Orbit 这种面向知识工作者的 Web 应用并不合适。Orbit 的用户更期望通过图形化的设置面板切换主题，而非编辑配置文件。此外，Web 应用没有直接访问本地文件系统的能力（除非走 Electron/Tauri），YAML 配置加载模式在浏览器环境中缺乏天然支持。

### 4.2 Hermes 的品牌定制深度

Hermes 允许用户自定义 agent_name、response_label、prompt_symbol 等品牌元素。这对于一个面向开发者的 AI Agent 平台有意义（企业可能需要白标定制），但对 Orbit 来说过度设计。Orbit 是一个有明确品牌标识的产品，用户不需要也不应该修改应用名称和品牌符号。设计系统应该聚焦在色彩、排版、间距等视觉变量的可切换性上，品牌层保持固定。

### 4.3 Claude Code 的终端特定渲染优化

Claude Code 投入了大量工程资源在终端渲染引擎上（自研 Ink、Yoga 布局、虚拟 DOM diff），这些都是为了弥补终端环境缺乏浏览器渲染能力的不足。Orbit 运行在浏览器或桌面端 WebView 中，天然拥有 CSS、DOM、Canvas 等完整的渲染能力，无需重复造轮子。不过，Claude Code 对渲染性能的极致追求（LRU 缓存、虚拟滚动、Token 缓存）在 Orbit 处理大量对象卡片渲染时值得参考。

### 4.4 Claude Code 的 sub-agent 颜色硬编码

Claude Code 使用 `_FOR_SUBAGENTS_ONLY` 后缀命名约定来限制 agent 专用色的使用范围，这是一种非常"约定优于配置"的做法，缺乏编译时强制保证。Orbit 应该采用更工程化的方式来管理对象类型专属色——例如通过独立的 `objectTypeColors` 配置模块，或通过 CSS 变量的 scope（如 `[data-object-type="project"]`）来实现作用域隔离。

### 4.5 两者的颜色空间局限

Hermes 和 Claude Code 均使用传统的 RGB/Hex 颜色表示。Orbit 方案已经选择了更先进的 OKLCH 色彩空间，这是正确的方向。OKLCH 的感知均匀性使得在生成一致的配色方案（尤其是同色相不同亮度的变体）时，比 RGB/HSL 更加可预测和和谐。这是 Orbit 方案的一个优势，不需要向两个参考项目对齐。

## 5. 具体建议

### 建议一：建立三层令牌架构 + TypeScript 类型守护

**问题**：当前方案的令牌是单层 CSS 变量，直接在 `:root` 下定义语义值。当主题数量增加或需要组件级覆盖时，单层结构会导致变量爆炸。

**建议**：建立三层令牌架构：

```
原始层 (Primitive)    →  语义层 (Semantic)        →  组件层 (Component)
--oklch-mint-50       →  --bg-el                  →  --card-bg
--oklch-mint-400      →  --text-primary            →  --card-title-color
```

同时为所有令牌创建 TypeScript 类型定义：

```typescript
// tokens/types.ts
interface SemanticTokens {
  bgBack: string;
  bgBase: string;
  bgFront: string;
  bgEl: string;
  textPrimary: string;
  textSecondary: string;
  textSubtle: string;
  borderBase: string;
  // ...
}

interface ThemeDefinition {
  name: string;
  tokens: SemanticTokens;
  objectTypeColors: Record<ObjectType, ObjectTypeColorSet>;
}
```

这样在代码中引用令牌时有类型检查，主题定义时有完整性保证，新增主题时不会遗漏令牌。

### 建议二：实现 ThemeProvider + 三态切换 + 预览

**问题**：当前方案仅支持明暗两态手动切换。

**建议**：借鉴 Claude Code 的 ThemeProvider 模式，实现完整的主题管理方案：

1. **三态切换**：`light` | `dark` | `system`（跟随系统偏好）
2. **React Context**：通过 `ThemeContext` 提供 `currentTheme`、`setTheme()`、`previewTheme()` 能力
3. **主题预览**：在设置面板中悬停/选择主题时实时预览，确认后持久化
4. **初始化优化**：在 `<head>` 中注入同步脚本读取 `localStorage`，避免首屏闪烁（FOUC）

关键实现细节：`data-theme` 属性继续作为 CSS 切换的机制，ThemeProvider 负责管理状态并同步到 DOM。两者配合而非替代。

### 建议三：为对象类型专属色建立独立管理模块

**问题**：对象类型专属色（Project/Quote/Atomic Note 等）是 Orbit 设计系统的独特需求，需要特殊管理策略。

**建议**：

1. **独立配置模块**：将对象类型色从全局令牌中独立出来，建立 `objectTypeTheme.ts`
2. **CSS 作用域**：使用 `[data-object-type="project"]` 选择器作用域，而非全局变量
3. **运行时扩展**：当用户创建新的自定义对象类型时，需要能动态生成合适的专属色。可以利用 OKLCH 色彩空间的特性，通过固定 L（亮度）和 C（色度）值、仅旋转 H（色相）来生成感知上一致的新配色
4. **恒定性保证**：明确这些颜色不随明暗主题变化的语义——通过将它们定义在全局作用域而非主题作用域来实现

### 建议四：构建设计系统基础组件层

**问题**：方案定义了 5 类核心组件的视觉规范，但缺少组件封装策略。

**建议**：借鉴 Claude Code 的 `design-system/` 目录模式，在 Orbit 中建立独立的设计系统包（或目录），包含：

1. **基础组件**：
   - `<ThemedBox>`：自动应用主题背景色、边框色、圆角的容器组件
   - `<ThemedText>`：自动应用主题文本色、字号、字重的文本组件
   - `<Divider>`：使用 `--border-base` 的分割线组件
2. **复合组件**：基于基础组件构建 Button、ObjectCard、ViewSwitcher、Modal、DetailEditor
3. **令牌 Hook**：`useTheme()` 返回当前主题信息，`useObjectTypeColor(type)` 返回对象类型色
4. **原则**：业务组件禁止直接使用 CSS 变量，必须通过设计系统组件或 Hook 间接引用。这确保了主题系统的一致性。

### 建议五：纳入可访问性和环境适配令牌

**问题**：当前方案聚焦视觉令牌，缺少对可访问性和用户环境偏好的考量。

**建议**：

1. **`prefers-reduced-motion` 适配**：方案中定义了 0.15s 过渡动效和 `cubic-bezier` 缓动曲线。当系统启用"减少动画"时，应自动将过渡时长降为 0 或极短（如 0.01s）。可以通过 CSS 媒体查询或定义 `--motion-duration` 令牌实现。
2. **`prefers-contrast` 适配**：为高对比度模式准备一套令牌覆盖，增强边框对比度、加深文本颜色。
3. **`prefers-color-scheme` 适配**：与建议二的"跟随系统"配合。
4. **焦点可见性**：方案中的组件规范缺少键盘焦点状态的定义。建议增加 `--focus-ring-color` 和 `--focus-ring-offset` 令牌，确保所有交互组件在键盘导航时有清晰的焦点指示。

## 6. 总结

通过对 Hermes Agent 和 Claude Code 设计系统实践的深入分析，可以提炼出对 Orbit「设计系统与换肤」方案的三个层面的启发：

**架构层面**：Orbit 的 OKLCH + 语义化令牌基础是扎实的，优于两个参考项目的颜色方案。但需要进一步工程化——建立三层令牌架构（原始 → 语义 → 组件）、TypeScript 类型守护、以及集中管理模块。Hermes 的"配置层与样式层分离"思想和 Claude Code 的"TypeScript 类型化主题"模式都值得融合采纳。

**机制层面**：主题切换不应仅是明暗二态手动切换。Claude Code 的 ThemeProvider 模式（三态 + 系统跟随 + 预览）提供了完整的参考实现。Hermes 的皮肤加载优先级链（用户自定义 → 内置）为未来的主题扩展留足了空间。Orbit 应该从第一天就把主题管理的 React 层基础设施建好，而非仅依赖 CSS 属性选择器。

**组件层面**：设计令牌的价值最终通过组件体现。Claude Code 的 `ThemedBox`/`ThemedText` 基础组件模式和 16 组件设计系统库展示了如何将令牌系统与组件库深度集成。Orbit 应该建立自己的设计系统组件层，让业务代码通过组件 API 而非直接引用 CSS 变量来使用设计系统，这既保证一致性，也为未来的主题扩展提供了稳固的抽象层。

最终，Orbit 的设计系统应该做到：**令牌够用且类型安全，主题切换优雅且可扩展，组件统一且易于使用，可访问性内建而非后补**。这四个维度是衡量一个现代 Web 应用设计系统成熟度的核心标准。
