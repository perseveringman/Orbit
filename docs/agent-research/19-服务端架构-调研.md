# 19. 服务端架构 — Hermes / Claude Code 调研报告

> **对应设计方案**：`docs/orbit-reboot/19-服务端架构.md`
> **分析来源**：Hermes Agent 源码分析 + Claude Code 源码分析（6 篇专项深度报告）
> **调研维度**：服务端设计、API 架构、认证授权、部署模式

---

## 1. Orbit 设计方案摘要

Orbit 的服务端定位为 **"加密中转站"**，核心原则是零知识——服务端永远不接触用户内容明文。其职责严格限定在六个方面：账号管理、设备管理、加密对象存储、同步游标、变更通知、GDPR 执行。

### 技术选型与架构特征

- **单体服务**：Hono 框架 + Node.js/Bun 运行时，不拆微服务
- **存储分离**：PostgreSQL 存元数据（账号、设备、游标），S3/R2 存加密 blob
- **API 设计**：RESTful 风格，分为 Auth / Device / Sync / Blob / Notification / Admin 六组
- **认证方案**：JWT（15 分钟有效期）+ refresh token（30 天），bcrypt/argon2id 密码哈希
- **通知机制**：WebSocket（桌面/Web）+ APNs（iOS），通知内容不含用户数据
- **部署模式**：单容器部署 Fly.io / Railway / VPS，月成本 $10-25
- **自托管就绪**：`docker compose up` 即可运行全套后端

### 关键设计决策

1. Agent 100% 运行在客户端，服务端不参与推理
2. 不做实时协作基础设施，WebSocket 仅用于通知
3. 单体架构直到用户量明确需要拆分
4. 客户端可配置 `syncServerUrl` 切换自托管实例

## 2. Hermes 可借鉴之处

Hermes 作为"开放式全栈 AI 基础设施"，虽然其主要关注点在 AI Agent 而非传统后端服务，但在以下方面对 Orbit 的服务端架构有启发意义。

### 2.1 多后端执行环境的统一抽象

Hermes 设计了 `BaseEnvironment` 抽象基类和 `ProcessHandle` 协议，统一 6 种执行后端（Local / Docker / SSH / Modal / Daytona / Singularity）的接口。每种后端都实现相同的 `_run_bash()` 方法，返回符合 `ProcessHandle` 协议的对象。

**对 Orbit 的启发**：Orbit 的对象存储层可以借鉴这种统一抽象思路。设计方案中已提到支持 S3、R2 和 MinIO（自托管），但缺少一个显式的存储适配器抽象。建议定义一个 `StorageBackend` 接口（`put` / `get` / `head` / `delete`），让不同云厂商的 SDK 实现同一接口，这样切换存储供应商和支持自托管 MinIO 时不需要改动业务代码。

### 2.2 凭证隔离与环境变量安全

Hermes 在环境变量管理上极为谨慎：`local.py` 中显式剥离 100+ 个 API 密钥变量，使用 `ContextVar` 实现会话级白名单隔离，防止凭证跨会话泄漏。`credential_files.py` 还实现了路径遍历防御（拒绝绝对路径、`..` 序列、符号链接逃逸）。

**对 Orbit 的启发**：Orbit 服务端在运维安全章节提到"数据库连接使用 SSL，密码走环境变量"，但缺乏对环境变量本身的安全管理细节。建议：

- 在服务启动时验证必需环境变量的存在（fail-fast），避免运行时空指针
- 对敏感环境变量（如 `DATABASE_URL`、`S3_SECRET_KEY`）使用 Vault 或 sealed secrets 管理，而非裸露在 `.env` 文件
- 日志中对敏感值做脱敏处理（Hermes 的做法是完全剥离，Orbit 至少应做掩码输出）

### 2.3 SSRF 防护与网络安全

Hermes 的 `url_safety.py` 实现了完整的 SSRF 防护：阻止已知内部主机名（`metadata.google.internal`）、DNS 解析后检查 IP 范围（阻止 RFC 1918 私有地址、回环地址、链路本地地址、CGNAT 范围），并采用"失败关闭"策略。

**对 Orbit 的启发**：Orbit 的服务端虽然不代理外部请求，但在以下场景仍需 SSRF 防护意识：

- `syncServerUrl` 自托管配置不应允许客户端将服务端指向内网地址并利用服务端做中转
- 如果未来增加 Webhook 通知功能，必须对目标 URL 做类似 Hermes 的 IP 范围检查
- GDPR 导出包的下载链接生成应使用签名 URL 而非直接暴露存储路径

### 2.4 多渠道网关的会话管理

Hermes 的 Gateway 层通过 `SessionStore` 集中管理 14+ 消息平台的会话，每个平台适配器都遵循统一的生命周期协议。Cron 调度器每 60 秒 tick 一次，使用文件锁防止并发执行。

**对 Orbit 的启发**：Orbit 的通知系统（WebSocket / SSE / APNs）也面临类似的多通道管理问题。建议设计一个 `NotificationDispatcher` 统一管理各通知通道的连接状态、重连逻辑和消息去重，而非在每个通道各自实现。特别是 WebSocket 的连接管理、心跳检测和优雅断连，可以参考 Hermes Gateway 中 `_SYNC_INTERVAL_SECONDS` 的定期巡检机制。

### 2.5 Docker 安全加固实践

Hermes 的 Docker 后端实践了严格的安全加固：`--cap-drop ALL` 丢弃所有 Linux capabilities，`--security-opt no-new-privileges` 阻止权限提升，`--pids-limit 256` 限制进程数，`--tmpfs` 挂载使用 `nosuid` 和 `noexec` 标志。

**对 Orbit 的启发**：Orbit 的自托管方案使用 `docker compose up`，应在 compose 文件中应用类似的安全加固。API Server 容器不需要特权，可以 `cap-drop ALL` 后仅添加 `NET_BIND_SERVICE`（如果需要绑定低端口）。PostgreSQL 容器也应限制资源和进程数。

## 3. Claude Code 可借鉴之处

Claude Code 是"精雕细刻的开发者协作工具"，在企业级安全管控、认证流程、结构化 API 设计等方面有大量可供 Orbit 服务端借鉴的实践。

### 3.1 OAuth 认证与令牌管理

Claude Code 实现了完善的 OAuth 令牌生命周期管理：`isOAuthTokenExpired()` 自动检测过期、`refreshOAuthToken()` 无感刷新、`isManagedOAuthContext()` 检测托管上下文以防止认证源回退。令牌存储使用 macOS Keychain 实现操作系统级安全存储，并有 `clearLegacyApiKeyPrefetch()` 和 `clearKeychainCache()` 进行遗留数据清理。

**对 Orbit 的启发**：Orbit 的认证方案（JWT + refresh token）基本合理，但可以在以下方面增强：

- **Token 轮换策略**：参考 Claude Code 的 `refreshOAuthToken()` 实现，在 refresh token 使用后立即轮换（rotation），防止 token 被截获后重放
- **设备绑定**：将 refresh token 与设备 ID 绑定，设备撤权时自动使对应 refresh token 失效（设计方案提到了撤权后立即失效，但未说明实现机制）
- **认证上下文检测**：借鉴 `isManagedOAuthContext()` 的思路，在自托管场景下检测认证上下文，防止客户端在切换 `syncServerUrl` 时误用旧服务器的 token

### 3.2 声明式权限规则引擎

Claude Code 的权限系统是声明式的：6 种权限模式（default / plan / acceptEdits / bypassPermissions / dontAsk / auto）、3 态行为（allow / deny / ask）、完整的规则来源追踪（配置文件 / CLI 参数 / 会话）。每条权限决策都记录来源，便于审计。

**对 Orbit 的启发**：Orbit 的 API 权限模型目前比较简单（JWT 验证 + 设备 ID 匹配），但随着功能扩展可能需要更细粒度的控制。建议：

- 为每个 API 端点定义声明式权限元数据（如 `@requireAuth` + `@requireTrustedDevice`），而非在每个 handler 中手写验证逻辑
- 设备信任状态（pending / trusted / revoked）转换应有状态机定义，参考 Claude Code 的 `TaskStatus` 显式多态设计
- 审计日志应记录权限决策的完整上下文（谁、从哪个设备、对哪个资源、做了什么操作、决策依据），而不仅是"事件类型 + 元数据"

### 3.3 集中式安全审计与遥测

Claude Code 通过 `logPermissionDecision()` 实现了集中式审计枢纽，扇出到 Statsig 分析、OTel 遥测、代码编辑计数器和上下文存储四个通道。每个事件包含 `messageID` 关联 ID、工具名、沙箱状态、用户等待时间等丰富数据点。

**对 Orbit 的启发**：设计方案中的 `audit_logs` 表设计偏简化（`event_type` + `metadata_json`），建议增强：

- 引入结构化审计事件类型（参考 Claude Code 的 `tengu_tool_use_granted_*` / `tengu_tool_use_rejected_*` 模式），而非自由格式 JSON
- 添加请求关联 ID（`request_id` 或 `trace_id`），方便追踪一次操作涉及的多条审计记录
- 对安全敏感事件（登录失败、设备撤权、GDPR 删除）设置告警阈值，参考 Claude Code 的拒绝追踪（`maxConsecutive: 3, maxTotal: 20`）自动触发预警

### 3.4 远程管理策略与熔断机制

Claude Code 通过 Statsig 功能门控实现了远程管理策略，可以远程禁用危险模式（`bypassPermissionsKillswitch`），支持组织级安全策略强制执行。托管设置变更需要用户显式确认，且有危险设置检测（Shell 设置、不安全环境变量、Hook 配置）。

**对 Orbit 的启发**：Orbit 作为个人产品当前不需要组织级管控，但在以下场景可以借鉴熔断思想：

- **速率限制熔断**：当某账号在短时间内触发大量速率限制错误，自动临时封禁并通知用户（可能是密钥泄露或恶意使用）
- **异常行为检测**：如果一个账号突然从大量不同 IP 发起请求，自动触发安全审查
- **GDPR 删除的冷静期**：设计方案已有 7 天冷静期，建议增加"在冷静期内有从新设备登录则自动延长冷静期"的防护，防止账号被盗后恶意删除

### 3.5 结构化类型系统与 API 契约

Claude Code 广泛使用 TypeBox / Zod schema 定义工具输入输出的类型契约。每个工具的 `inputSchema` 都是强类型的，配合 TypeScript 的类型推导实现端到端类型安全。任务系统的 7 种 `TaskState` 通过 discriminated union 实现多态，状态转换有严格定义。

**对 Orbit 的启发**：Orbit 设计方案中已经定义了 `SyncChange`、`SyncPushRequest`、`SyncPullResponse` 等 TypeScript 接口，方向正确。进一步建议：

- 使用 Zod 定义 API 请求/响应 schema，同时用于运行时验证和 OpenAPI 文档生成
- 将共享类型放入 `packages/api-types`（设计方案已提到），并通过 `zod-to-openapi` 自动生成 API 文档
- 设备信任状态（pending / trusted / revoked）和 GDPR 工单状态（pending / cooling / processing / completed）建议使用 discriminated union + 状态机守卫，防止非法状态转换

## 4. 不适用的设计

### 4.1 Hermes 的多模型适配层

Hermes 支持 10+ LLM 提供商的统一适配，包括 OpenAI、Anthropic、Google、Ollama 等。这对 Orbit 服务端完全不适用——Orbit 的设计决策是 Agent 100% 运行在客户端，服务端不参与任何 LLM 推理。引入模型适配层只会增加服务端复杂度，违背"服务端是加密中转站"的核心定位。

### 4.2 Hermes 的多平台消息网关

Hermes 的 Gateway 层支持 14+ 消息平台（WhatsApp、Telegram、Discord 等）。Orbit 是桌面优先的个人笔记/知识管理工具，不需要接入即时通讯平台。通知机制只需 WebSocket + APNs 即可满足需求。

### 4.3 Claude Code 的 ML 安全分类器

Claude Code 在自动模式下使用 ML 分类器判断操作安全性。这对 Orbit 服务端不适用，原因是：Orbit 服务端的操作集极其有限（存密文、查游标、管账号），没有需要 ML 分类的复杂决策场景。简单的速率限制 + IP 封禁足以满足安全需求。

### 4.4 Claude Code 的 Coordinator 编排模式

Claude Code 的 Coordinator 通过 370 行系统提示词定义了 4 阶段工作流（Research → Synthesis → Implementation → Verification），用 LLM 语义调度子 Agent。这是面向 AI 开发者工具的编排模式，Orbit 服务端不涉及任何 Agent 编排——它只做数据中转。

### 4.5 Claude Code 的企业级远程管理

Claude Code 的 Statsig 功能门控、远程托管设置、团队记忆同步等功能面向企业 SaaS 场景。Orbit 是个人产品，没有团队/角色/组织的概念。引入这些机制只会增加不必要的复杂度。

### 4.6 Hermes 的 RL 训练闭环

Hermes 内置强化学习训练工具，支持数据生成与训练闭环。这完全超出 Orbit 服务端的职责范围。

## 5. 具体建议

### 建议 1：设计存储后端抽象层（StorageBackend Interface）

**问题**：设计方案中直接引用 S3/R2，但自托管需要 MinIO，未来可能需要支持更多存储供应商。缺少统一抽象会导致切换成本高。

**建议做法**：

```typescript
// packages/api-types/src/storage.ts
interface StorageBackend {
  put(key: string, data: Buffer, metadata?: Record<string, string>): Promise<void>
  get(key: string): Promise<{ data: Buffer; metadata: Record<string, string> }>
  head(key: string): Promise<{ exists: boolean; size: number; lastModified: Date }>
  delete(key: string): Promise<void>
  deletePrefix(prefix: string): Promise<number>  // 用于账号删除
  presignGet(key: string, expiresIn: number): Promise<string>  // 签名下载 URL
}
```

实现 `S3StorageBackend`、`R2StorageBackend`、`MinIOStorageBackend`（后两者可能共用 S3 兼容实现），通过配置选择。借鉴 Hermes `BaseEnvironment` 的统一接口理念，确保所有实现遵循相同的错误处理和重试策略。

### 建议 2：增强认证安全——Token 轮换 + 设备绑定

**问题**：设计方案提到 JWT 15 分钟过期 + refresh token 30 天，但未详细说明 refresh token 的安全管理。

**建议做法**：

- **Refresh Token 轮换**：每次使用 refresh token 获取新 access token 时，同时颁发新的 refresh token 并使旧的失效。参考 Claude Code 的 `refreshOAuthToken()` 流水线。
- **设备绑定**：在 `devices` 表中增加 `refresh_token_hash` 字段，refresh token 与特定设备绑定。设备撤权时同步使该 token 失效，无需遍历 token 黑名单。
- **Token 指纹**：在 JWT payload 中包含 `device_id` 和 `token_family`（用于检测 refresh token 重放攻击），如果检测到旧 token 被使用，立即撤销该 token family 下的所有 token。

### 建议 3：结构化审计日志 + 安全告警

**问题**：设计方案的 `audit_logs` 表用 `event_type TEXT` + `metadata_json TEXT` 存储事件，结构过于松散，难以查询和告警。

**建议做法**：

```sql
CREATE TABLE audit_logs (
  id TEXT PRIMARY KEY,
  account_id TEXT,
  device_id TEXT,                    -- 关联发起操作的设备
  request_id TEXT NOT NULL,          -- 请求关联 ID（trace_id）
  event_type TEXT NOT NULL,
  event_category TEXT NOT NULL,      -- auth | device | sync | admin
  severity TEXT NOT NULL DEFAULT 'info',  -- info | warning | critical
  ip_address INET,                   -- 请求来源 IP
  metadata JSONB NOT NULL DEFAULT '{}',  -- 结构化元数据
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX idx_audit_account_category ON audit_logs(account_id, event_category, created_at);
CREATE INDEX idx_audit_severity ON audit_logs(severity, created_at) WHERE severity != 'info';
```

参考 Claude Code 的 `logPermissionDecision()` 集中枢纽模式，在 Hono 中间件层实现统一的审计日志记录，而非在每个 handler 中手动插入。对 `severity = 'critical'` 的事件（连续登录失败、设备撤权、GDPR 删除发起）设置定时巡检或实时告警。

### 建议 4：Docker Compose 安全加固

**问题**：设计方案提到自托管只需 `docker compose up`，但没有提及容器安全配置。

**建议做法**（参考 Hermes Docker 安全实践）：

```yaml
# docker-compose.yml
services:
  api:
    image: orbit/server:latest
    security_opt:
      - no-new-privileges:true
    cap_drop:
      - ALL
    cap_add:
      - NET_BIND_SERVICE
    read_only: true
    tmpfs:
      - /tmp:rw,nosuid,noexec,size=64m
    deploy:
      resources:
        limits:
          memory: 512M
          pids: 128
    environment:
      - DATABASE_URL=${DATABASE_URL}
      - S3_ENDPOINT=${S3_ENDPOINT}
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:3000/health"]
      interval: 30s
      timeout: 5s
      retries: 3

  postgres:
    image: postgres:16-alpine
    security_opt:
      - no-new-privileges:true
    volumes:
      - pgdata:/var/lib/postgresql/data
    deploy:
      resources:
        limits:
          memory: 256M

  minio:
    image: minio/minio:latest
    command: server /data --console-address ":9001"
    volumes:
      - miniodata:/data
```

### 建议 5：API Schema 验证 + OpenAPI 文档自动生成

**问题**：设计方案定义了 TypeScript 接口，但没有运行时验证和 API 文档生成策略。

**建议做法**：

- 使用 Zod 定义所有 API 请求/响应 schema（替代纯 TypeScript interface），Hono 生态有 `@hono/zod-validator` 中间件直接集成
- 通过 `@hono/zod-openapi` 自动生成 OpenAPI 3.1 规范文件
- 将 schema 定义放在 `packages/api-types` 中，客户端和服务端共同消费（设计方案已规划）
- 参考 Claude Code 的 Zod discriminated union 模式定义设备状态和 GDPR 工单状态的合法转换

示例：

```typescript
// packages/api-types/src/sync.ts
import { z } from 'zod'

export const SyncChangeSchema = z.object({
  changeId: z.string().uuid(),
  objectId: z.string(),
  channel: z.enum(['object', 'document', 'blob']),
  version: z.number().int().positive(),
  timestamp: z.number().int(),
  encryptedPayload: z.string().base64(),
  checksum: z.string().length(64),  // sha256 hex
})

export const SyncPushRequestSchema = z.object({
  deviceId: z.string(),
  changes: z.array(SyncChangeSchema).min(1).max(100),
})
```

## 6. 总结

Orbit 的服务端架构设计方案定位清晰、约束合理。"零知识加密中转站"的核心原则和"单体 + 对象存储"的极简架构，让整个方案保持了难得的简洁性。通过对 Hermes 和 Claude Code 的深度分析，我们识别出了五个值得融入的增强方向：

| 建议 | 来源启发 | 核心价值 |
|------|---------|---------|
| 存储后端抽象层 | Hermes `BaseEnvironment` 统一接口 | 降低存储供应商切换成本，提升自托管体验 |
| Token 轮换 + 设备绑定 | Claude Code OAuth 令牌管理 | 堵住 refresh token 重放攻击漏洞 |
| 结构化审计日志 | Claude Code `logPermissionDecision()` | 从松散 JSON 升级为可查询、可告警的结构化事件 |
| Docker Compose 安全加固 | Hermes Docker `cap-drop ALL` | 自托管方案开箱即安全 |
| Zod Schema + OpenAPI | Claude Code 强类型工具定义 | 运行时验证 + API 文档 + 前后端类型共享一体化 |

这些建议的共同特点是：**不改变原始架构的简洁性，但显著提升工程质量和安全性**。它们都可以在 MVP 阶段就落地，不增加架构复杂度。

值得强调的是，Hermes 和 Claude Code 中大量有趣的设计（多模型适配、ML 分类器、Coordinator 编排、RL 训练闭环等）对 Orbit 服务端**完全不适用**。这恰恰验证了设计方案中"服务端不负责"列表的正确性——当一个系统清楚地知道自己不做什么时，它反而能把该做的事做到极致。Orbit 服务端最大的优势就是它的简单，任何增强都不应牺牲这一点。
